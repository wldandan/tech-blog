###### PART 01
# Agent故障与传统故障，本质上不是同一种东西

传统软件故障是**确定性**的：一段代码、一个空指针、一行越界访问，栈帧里写得清清楚楚，复现一次就能定位。它是**点状**的，错误发生在一个具体的位置，修好那一行，问题就消失。

Agent故障则完全不同。一个Agent的一步行动，背后是「回忆历史 → 评估进展 → 规划下一步 → 调用工具 → 接收环境反馈」这条认知链。链上任何一环出问题，如记错了上下文、误判了上一步结果、规划违反了约束，都不会立刻报错，而是被**悄悄带入后续每一步**，在十几轮之后才以一个面目全非的症状爆发出来。

更棘手的是，Agent故障往往**没有「崩溃」**。进程活着、工具在调、Token在涨、日志在刷，一切「健康指标」都是绿的，但任务就是**永远不收敛**。这种「一直在动，却永远到不了终点」的形态，是传统 APM、日志告警、健康检查的彻底盲区。

# Agent故障诊断，要先翻过三道技术坎

正因为故障形态变了，诊断的难度也跟着变。这件事到底有多难？学术界第一个系统研究自动故障归因的工作（Who&When，宾州州立 / 杜克）给出了一组冷峻的数字：即便用最强的方法，**识别「是哪个Agent出了错」也只有约53.5%的准确率，而精确定位「决定性错误步」只有约14.2%**。换句话说，把一条出错的trace丢给一个再聪明的模型直接问「哪儿错了」，十次有八九会指错地方。把一个出错的Agent trace摆在面前，至少有三道坎挡在「自动化根因定位」前面：

- **第一坎 · 上下文太长，又不能不看。** 一次长程任务动辄数百个step、几十万Token。全量喂给LLM做推理，成本爆炸且注意力涣散；只看摘要，又会漏掉藏在细节里的真因。「省Token」和「看全貌」天然矛盾。
- **第二坎 · 单步看都「没大错」，合起来才是灾难。** 死循环里的每一步，单独看可能只是一次「冗余调用」或「低效计划」，严重度都不高。逐step检测会把它们打散成一堆低分小问题，真正的「不收敛」反而消失在统计噪声里。
- **第三坎 · 因果会级联，根因常被症状盖住。** 第2步的一个错误规划，会引发第3步参数错、第4步工具失败……到最后报错的位置，离真正的根因可能隔着几十步。「找最显眼的错」往往等于「找错了根因」。

:::info
核心矛盾

诊断既要**微观**到看清「哪一步、哪个认知模块第一个出错」，又要**宏观**到看出「整体为什么不收敛」。单靠一把尺子量不出来——这正是Agent Insight用**两套并行引擎**的原因。
:::

## PART 03 Agent Insight智能诊断Agent的解法

Agent Insight的智能诊断能力（内部代号`AgentDebug`），核心是一条 **「逐step认知错误检测 + 单点根因定位」**的流水线，由确定性Python脚本先做硬规则判错，再由智能诊断Agent补充语义判断，最后聚合为关键发现与根因。它的认知模型并非凭空设计，而是吸收了学术界对Agent失败的最新归纳，把一次行动失败归因到某个具体的「认知模块」，再沿因果链找到最早的那一步。

### 把每一步行动，拆成五个认知维度

诊断的最小单元，是Agent的一个 **「轮次」**。每个轮次被拆解为五个模块——这正好对应前文那条认知链：

:::info
📖Agent轮次

可以把它理解为Agent的**「一个回合」**：模型读入当前上下文 → 思考 → 决定下一步 →（通常）调用一个工具 → 拿到结果。一次长程任务就是几十、几百个这样的回合首尾相接。诊断不是去看每一行底层日志，而是以「轮次」为最小单元逐个体检——因为认知偏差正是在一个个回合之间被悄悄传递下去的。
:::

### 拆完之后，凭什么说这一步「错了」

拆成五维只是第一步。真正决定可信度的，是**判错** —— 而Agent Insight在这里没有把一切交给大模型，而是按模块类型走两条不同的路：

![alt text](images/fig03-两条判错路径.png)

图3 ｜ 两条判错路径：客观事实（工具/环境）交给脚本，主观正误（记忆/反思/计划）才交给大模型。

关键在于：**凡是能从真实证据确定读出来的，绝不交给大模型猜**。工具调用对不对、命令报没报错、上下文有没有溢出，这些是**事实**，脚本说了算；而「这一步是不是记错了上下文」「是不是把失败误判成了成功」，这些是**语义判断**，才轮到LLM出场。每条结论还都带一个**置信度**，让整份报告是分级可审计的，而非一锅黑箱：

| 证据来源 | 含义 | 置信度 |
|---|---|---|
| 真实工具调用 | 从raw tool call确定性提取 | 0.95 |
| 外部系统证据 | status=error、超时、上下文溢出等硬信号 | 0.95 |
| 明确标签 | 带显式动作 / 状态标签 | 0.85–0.9 |
| 可见文本 / 规则抽取 | 正则命中或可见输出推断 | 0.65–0.8 |
| LLM语义补充 | 记忆 / 反思 / 计划的内容正误判断 | 0.5–0.75 |
| 留白 | 没判到，不算错 | 0 |

### 诊断者自己，也被上了「镣铐」

「让AI来诊断AI，它会不会自己也乱编一通？」这是最该被回答的疑虑。Agent Insight的智能诊断Agent 被 skill严格约束，它**必须**先跑静态脚本、**必须**以脚本输出为主事实源，并且：

- **禁止编造Action。** 动作必须来自真实工具调用或明确动作标签，不能凭空想象一个没发生过的操作。
- **禁止反推。** 不能因为Action / System失败，就倒推「Planning一定也错了」，每个模块独立判定。
- **禁止修改项目文件。** 它只跑分析与校验脚本，绝不执行用户的原始任务，诊断过程对被诊断系统零副作用。
- **输出必须是纯结构化 JSON。** 不许夹带自然语言碎碎念，每条finding必须引用真实存在的issue。

### 只报真问题：把噪声挡在报告之外

逐step全量检测会产出海量低级问题，如果一股脑全上报，就成了「告警轰炸」。Agent Insight设了一道**降噪闸门**：一个问题只有满足下列条件之一，才会从普通issue**提升**为需要关注的finding—：

- 直接导致最终失败或结果偏差；
- 问题没有被恢复，或恢复代价明显；
- 造成额外轮次、额外工具调用、额外成本或明显延迟；
- 违反硬约束、安全约束、schema约束或关键用户约束；
- 重复出现、具备明确优化价值。

反过来，已被自动修复且不影响结果的、正常探索中的低影响问题、仅仅是别人下游症状的，一律留作普通issue不打扰你。报告里出现的，都是值得你花时间的。

### 两套并行引擎：认知诊断回答「哪里开始错」，轨迹诊断回答「为什么不收敛」

这是Agent Insight最关键的设计。同一份原始交互记录，分两条互不干扰的通道处理：

- **认知诊断（cognitive）。** 走 `Phase 0系统预检 → Phase 1逐step五模块检测 → Phase 2根因归因` 三阶段。原则是「**找最早的、修复后最可能改善轨迹的错误**」作为根因，级联错误作为下游，找因不找果。它擅长**单点定位**。
- **轨迹诊断（trajectory）。** 由确定性代码**直接读取原始交互记录的结构化元数据**，零LLM成本、不受数据截断影响。它专门检测跨区间的时序与模式信号，同一文件反复读取、Token单调上涨却无进展、任务状态长期不更新。它擅长**宏观模式**。

### 回到「第一坎」：长上下文怎么省着看又不漏

还记得Part 02那个「省Token和看全貌天然矛盾」的难题吗？Agent Insight用一套**三层数据策略 + 按需点读**把它化解掉：

- **第一层 · 执行摘要**（≤8000 字符）：让 LLM 快速建立任务背景；
- **第二层 · 归一化Step摘要**（约 40000 字符，长trace每轮次仅压到约 85 字符）：作为逐step分析的主事实源；
- **第三层 · 完整Trace资料包**落盘到磁盘：证据不足时才按需点读具体节点，不进主推理上下文。

这套取舍对单步认知诊断刚好够用，省token又看得见局部。而它也顺带解释了**为什么必须再来一个轨迹引擎**：认知诊断看的是被压到约 85 字符的截断视图，天生看不清跨越数百轮次的循环；轨迹引擎绕开这层截断，直读未压缩的元数据，才补上了这块盲区。

:::info
**和你已有的Trace工具是什么关系**

市面上的可观测性平台（如Langfuse、LangSmith、Arize Phoenix）大多止步于「**告诉你发生了什么**」，采集trace、画时间线、打评分，根因还得人盯着日志一步步看。Agent Insight不是又一个trace查看器，而是**站在trace之上的「诊断层」**：在你已有的可观测性数据之上，自动产出认知根因 + 轨迹故障机制，把「发生了什么」推进到「为什么、从哪开始错」。两者是叠加，不是替代。
:::

## PART 04 认识JiuwenSwarm

要讲清这次故障，得先认识它跑在哪。**JiuwenSwarm**是openJiuwen开源的蜂群智能体框架，它的出发点是一个公式：`Agent = Model + Harness`。模型提供「大脑」，而 Harness（驾驭层）提供「躯干和四肢」，把一个可对话的推理单元，驾驭成一个能规划、能调工具、能管上下文、能受权限约束、能委派子任务、能持续演进的**任务执行系统**。

更进一步，JiuwenSwarm把单兵Agent编织成 **「蜂群」**：让一群专业化Agent像蜜蜂一样围绕同一目标分工协作。结构上是「**分级自主协同**」：**Leader Agent**负责动态组队、任务规划、关键审批与结果汇总，**Teammate Agent**各自认领专长任务、在独立工作空间执行。协同由事件驱动机制推动，某个成员卡住，Leader会自动收到事件决定重试还是换人。

把单兵和蜂群真正跑成「可控、可复用、可持续」的生产系统，靠的是JiuwenSwarm的一组关键能力：

- **团队工作空间（Team Workspace）。** 每个成员在隔离的独立空间里读写文件、调用工具，上下文互不污染，天然适配大规模并行协作，也让单个成员的故障可以被局部隔离。
- **模型路由（Model Routing）。** 按任务难度为不同成员与步骤动态分配合适的模型，难的上强模型、简单的上轻量模型，在保证质量的同时显著压低成本（实测Token用量可降低约三成）。
- **安全护栏与停止条件（Rails / Stop-Condition）。** 借助贯穿全生命周期的Rail钩子，对权限、预算、调用边界施加约束，并以显式停止条件防止任务无限发散，这正是与故障诊断强相关的一层。
- **技能复用与自演进（Swarm Skills Hub + 自演进）。** 一次成功的团队协作可沉淀为标准化技能包，在共享市场里被检索、复用，并能从实战反馈中持续自我改进，让团队「越用越强」。

得益于这套设计，JiuwenSwarm即便搭配较小的模型，也能在PinchBench、LOCOMO等智能体与长程记忆基准上取得有竞争力的成绩，兼顾了能力、成本与可控性。

## PART 05 实战：一次「设计评审死循环」，与Agent Insight的自动化定位

### 故障现场：aet-engineering-team的评审员卡死了

`aet-engineering-team-swarm`是一个基于JiuwenSwarm实现的**多Agent研发团队**，由五个专业Agent串成一条带隔离质量门的研发流水线：**需求分析师 → 分析评审 → 系统设计师 → 设计评审 → 实现者**。其中`design-reviewer`（设计评审员）是一个严苛评审者式的独立质量门，它必须读取完整的《功能设计说明书》，才能逐项打分、给出评审结论。

![图6 故障现场截图：JiuwenSwarm界面任务停在Step 4设计Review](images/fig6-jiuwenswarm-ui.png)

图6 ｜ 故障现场截图：Step 4设计Review长时间停在design-reviewer「正在评审 / 正常工作」，下游Step 5、6持续blocked，界面一片绿灯、显示「当前状态正常，无需干预」，实则已陷入死循环。

诡异之处正在于此：界面上每一项「健康指标」都是绿的，G1/G2/G3全部PASS、Step 4显示「刚刚认领、正常工作」、流水线「按预期推进」，可任务就是**永远卡在G4评审、永远等不到结果**。这正是Part 01说的那种「一直在动却不收敛」的Agent故障：没有崩溃、没有报错，却也没有进展。那么，问题究竟出在哪？

根子在框架的一个善意机制上。JiuwenSwarm的`MessageSummaryOffloader`会把超过阈值的大工具结果**自动压缩为摘要、并offload到磁盘**，以节省上下文。于是死循环诞生了：

![alt text](images/fig07-设计评审死循环.png)

图7 ｜ 读取-被压缩-重读的闭环（进程不崩、工具在调，下游任务却被永久阻塞）

reviewer拿到摘要 → 判定被截断需要全文 → 重新读取文件 → 又被offloader压缩 → Leader催促后再重读……**无限空转，下游任务永久阻塞**。至此，前面截图里那片「绿灯下的卡死」就有了解释：进程在动、工具在调，所以一切监控正常；可它每一圈都在原地打转，所以永远到不了G4。代价是惊人的——**这一圈圈空转整整持续了十多个小时、跑出600多步、烧掉约 900 万Token**，却没有产出任何有效进展，而传统监控自始至终都显示一切正常。

这绝不是个孤例。学术界对 1600+ 条多智能体失败trace的归纳（MAST，UC Berkeley）显示，**「系统设计类」故障占到 44.2%，其中「步骤重复 / 无进展地重复已执行步骤」单独就占 15.7%**，是出现频率最高的故障族之一。换句话说，本文这种死循环，正是多Agent系统最常踩的坑，也最值得被自动诊断盯死。

### 为什么「认知诊断」单独上阵会扑空

如果只用逐step的认知诊断，会发生什么？循环中的每一步，都会被各自判为**独立的单步小问题**，这一步是 `低效计划`，那一步是 `工具执行错误`，但「五步窗口内同工具同参数 ≥3 次」的 `冗余调用` 条件往往还不满足。这些低严重度问题被聚合计数后，呈现为一堆统计量；而Phase 2「取最早错误」的归因原则，最终很可能把根因落在执行轨迹**前段**某次显眼的启动预检低效探索上，而不是后段真正的死循环。

### 轨迹诊断介入：直读元数据，一眼看穿循环

这时轨迹诊断器登场。它**不经过任何数据截断**，直接读取原始交互记录的结构化元数据，于是那些被认知诊断打散的信号，在它眼里立刻连成一片清晰的模式：

| 检测信号 | 在本案例中的表现 |
|---|---|
| 同一文件反复读取 | `read_file(设计说明书)`在后段trace中出现**数十次** |
| 任务状态不前进 | `update_task`在大量step中始终为`false` |
| input token单调上涨 | 每次offload的摘要不断累积进输入上下文 |
| 同一offload主题反复出现 | `compression + summary`模式在30+ step持续重复 |
| 无净进展时间跨度 | 从首次读取到最终中断，跨越数百个节点 |

> **一个常见的误解**
> 
> 有人会问，检测循环不就是 **grep一下重复读取**吗，至于上「引擎」？难点恰恰不在「发现重复」，而在判定「**无净进展**」—— 要把**空转死循环**和正常的重试、轮询、迭代区分开。这需要联合判定任务状态是否前进（`update_task`）、input token是否单调上涨、offload主题是否重复，再结合循环span的长度，才能得出「在动，但没往前」的结论。grep抓得到重复，抓不到空转。

基于这些信号，轨迹诊断器产出一条`kind=trajectory`的finding，它不会被硬塞进某个单一的关键步骤，而是以「故障机制 + 故障链 + 代表性证据节点」的形式，把整个循环描述出来：

**轨迹诊断结论 · 设计评审死循环**（`kind = trajectory`）

| 字段 | 内容 |
|---|---|
| 起点Span | 节点 #68（首次读取设计文档） |
| 终点Span | 节点 #470（最终中断） |
| 循环周期数 | 37 次完整「读取-被压缩」循环 |
| 无进展度量 | 任务状态 = 未更新（最近 400 个节点） |
| 关键发现 | 设计评审员因压缩卸载机制反复压缩文档内容，陷入读文档-被截断-重读的无限循环，持续 400+ 节点无净进展 |
| 证据节点 | #68、#78、#91 …（代表性循环锚点，界面可逐一跳转） |
| 修复建议 | 在评审员提示词中加入防循环守卫：检测到连续 3 次相同文件读取且任务状态未更新时，主动跳过并输出诊断摘要 |

这张卡片背后，是一份**结构化、可消费、可审计**的 JSON——不是一段自然语言总结。每条finding都带kind、严重度、证据节点锚点与置信度，下游可以直接拿去渲染、告警或喂回修复流程：

```json
{
  "kind": "trajectory",
  "severity": "high",
  "impact": "result_blocking",
  "span": { "fromNode": 68, "toNode": 470 },
  "cycleCount": 37,
  "noProgressMetric": "task_updated=false in last 400 nodes",
  "summary": "设计评审员陷入读文档-被截断-重读的无限循环…",
  "evidence": [68, 78, 91],
  "correctionGuidance": "在 reviewer prompt 中加入 anti-loop guard…",
  "confidence": 0.92
}
```

### 两种诊断协作：一份报告，看清全貌

认知诊断与轨迹诊断**共享同一份`findings[]`**，UI按`kind`渲染不同卡片，两者互不干扰：

- **认知诊断结论**锚定前段启动预检低效探索的单点根因，回答「哪里开始错」；
- **轨迹诊断结论**描述后段无限循环的故障机制，回答「整体为什么不收敛」。

![alt text](images/fig08-微观宏观协作.png)

图8 ｜ 微观 + 宏观：认知诊断锚定起点，轨迹诊断框定整段不收敛区间，互补成完整因果。

### 定位效果：从「数小时人工排查」到「一键 10 分钟」

把这次「设计评审死循环」放到真实排障场景里，最直观的收益是**时间**。没有智能诊断时，工程师只能凭经验逐行翻日志、手工排查调用链中存在的错误；有了Agent Insight，一次点击就能拿到根因与修复建议：

![alt text](images/image.png)

除了时间，更关键的是**定位质量稳定**——不再因人而异、不会被前段症状带偏：把数百节点的空转收敛成一条轨迹诊断结论，精确框定 `#68→#470` 的循环区间，轨迹检测零大模型成本，还直接给出可执行的防循环修复建议。

:::info
一句话总结

认知诊断回答了**「哪里开始错的」**，轨迹诊断回答了**「整体为什么不收敛」**。两者互补，覆盖从微观到宏观的完整诊断视角 —— 这正是Agent时代根因定位该有的样子。
:::

图9 ｜ 诊断实况（动图占位）**：Agent Insight 对「设计评审死循环」的自动化定位演示

### 诊断之后：把根因喂回进化飞轮

定位到根因、给出修复建议，并不是终点。Agent Insight本身就构建了**「观测 → 评测 → 诊断 → 优化」**的完整闭环，诊断产出的根因与 `correctionGuidance`（如本例的「防循环守卫」）**直接作为「优化」环节的输入**，让平台不止于「告诉你哪坏了」，而是推动「下次别再坏」。这个进化飞轮有两种闭合方式：

![alt text](images/fig10-进化飞轮双闭合.png)

图10 ｜ 双闭合的进化飞轮：诊断结果既驱动Agent Insight自身的「观测→评测→诊断→优化」内循环，也可外溢给自进化框架。

**闭合方式一：Agent Insight平台内部闭环。** Agent Insight把诊断结论沉淀为可复用的优化资产，固化的错误模式、收敛性基线、针对性的修复建议，回流到「观测」与「评测」两端：让下一轮观测更有的放矢、评测更贴合真实故障族。同一类「设计评审死循环」被诊断过一次，平台就记住了它的特征，后续同类问题能更快被识别、甚至在评测阶段提前预警。这是一条不依赖外部框架、Agent Insight自给自足的飞轮。

**闭合方式二：外溢给自进化的Agent框架。** 更进一步，诊断结果是**结构化、可消费**的 `findings[]`，因此可以作为**优化输入直接喂给支持自进化的Agent框架**，JiuwenSwarm正是这样一个消费方。它接到诊断给出的`correctionGuidance` 后，由自身的自演进机制把改进归因到具体Skill、生成改进并经审批固化回 `SKILL.md`（在团队技能层增补角色/约束/流程，在成员技能层沉淀实战经验，且采用补丁式追加、可单独审查与淘汰）。这里的边界很清楚：**Agent Insight负责「发现与给出建议」，框架负责「采纳与固化」**，两者通过标准化的诊断产物对接，而非互相耦合。

:::info
为什么这件事重要

诊断由此从「事后报告」升级为**「优化的起点」**：对内驱动Agent Insight自身闭环越用越准，对外则让JiuwenSwarm这类框架把一次性的故障修复，沉淀成**不会再踩第二遍**的团队经验。观测让Agent可见，评测让它可衡量，诊断让它可归因，而优化，无论闭在平台内还是外溢给框架，才让它真正**可自我进化**。
:::

## 结语：让每一个Agent都可被观测、可被评估、可自我进化

Agent不再单兵作战，故障也不再是孤立的一个坏点。当蜂群规模化协同，诊断需要同时具备「读懂单步认知」和「看穿整体轨迹」的能力。这就是**Agent Insight**，openEuler孵化的AgentOps平台，想为每一个Agent提供的底座能力。

而**JiuwenSwarm**把单个智能体驾驭成系统、把系统编织成蜂群；Agent Insight则在蜂群之上，为它们装上一双能看清因果的眼睛。两者一个负责「跑得更稳」，一个负责「出问题时找得到根因」，恰好互为表里。
