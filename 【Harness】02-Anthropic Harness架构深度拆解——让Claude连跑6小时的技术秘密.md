## 引言

在本系列第一篇中，我们介绍了[Harness](https://zhida.zhihu.com/search?content_id=272587044&content_type=Article&match_order=1&q=Harness&zhida_source=entity)工程作为AI开发新范式的核心价值。而[Anthropic](https://zhida.zhihu.com/search?content_id=272587044&content_type=Article&match_order=1&q=Anthropic&zhida_source=entity)无疑是Harness技术的先行者，他们不仅在理论上提出了完整的设计框架，更在生产环境中大规模验证了这套架构的可行性——通过精心设计的Harness系统，[Claude](https://zhida.zhihu.com/search?content_id=272587044&content_type=Article&match_order=1&q=Claude&zhida_source=entity)可以连续运行6小时，独立完成从需求分析到上线部署的全栈应用开发，代码质量甚至超过了拥有3-5年经验的人类工程师。

更令人兴奋的是，随着2026年3月Anthropic官方Harness设计文档的公开，以及51万行Claude Code源码的意外泄露，我们得以一窥这个工业级Harness系统的完整设计细节。本文将深度拆解Anthropic Harness的核心架构、工作流程和关键技术创新。

## 一、为什么简单的智能体实现走不远

在讲 Harness 架构之前，得先说说普通的单智能体模式为什么遇到复杂任务就不行。Anthropic 团队在之前的实验里已经发现，把任务拆解成步骤、让智能体每次做一个小功能，再通过上下文传递状态，确实能提升长任务的完成度，但还是会遇到两个绕不开的瓶颈。

第一个问题是**上下文焦虑** 。很多大模型在[上下文窗口](https://zhida.zhihu.com/search?content_id=272587044&content_type=Article&match_order=1&q=%E4%B8%8A%E4%B8%8B%E6%96%87%E7%AA%97%E5%8F%A3&zhida_source=entity)快满的时候，会出现一种奇怪的行为：明明还有空间，却开始着急收尾，把没做完的工作草草交差。之前有人尝试用上下文压缩的方式，把前面的对话摘要之后继续用，但效果并不好——摘要会丢失细节，而且模型的焦虑感并没有消失，还是会提前结束。Anthropic 团队试过，对于 Claude Sonnet 4.5 来说，单纯的压缩根本不够，必须定期彻底重置上下文，用结构化的交接文档把状态传给下一个新的智能体实例，才能解决这个问题。但这么做又会增加编排复杂度、token 开销和延迟。

第二个问题更严重：**智能体不会自我批评** 。你让 AI 评价自己写的代码或者做的设计，它基本只会说好话，哪怕明显有问题。这种现象在主观任务里尤其突出，比如设计一个界面，没有客观的对错标准，AI 对自己的产出打分总是偏乐观。就算是有明确对错的编程任务，AI 也经常会忽略自己的 bug，觉得”差不多能用就行”。

你让同一个智能体既做执行又做审核，本质上是让球员兼任裁判，肯定会出问题。把生成和评估拆开，是解决这个问题的核心思路。

## 二、Anthropic Harness的核心架构

有了前端设计的成功经验，团队把这个生成-评估的模式扩展到了全栈应用开发领域。毕竟软件开发里的代码评审和 QA 环节，本来就和设计评估是一个逻辑。

他们在之前的长任务 Harness 基础上，搭了一个三智能体的系统，每个智能体只负责一个特定的环节：

### 三个核心角色

**[Planner](https://zhida.zhihu.com/search?content_id=272587044&content_type=Article&match_order=1&q=Planner&zhida_source=entity) （规划者）**：之前的 Harness 需要用户提前写好详细的产品需求文档，这个门槛太高了。Planner智能体的作用就是把用户的一两句话需求，扩展成完整的产品规格说明。团队特意要求Planner只关注产品上下文和高层技术设计，不要写太细的技术实现细节——如果一开始就把实现方案定死了，后面写错了会连锁反应出一堆问题。Planner还会主动在产品里找可以加AI功能的地方，比如给编辑器加自然语言生成能力。

**[Generator](https://zhida.zhihu.com/search?content_id=272587044&content_type=Article&match_order=1&q=Generator&zhida_source=entity) （生成者）**：生成智能体沿用了之前”一次做一个功能”的模式，按照Planner写的规格，一个sprint接一个sprint地实现功能。技术栈固定用React、Vite、FastAPI 和 SQLite（后面换成PostgreSQL），每个sprint做完之后先做自我评估，再交给QA。生成智能体有git 权限，可以自己做版本控制。

**[Evaluator](https://zhida.zhihu.com/search?content_id=272587044&content_type=Article&match_order=1&q=Evaluator&zhida_source=entity) （评估者）**：评估智能体同样用Playwright工具模拟真实用户操作，点击运行中的应用，测试UI功能、API接口和数据库状态。它会从产品完整性、功能性、视觉设计、代码质量四个维度打分，每个维度都有硬门槛，只要有一个不达标，这个sprint就不过关，生成智能体会收到详细的问题反馈。

### 协作机制：先签合同再干活

在每个 sprint 开始之前，生成智能体和评估智能体会先协商出一个”sprint 合同”：明确这个sprint要做什么，”完成”的标准是什么，要通过哪些测试来验证。因为Planner写的产品规格是高层的，这个合同就是用来把用户故事翻译成可测试的实现要求。生成智能体先提方案，评估智能体审核，两个智能体来回迭代直到达成一致。

所有沟通都通过文件进行：一个智能体写文件，另一个读了之后要么在同一个文件里回复，要么写新的文件。这样既能保证工作符合规格，又不会过早限制实现细节。

## 三、核心工作流程：生成-评估-迭代循环

Anthropic Harness的核心运行模式是一个闭环的迭代过程，每个完整的迭代包含以下步骤：
    
    
    ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
    │  规划器设计  │────▶│  生成器产出  │────▶│  评估器评审  │
    └─────────────┘     └─────────────┘     └─────────────┘
           ▲                    │                   │
           │                    │                   ▼
           │                    │             问题是否严重？
           │                    │                   │
           │                    │              是 /      \ 否
           │                    │             /          \
           │                    ▼            ▼            ▼
           │               反馈引擎处理    任务达标？    输出结果
           │                    │            │
           │                    │            ▼
           │                    │        是 /   \ 否
           │                    │       /       \
           └────────────────────┘      ▼         ▼
                                       完成    继续迭代

### 迭代过程的关键设计要点：

  1. **小步迭代** ：每次迭代只完成一个小的功能点，避免一次改动过大导致问题难以定位。
  2. **快速反馈** ：生成器每完成一次产出，立即进入评估环节，得到及时反馈。
  3. **可控的迭代深度** ：针对每个问题设置最大迭代次数，避免在单个问题上消耗过多资源。
  4. **升级机制** ：如果同一问题多次迭代无法解决，自动升级到更复杂的智能体或人工介入。



## 四、关键技术创新

Anthropic的Harness系统有几个非常精妙的技术设计，是其强大能力的秘密所在：

### 1\. 上下文窗口的动态管理

传统的智能体应用通常采用固定的上下文窗口，要么上下文不足导致信息丢失，要么上下文过长导致注意力分散。Anthropic采用了[动态上下文管理](https://zhida.zhihu.com/search?content_id=272587044&content_type=Article&match_order=1&q=%E5%8A%A8%E6%80%81%E4%B8%8A%E4%B8%8B%E6%96%87%E7%AE%A1%E7%90%86&zhida_source=entity)策略：

  * **分层上下文** ：将信息分为全局上下文、任务级上下文、迭代级上下文三个层次
  * **按需加载** ：每个智能体在执行任务前，只加载当前需要的上下文信息
  * **智能裁剪** ：使用专门的模型对历史上下文进行摘要和压缩，保留关键信息的同时大幅减少长度
  * **最近最少使用（LRU）淘汰** ：自动淘汰长时间未使用的上下文信息



这种设计使得即使是长达数小时的任务，上下文窗口也能保持在最优长度，既不会丢失关键信息，也不会因为信息过载导致性能下降。

### 2\. 智能体角色的专业化训练

Anthropic并没有使用通用的Claude模型来做所有事情，而是针对不同的智能体角色进行了专业化微调：

  * **Planner Agent** ：将用户简略需求扩展为完整产品规格，聚焦产品上下文与高层设计以避免实现细节过早定死，并主动挖掘可添加AI功能的环节。
  * **Generator Agent** ：在高质量代码数据集上微调，编程能力大幅优于通用模型。
  * **Evaluator Agent** ：在代码评审、漏洞检测的数据集上微调，发现问题的能力甚至超过了很多资深工程师。



### 3\. 错误恢复的闭环设计

任何系统都会出现错误，关键是如何快速从错误中恢复。Anthropic的Harness设计了多层错误恢复机制：

  1. **自动重试** ：对于临时性错误（如API调用失败、网络波动），自动重试最多3次
  2. **回滚机制** ：如果当前迭代出现无法解决的问题，自动回滚到上一个正确的状态
  3. **路径切换** ：如果某种解决方案行不通，自动尝试其他技术路径
  4. **人工介入** ：当错误连续出现超过阈值时，自动暂停任务并通知人类工程师介入



这种多层防护机制使得系统的鲁棒性极强，即使出现问题也能快速恢复，不会导致整个任务失败。

### 4\. [质量门控机制](https://zhida.zhihu.com/search?content_id=272587044&content_type=Article&match_order=1&q=%E8%B4%A8%E9%87%8F%E9%97%A8%E6%8E%A7%E6%9C%BA%E5%88%B6&zhida_source=entity)

为了确保最终输出质量，Anthropic在整个流程中设置了多道质量闸门：

  * **迭代门控** ：每次迭代的产出必须通过评估器的检查才能进入下一个环节
  * **里程碑门控** ：每个重要里程碑完成后，进行更全面的检查
  * **最终验收门控** ：任务全部完成后，进行完整的功能、性能、安全测试
  * **发布门控** ：上线前的最后一道检查，确保所有指标符合发布标准



只有通过了所有门控检查的产出，才会被认为是合格的。

## 五、实际效果与性能数据

团队拿”做一个带关卡编辑器、精灵编辑器、实体行为系统和可玩测试模式的 2D 复古游戏制作工具”这个需求做了对比测试：

  * 单智能体模式：跑了 20 分钟，花了 9 美元，做出来的东西看起来像模像样，但实际运行的时候实体根本不响应输入，核心功能是坏的，界面布局也浪费空间，流程不清晰。
  * 完整 Harness 架构：跑了 6 个小时，花了 200 美元，Planner 直接把需求扩展成了 16 个功能点，分 10 个 sprint 实现，除了核心功能之外还加了精灵动画系统、行为模板、音效音乐、AI 辅助精灵生成和关卡设计，还有游戏导出分享功能。做出来的应用界面有统一的视觉风格，画布占满整个视口，精灵编辑器功能更丰富，最关键的是游戏能实际运行，玩家能控制实体移动。



评估智能体抓 bug 的能力很强，比如有一次 sprint 合同里要求”矩形填充工具支持点击拖拽填充选中的 tile 区域”，评估智能体测试发现工具只在拖拽开始和结束的地方放 tile，根本不填充中间区域，还直接定位到了问题：fillRectangle 函数存在，但 mouseUp 事件没正确触发。还有一次发现 FastAPI 路由定义顺序错了，把 PUT /frames/reorder 路由写在了 /{frame_id} 后面，导致框架把”reorder”当成了 frame_id 参数，返回 422 错误。

不过评估智能体也不是天生就好用，一开始它经常会发现问题之后又自我安慰”这个问题不大”，然后就给过了，测试也只测表面功能，很多边缘 case 的 bug 都漏了。团队花了好几轮调优，每次发现它的判断和人不一样就修改提示词，才让它的审核标准和人类对齐。

## 六、架构设计的启示

Anthropic的Harness架构给我们带来了很多宝贵的启示：

  1. **架构 > 模型**：很多人认为AI应用的核心是模型能力，但Anthropic的实践证明，优秀的架构设计可以让现有模型的能力发挥到极致，带来的提升甚至比换一个更强大的模型还要大。
  2. **专业化 > 通用化**：通用大模型虽然能力全面，但在特定任务上往往不如经过专业化训练的小模型。Harness架构通过角色分工和专业化训练，充分发挥了每个模型的优势。
  3. **工程化 > 算法**：AI应用的落地，80%的问题是工程问题，而不是算法问题。Anthropic的Harness系统并没有提出什么新的算法，而是通过工程化的方法，系统性地解决了智能体应用的各种痛点。
  4. **闭环迭代 > 一次性产出**：软件研发本身就是一个迭代优化的过程，让AI也遵循同样的迭代模式，才能产出高质量的结果。
