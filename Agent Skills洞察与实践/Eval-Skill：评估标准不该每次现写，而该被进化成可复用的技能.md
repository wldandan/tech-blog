# Eval-Skill：评估标准不该每次现写，而该被进化成可复用的技能

> LLM-as-a-Judge 越来越依赖在线生成 rubric 来增强评估质量。每次查询来，先生成一套评估标准，再让 Judge 模型按标准打分。直觉上这条路没问题，"写清楚标准再打分"听起来比"直接打分"可靠。但实验数据是反直觉的：**生成的 rubric 经常让 Judge 模型的表现还不如裸评。** Eval-Skill 换了一个方向：把评估标准做成可复用的领域级技能，离线进化，一次生成，跨查询重复使用。

## rubric 生成的三个坑

在 RewardBench 2 上，Qwen3-8B 裸评的准确率是 57.04%。给它加上自生成的 rubric 之后，跌到 50.63%。**不是微跌，加了更详细的评估标准反而让效果掉了 6.4 个百分点。**

浙江大学和小红书联合团队的论文翻查了 200 个 rubric 引导下的失败案例，用 RIFT 分类法归因，发现三个系统性故障模式。

第一类，**缺失判别标准（34.2%）。** rubric 生成时只看查询文本，看不到实际候选回答之间的差异。如果两个回答在查询层面看起来都应该覆盖的维度上都很完整，rubric 就无法感知"实际区分它们的不是那些通用维度，而是一个查询文本中没有提到的细节"。生成出来的 rubric 列了一堆正确但无用的标准，Judge 按这些标准打分只能给平局。

第二类，**错位或僵化（24.1%）。** rubric 在查询级别过度指定了表面要求。比如查询提到"请回答应简洁"，rubric 就把"简洁"列为最高权重的判别条件。但实际两个候选回答的长度非常接近，真正的区别在逻辑完整性上。Judge 被 rubric 的权重结构引导去关注一个几乎没有区分度的维度，漏掉了真正有效的判别点。

第三类，**criterion-list 格式本身的表达力瓶颈。** rubric 的标准格式是一组独立、无条件的准则列表，逐条打分然后加权加起来。但在实际的偏好判断中，很多推理是条件化的："如果候选 A 包含事实错误，优先选 B；如果两者都有事实错误，再比较格式规范。" 这种 if-then 的决策逻辑在列表式准则中无法自然表达，Judge 只能在自由推理和 rubric 之间来回摇摆。

**这三个故障的根源都指向同一件事：rubric 是从查询生成的，而不是从实际候选回答的差异空间中学习出来的。** 查询只能告诉 rubric 该看什么，不能告诉它不同候选回答之间的区别在哪里。

## 从 rubric 到 skill：评估指导可以只做一次

Eval-Skill 的核心思路是把评估标准从"每个查询生成一次"变成"每个领域进化一次"。一个领域用 100 个进化样本跑搜索和筛选，产出一份 Markdown 技能文件。这份技能不是 criterion-list，而是两部分内容的组合。

**上半部分是 Workflow（评估工作流）。** 告诉 Judge 怎么做评估：先看什么、后看什么、当候选回答在某维度不相上下时怎么分岔、优先级怎么排、平局怎么处理。这不是一列"检查要点"，是一套"操作流程"。

**下半部分是 Principles（评估原则）。** 列出这个领域里反复出现的关键判别维度，每条原则是从实际候选回答对的差异中提炼出来的，而不是从查询文本中推断出来的。一个维度在 100 个样本中频繁出现并有效区分了候选回答，它就会被写进 Principles。

Workflow 和 Principles 在同一个阶段分别进化。Workflow 先被生成和筛选：多分支搜索不同的评估路径结构，在每个分支上用进化集评分选优。然后 Principles 从最好的 Workflow 路径的 rollout 中提取，再经过一轮筛选加回技能组。**两层在进化集中交叉迭代：Workflow 决定了"按什么步骤比较"，Principles 决定了"在每一步中看什么"。**

整个过程是离线的一次性计算。产出的 .md 技能文件被缓存在本地。推理时技能直接注入 Judge 的 prompt 上下文，每轮评估不需要额外的 rubric 生成步骤，不需要额外模型。

## 实验

三种 Judge 底座：Qwen3-4B、Qwen3-8B、DeepSeek-V4-Flash。三个 RM 基准：RewardBench 2、RewardBench、RM-Bench。

**Eval-Skill 在所有底座-基准组合上一致优于裸评和 rubric 引导的基线。** RewardBench 2 上的提升：Qwen3-4B +11.75 个百分点，Qwen3-8B +13.44 个百分点，DeepSeek-V4-Flash +18.51 个百分点。

**进化时间的缩放行为说明了一件事：离线计算投入越多，推理时的收益越大，而且这条曲线没有在 100 个样本处触顶。** 增加进化样本数（从 50 到 100 到 200），技能质量持续提升。对计算预算敏感的场景来说，离线搜索上多花时间可以直接转化为部署时的精度提升，而推理成本完全不变。

跨底座迁移实验更有意思。**用 Qwen3-8B 的 rollouts 进化出来的技能，直接注入 DeepSeek-V4-Flash 的推理上下文，效果比 DeepSeek-V4-Flash 裸评更好。** 反过来也成立。进化出来的技能编码的是"对一个领域的好评估标准"，而不是"某个模型在某个 prompt 下的偏好"。技能的跨底座可迁移性让离线进化的计算成本可以被多底座共享。

下游应用的验证是 best-of-N 推理，用 Eval-Skill 增强的 Judge 来从 N 个候选回答中选最佳。**Eval-Skill 引导的选择在胜率、平局减少和质量一致性三个指标上都优于裸评和 rubric 引导的选择。** 评估技能的改进对下游策略优化有直接的信号价值。

## 几个问题

**每个领域只用了 100 个进化样本。** 这是算法设计和计算预算折中的结果，不是技能性能的上限。如果某个领域的实际评估需求非常细粒度（比如同一个领域内有两个完全不同的子任务类型），100 个样本中可能只有少数覆盖了其中一种子类型，那对应的 Workflow 分支和 Principles 就会偏稀疏。

**Workflow 的候选分支数是手动设定的探索参数。** 论文在 Workflow 生成阶段跑多分支搜索，分支数量是预定义的。在这个参数上做进一步消融（分支太少 vs 分支太多各自的收益曲线）可能揭示一些关于"什么类型的评估任务需要更宽的 Workflow 搜索"的 insights，但论文未展开。

**技能在跨领域的泛化能力边界还不清楚。** 论文验证了同领域内跨底座迁移和跨子领域泛化，但没有测试"从安全领域进化出的技能直接用于对话质量评估"这种大跨度的迁移。不同领域的评估结构可能完全不同，技能的泛化期望在跨领域时应该收敛到多少，目前还是开放的。

## 最后

Eval-Skill 指出了一条跟"每次查询都在线生成 rubric"不同的路：**把评估知识固化为一组领域级可复用技能，离线进化，在线加载，跨模型可迁移。**

100 个样本加一些离线计算，就能让 Qwen3-8B 的评估准确率涨 13 个百分点、DeepSeek-V4-Flash 涨 18 个百分点。这不只是 benchmark 的提升：目前 LLM 评估系统在"知道该怎么评"这件事上远没到收益递减点。问题是 rubric 方式的设计假设（"标准应该从查询中推断"）限制了改进空间。Eval-Skill 的设计假设是"标准应该从实际候选回答差异的分布中归纳出来"。哪条假设更好，数据说了算。
