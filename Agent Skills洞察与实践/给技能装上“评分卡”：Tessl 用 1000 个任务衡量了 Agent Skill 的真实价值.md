# 给技能装上"评分卡"：Tessl 用 1000 个任务衡量了 Agent Skill 的真实价值

> 技能值不值钱，不看它写了多少行 Markdown。看它让 Agent 的行为偏移了多少——以及偏移的方向对不对。

## 500 个技能，一个老问题没人答

Agent Skill 正在变成标配。Anthropic 最早提出这个概念之后，skills.sh、SkillsMP、Tessl Registry 等社区注册中心上已经有几千个技能在流通。这件事本身不奇怪——给 Agent 配领域知识、工作流和规范，比每次都从头写 prompt 高效得多。

但有个问题一直没人系统性地回答：你装了一个技能，然后呢？Agent 的行为真变了吗？是往好的方向变了，还是只是表面换了个格式？变了多少？

Tessl 团队在《A Framework for Evaluating Agentic Skills at Scale》这篇论文里做的就是这件事。他们的目标不是评测 Agent，是**评测技能对 Agent 行为的因果影响**。

做法很直接。从社区找了约 500 个真实开源技能（来自 Anthropic、Google、ElevenLabs、Shopify 等约 100 个组织），自动化生成了约 1000 个可执行评测任务，在 19 个 Agent-模型组合上跑了约 38,000 条有效轨迹——每个任务跑两遍，一遍有技能，一遍没有。

## 评测框架怎么搭的

框架的核心逻辑不复杂。给定一个技能，先生成一批"应该需要这个技能才能做好"的任务，然后同一个 Agent 在有技能和没技能两种条件下分别解，对比差异。分了几步。

第一步，读技能。解析 SKILL.md 和支持文件，搞清楚技能教 Agent 做什么。如果用户给了使用意图（比如"这个技能用在 Python CI 流程里"），也纳进去。全自动模式下，系统靠一组模型推断技能的使用场景。

第二步，搞环境。技能可能依赖 CLI 工具、MCP 服务器、外部 API、认证凭据、运行时、浏览器、数据库甚至特定的 Git 状态。论文把依赖分了 16 类。最常见的三类：工具和 CLI（70.4%）、认证凭据（66.0%）、特定语言运行时（65.2%）。环境工程 Agent 逐一检查，能自动满足的自动满足，满足不了的在全自动模式下就丢弃。

第三步，出任务。从技能内容反推用户会怎么用这个技能，生成任务描述和输入文件。每项技能最多出三个任务。任务描述不透露技能要教的具体步骤，也不泄露评分维度——防止 Agent 看到任务描述就猜出答案。

第四步，验质量。质检 Agent 查四件事：环境可达吗、任务有歧义吗、输入齐全吗、评分标准有没有漏到任务描述里。不合格的直接丢。

最后一步，跑两遍加评分。Agent 不装技能解一遍，装技能再解一遍。两次产物交给评判 Agent（固定用 Sonnet 4.6），对照两套隐藏评分标准打分：目标完成度（产物对不对）和指令遵循度（做法对不对），各 100 分。这两套标准问的是完全不同的事。目标完成度问"输出文件在不在、结果是否正确"。指令遵循度问"有没有用我指定的库、有没有按我的命名规范走、有没有避开我禁掉的写法"。

## 数据说：技能确实管用，但方差巨大

加技能后每个模型都有提升，幅度从 5.5 到 22.1 分。看起来不错。但拆开看有几个更有意思的发现。

**技能改变的是"怎么做"，不是"能不能做"。** 目标完成度这个维度，几乎所有前沿模型在有技能和没技能的条件下都能过 90%，已经饱和了。真正的差距在指令遵循度上。Opus 4.8 得 88.0，Nemotron 3 Nano 30B 只有 25.2。Kimi K2.6 只有 60.1，Qwen3-Coder-Next 只有 57.3，MiniMax 2.7 是 59.0。这些都不是小模型，但指令遵循上被头部甩开 20-30 分。论文团队推测跟训练数据的配方有关，有些模型在训练时缺了足够的"跟着指令走"的数据。

这一点其实挺反直觉的。通常大家觉得模型越大越能"理解"技能内容，但数据说的是：**理解是一回事，遵从是另一回事**。有些模型看懂了技能，但该用自己的老习惯还是用自己的。

便宜模型加技能逼近旗舰模型，这可能是有实际部署意义的发现。GPT-5.4 mini 加技能 84.5，GPT-5.4 加技能 88.2，差距不到 4 分。DeepSeek V4 Flash 加技能 83.9，V4 Pro 加技能 86.6。最夸张的是开源模型 GLM 5.1，加技能后到了 91.1，跟 Sonnet 4.6（91.5）和 Opus 4.8（92.7）基本站在一条线上。再叠上成本：GLM 5.1 每个场景约 $0.89，Sonnet 4.6 要 $1.46，Opus 4.8 要 $3.26。**技能加便宜模型，成本能压到旗舰模型的 1/3 以下。** 对于技能需求明确的生产负载，这个组合比直接砸大模型划算得多。

哪些技能最能打？论文按领域做了统计。前三名是媒体与文件处理（+38.1）、生产力与通信（+32.5）、安全与合规（+30.3）。共同点是技能内容高度程序化，教的是具体操作序列、命令行调用、文件格式规范。Agent 照着抄就行。垫底的是测试与代码质量（+16.7）、科学计算（+17.0）、数据处理（+19.0）。这些领域的技能大多是原则建议和最佳实践，不是操作清单。一个实用的启发是：**能被写成操作清单的知识最适合做成技能；泛泛的指导原则，技能的表达力有限**。

## 不只看总分，还能看每条技能哪里不行

论文区别于已有技能基准（SkillsBench）的一个设计是：框架能诊断单个技能，不只看总分。

以 HuggingFace 的 hf-cli 技能为例。它教 Agent 把旧版 `huggingface-cli` 命令换成新的 `hf` 命令。没技能时 Agent 每次都回退到 `huggingface-cli`。有技能后所有 Hub 操作都换成了 `hf` 前缀。论文把每个评分维度的前后差异列清楚了，你能精确看到：`hf auth whoami` 这条指令被完美遵循（+10 分），`hf models list` 也换过来了（+5 分），哪条指令 Agent 照做了、哪条没理会、哪条写太模糊了 Agent 理解错了——一目了然。

对技能作者来说这个功能挺实在。技能上线前跑一遍评测，你就知道哪些内容 Agent 真的吸收了、哪些是冗余的（Agent 本来就会）、哪些指令得改。以前没有这套工具的时候，这些问题全靠主观判断。

## 局限也摆在那

单评判模型（Sonnet 4.6）是个风险。虽然评分标准写得具体、论文团队做了人工抽查，但在风格审美类的指标上，偏差难免。

数据偏软件工程。500 个公开技能绝大多数是编程、DevOps、Web 开发相关。框架可以用于其他领域，但论文的实证结论不能直接外推。

有技能条件里告知了 Agent "这个技能可用"。真实部署中技能可能装了但 Agent 没意识到——这种情况下的实际收益会低于论文数字。技能选择能力是个独立问题。

约 20% 的技能因为环境依赖太复杂被过滤掉了（要数据库的、要 MCP 服务器的、要预填充外部状态的）。这部分技能的覆盖率是下一步要解决的。

## 一个有意思的转向

这篇论文最让我印象深刻的反而不是某个数据，是它把技能评测的核心问题悄悄转了个向：**从"Agent 能不能完成这个任务"转成了"Agent 是不是按你指定的方式完成了这个任务"**。

2026 年的前沿模型在纯任务完成度上已经很难拉开差距。但指令遵循度的方差大到离谱。这说明"把事做了"和"按你的规矩把事做了"是两码事。技能的价值恰恰在后半句。

这个转向在工程上很实在。维护 Agent 系统的人迟早会遇到这种场景：任务确实完成了，但库不对、命名不对、错误处理不对、部署流程不对。当下看不出来，后面维护全是坑。一个靠谱的技能评测框架应该能在上线前就抓住这些问题。

Tessl 的框架和数据集已经开源了。技能作者可以拿它验证自己写的技能到底有没有用，平台运维者可以用它算"技能 + 模型"的最优组合，研究者可以用它系统性地研究技能对 Agent 行为的因果效应。
