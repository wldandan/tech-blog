# 当AI Agent学会了"肌肉记忆"：Agent Skills技术全景解读

> 你有没有遇到过这种情况：让AI Agent帮你抓个网页数据、生成一份分析报告，结果它反复调用同样的工具，在同一个坑里摔好几次，每次都要从零开始推理。
>
> 这不是你的prompt写得不好。这是 **"程序性鸿沟"（Procedural Gap）**：Agent手里有工具，却不知道怎么用好它们。
>
> 香港中文大学（深圳）的团队最近发了一篇综述 *A Comprehensive Survey on Agent Skills: Taxonomy, Techniques, and Applications*，把这个问题掰开揉碎讲了一遍。下面是我读完之后的梳理，聊聊**Agent Skills（智能体技能）**到底是怎么回事。

# 一、从"能用工具"到"会用技能"

2023年到现在，LLM Agent的标配一直是**工具调用（Tool Use）**。OpenAI的Function Calling，Anthropic的MCP（Model Context Protocol），本质都是在做一件事：**让 Agent能摸到外部能力**。

但"摸得到"跟"用得好"是两码事。实际用起来，槽点一大堆：

- Agent每次接到类似任务都从头推理一遍，token和时间哗哗地流；
- 多个工具的调用顺序、异常处理全靠prompt死撑，任务一长就翻车；
- 团队之间没法共享已验证的工作流，经验全烂在聊天记录里。

论文给Agent Skill下了个很明确的定义：

> **Agent Skill（智能体技能）是一种可复用的程序性工件（Reusable Procedural Artifact）。它编码了"怎么做"的完整知识：不只是做什么，还包括什么时候做、怎么协调多个工具、失败了怎么处理、怎么判断做完了没有。**

形式化地，一个Skill是一个三元组：

```
S = (M, R, C)
```

- **M（Main Instruction）：** 核心指令文档，Agent加载后直接照着执行；
- **R（Auxiliary Resources）：** 辅助资源，参考文档、模板、可执行脚本都算；
- **C（Applicability Conditions）：** 触发条件，告诉系统"什么场景下该用这个技能"。

一句话：**Tool暴露的是原子能力，Skill封装的是上下文里的操作诀窍。** 论文打了一个我很喜欢的比方——Agent负责高层推理和规划，**Skills是它的"肌肉记忆"**。脑子想清楚要干什么，肌肉负责把它干漂亮。一旦程序性知识被外化为技能，Agent就不用在每次执行时重新推理"如何做"，而是直接从技能库调取已验证的行动模式。

这听起来像是"把prompt存起来复用" —— 但两者有本质区别。Prompt模板是静态文本，而Skill是一个**有边界、可版本化、可组合、可治理的工件**。它有自己的生命周期。

# 二、技能的生命周期：四个环节，一个闭环

论文把Agent Skills的生命周期拆成了四个阶段 —— **表征 → 获取 → 检索与选择 → 进化**，形成一个完整的反馈闭环。

```
表征 → 获取 → 检索与选择 → 进化
  ↑                         ↓
  ←—— 执行反馈回到修订阶段 ←——
```

下面一个一个说。

## 2.1 表征（Representation）：技能长什么样？

技能不仅仅是"一段prompt"。论文根据辅助资源R的类型，把技能表征分成三种：

| 类型 | R是什么 | 好用在哪 | 麻烦在哪 |
|------|---------|---------|---------|
| **Text-backed** | 文本参考、模板、评分标准 | 人类友好，无执行依赖 | 执行确定性偏弱 |
| **Code-backed** | 脚本、函数、notebook | 确定性强，可重复跑 | 需要版本管理、测试 |
| **Hybrid** | 文本 + 可执行代码 | 可读性和执行可靠性兼顾 | 维护成本最高 |

三种方案没有绝对的优劣之分。选择取决于场景在"语义可检索性"和"执行可靠性"之间的权衡。

<center>

![alt text](image-1.png)

Skill的典型示例</center>

几个真实世界里的例子：

**Voyager**（Minecraft里的Agent）把成功的游戏操作轨迹存成可执行代码，同时配一段文本描述方便以后检索。典型的Code-backed。

**Claude Code**的skill文件（SKILL.md）更接近Hybrid：一个主指令文档，附带可选的脚本、模板和参考文件。人看得懂，机器跑得动。

## 2.2 获取（Acquisition）：技能从哪来？

这是目前研究最热闹的环节。论文归纳了四条路线，它们之间是互补关系，不是谁替代谁。

<center>

![alt text](image-2.png)

Skill获取方法概览</center>

**① 人类驱动（Human-Derived）：**领域专家直接把经验写成技能。精度高、可控，但扩展慢。目前**SkillNet**（30万+技能）、**ClawHub**（4万+技能）这些平台的种子技能，最初都来自人肉编写。

<center>

![alt text](image-3.png)

人类生成Skill的累计数量随时间变化趋势</center>

**② 经验驱动（Experience-Derived）：** 从Agent自己的执行历史里提炼技能。这条路研究最多，核心区别在于"怎么处理原始经验"：

- **筛选：** Voyager只保留成功的轨迹，SkillCraft过滤出有用的工具使用序列；
- **抽象：** Reflexion把失败转化成纠正性规则，ExpeL从一堆成败里提炼跨任务的经验教训；
- **记忆组织：** G-Memory构建层级记忆图，Nemori把对话历史固化成更稳定的语义记忆；
- **程序化打包：** Agent Workflow Memory（AWM）直接从交互记录里归纳出带触发条件和参数槽的可复用工作流。

<center>

![alt text](image-4.png)

</center>

**③ 任务驱动（Task-Derived）：** 遇到新任务现场造技能，跑通了就留下。CREATOR和ToolMakers把"造技能"和"用技能"拆开，一次创建，后面反复用。

**④ 语料驱动（Corpus-Derived）：** 从文档、代码仓库、接口说明里挖掘技能。AppAgent从UI界面结构里提取操作模式，DS-Agent从Kaggle竞赛报告中提取可复用的解决方案套路。

> 四条路线各有各的用：人类驱动贡献精度和信任，经验驱动贡献接地气的行为模式，任务驱动保证遇到新问题不卡壳，语料驱动提供冷启动的覆盖面。**真正能打的技能库，四条路线都得走。**

## 2.3 检索与选择（Retrieval & Selection）：在正确的时间找到正确的技能

技能库一旦膨胀到成千上万条，"找到合适的技能"本身就变成了瓶颈。论文把这一步精细地拆成**检索（缩小候选集）**和**选择（决定最终执行谁）**。

<center>

![alt text](image-5.png)

技能检索与选择</center>

### 检索：从海量技能中筛出候选

- **稠密嵌入检索（Dense Retrieval）：** 把任务描述和技能文本映射到同一个向量空间，靠语义相似度匹配。Voyager就是这么干的。但"语义最近"不等于"最适合执行"，这是它的软肋。
- **稀疏关键词检索（Sparse Retrieval）：** 靠名称、接口字段、元数据做精确匹配。SAGE和SkillNet用了这条路——当技能元数据稳定时，精确匹配比语义搜索靠谱得多。
- **生成式检索（Generative Retrieval）：** 模型直接生成目标技能的ID，跳过独立的检索索引。ToolGen是代表。
- **结构感知检索（Structure-Aware Retrieval）：** 利用技能的层级结构或依赖关系来引导搜索。SkillRL和AgentSkillOS从粗到细逐层缩小范围；SkillWeaver通过前置条件和依赖过滤，直接排除跑不起来的候选。

### 选择：从候选里挑出该用的

- **上下文感知：** 不搞一次性选择，而是根据环境状态和执行历史动态调整。AutoGuide、MemSkill在做这个。
- **技能组合：** 复杂任务需要的往往不是一个技能，而是一组技能串起来。SkillWeaver、AWM、ASI都支持构建多步技能计划。
- **成本/效用感知：** 选技能时同时看预期收益和执行代价。SkillsBench的实验挺扎心的：即使是精心编排过的技能，在某些任务上也会产生负效用。
- **反馈驱动重排序：** 历史的执行成败直接影响后续的偏好排序。今天犯的错，就是明天排序的信号。

## 2.4 进化（Evolution）：技能也得持续更新

API会变，任务分布会漂移，新的失败模式会冒出来。技能不能写死就扔那不管。论文把技能进化拆成五个子阶段：

<center>

![alt text](image-6.png)

基于分阶段优化的Skill演进机制</center>

| 子阶段 | 要解决的问题 | 谁在做 |
|--------|------------|--------|
| **技能修订（Revision）** | 怎么根据反馈改技能？ | EvoSkill、Memento-Skills、AutoSkill |
| **技能验证（Validation）** | 改完之后真的变好了吗？ | SkillWeaver、ASI、TroVE、PSN |
| **策略耦合（Policy Coupling）** | 技能库和决策策略怎么一起优化？ | SkillRL、ARISE |
| **仓库进化（Repository Evolution）** | 改动怎么在团队/生态里同步？ | Uni-Skill、SkillNet、SkillClaw |
| **运行时治理（Runtime Governance）** | 检索到的技能安全吗、可信吗？ | SkillRouter、AuditedSkill-Graph |

一个典型的进化流水线长这样：技能执行失败 → EvoSkill定位问题并修订 → Memento-Skills跑单元测试门控验证 → PSN通过成熟度门槛决定要不要纳入技能库 → SkillRouter下次任务时检索已更新的版本。

论文特别指出了一个我感触很深的问题：**非对称修订（Asymmetric Revision）**。现在几乎所有系统都擅长"加新技能"，但"安全地改掉或退役旧技能"是明显短板。增长容易，清理太难。SkillClaw搞同步校验、Memento-Skills搞回滚机制，都是在填这个坑。

# 三、生态现状：70万+技能的行业图景

Agent Skills不是纸上谈兵，应用场景已经很广：

| 平台 | 技能规模 | 特点 |
|------|---------|------|
| **SkillsMP** | **70万+** | 最大的公开技能市场 |
| **SkillNet** | **30万+** | 带本体论和多维评估的仓库 |
| **Skills.sh** | **9万+** | 社区驱动，快速迭代 |
| **SkillHub** | **8万+** | 聚焦质量和审查 |
| **ClawHub** | **4万+** | 与 Claw 生态系统深度集成 |

应用场景也远不止聊天和写代码。论文列出了八个主要方向：**软件工程**（调试、重构工作流）、**Web/GUI自动化**（多步界面操作）、**聊天机器人**（记忆更新策略、故障恢复流程）、**机器人**（控制例程的感知-动作耦合）、**金融**（市场分析决策启发式）、**医疗**（诊断推理和治疗规划）、**游戏**（长周期行为的组合复用）、**社交模拟**（约束下的规划-行动-反思循环）。

# 四、还没解决的问题

论文在最后用相当篇幅讨论了开放问题，不是那种"展望未来一切美好"的套路，而是实实在在的工程和研究难题：

1. **抽象质量：** 从乱七八糟的执行轨迹里到底该保留什么、扔掉什么？太细了变成情景记忆，太粗了丢掉操作价值。这个度很难拿。
2. **触发条件薄弱：** 很多技能本身是好用的，但"什么时候该用它"写得不清楚。结果是技能没问题，路由选错了，照样翻车。
3. **多目标选择：** 成功率、成本、延迟、安全性、用户偏好，这些维度的取舍现在全靠拍脑袋，没有一个统一的决策框架。
4. **非对称修订：** 前面说了，加技能容易，删技能和改技能难。
5. **仓库级治理：** 技能可以被人发布、共享、修改——那谁值得信任？一个技能什么时候该退役？出了错谁负责？这些问题比技术本身更难。
6. **因果诊断：** 失败可能出在检索、选择或执行的任何一层，但现在的系统基本没法从"表面日志"追溯到真正的根因。

# 五、当学术框架碰上工程实践：一个正向的相互印证

读完这篇综述，一个感受越来越强烈：**论文从学术角度梳理出的全生命周期框架，跟我们此前从企业落地一线总结出的判断，在每个关键环节都形成了呼应**。

在今年4月的 [《Agent Skills 的全景与思考》](https://zhuanlan.zhihu.com/p/2025509340916794370) 中，我们基于跟一线Agent开发团队的深度交流，提出了一个核心判断：**Agent Skills遇到的不是某一环的单点故障，而是覆盖开发、部署、运行到迭代下线的全流程系统性问题 —— 零散的单点优化治不了根，必须建立技能全生命周期的标准化治理体系。**

这不是巧合。论文从2023年4月到2026年4月的海量文献中梳理论证，我们从企业级Agent项目的真实落地中提炼经验 —— **两条路径相互独立，却在"技能全生命周期管理"这个核心命题上形成了正向的相互印证**。这种产学界的同时发声，本身就说明了一件事：Agent Skills从"能用"到"好用"的跨越，已经到了必须系统性解决的临界点。

### Skill-insight：把生命周期管理做成产品

基于上述判断，我们做了一件事：把技能全生命周期管理的理念落地成一个可用的工具——[**Skill-insight**](https://zhuanlan.zhihu.com/p/2032150871014895927)。

![alt text](image-7.png)

Skill-insight的核心理念很简单：**让Skill从"写完就丢"变成可观测、可量化、可持续改进的闭环**。恰好覆盖了论文框架中"获取 → 检索与选择 → 进化"这几个最关键的阶段：

- **生成阶段**：支持自然语言一句话生成和文档批量提炼（Doc-Pipeline），对应论文的"任务驱动"和"语料驱动"获取路径。特别在批量场景下，Merge Mode能从N份相似文档中提取共性模式、归纳为一个泛化Skill，避免技能库膨胀导致的召回率崩跌——这直接回应了论文中"技能库规模膨胀后检索失效"的核心挑战。
- **评测与优化阶段**：自动生成执行流程图，跟预期流程逐节点比对——偏离、冗余、跳过的步骤一目了然。多维指标面板覆盖准确率、召回率、时延、Token 消耗，并能自动归因定位瓶颈。这跟论文提出的"执行中心化评估"和"因果驱动技能诊断"方向完全一致。
- **质量管控**：L1结构检查 + L2内容检查的自动化验收机制，本质上就是在落地论文所强调的"技能验证"环节——在技能进入可复用仓库之前，先过一道生存考核。

如果用一句话来说 Skill-insight做了什么：**它把论文里讨论的那些演化闭环 —— 修订、验证、再路由 —— 从学术概念变成了开发者打开终端就能跑起来的东西。** 项目地址在 [gitcode.com/openeuler/witty-skill-insight](https://gitcode.com/openeuler/witty-skill-insight)。

### Skill Radar：给技能技术画一张"活地图"

![alt text](image-8.jpg)

如果说Skill-insight解决的是"怎么做"，那[**Skill Radar**](https://zhuanlan.zhihu.com/p/2038679645248282694) 解决的就是"看得清"。

Skill Radar是一个 **Agent技能全生命周期技术的全景雷达图**，Slogan叫"追踪Skills技术，让Agent能力进化有迹可循"。它把技能相关的所有技术按照完整的生命周期脉络整理成了一个结构化的知识体系，覆盖了**Skill 生成、召回、执行、评测、优化、管理**六大核心环节。

这跟论文的分类框架形成了互补 —— 论文给出的是学术视角的taxonomy和方法对比，Skill Radar给出的是一张可以随时查阅的技术演进地图。每项技术都标注了提出时间、适用场景、落地案例、相关开源项目和论文，还设计了行业首个针对Skill全链路的**量化成熟度评估模型**，从技术成熟度、创新性、落地程度、生态活跃度四个维度综合评分。

对于正在做Agent开发、尤其是正在搭建技能体系的团队来说，Skill Radar可以当作论文的"配套工具"来用——论文告诉你每种方法的设计逻辑和trade-off，Skill Radar告诉你每种方法现在到底能不能用、谁在用了、用了之后效果怎么样。完整图谱在 [mangooai.github.io/skills-radar](https://mangooai.github.io/skills-radar/)。

# 六、Skills Are the New Apps

读完这篇综述，我脑子里一直回荡一句话：**LLM Agent正在从"每次从零推理"走向"积累并复用可执行技能"**。

这不是工程优化，是范式变化。

想想看：人类的技能是怎么演化的？从手艺人的体感经验，到工业时代的标准化操作规程，到数字时代的可编程工作流，现在到了 **Agent-Native技能生态**这一步。智能手机时代说"Apps are the new websites"，Agent时代可能是"**Skills are the new Apps**"。

论文提出的四阶段框架 —— 表征、获取、检索与选择、进化——把这块快速膨胀的领域收进了一个统一视角。而我觉得当下最大的机会，不是又搞一个新的技能获取算法，而是**让这四个阶段形成真正正向旋转的飞轮**：获取产出高质量技能，检索在正确时刻激活它们，进化让它们随环境变化持续自我修正。最终的状态是：Agent 像一个经验丰富的工匠，不再需要为每一次挥锤思考怎么握锤子。

回头再看前面聊过的Skill-insight和Skill Radar —— 如果说论文给这层"肌肉记忆"提供了理论框架和学术论证，那Skill-insight 做的事就是**给它装上了视觉神经**，让每一步执行都可见、可测、可追溯；Skill Radar做的事就是**画出了整个技术赛道的导航图**，让每个技术决策都有据可依。这三者放在一起，恰好构成了一个完整的拼图：**理论框架（论文）→ 工程工具（Skill-insight）→ 生态视野（Skill Radar）**。

论文说，Agent Skills是"外部化的程序性知识"。这条路走下去，技能生态可能会成为AI Agent时代的"npm"或者"应用商店"——但在那之前，质量管控、安全审查和互操作性标准还有很长的路要走。**方向已经清楚了，剩下的就是一行一行代码、一个一个 Skill地做下去。**
