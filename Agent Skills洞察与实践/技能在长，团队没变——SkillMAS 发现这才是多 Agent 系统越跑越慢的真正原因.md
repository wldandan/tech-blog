# 技能在长，团队没变——SkillMAS 发现这才是多 Agent 系统越跑越慢的真正原因

一个多 Agent 系统上线之后，会发生什么？

技能库在长。每次成功执行都可能提炼出新技能。每个 Agent 管的技能越来越多，做选择时越来越犹豫。**然后某一天你发现：技能多了，Agent 多了，但系统的成功率反而降了。**

上海交大、中南大学和 OPPO 的联合工作 SkillMAS 给这个现象起了个名字：**适应解耦（Adaptation Decoupling）。** 技能进化是一件事，多 Agent 系统的组织重构是另一件事。现有方案把它们分开做。技能在外面生成好了塞进来，Agent 组织结构什么时候调、怎么调，跟技能完全没关系。**但技能变了，Agent 的职责边界也应该变。不一起变，就会互相拖累。**

SkillMAS 做的事就是把这两件事拧在一起：**从同一条验证过的执行轨迹出发，同时更新技能和更新 Agent 组织结构。**

---

## 一、适应解耦：为什么分开管会出问题

先看一个真实的数字。SkillMAS 在 ALFWorld 上的移植应激测试触目惊心。

系统经过多轮适应后，成功率到达了 94.0%。然后他们把**最终版本的技能库**塞回**初始版本的 Agent 组织**里。技能在，但 Agent 的职责分工没变。成功率从 94.0% 直接崩到 **68.7%，甚至比什么都不做的种子基线（76.1%）还差 7.4 个百分点。**

反过来也崩。把**最终的 Agent 组织**配上**初始版本的技能库**，成功率掉到 **50.0%。**

**技能单独涨，组织不跟着调，涨出来的技能不是在帮忙，是在添乱。** Agent 的路由没有跟着重新规划，每个 Agent 面对的技能池越来越大，选择越来越难，互相之间的职责边界越来越模糊。

这就是适应解耦。两条演化线各自优化各自的，但实际上它们在互相改变对方的运行条件。技能变了，MAS 需要重新规划路由和维护边界；MAS 变了，技能能不能被复用取决于新的 Agent 结构和上下文负载。**分开管，就是在两条各自在跑的传送带上同时作业，迟早踩空。**

---

## 二、SkillMAS 怎么把两件事拧在一起

SkillMAS 的核心理念很简单：**技能进化和 MAS 重构共享同一份证据源，也就是验证过的执行轨迹。** 不是技能一份数据、组织一份数据，而是从同一批 Trace 里同时驱动两个更新。

### 2.1 效用学习：只有真正用到的技能才记功劳

SkillMAS 维护两张效用表：**Skill Utility**（技能在某种任务上的成功率）和**Executor Utility**（某个 Agent 在某种任务上的成功率）。两个效用值**只从验证过的执行轨迹里更新**，不是从检索相似度、不是从 LLM 打分。

这里有一个关键的过滤规则：**检索不等于使用。** 在这次执行中，检索模块返回了哪些技能，这不算。Agent 实际调用了哪些技能、执行轨迹里出现了哪些技能的标识符或匹配模式，这才算"使用"。只有真正被使用的技能才能拿到正面信用。

这个规则的目的是**防止技能效用被"检索到了但没用到"的噪声污染。** 一堆相似技能被检索出来塞进 Prompt，但 Agent 只用了其中一两个。剩下的那些不该因为"被检索到了"就涨分。

效用表用 Monte Carlo 规则更新，更新率随证据累积自动衰减。见过的组合证据越多，新证据的权重越小。这是一个非常轻量的非参数方案，不引入第二个优化器。

### 2.2 有界技能进化：不是所有的失败都值得改

SkillMAS 的技能进化**不是每次失败都触发技能更新。** 它先从保留证据中诊断失败原因：缺了前置条件？步骤顺序错了？检索范围太宽？技能职责太臃肿？只有那些**原因单一、能从轨迹中明确识别、改动范围有限**的失败，才会触发技能修复。

修复类型被限定为四类：**加前置守卫**（技能在什么条件下才该触发）、**重排步骤**、**收紧检索范围**、**拆分过载技能**。不是重新生成一份技能，是在现有技能的基础上做一次手术级的精准修改。

对于成功轨迹，处理同样克制：提取可复用模式，但需跟已有技能和策略卡做去重对比。**新建或大幅改写的技能不会立刻入库，而是先进验证池，后面的轮次有了足够使用证据才会正式接收。**

这套机制的核心目的是**控制技能库膨胀。** 无过滤的技能增长会让每个 Agent 的选择空间指数级变大，路由越来越不准。

### 2.3 证据门控的 MAS 重构：不是技能修不好就要改组织

技能修了一轮，还有残留的失败。这时候是不是该改 Agent 组织？SkillMAS 的做法是：**拿证据说话。**

系统先构建结构化的诊断工件：被牵连的 Agent 是谁、它们的效用值最近在涨还是在跌、技能在它们之间是不是有重叠、刚做完的技能修改有没有改善这些失败。然后判断：**当前的残留失败到底是因为技能还没修好，还是因为 Agent 的组织结构本身就是瓶颈？**

如果证据指向后者，比如两个 Agent 在同一个任务家族上效用值都很低且波动大、大量技能在这两个 Agent 之间共享但路由不稳定，MAS 重构才会被触发。

重构操作同样是有界的，每次只做一件事：**保持不变、新增一个专业 Agent**（把某个任务家族从过载 Agent 上拆出来）、**删除或合并冗余 Agent**、**修改 Agent 的职责边界和技能归属**。

**不拍脑袋，不看任务复杂度，只看证据。** Agent 结构调整的理由必须能在 Trace 里找到对应的失败模式和效用数据。没有证据，就不改。

## 三、实验说了什么

ALFWorld（具身家居，134 个未见任务）、Lifelong AgentBench OS Task（命令行操作，150 个任务）、τ-Bench Retail（零售服务，114 个任务），都用一个 GPT-4o-mini 跑。

### 三领域全部最高

**ALFWorld 94.0%**，跟最强的已报告分数（Traj-Bootstrap 93.0%）持平。**Lifelong AgentBench 76.7%**，超过 CDMem（68.0%）和 Traj-Bootstrap（70.0%）一大截。**τ-Bench Retail 70.2%**。

### 适应是定向的，不是均匀的

ALFWorld 六个任务家族分开看：**examine 从 27.8% 涨到 94.4%，净增 12 个成功。** 这个涨幅跟搜索和检查导向的专业化直接相关。系统发现 examine 是最弱的环节，围绕它生成了专门的搜索技能和 Agent。其他五个任务家族的种子基线本身就不低（76-87%），涨得不多但稳定。

### 系统在变得更紧凑，而不是更臃肿

ALFWorld 的技能数从 5 个长到 13 个，没有指数爆炸。OS Task 上技能在第 3 轮到达 10 个、成功率 78%，第 4 轮技能数不变但成功率反降到 75%，说明**技能够用就行，多了反而干扰。**

τ-Bench 的轨迹更有意思：终点 checkpoint 只保留了一个 Agent，成功率从 43/74 涨到 51/74。如果把 checkpoint 扩展成多 Agent，性能反而降到 32/74。**在这个领域，增益来自更好的技能使用而非更多的 Agent。** 两种领域，两种结构模式，SkillMAS 都能自适应地收敛到对应的最优结构。

## 四、局限与总结

论文坦诚：当前版本没有做控制变量的策略冻结消融实验（比如冻结策略参数、只让技能或只让 MAS 进化），所以无法精确量化每个组件的因果贡献。诊断和重构的判定规则目前是领域局部的启发式，跨领域迁移需要重新定义。实验规模（每个领域一个基座模型、固定任务集）限制了结论的泛化范围。

但 SkillMAS 最值得记住的不是它的数字。是它识别出的那个系统级矛盾：**技能进化和 Agent 组织结构是一对耦合变量。你把它们拆开管，每个单独看都在变好，放在一起反而更差。** 技能涨了组织不调，新技能抢旧技能的路由，Agent 不知道该听谁的，最终不如什么都不做。那个移植应激测试的数据（94.0%→68.7%，低于种子基线的 76.1%）是这篇论文最重要的数字。

如果你在维护一个多 Agent 系统，这篇论文给了一条简单但反直觉的建议：**别只盯着技能库有没有在长。定期检查你的 Agent 组织还是不是适配当前的技能分布。不适配的话，加技能不如减 Agent、改职责。** 技能和 Agent 一起进化才是进化，单独一个长了另一个不动，那不是进化，是内耗。
