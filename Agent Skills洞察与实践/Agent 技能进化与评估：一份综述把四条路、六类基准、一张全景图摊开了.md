# Agent 技能进化与评估：一份综述把四条路、六类基准、一张全景图摊开了

> 技能生态在膨胀，但评估和进化框架没跟上。一篇来自罗格斯大学的综述把技能进化拆成了四条路（执行反馈、轨迹蒸馏、压缩增强、强化学习），把技能基准拆成了六类（验证类、效用类、安全类、检索路由类、生成类、真实环境类），然后指出了一件之前没人系统说过的事：**现有基准能测"这个技能好不好"，但测不了"这个技能为什么不好"以及"怎么让它变好"。** 进化策略的粒度卡在基准提供的反馈信号密度上：基准只给一个 outcome，进化策略就只能在 outcome 级别做优化。

## 两条主线和四个分支

Agent 技能已经不是早期单纯的 prompt 片段。一条技能的形式化定义是 S = (C, π, T, R)：适用条件 C 决定了什么时候激活这个技能，执行策略 π 编码了该怎么做，终止标准 T 定义了什么时候算做完，可复用接口 R 决定了这个技能怎么和其他技能组合。

技能的生命周期横跨三个环节：创建、使用、进化。创建把人的经验或模型的生成结果封装成技能包；使用包括检索（从大库里挑小集）、路由（在运行时决定这一步该调哪个技能）和管理（保持技能库有序、新鲜、安全）；进化是这篇综述的重点：**技能上线之后怎么持续变好。**

综述把技能进化拆成了四条路，按学习信号的来源和粒度区分。

**第一条路：执行反馈（Execution Feedback）。** 拿技能跑一遍、看哪一步出错了、根据错误改技能。SkillForge 把失败记录结构化为根因分析和修订建议，CO-EvoSkills 用验证器定位为什么失败并生成修改方案，Skills-Coach 在合成案例上执行技能以产生进化反馈。**这条路上一个关键的设计分叉是"结构化诊断 vs 原始轨迹"：做了显式故障分离的系统（SkillForge、CO-EvoSkills）在跨任务结果上往往优于直接在原始轨迹上工作的系统（AutoSkill、SkillClaw）。** 但头部对比目前还是缺失的：各家用各家的基准、各家的指标。

**第二条路：轨迹蒸馏（Trajectory Distillation）。** 不是只看单次执行的报错，而是从多条成功和失败轨迹中提取可复用的模式。Trace2Skill 从多条轨迹中生成定向 patch，将多个修复合并为一份无冲突的技能文件。Memento-Skills 引入了一整套读写反射循环：router 检索相关技能给 Agent 执行，Agent 根据执行反馈更新技能，形成迭代精炼和长期行为记忆。XSkill 跨到了视觉领域：从视觉观察中提炼技能，同时做任务级技能抽象（通过视觉摘要）和动作级经验提取（通过跨 rollout 的优劣对照）。

**第三条路：压缩与增强（Compression & Augmentation）。** 技能库越滚越大，重叠和冲突导致冗余探索和泛化退化。SkillNet、SkillReducer、SkillX 在做的事就是把技能压缩到保持关键信息的同时减少 token 开销。SkillFoundry 是增强方向，补充缺失的知识和能力维度。这条路的一个可用线索是**让压缩和增强的决策锚定在实时信号上（检索频率、运行时失败率），而不是只靠离线设计。**

**第四条路：强化学习（Reinforcement Learning）。** 用 RL 来对齐技能执行质量和任务奖励。但标准 RL 每次更新只奖励单个任务，而技能的价值在跨任务的可复用性。D2Skill 用 GRPO 的多 rollout 策略为每个任务跑两次（有技能和无技能），两次的成功率差作为这个技能在当前上下文中的边际效用估计，比单次 reward 信号更稳定。SkillRL 同时在成功和失败信号上做策略更新，把失败模式诊断和技能检索的联合优化融进了一个 RL 循环。

**四条路不是互斥的。** 实际上，RL 路经常依赖执行反馈路和轨迹蒸馏路来做奖励塑形，压缩增强路又需要 RL 路来评估压缩决策在任务质量上的影响。但综述的一张对比表（Table 1）暴露了一个结构性问题：绝大多数进化方法在同一张基准上没有被头对头测过。方法之间的相对优势目前更多来自设计分析，而非实证对比。

## 六类基准，每类测一个维度的东西

综述把技能基准按测量目标分成了六类，每类的覆盖面有一个清晰但有限的半径。

**验证类基准。** 测功能正确性。SkillsBench 是目前最系统的：94 个任务，8 个领域，确定性验证器。它能告诉你技能能不能完成任务，但不能告诉你为什么完不成。

**效用类基准。** 测效率和质量之间的 trade-off。SkillFlow 评估技能检索的精度和召回率，SRA-Bench 评估技能增强对下游成功率的边际贡献。Zhong et al 的工作直接测量不同模型骨架下同一技能的效果差异。

**安全类基准。** 测技能的恶意和脆弱性。SkillInject 是提示注入的对抗样本集，MalSkillBench 是目前唯一做了 runtime 验证的恶意技能基准。

**检索和路由类基准。** 测技能选择和匹配能力。Skill-Router 在 8 万技能上做检索精度测试，R3-Skill 测跨语言和跨模态的技能兼容性。

**生成类基准。** 测技能构造质量。WSASkill 用行为重放一致性来衡量技能生成是否保留了原始轨迹的关键操作逻辑。

**真实环境类基准。** SWE-Skills 在真实 GitHub 仓库上做软件工程技能评估，SWE-bench 和 Terminal-Bench 间接评估嵌入在 Agent 流程中的技能有效性。

**一个跨基准的 meta-observation 是：现有基准几乎全部聚焦在"技能的效果测量"，没有一个基准被设计用来诊断"技能的哪个部分在执行中失效了"。** 当前生态中，进化策略可以知道你该改技能，但基准本身不提供细粒度的诊断信号让你知道该改哪里。进化方法的粒度上限被基准的反馈粒度锁住。

## 四类进化分别在哪些基准上能被有效评估

综述做了一张进化策略和基准类别的对齐表。执行反馈类方法在验证类和效用类基准上有直接的信号映射（执行结果就是基准输出）。轨迹蒸馏类在生成类和真实环境类基准上能找到最接近的信号（轨迹覆盖面广、场景差异大）。压缩增强类在效用类基准上做成本-质量 trade-off 评估最合理。RL 类需要验证类基准的确定性奖励信号，但 RL 方法真正的优势（跨任务可复用性）在效用类基准上才能被充分测量。

**一条关键 design recommendation 是：基准给出的信号粒度应该匹配进化策略的操作粒度。** 如果一个进化方法在单步执行级别做修正（如 Skills-Coach），它需要基准能输出步骤级的反馈而不是任务级的 binary pass/fail。如果基准只能输出最终成功率，那进化策略就只能做 trajectory-level 或 library-level 的优化，步骤级的精确修改在缺乏对应信号的情况下容易变成随机扰动。

## 最后的开放方向

综述列了几个当前生态中的硬缺口。

**多模态技能的评估缺口。** XSkill 已经在做视觉-语言技能的学习，但所有的 skill benchmark 目前都是纯文本的。图像理解、GUI 操作、视频分析技能的评估基准几乎为零。

**安全基准的覆盖率差异是结构性的。** 代码注入检测的基准已经有 MalSkillBench，但提示注入检测、技能依赖链投毒、跨技能的权限泄漏，这些目前没有一个统一的评估维度。不同类型的攻击在同一个检测工具上的表现差异可以达到 66 个召回点，但当前的基准生态无法让用户感知到这个差异。

**长期进化的漂移检测基本被忽略了。** 技能在连续进化中发生的内容漂移、跟原始意图的逐渐偏离、跟部署环境的不适配，这些没有自动化的检测机制。技能版本管理目前靠人工 review，而技能库的规模让这种方式不可持续。

## 最后

这篇综述的价值不在发现新东西，而在于把一堆分散的论文码成了两条清晰的轴线：**四条进化策略讲的是"怎么改"，六类基准讲的是"改完之后怎么测"，两者之间的匹配关系决定了一个领域当前能做多好的工程。** 目前信号最密的地方是执行反馈 → 验证类基准（最近的一条通路），信号最弱的地方是 RL 进化 → 安全基准（几乎没有通路）。

对正在做技能系统的团队来说，一个直接的用途是拿来对表：进化策略属于哪一类，基准覆盖了哪一类，中间的信号粒度匹不匹配。
