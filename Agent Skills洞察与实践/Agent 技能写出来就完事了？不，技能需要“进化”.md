# Agent 技能写出来就完事了？不，技能需要"进化"

> 大多数方法把构建技能当成一次性提取，轨迹里抽出来、存进库、完事。但太具体的技能迁移不了，太抽象的又没有指导力。浙大和通义实验室的SkillComposer把技能构建拆成了三个可学习的操作：创建、改进、合并。技能不是写出来的，是进化出来的。

# 技能好造，好技能难造

技能（Skill）已经是Agent生态的标配。Claude在 2025 年 10 月推出Agent Skills机制后，社区在不到半年内贡献了超过 28 万个公开技能。通过给Agent加载一个SKILL.md文件就能注入领域知识和操作流程，这种方式比微调轻量，比few-shot prompt稳定，迅速成为主流。

但热闹背后，一个基础问题很少被讨论：**这些技能本身是怎么造出来的？质量由谁兜底**？目前两条路，各有各的死穴。

第一条路是专家手写。质量有保证，但太贵太慢，28 万个技能、每天还在涨，靠人写根本追不上。

第二条路是让模型从成功轨迹里自动提取。Agent 跑通一个任务，把执行轨迹丢给 LLM，让它蒸馏成一个 SKILL.md。这条路可规模化，但实际效果极不稳定。SkillsBench 的评测数据很直接：模型自己生成的技能，**平均增益是零，甚至可能为负**。不加控制地自动提取，造出来的技能经常比没有还糟。

问题出在哪？论文的诊断切中要害：**技能构建存在一个根本性的张力——专化（specification）和泛化（generalization）是正交的两个质量维度，但现有方法把它们当一件事做。**

从具体任务轨迹里直接提取的技能，天然偏"专"：它沾染了那个任务的特定上下文，换一个看似相似的场景就失效。反过来，如果试图写得更抽象、覆盖面更广，又会丢失足够落地的细节指导力，Agent 看了等于没看。一个追求专化，一个追求泛化，但现有方法既没有显式的泛化机制（只会一对一提取），也没有显式的专化机制（提取出来什么样就是什么样），两个方向都做不到位。

SkillComposer的核心贡献，就是把这个张力拆开，用三个可学习的操作分别处理 —— Create负责初始提取，Merge驱动泛化，Improve驱动专化。


# 三个操作，一个进化生命周期

SkillComposer的三个操作不是简单的功能罗列。它们分别对应技能生命周期中的三种状态转换，各自解决一个特定的质量问题。

## Create：从轨迹到技能，完成第一次提取

Create是整个生命周期的起点。输入是一条没有技能指导的原始执行轨迹，也就是Agent纯靠自身能力跑完一个任务的完整记录。LLM审视这条轨迹，从中抽取出**可复用的过程性知识**：先做什么、再做什么、遇到什么情况该怎么处理、有哪些坑要避开。

这里的关键约束是**可复用**。Create提取的技能不能只适用于当前这条轨迹，而是要能指导同一类任务。这意味着LLM必须区分"这条轨迹特有的偶然操作"和"这类任务通用的有效策略"，只抽出后者。生成的每个技能有三个组件：名称（name，一旦确定不可修改，要求尽量抽象）、描述（description，触发条件和功能概要）、正文（body，markdown格式的操作指令）。

## Merge：从多到精，驱动泛化

技能一旦开始积累，库就会迅速膨胀。相同类型的任务跑了多次，产生了多条轨迹、多个技能，这些技能之间可能存在大量语义重叠。比如三个不同任务各自Create出来的技能，本质上都在讲"二次代价函数下的贪心资源分配"，只是表述不同、侧重点不同。

Merge解决的就是这个问题。它通过**多视图语义相似度**来识别可合并的技能对，分别在name、description、body三个维度上计算相似度然后取均值。超过阈值 δ 的配对才触发合并。合并后的技能要覆盖原有两个技能的全部有效指导，同时去除冗余和矛盾。合并后的技能更宽、更泛化，适用范围更广。

这一步是整个框架中**驱动泛化（generalization）**的核心机制。它让技能库从"一堆高度特化的补丁"收敛为"一套可跨任务迁移的原则性指导"。

## Improve：从粗到细，驱动专化

泛化技能虽然覆盖面广，但面对一个具体的新任务时可能不够精细。比如一个泛化的"二次代价资源分配"技能，指导Agent解决某个特定约束下的变体问题时，可能漏掉了关键的边界条件。

Improve解决的是这个问题。Agent拿着已有技能去执行新任务时，出来的新轨迹会暴露原技能没覆盖到的模式和坑。Improve操作审视这条新轨迹，把它跟原技能做对比，找出"轨迹里做到了但技能没写"和"轨迹里犯了错但技能没预警"的部分，注入原技能生成一个改进版。

**值得强调的是Improve的防退化机制。** 如果新轨迹没有提供有用的改进信号，比如任务本身就很简单、轨迹里没有任何新信息，Improve被设计为直接返回原技能不做修改。这避免了"为了改而改"引入噪声。

这一步是框架中**驱动专化（specification）**的核心机制，让泛化技能在具体任务上真正落地。

## 如何部署：三种模式的组合逻辑

三个操作的不同排列，对应三种部署场景：

**Offline模式**走 Create → Merge 路径。在训练集上批量Create原始技能，然后迭代Merge直到没有技能对的相似度超过阈值。产出一个紧凑的泛化技能库。推理时对每个新任务检索top-k技能，executor自己判断是否适用。适合有历史数据积累、追求通用覆盖的场景。

**Online模式**走 Create → Improve 路径。每个新任务来了，先从原始轨迹Create一个初始技能，然后反复执行"跑任务 → 看轨迹 → Improve 技能"的循环（默认 5 轮）。零先验数据，完全在线迭代。适合开放域或流式场景。

**Hybrid模式**将两者串起来。先用Offline的泛化技能库做冷启动，再对每个新任务跑Online的迭代Improve。泛化库给了一个好的起点，专化循环让它精准落地。两边之长都占。

# 怎么让模型学会这三个操作？拒绝采样的设计逻辑

任何LLM都可以zero-shot执行Create、Merge、Improve，但质量方差巨大。核心问题是：**没有ground-truth技能来监督训练。** 你不知道一个"好"技能长什么样。

SkillComposer的做法是绕开"定义好技能"这个问题，转而问一个更可操作的问题：**"用了这个技能之后，Agent的表现比不用的时候好了多少？"**

这就是**Delta Pass Rate过滤**的逻辑。关键设计在于，三种操作的基线各不相同：

- **Create的基线是"不用技能"**：Executor不带技能跑一次任务，记下pass@1；带候选技能再跑一次。带技能的pass@1减去不带技能的pass@1必须超过阈值$ε_c$，否则候选技能被丢弃。这保证了每个入库的技能有可量化的正向贡献。
- **Merge的基线是"各自原来的技能"**：合并后的技能在两个源任务上的pass@1都要优于各自原来的技能。取平均值，超过$ε_m$才保留。这保证合并不会丢失信息，只会增强泛化。
- **Improve的基线是"改进前的原技能"**：改进后的技能在源任务和新任务上的pass@1都要优于原技能。取平均，超过$ε_i$才保留。同时包含同任务对和跨任务对，让训练信号同时覆盖专化（同一个任务做得更好了）和泛化（换了任务也更好）。

这个设计的精巧之处在于：**完全不需要人类标注。**质量信号的唯一来源是executor的表现变化。最终用约7000条通过拒绝采样的高质量样本，SFT一个Qwen3.5-4B模型。训练出来的SkillComposer-4B比zero-shot的Qwen3.5-4B在τ2-Bench Online模式上高出 2.0 个点，确认了训练的有效性。


# 实验讲了什么

三个 benchmark：τ2-Bench（客服 Agent，三领域）、LiveCodeBench v6（代码生成）、AppWorld（训练中完全未见）。

### 整体表现

![alt text](image-1.png)

SkillComposer-4B 在所有 benchmark 上跑赢 No Skill 和 MemP 基线。τ2-Bench 上 Hybrid 模式用 4B executor 到 85.7（+6.2），用 27B executor 到 88.0（+4.5）。LiveCodeBench 上 Online 模式用 4B 到 59.1（+2.5），用 27B 到 83.1（+3.4）。对照组 MemP 在 LiveCodeBench 上从 56.6 掉到 47.4，比 No Skill 还差一截，印证了不加控制的技能提取可能帮倒忙。

### 消融实验验证了Merge和Improve的正交性

Offline 模式下：只有 Create 时 τ2-Bench 73.8；加上 Merge 到 77.5（+3.7），Merge 是 Offline 模式下贡献最大的操作，因为它驱动泛化；加上 Improve 到 75.6（+1.8），但 LiveCodeBench 降了 1.2，光专化不泛化反而损伤迁移性；三个全上的效果略低于 Create+Merge，说明在离线阶段无差别做专化会部分抵消 Merge 的泛化收益。t-SNE 可视化佐证了 Merge 后技能分布明显变稀疏。

### 跨模型、跨领域、跨任务类型全部有效

一个用 4B 训练的 SkillComposer 产出的技能直接给 27B executor 用，效果不减反增。训练数据只来自 τ2-Bench 的 Retail 领域，Airline（+13.7）和 Telecom（+2.5）照样涨。最值得关注的是：**只用 Agent 数据训练的 SkillComposer，在代码任务上也涨了**（27B executor 从 81.4 到 82.9），说明技能编排能力是一种可以跨任务类型迁移的元能力。在完全未见过的 AppWorld 上，SkillComposer 同时提升了准确率（+2.5）和通过率（+0.2），而未训练的 Composer 虽然准确率也涨了但**通过率下降了 2.1 个百分点**，拒绝采样的过滤效应在这里很直观。

### 迭代进化持续跑赢暴力重复采样

相同推理预算 k 轮，SkillComposer 在线进化一直领先于 vanilla pass@k，而且 k 越大差距越明显。k=5 时 73.71 vs 69.71（4B），差 4 个点。说明每轮改进是在上一轮的肩膀上往前走，不是每次从零开始。

# 所以这到底意味着什么

SkillComposer把"技能构建"从一个一次性动作重新定义为**由三个可学习操作驱动的持续进化过程**。在这之前，大家对技能的态度基本是"写好了放那就行"。这篇工作给了一个更动态的视角：**技能应该像代码一样被重构，合并重复（Merge）、补充细节（Improve）、持续迭代。** 而且这个过程可以训进模型本身。

拒绝采样 + delta pass rate这套训练recipe也值得单拎出来。它不需要人类标注，完全靠executor的表现反馈决定技能好坏，为技能质量控制的自动化提供了一条切实可行的路径。

没解决的问题主要在工程侧：拒绝采样需要每任务多次推理来算delta pass@k，训练数据收集的计算成本不低；实验目前只覆盖了Qwen系列。但这些都不影响核心思路的有效性。

对正在搭技能系统的团队，这篇论文最值得带走的是：**别把技能当静态资产。给系统加上Create、Merge、Improve三个操作，让技能生态自己进化。结构化的进化，比堆积更多技能有用得多。**
