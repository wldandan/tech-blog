# SkillWiki：当 Agent 技能需要一个"GitHub"

> 知识管理有维基百科，软件工程有 GitHub。Agent 技能呢？论文 2606.16523v1 给出的答案是 SkillWiki。

## 技能越多，问题越大

LLM Agent 这几年发展很快。从最早的单一工具调用，到现在能编排几十个步骤完成一个任务，Agent 需要掌握的技能数量也在同步暴涨。工具使用、界面操控、工作流执行、多 Agent 协调，再加上每个垂直领域的专业能力——一个稍微复杂的 Agent 系统，技能库里塞几百条技能很正常。

更麻烦的是，技能来源也在变多。以前技能靠人手写 prompt 和工具描述，现在 Agent 还能从执行轨迹里学、从环境交互里学、从文档里提取、从历史经验里归纳。技能的生产方式从"手工打造"变成了"机器批量"。

但技能管理的方式几乎没变。大部分团队的做法是：把 prompt 模板扔进一个文件夹，需要的时候靠关键词搜。技能之间什么关系、某条技能是从哪个知识源来的、改了哪些版本、被谁调过、成功率怎样——这些问题没人回答得了。

学术界的情况也没好到哪去。过去两年的研究集中在技能生命周期的单个环节上——有人做技能获取，有人做技能记忆，有人做技能演化——但都是把技能当成单个 Agent 的能力组件来优化。当技能从几十条涨到几千条时，真正让人头疼的已经不是"怎么生成新技能"，而是**组织、溯源、治理和持续演化**这些元问题。

哈工大、腾讯和南洋理工的联合团队在 SkillWiki 这篇论文里把这个问题讲得很直白：Agent 技能领域缺的不是算法，**缺的是一套像 Git 和 GitHub 之于代码那样的基础设施**。

## SkillWiki 的思路

SkillWiki 把自己定位成一个**技能生产与治理系统**，而不是又一个技能库。这个区别很关键。

技能库只管"存"，SkillWiki 管的是"生产—组织—治理—演化"的完整闭环。你可以把它想象成一条持续运转的流水线：一头不断喂入各种知识原料（执行轨迹、技术文档、API 规范、脚本、历史技能、Agent 执行记录），另一头持续产出可复用、可验证、可追踪的技能资产。技能上线后，系统还会监控它的运行状态，发现退化就触发修复流程，修完发布新版本，整个过程自动完成。

具体来说它做了几件事：

**知识到技能的转化。** 这一点和常见的"把文档喂给 LLM 生成 prompt"完全不同。SkillWiki 做的是从原始知识材料中抽取可复用的操作模式和工作流模式——它关注的是"怎么做"，不是"怎么说"。原始知识材料在整个过程中保持不动，作为长期证据资产保留，技能和它的知识来源之间始终维持一条显式的可追溯链接。

**技能溯源图谱。** 这是 SkillWiki 最核心的数据结构。系统维护一张异构图，把知识来源、技能、工具、验证记录、执行历史、版本沿革全部连在一起。这意味着你点开任何一条技能，都能看到它完整的"履历"：从哪个文档或轨迹产生的、经过了哪些验证、被哪些 Agent 调用过、从 1.0.0 到 1.1.0 改了什么、谁做的修改决策。对于大规模技能仓库来说，没有这种溯源能力基本上就是盲飞。

**执行驱动的闭环演化。** 技能上线后，SkillWiki 的自管理 Agent 会持续收集四类运行时信号：执行次数和成功率、失败证据和重复故障模式、从失败中生成的反思记忆、以及生命周期状态变更。当系统发现某条技能成功率跌到 90% 以下或者反复出现同一个失败模式，它会自动生成修复提案，推送到治理工作流，走完审查、验证、发布的全流程。论文里有一个具体案例：系统检测到 2 次执行失败，聚类出 1 个重复故障模式，自动生成了修复方案，然后发布了 1.1.0 版本。

## 一项技能在 SkillWiki 里长什么样

在 SkillWiki 的建模里，技能不是一段 prompt 文本。每项技能封装了八个维度的信息：身份标识和分类、生命周期状态、输入输出接口、可执行实现体、评估合约（验证标准）、溯源信息（关联的知识来源）、图谱关系（和其他技能与工具的连接）、以及运行时指标（调用次数、成功率、延迟）。这八个维度合在一起，技能就从一个"文本片段"变成了一个**可以被版本化、验证、治理和持续演化的能力资产**。

论文还给技能串了一条完整的生命周期线：S0 原始经验 → S1 候选 → S2 草稿 → S3 已验证 → S4 已发布 → S5 已退化 → S6 已弃用 → S7 已归档。这八个状态不是拍脑袋分的——每个状态之间都有明确的转换条件和验证门禁。从 S2 到 S3 要过验证门禁（Harness Verification Gate），从 S4 到 S5 由执行反馈驱动（连续失败或性能退化触发），从 S5 到 S4 的修复路径需要走完整的治理评审流程。

技能还有一个三层能力分类。原子技能负责基础操作（单次 API 调用、DOM 元素定位），功能技能负责可复用的任务（填写表单、生成报告），战略技能负责高层规划（多 Agent 任务分配、工作流编排）。

## Git 风格的治理，但对象是技能

如果你做过代码评审，SkillWiki 的治理工作流你一看就懂。技能的创建、修复、分解、合并和版本更新，全部以"候选变更"的形式提交，然后依次经过快照、结构化差异分析、审查和发布。内置的元技能和自管理 Agent 会自动识别接口变了没有、实现改了哪里、依赖有没有断裂、评估合约需不需要更新——有破坏性变更的时候会标记出来。

比较有意思的是它的权限设计：默认情况下治理流程完全自主运行，不需要人盯着。但你随时可以介入任何一个阶段，而且**人的决策永远覆盖自主治理行为**。论文团队显然意识到了全自动治理在关键场景下的风险，所以留了这个覆盖机制。

## 实验数据

评测用了一个 125 件制品的语料库，覆盖五种类别：执行轨迹、技术文档、API 规范、脚本和历史技能。走完完整的知识到技能生产流水线后，99 件被成功转化为受治理的技能候选（79.2%）。转化失败的集中在文档类和历史技能类，这两类的输入结构比较松散，自动化抽取的难度更高。

生命周期治理方面，团队做了一个全链路案例：一件 API 文档制品走完了从 S0 到 S7 的全部八个状态。在这个过程中系统记录了 7 次状态转换、3 次执行（2 成功 1 失败）、1 个重复故障聚类、1 次版本更新（1.0.0 → 1.1.0，成功率从 90% 修复到 100%），最后是弃用和归档。整个过程产生的快照差异、验证记录、溯源链接全部保留在系统中。

## 坦白局：局限

论文没有回避当前系统的短板。

最明显的是规模问题。现有实验在百级技能仓库上跑的，几万条技能的场景会怎样——依赖会不会爆炸、图谱查询会不会卡、治理工作流的并发冲突怎么处理——论文直接说"还没系统性评估"。这是诚实的，也意味着 SkillWiki 目前更像一个原型验证，离真正的生产级基础设施还有距离。

第二个点是下游影响没测。论文评估了"能不能把知识转成技能""能不能管好技能的生命周期"，但没有测 SkillWiki 治理过的技能仓库到底能不能让 Agent 的任务完成率更高。这个因果关系不建立起来，工程团队很难判断迁移成本值不值得。

第三是长期稳定性的问题。一个持续演化的技能生态系统，跑了半年一年之后会收敛还是漂移？技能版本不断滚动，整个生态的兼容性和一致性怎么保证？论文把这个问题标注为"开放研究问题"——确实目前没人能回答。

## 最后

我对这篇论文的兴趣不在于它又刷新了什么基准分数。它做了一件更基础的事：**把技能管理从一个"算法优化问题"重新定义成了"基础设施问题"**。

这个视角转换是务实的。Agent 在往生产环境走，技能正在变成一种需要长期维护的能力资产。代码有 Git，数据有数据库，配置有配置中心，而技能——目前大部分团队还在用文件夹管理。SkillWiki 是不是最终答案不好说，但它把"技能基础设施"这个问题本身讲清楚了。论文和代码都公开了，无论你是正在搭 Agent 系统还是在头疼技能仓库的膨胀问题，都值得翻一翻。
