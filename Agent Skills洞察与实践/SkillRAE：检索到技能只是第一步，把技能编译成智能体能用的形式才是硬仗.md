# SkillRAE：检索到技能只是第一步，把技能"编译"成智能体能用的形式才是硬仗

> 来自港中深的RAE新工作。他们没有发明更好的检索算法，而是在检索和执行之间塞了一个"编译器"——把搜出来的技能和子单元重新组织成紧凑、可执行的上下文。这个看似不起眼的中间环节，带来了 11.7% 的提升。

# 一、RAE的"最后一公里"：搜到了，然后呢？

LLM智能体越来越依赖**可复用的技能库**来处理复杂任务。OpenClaw、Manus这些系统跑的不是简单的API调用，而是打包好的技能。每个技能是一套完整的工作流：执行步骤、文件约定、工具脚本、输出格式要求，全塞在里面。SkillsBench里有**200+**个这样的技能，AgentSkillOS更是把技能组织到了生态系统级别。

技能多了，自然需要一个检索机制。这就是**Retrieval-Augmented Execution（RAE）** —— 先检索相关技能，再把它们喂给执行器去完成任务。

问题出在哪？**现有工作几乎全部精力都花在"搜得更准"上了。** 技能路由方法（SkillRouter）专注于从大规模技能池里挑出相关的技能；仓库感知检索（RepoCoder、RepoGraph）利用代码结构和数据流提升证据访问；执行规划方法（HuggingGPT、ToolPlanner）把技能编排成多步执行计划。

但有个关键环节被跳过了：**搜出来的技能怎么变成执行器能直接用起来的上下文？**

这个gap不是咬文嚼字。假设你要生成BibTeX引用，检索系统返回了`citation-management`这个技能。问题是，这个技能是为学术写作全流程设计的，从搜索论文到格式化参考文献，几百行内容。你的任务只需要其中"验证DOI并生成BibTeX"那一小段逻辑。把整个技能原封不动丢给执行器，噪音直接淹没了关键信息。

反过来也一样。有些任务天然跨技能边界。你可能需要`citation-management`里的BibTeX格式化子单元、`latex-writing`里的编译约定、`figure-creation`里的文件命名规范。单独哪个技能都不够，但它们的**子单元拼在一起刚好解决问题**。

**RAE的真正瓶颈不是检索精度，而是检索和执行的中间地带没人管。** 这篇工作的核心主张很简单：需要一个编译层，把"一组相关的技能"变成"一段紧凑、有依据、可以直接用的上下文"。

# 二、SkillRAE：在检索和执行之间加一个编译层

SkillRAE是一个**两阶段RAE方法**。离线建图，在线检索+编译。

## 2.1 离线阶段：建一张三层技能图

SkillRAE在离线阶段构建一张**多层级技能图**，把技能仓库拆成三个粒度：

- **技能社区（L2）**：把相关技能聚成社区。从每个技能里提取高 IDF 的子单元做文本表示，embedding后用KMeans聚类。一个社区里的技能解决同一类问题。
- **技能节点**：保留原始技能作为执行器兼容的基本单位。不拆碎，执行器仍然以技能为消费单元。
- **子单元节点**：从技能里提取的细粒度可复用片段，包括三类：过程性代码行、文件/命令模式引用、约束类使用说明。这些子单元被标准化、去重后通过**提取边**连回来源技能。

三个层级各有分工：**社区负责粗粒度的领域匹配，技能保留了执行器能消费的程序，子单元暴露了技能级别看不见的局部证据。** 技能和子单元之间的多对多提取边是关键 —— 同一个子单元可以跨多个技能共享，这让后面的自底向上证据聚合成为可能。

## 2.2 在线阶段：两个方向检索，四步编译

在线阶段收到一个任务请求后，分两大步走。

### 检索：从上往下 + 从下往上

SkillRAE的检索同时用两个信号：

- **自上而下**：把任务请求和技能社区做embedding匹配，得分最高的社区里的技能获得一个**社区加分（community bonus）**。这保证了领域一致性——你不会在做CV任务时被塞进 NLP 的技能。
- **自下而上**：把任务请求和子单元做匹配，然后将子单元的证据反向投射到源技能上。投射公式考虑了子单元-任务相似度和子单元在技能间的分布（一个子单元被越少技能共享，它的区分度越高）。

最终的技能得分是**技能级相似度 + 子单元证据 + 名称匹配 + 社区加分**的加权和。这个得分决定了哪些技能进入**K(q)**。

检索阶段的输出不只是选中的技能。系统还会为每个选中技能导出**高亮子单元H(q)**——这些是技能里和任务最相关的局部程序，不替代技能本身，但让编译阶段知道该强调什么。同时保留**未被选中但子单元相关的技能记录R(q)**，供编译阶段做rescue。

这个设计很讲究。**检索的输出对执行器友好（仍然是一组技能），同时给编译器提供了图结构支撑的子单元证据。** 一个输出喂饱两个下游。

### 编译：四步把技能变成可执行上下文

编译是SkillRAE最核心的差异化贡献。整个过程不改源技能、不改规划器、不改执行器。**它只改变智能体在执行前看到的东西。**

**Step 1：过滤子单元（Rescue）**

有些技能整体没被选中，但它们包含的某个子单元和任务高度相关。编译器的第一步是扫描这些"落选"技能的候选子单元，挑出一小批满足相关性、非冗余、源技能对齐和预算约束的进入**Z(q)**。这些叫"救援子单元"。差一点就因为整技能落选而被埋没。

**Step 2：把子单元挂到选中技能上（Affiliate Attachment）**

救援子单元不能单独暴露给执行器。那等于引入新执行单元，改变了执行面的定义。SkillRAE 的做法是把每个救援子单元**附着到一个已选中的技能上**，选择标准包含五个信号：子单元和任务的相关性、子单元和候选技能的文本重叠、候选技能的检索置信度、技能社区对齐。

这一步的本质是把散落子单元变成**已选技能的局部提示**。救援来的证据不独立存在，挂靠在执行器已经认得的技能下面。

**Step 3：生成任务特定指导**

不是所有附着成功的子单元都值得渲染进上下文。编译器会再做一次筛选——要求子单元必须**具体（concrete）、任务对齐（task-aligned）、和已有证据不冗余**，并且能在上下文预算（384 tokens）下塞得进去。过于泛化或低亲和度的子单元会被丢弃。

**Step 4：组织上下文**

最后一步把选中的技能、高亮子单元、任务-输出约束、以及经过筛选的附属子单元打包成一个**紧凑的上下文包C(q)**。这个包的格式是任务优先的——先明确要干什么，然后给出要用哪些技能、每个技能里的关键子单元是什么、有哪些跨技能的补充提示。执行器拿到的是**一个编排好的上下文，不是一堆技能dump**。

## 2.3 编译器 vs 规划器：一个关键的区分

这里有必要区分一下SkillRAE和那些执行规划方法（HuggingGPT、ToolPlanner 等）。规划器的工作是**改变执行过程**：多步分解、工具编排、控制流。它们改的是"智能体怎么做"。

SkillRAE的编译器是**advisory的**。它只改变"智能体在执行前看到什么"，不碰执行逻辑本身。这意味着它可以叠加在任何现有规划器或执行器上。论文验证了这一点：把 SkillRAE的编译层加在vanilla retrieval和LLM-based retrieval上，各自都稳定提升。

# 三、实验：编译这步，到底值多少？

实验在两个benchmark上跑：**SkillsBench**（87个任务，执行导向，有确定性verifier）和**AgentSkillOS**（30个任务，生态系统级别的技能组织和编排）。下游执行器用的是Codex CLI + GPT-5.2。

### 整体结果

![alt text](image-1.png)

SkillRAE在两个benchmark上都拿到最优。SkillsBench上比人工指定的curated skills高**3.06**个百分点，比SkillRouter高**7.22**个百分点。**最值得盯着的对比是和 curated skills —— 这些是benchmark里人工为每个任务标注的正确技能。** 技能已经选对了，但靠更好的上下文编译居然还能再提3个点。**光选对技能还不够，怎么组织同样关键。**

### 消融实验

![alt text](image-2.png)

- **去掉自顶向下检索伤害最大**，从**29.26%**直接掉到**16.61%**。技能社区这个设计不是锦上添花，是整套方法里最承重的柱子。没有社区级别的粗粒度匹配，细粒度子单元信号就像没了骨架的肉。
- 去掉上下文编译掉了约**6.7**个百分点。这个差距就是**编译层的净价值**，在检索完全一样的情况下，只改变了证据的组织方式。
- 两种模型时，两边趋势一致，绝对值接近（29.26 vs 28.85）。**编译层的收益不绑定于某个特定模型或 agent 框架。** Gemini那边去掉编译掉的分数少一些（1.05 vs 6.67），更强的模型可能部分弥补了编译信号缺失，但自顶向下检索在两个 backbone 下都是最大单一贡献者。

### 编译层的"外挂"效果

把SkillRAE的编译层加在vanilla retrieval和LLM-based retrieval上，不换检索器：

![alt text](image-3.png)

两个外部检索器加了编译后都有提升。最有意思的是LLM-based retrieval + 编译（**23.36%**）居然超过了裸SkillRAE检索（22.59%）。**一个好的编译器甚至能把平庸检索器的输出变得有用。** 当然，图原生检索 + 附属感知编译的完整版还是最强，编译器能利用图里的子单元导出和社区对齐信号。

# 四、这东西为什么重要，以及还不能做什么

SkillRAE的核心贡献一句话：**它为RAE补上了缺失的编译环节。** 在检索和执行之间插一个不碰执行面的advisory编译器，不改下游任何东西，只优化智能体看到的信息组织方式。这个做法之所以有效，是因为它抓住了一个结构性的盲区：大家都在卷检索精度，没人管搜出来之后怎么组织。

局限也写得诚实。**SkillRAE强依赖显式的过程性文本。** 技能的SKILL.md里得有清晰的步骤、文件约定、约束声明，编译器才能提取子单元。如果关键依赖藏在黑盒工具、无文档代码或运行时状态里，编译器就无从下手。作为 advisory 编译器而非planner，它不保证智能体一定会正确使用编译后的上下文，也不负责运行时失败。

覆盖面上，目前只在两个benchmark的**117**个任务上做了验证。技能社区的构建用的是简单的KMeans，对技能数量和质量敏感。编译器的上下文预算只有**384 tokens**，在需要大量上下文才能说清的任务上可能需要调整。

但这些局限不影响核心论点的成立。**RAE的下一次进步不会只来自更好的检索。把检索到的信息编译成真正可用的形式，是至少同等重要的方向。** SkillRAE用**11.7%**的提升把这件事摆在了台面上。
