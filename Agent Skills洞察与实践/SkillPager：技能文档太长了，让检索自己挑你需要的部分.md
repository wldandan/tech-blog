# SkillPager：技能文档太长了，让检索自己挑你需要的部分

> 技能文档动辄几百行 Markdown，每次调用都全文注入，相当于每次做菜翻完整本菜谱。SkillPager 把技能文档拆成有语义标签的节点，调用时按查询挑出真正需要的部分。**效果是不损失执行质量，同时节省 47% 的 token。**

## 全文注入不是免费的

技能正在成为 LLM Agent 的标准接口。每个技能是一份 Markdown 文档，里面有执行步骤、参数说明、代码示例、前置条件、错误处理指南、概念背景。把这些全部注入 prompt，Agent 每次推理都在翻几百行的完整文档。

**注入消耗 token 只是账面上的代价，更要紧的是关键信息被稀释。** 一个多步执行任务真正需要的可能只是文档里几十行关键段落，其余的是参考性内容，或永远不会被当前查询触发的边缘情况。全量注入下，Agent 必须在大段外围文本里自行定位关键信息，既消耗推理预算，也容易漏掉散在不同章节里的执行约束。

上海交通大学和上海创新研究院的团队在 **SkillPager** 里定义了一个新问题：**技能内检索（intra-skill retrieval）。** 跟跨技能路由和通用 RAG 都不同。已知该用哪个技能，问题不是找技能，而是在这个已知文档里，根据当前查询选出最小但够用的上下文子集。

## 两阶段：离线拆，在线挑

SkillPager 分两阶段。离线把每个技能拆成有语义标签的节点存进缓存。在线用 MMR 做查询条件化的全局选择。

### 离线解析：六种节点类型

离线阶段，一条流水线自动把原始 Markdown 拆成六种语义类型。

**step** 是可执行的操作或指令，直接对任务完成有贡献。**example** 是使用演示、代码片段或输入输出对。**param** 是可配置的参数、选项或参数规格。**precondition** 是执行前必须满足的需求或约束。**error_handling** 是失败模式引导或恢复步骤。**concept** 是定义性或解释性段落，提供背景语义但不属于可执行动作本身。

前五种被视为可执行节点。概念节点不可执行但留在候选池里，当某个概念性说明确实有助于当前查询的推断时，它也应该被选中。

解析阶段还会自动推断四种边：sequence 是文档顺序上的相邻连接，semantic 是嵌入相似度超过阈值（τ = 0.85）的节点对，reference 是节点之间明确提到的关键词引用，condition 是前置条件节点和被约束步骤之间的依赖边。这些边定义了解析后的节点图，**但 SkillPager 在在线检索阶段完全不依赖这张图，它只做全局打分，不走局部遍历。**

整条离线管线对 395 个技能做一次解析，四个并行 worker 跑了 9.83 小时，每个技能中位数约 3.2 分钟。解析一次，永久缓存。在线阶段只需要做一次查询嵌入和 MMR 循环，不再对技能正文发起 LLM 调用。

### 在线检索：全局 MMR，不看图

给定一个查询 q 和缓存的节点集合 V(D)（全部类型都在候选池中），MMR 从一个空的已选集 S 开始，每次迭代选一个节点 n，最大化：

```
MMR(n) = λ · sim(e_q, e_n) - (1-λ) · max sim(e_s, e_n)
                                                s∈S
```

**第一项把跟查询最相关的节点往前推。第二项把在已选集里已经有高相似替代品的节点往后压。** λ = 0.7 来自灵敏度分析。当最好的剩余候选 MMR 值变为负时算法提前终止。

选完之后，节点按它们在原文中的原始顺序重新排列，再拼成上下文。这一步恢复了文档层面的连贯性和技能作者想要保留的过程流。

预算不是固定值。**节点总数 ≤ 100 的技能，上限取 20 个节点。更大的取 60 个。** 这个两档设计来自对 k 的扫描分析：大技能确实需要更多节点才能保持覆盖，但小技能用更大的预算只会多选进冗余内容。

离线嵌入全缓存。在线阶段只做一次查询嵌入（中位数约 498 token），MMR 循环是纯数值计算，不再调用任何 LLM。

## 为什么放弃图

论文里有一个关键的经验观察：**从文档中自动解析出的图是稀疏的、不可靠的。** 执行关键节点之间经常没有被连起来，或者距离文档入口点的拓扑距离极远却没有被边表示。局部图遍历在这个场景里只能拿到 66.73% 的上下文充分性，而全局 MMR 检索拿到 78.89%，差了 12.16 个百分点。

论文还测试了另一个方向：在 MMR 已经选出的节点集合上，再加一阶图扩张，把选中节点的依赖前驱也加进来。结果上下文充分性从 69.8% 掉到 67.4%，平均选中节点从 17.9 个膨胀到 28.1 个，token 节省率从 62.4% 缩到 48.0%。**图对这个场景不是信号源，是噪音源。** 一旦节点被正确地按语义类型标注，扁平化的全局 MMR 比任何基于局部图遍历的方案都更准确也更省 token。

## 消融实验：收益从哪来

**把同一套 MMR 检索算法用在原始固定大小的文本块（chunk）上，充分性也能到 81.77%，接近 SkillPager 的 78.89%，但 token 成本高了 28.81%。** MMR 本身选得不错，但选出来的东西粒度太粗。chunk 里混着可执行内容和不必要的外围文本，语义类型标注把选择粒度压细，用更少的上下文拿到了接近的充分性。

概念节点的去留也有讲究。在 195 个技能、975 个 ground-truth 查询的子集上，**动态保留概念节点在候选池中让 MMR 自适应选择，比静态删掉所有概念节点表现更好。** 概念内容确实有时不需要，但什么情况下不需要，不应该是一个静态规则决定的事。让检索在查询级别做判断比一刀切合理。

## 实验

自建 IntraSkill-Bench：395 个技能，1,975 个查询，LLM 判断加人工验证评估。

| 方法 | 上下文充分性 | Token 节省 |
|------|------------|-----------|
| 全文注入 | 82.23% | 0%（基准） |
| SkillPager | **78.89%** | **47.04%** |
| MMR+chunk | 81.77% | 18.23% |
| 图遍历 | 66.73% | — |

**充分性接近全文注入，token 几乎砍半。** 而且用 chunk 的时候，充分性确实更高（81.77%），但这多出来的不到 3 个百分点是用 28.81% 的额外 token 成本换的。语义节点的细粒度才是效率提升的来源，不是 MMR 算法本身。

## 最后

SkillPager 做了一件很具体的事：**在已知该用哪个技能的前提下，别把整个文档扔进去，而是按查询挑出最小够用的部分。** 这件事在当前 Agent 技能生态中几乎没有被当作一个独立问题来研究，大部分的注意力在"该用哪个技能"和"技能应该怎么写"上，不在"技能文档太大怎么办"上。

把技能文档拆成有类型标签的语义节点，用全局 MMR 做查询条件化的选择，结果是 47% 的 prompt token 节省，同时充分性只比全文注入低了 3.34 个百分点。
