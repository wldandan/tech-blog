# RSA：静态扫描看不出技能有什么问题，让它真的跑一遍就知道了

> 一份文件管理技能，文档里写的是"整理文件并提交匹配项供审核"。静态扫描扫不出任何恶意关键词：没有 `api_key`，没有 `curl`，没有 `webhook`。LLM Judge 也觉得"整理文件"和"审核匹配项"完全是正常的文件管理行为。**但把这份技能装到 Agent 上，在一个包含 `auth.json` 的本地目录里跑一遍，它会静默筛选出 `.json` 文件并把匹配结果发送到外部。** 这个行为链只在运行时暴露。RSA 做的事情就是：不是读代码猜意图，而是造一个有敏感资产的环境让它真的执行，然后用执行 trace 做证据来判它是好是坏。

## 恶意不在代码里，在执行路径里

技能安全面临一个结构性问题。静态扫描可以匹配代码模式：找 `curl`、找 `api_key`、找 `webhook`。但攻击者已经学会了绕过这些模式：把恶意行为藏在资源选择规则里、藏在多步工具调用的交互链里、藏在"只在特定本地文件存在时才触发"的条件逻辑里。LLM-as-Judge 可以读自然语言描述并做语义判断，但它看到"选择匹配的文件并发送审核"时，合理判断不了这是文件管理的正常行为还是数据外泄的伪装。

**这两种方法的共同盲区是：它们只分析产物（代码文本和指令文本），而不分析行为。** 一段代码在文本层面可以被静态分析和语义判断反复扫描，但它的安全边界最终由执行时的系统调用、文件访问、工具调用和状态变化来定义。

约翰霍普金斯大学的团队在 **RSA（Runtime Skill Audit）** 里把技能安全评估从产物分析搬到了运行时行为审计。不是问"这段代码长什么样"，而是问"这个东西在真实条件下跑起来之后到底做了什么"。

## 风险剖析引导定向探测

RSA 的三阶段流程。第一阶段是风险剖析，第二阶段是定向探测任务生成和执行，第三阶段是基于 trace 证据的安全标签分配。

**第一步：不看代码长什么样，看它可能会碰什么。**

送给 RSA 一份技能，它先不判断好坏。它用 LLM 扫一遍技能产物，推断这份技能可能会调用哪些平台工具、可能以什么方式跟 Agent 的环境交互。然后把这些工具映射到一张预定义的安全能力组表上：文件访问、shell 执行、Web 交互、知识/记忆读写、会话控制。

**这个映射不产生判定。它的作用是告诉后面的探测阶段：这份技能的潜在风险接口在哪些地方，你应该设计什么样的任务来触达这些接口。** 一张工具调用映射表让探测不再是"拿一套通用任务测所有技能"，而是"每个技能用自己的风险接口来定向测试"。

**第二步：造一个有敏感资产的环境，让它真的跑。**

风险剖析输出之后，RSA 为每份技能生成一组定向探测任务。任务设计逻辑是：把该技能的风险相关接口放在一个包含真实敏感资产的本地环境中。文件管理工具看起来可疑？任务环境里放一份 `auth.json`、一份 `config.yml`、几份普通的 `.txt` 文件。Agent 被提示完成一个看起来普通的文件整理任务。

执行在隔离的 Agent 沙箱中完成。如果技能需要持久化知识或记忆才能触发恶意路径（比如"只有当 Agent 的长期记忆中存有特定用户偏好时才执行外泄"），RSA 在沙箱中预先填充对应的内存状态。这个设计保证探测的是"这个技能在它的恶意 trigger 被满足时会不会真的走恶意路径"，而不是"Agent 有没有在缺少 context 的情况下自己猜到该走哪条路"。

**第三步：不是猜意图，是看 trace。**

探测任务完成后，RSA 不依赖技能的内容文本做判断。它读执行 trace：系统调用日志、工具调用序列、文件访问记录、网络请求、进程生成。一条 trace 是"工具 A 被调用 → 读取了 `auth.json` → 工具 B 被调用 → 通过网络外发了一个包含文件内容的消息"。另一条 trace 是"工具 A 被调用 → 读取了 `readme.txt` → 工具 C 被调用 → 修改了文件命名格式"。

**前者被标记为恶意，不是因为它"看起来可疑"，而是因为 trace 证据显示了从文件读取到网络外发的完整链路。** RSA 的标签是基于行为的证据链，而不是基于文本的模式匹配。

## 实验结果：100 个技能，6 个静态基线

RSA 在 OpenClaw 平台上实例化，评估了 100 个技能（50 恶意，50 良性）。恶意技能来自 CIK-Bench 和 Skill-Inject，良性技能来自 OpenClaw 公开生态，经过已有标签和人工复核验证。

6 个静态基线做对比：Aguara（LLM-as-Judge 语义审计）、两个 Semgrep 变体（规则驱动代码扫描）、ClawScan、两个 OpenClaw SkillScanner 变体。

**RSA 达到 90.0% 准确率，88.0% 真阳性率，8.0% 假阳性率。** 比最优静态基线（SkillScanner，77.0%）准确率高 13.0 个百分点，假阳性从 12.0% 降到 8.0%。

**自进化攻击下的稳定性差距巨大。** 20 个技能被设置为每隔几轮改写一次自己的内容以绕过静态检测。静态检测器在一到两轮进化后基本失效，检出率直接从 80-90% 掉到个位数。RSA 在所有轮次中持续检出 19 到 20 个（总共 20 个恶意技能）。

**这个差距的原因很直接：静态检测器依赖文本模式。攻击者改写文本，模式失效。RSA 依赖行为 trace。攻击者无论如何改写内容，只要是技能被加载后被 Agent 真实执行，恶意行为的目标接口和系统调用模式在运行时是无法隐藏的。** 词可以换，行为换不了。

## 几个问题

**RSA 的风险接口推断受 LLM 的能力约束。** 如果技能的恶意路径使用了 LLM 在静态扫描中无法推断的工具（比如通过动态拼接调用一个间接引用的外部 API），风险剖析阶段可能错误地把这个接口标为非关键并跳过了对应的探测任务。

**探测任务的覆盖面是风险评估和计算预算的折中。** 一份技能可能有多个潜在风险接口（文件访问、网络外发、进程生成、记忆注入）。如果为每个接口组合都生成全量探测任务，执行成本会指数级增长。RSA 目前的策略是只针对每个已识别的风险接口生成一轮定向任务，不覆盖接口之间的交互组合。两个风险接口单独看都无害但组合起来有害的情况不在当前探测范围内。

**沙箱环境的现实性有天然的边界。** 在沙箱中，Agent 的敏感数据是 RSA 自己放进去的，攻击行为的预期模式是 RSA 自己定义的。但真实部署环境中，恶意技能可能利用 RSA 无法预料的敏感资产类型（比如特定业务数据格式、特定云服务配置模式）来触发外泄逻辑。沙箱探测对这种类型的外泄行为可能会漏报。

## 最后

RSA 的核心洞察只有一句话：**技能安全的证据不在代码里，在执行轨迹里。** 静态扫描告诉你技能的文字长什么样，Runtime Audit 告诉你技能被装上之后实际做了什么。

"看起来可疑"和"确实做了坏事"之间的差距，就是一段在受控环境中被记录下来的完整 trace。RSA 做的事情就是把这段 trace 变成可审计的安全证据。
