#  GOSKILLS：把技能编成组再交给Agent，而不只是"找到就行"

> **摘要：** 技能增强型 Agent 依赖的技能库越来越大，但检索相关技能只是第一步——怎么把这些技能**组织成可执行的上下文**才是真正的瓶颈。GOSKILLS 提出了一种推理时组结构检索方法，把给 Agent 看的技能从"扁平列表"变成"带角色标签的执行契约"。在 SkillsBench 和 ALFWorld 上，GOSKILLS 在 6 个主流模型下都跑赢了现有的技能检索基线。


# 一、问题：检索到了 ≠ Agent 会用

现在的技能增强型Agent系统，技能库动辄几百上千条——代码片段、工具调用指令、检查器、格式约定，什么都有。技能多了就需要检索。

市面上的三种做法，各有各的毛病：

| 方法 | 做法 | 问题 |
|------|------|------|
| **VanillaSkills** | 把整本技能库塞进prompt | 上下文爆炸，token成本高 |
| **VectorSkills** | 语义top-k检索 | 可能漏掉功能上必需但字面上不相似的技能 |
| **GraphofSkills** | 基于依赖图做结构化检索，还原依赖感知的技能包 | 检索到的技能**角色不清晰** ——Agent还是得自己琢磨从哪开始、哪些是辅助、哪些绝对不能碰 |

问题在于：**检索相关技能和呈现可用上下文是两回事。** 就算你找全了所有相关技能，如果扔给Agent时顺序是乱的、角色没标、约束藏在字里行间，Agent还是得自己推理半天怎么用。

举个例子：一个代码格式化任务，格式化器是主角，解析器是辅助，但解析器本身可能看起来更"重要"（更长的描述，更多的API调用）。如果不显式标注角色，Agent可能会先调解析器再想格式化器，绕一大圈。

GOSKILLS解决的就是这个"检索到 → 用起来"中间缺失的一环。

# 二、核心思路：检索单元从"技能"变为"技能组"

GOSKILLS 的关键决策很简单：**不要给Agent看原子技能，给它看带有明确角色的技能组。**

一个技能组不是随意堆砌的语义聚类，而是**以锚点技能为中心的局部有界模式**。组里的每个成员都有明确角色 —— 谁主攻、谁辅助、谁检查、谁绝对不能碰。

这样设计，Agent不再需要猜"从哪开始"。锚点技能就是入口，START标签直接指名。支持组标为SUPPORT，辅助技能不会被埋没。输出格式、必须通过的测试、确定性要求作为CHECK和AVOID标签被显式保留——可见约束丢不掉。

最终交付给下游Agent的不是一串技能描述，而是一份**执行契约（Execution Contract）**：

```
START: 做什么，哪个技能负责
SUPPORT: 辅助技能，什么时候用
CHECK: 必须满足的可见条件
AVOID: 绝对不能做的事
```

## 3.1 离线阶段：从技能图中挖出可复用的技能组

首先，先把技能库表示成**带类型标签的技能图**（Typed Skill Graph）。图的边包含了依赖、工作流、语义、构件和替代关系，分门别类标清楚。

然后，从这张图里提取**锚点中心的技能组（anchor-centered skill group）**。一个组不是一个随机的语义簇，而是一个有明确内部结构的局部模式。每个组包含：

- **锚点技能**：组的核心，执行入口，定义了"这个组是干什么的"。
- **支撑成员**：至多两个辅助技能，各自有明确的角色 —— 预处理器、解析器、格式化器、检查器、回退方案。
- **适用性标签**：必须条件、可选条件、反向条件 —— 告诉系统什么场景该用、什么场景不该用。
- **可见需求线索**：组能产生什么输出、覆盖哪些可见的验证需求，如"输出必须是 JSON"、"必须通过单元测试"。
- **局部拓扑**：组内成员的关系结构。

组不是可执行的隐藏程序。它是一个有界的角色模板，其成员可以在后续的扩展、压缩和渲染中被选中、裁剪或舍弃。

最后，在技能组之间再建一张**组图**（Group Graph）。组图的边标记为：support（支持）、artifact（共享产出物）、visible-check（共享可见检查）、fallback（回退）、incompatibility（互斥）。这张组图的价值在于：它把原子技能间的关系**提升为可复用组级别的检索基板**，推理时不用再回到原子图上重新发现依赖。

## 3.2 在线阶段：检索 → 扩展 → 瓶颈 → 回填 → 签约

推理时，给定一个任务查询q，GOSKILLS的执行路径是五步流水线：

**第1步：查询模式提取**

把任务查询映射成一个确定性的查询模式，包含7个维度：核心目标、技术栈锚点、操作类型、产出物、约束条件、失败线索、检查语义。所有信息仅来自任务prompt和技能元数据，不触及隐藏测试或历史轨迹。

**第2步：锚点组选择**

候选组通过两种方式激活：直接查询-组匹配，以及种子技能重叠匹配。激活后用三组线性评分函数（群选择、支撑拓展、技能挑选）排序。评分策略**固定且确定**，跨任务、跨基准、跨模型都用同一套权重，不依赖学习。

**第3步：支持组扩展**

从锚点组出发，在保留的候选短列表和组图邻居中贪心添加支持组。扩展停止条件：边际增益不足、组预算用完、或上下文预算超限。实现上最多选3个组（1锚点 + 2支持）。

**第4步：瓶颈化（Bottlenecking）**

选组不等于全量呈现。这一阶段从选中组里**精筛**要呈现的原子技能负载，保证token不超预算。技能的选择仍然靠那组固定权重的线性评分进行排序，先插入所有lead技能，再逐个添加得分最高的剩余项。

**第 5 步：覆盖回填 + 契约渲染**

计算**覆盖债务（Coverage Debt）** —— 查询中必须覆盖的高置信度特征（框架名、文件扩展名、具名 API、输出格式等），减去已呈现技能覆盖的部分。剩余债务**不是静默修复，而是显式报告**。符合条件的活跃支持组技能会被回填来补债务。

最终，把所有东西渲染成固定格式的执行契约。

# 四、实验结果：更好的奖励，更短的运行时间

GOSKILLS在两个基准上做了评估，覆盖6个主流大模型：

**SkillsBench**（技术技能选择 + 确定性检查）和 **ALFWorld**（多轮交互决策）。

### 主要结果

在SkillsBench上，GOSKILLS在所有模型下都拿到了最高的平均奖励：

![alt text](image-2.png)

关键看点：
- **全面优于无技能基线（NoSkills）**：所有模型下都有显著提升，说明组结构上下文确实提供了有用的任务引导。
- **超过全库暴露（VanillaSkills）但不用全库token成本**：奖励更高，token 数可控。
- **优于依赖感知恢复（GraphofSkills）**：在GPT-5.4 下，奖励从 36.4 提升到 48.9，运行时间从 603.7s 降到 352.9s；在 Claude Sonnet 4.5 下，奖励从 38.7 提升到 46.8，运行时间从 571.9s 降到 347.6s。

ALFWorld上GOSKILLS也拿到最高奖励，但幅度比 SkillsBench 小。这符合预期——ALFWorld 更依赖多轮规划能力而非技能匹配精度。运行时间的优势仍然在，说明**显式的锚点、支持、检查字段确实让 Agent 在组织信息上省了不少力气**。

### 检索闸门测试

为了把检索质量与下游执行解耦评估，论文设计了一个"检索闸门"：在Agent执行前检查呈现的技能是否覆盖了所有标注的must_have技能。结果：

![alt text](image-3.png)

GOSKILLS在40个可见需求项目上**全部通过**，must-hit率达到**1.00**，且平均只用了不到4个技能负载。相比之下，GraphofSkills只有0.84。

这直接验证了**覆盖回填（Coverage Backfill）在预算紧张时确实能保住关键技能。**

# 五、消融实验：哪个组件最重要？

论文在 GPT-5.4 + SkillsBench 下做了组件消融：

![alt text](image-4.png)

几个重要发现：

1. **契约格式本身就有价值，但不是全部。** GraphofSkills + Contract 把奖励从 36.4 拉到 41.2，运行时间从 603.7s 降到 516.2s。但离完整 GOSKILLS 还有明显差距。核心收益来自**组级检索 + 扩展 + 瓶颈化**，不只是prompt格式换皮。
2. **锚点选择、组扩展、组图是三大支柱。** 去掉任何一个，奖励都掉到 40 出头，退回到仅加契约的GraphofSkills水平。三者缺一不可：没有可靠的入口点，或没有结构化的支持扩展，或没有组级关系基板，GOSKILLS就跟普通结构检索没区别了。
3. **覆盖回填是must-hit率的命门。** 去掉后must-hit从 1.00 暴跌到0.73。这意味着任务关键技能可能根本就没被呈现给Agent。
4. **扁平呈现就是不够用。** "仅检索技能"变体不改动检索到的技能集，只去掉组级检索、瓶颈化和契约渲染，奖励直接掉到 38.7，must-hit 0.76。**检索到了，但用不起来——问题出在呈现方式。**
5. **角色标签和 AVOID字段是锦上添花。** 去掉它们不影响must-hit，但奖励和运行时间会变差。显式标注角色和禁止项确实帮Agent省了推理开销。

### 灵敏度分析

论文还测试了技能库从 200条膨胀到2000条时各方法的表现。VanillaSkills全库暴露，token跟着线性涨；VectorSkills被检索噪声拖累，奖励明显下滑。**GOSKILLS的奖励和运行时间在库膨胀时都保持最稳**——组级激活和有界扩展让检索质量对库大小不敏感。


# 六、边界在哪

说清楚GOSKILLS不是万能的：

- **它不训模型，不改环境**。库里没有的能力，只能写进 DEBT，不能无中生有。
- **依赖离线阶段的好 metadata**。技能没有tag、文档、类型标注，建群效果打折扣。
- **长链条交互任务提升有限**。ALFWorld上的差距就比SkillsBench小得多——需要的不只是"知道从哪开始"，而是执行中的多轮动态决策。
- **不保证每个任务都碾压 Graph of Skills**。论文也说了，在某些匹配切片上可能不如。

但它的定位很清晰：**只改变检索结果的组织方式**。这个约束下，数字没话说。


# 七、总结

GOSKILLS 做的事情其实很简单：**把技能检索的交付物从"你可以用这些技能"变成了"你应该这样用这些技能"。**

如果你在搭Agent系统，下面是比架构图更值钱的三条：

**第一，检索单位本身就是设计决策。** 检索相关技能只解决了"召回"。把召回结果组织成能直接用的上下文，才是Agent实际在等的东西。GOSKILLS用群和合约填上了这道缝。代价是离线建群的一次性投入，换来的是在线阶段的可预测性。

**第二，角色标签是压缩后的关系知识。**parser、formatter、checker、prerequisite —— 这些不是可有可无的注释。它们省掉的是Agent从技能描述里反向推断调用顺序的整段推理。合约对照实验（+4.8分）就是证据：同样的技能，带上角色标签就是比不带好用。

**第三，确定性在工程上比"更好的模型"靠谱。** GOSKILLS的评分权重开发阶段定好，跨任务不动。Agent表现有波动的时候，你知道不是检索层在作祟。可预测意味着半夜三点被叫起来的时候能定位问题——这对生产系统来说，比多几个百分点的 Reward 重要得多。

对正在构建Agent Skill系统的团队来说，这篇论文最有价值的启发：**"把技能找出来"和"把技能用起来"之间的鸿沟，可能比你想象的大。填补这个鸿沟不需要重训模型，只需要换一种组织和呈现技能的方式。**
