# Notes2Skills：把实验记录变成 Agent 技能时，最不能丢的是作者的"确定性"

> 实验记录本里同时塞了三样东西：已经确认的观测事实（baseline drifted slightly），作者自己也不确定的判断（not sure the second read is reliable），和留给下一次实验的建议（try a fresh buffer next time）。如果把整个记录本不加区分地编译成 Agent 执行指令，**两种致命的错误会同时出现：不确定的判断被当成确定结论来执行（不确定性漂白），确定的指令被当成了就随便提提的建议（指令丢失）。** Notes2Skills 在两个阶段中间加了一层确定性标签，把事实、判断和建议拆开，然后只让事实驱动强操作、让判断默认退回人工复核、让建议保持建议。

## 实验记录不是操作手册

过去几十年里，程序文本提取走过了三个阶段。食谱年代的 flow-graph 提取，从烹饪教程里找到"加面粉→搅拌→烘烤"的结构。湿实验协议年代的 action 提取，从发表的标准操作流程里找到"12000×g，离心 10 分钟，4°C"的指令。LLM 年代的技能编译，把标准流程文档直接转成 Agent 可加载的技能包。

**这三个阶段共享一个隐含假设：输入的文本是规范性的。** 食谱在出版之前已经被作者反复摘过刺，湿实验协议在发表之前已经被审稿人和引用者验证过，标准操作流程本身就是一份权威文档。"离心 10 分钟"不是一个猜测，是你如果少离心了 2 分钟产物会不对的确定指令。

**实验记录打破了这个假设。** "read2 @ 200mV: events unclear, uncertain whether a signal was detected" 和 "this batch is unusable as-is; setting it aside" 和 "next time: double the catalyst loading" 在同一页上，同一段时间写的。它们表面上的语气区别极小，但第一条是判断，第二条是指令，第三条是建议。把三条都当指令来执行，Agent 会基于一个作者自己都不确定的判断去做不可逆的数据操作。

南方科技大学和港科大（广州）的联合团队在 **Notes2Skills** 里把作者确定性定义成了安全边界。一个独立撰写的实验记录本，把作者的确定性信号从文本中提取出来，让它一路跟到 Agent 的执行决策点。

## 确定性标签是一个操作级信号

Notes2Skills 的两阶段框架里，Stage 1 叫 Epistemic Directive Extraction。入口是一段实验记录文本，出口是一组标注了确定性标签的指令。三个标签：**FACT、JUDGMENT、SUGGESTION。**

每条指令带了五类型标签中的一种（FLAG_DATA、CONDITION_CHANGE、ANALYSIS_SUGGESTION、PROTOCOL_CHANGE、PARAMETER_SHIFT），然后叠加一个确定性标签。

**确定性标签控制 Agent 能做什么。** FACT 允许强操作（在策略和信号证据一致的前提下执行不可逆的数据处理）。JUDGMENT 默认退回到审查保留模式（数据不被修改，但被标记需要人工复核后再决定）。SUGGESTION 是纯粹的建议级别（不触发任何文件级操作，只在报告中作为潜在后续方向被提及）。

**Stage 2 叫 MetaSkill 编译。** 输入是带标签的指令集，输出是一个符合 SKILL.md 标准的技能文档。确定性标签作为 action commitment level 跟每个操作指令一起被封入技能胶囊。每个胶囊还带了原文片段的加密链接：当 Agent 执行一个强操作时，可以回溯到实验记录原文中对应的语句，验证这条指令确实来自作者确认过的事实，而不是被处理过程中错误升级的不确定判断。

**编译过程是完全确定的。** 不是让 LLM 再读一遍 Stage 1 的输出然后"理解"一下写个技能。没有模型参与。胶囊的每个字段要么从 Stage 1 的 EDE 记录中继承，要么由领域配置固定。论文视可审计性和保守性原则为硬约束：如果让 LLM 做编译，它有可能在不确定判断和确定指令之间自行决定"这段文字感觉更像是指令"，这个决定本身就会引入一个新的不确定性源。

## 执行时确定性怎么被用上

49 个文件级操作被编译进 MetaSkill。当 Agent 加载技能执行数据处理时，每条胶囊的确定性标签跟该文件的实际信号证据做 joint check。

如果胶囊标记为 FACT 且该文件的数据特征跟作者描述的事实一致，Agent 被授权执行强操作（标记数据段无效、设置截断边界等）。如果胶囊标记为 JUDGMENT，除非该文件的信号证据完全独立地支持同样的结论，否则 Agent 只能标为待人工复核。如果胶囊标记为 SUGGESTION，Agent 不会对文件做任何永久性修改。

**这个执行门控本质上是贝叶斯决策逻辑的一个简化实现：在没有观察到独立验证信号之前，不对基于不确定判断的结论做不可逆操作。**

## 实验中，七种配置只有一种没犯错

实验设计故意选了三个确定性特征不同的实验记录会话。一个 FACT 主导（作者的记录以确定性结论为主），一个 JUDGMENT 主导（大量不确定性判断），一个混合型。

七种配置被比较：裸 Agent 不看记录直接跑、只基于原始文本不加加工的 LLM 读取、只提取动作丢弃确定性标签、确定性标签由 LLM 推断但无硬约束……加上完整的 Notes2Skills。

**在 JUDGMENT 主导的会话上，不保留确定性标签的配置一致出现了不确定性漂白：Agent 把作者标记为不确定的判断当成确认事实来执行数据删除和截断，导致本应保留的测量数据被错误丢弃。** 在 FACT 主导的会话上，保留原始文本但不区分确定性的配置一致出现了指令丢失：作者明确写的"这个数据段无效，丢弃"被淹没在周围大量的不确定陈述里，Agent 在文本中找不到足够强的指令信号来执行必要的清理。

**只有 Notes2Skills 在两种偏误上都没有犯错。** 在 JUDGMENT 主导的会话上，它保留了所有不确定数据段标注为待人工复核。在 FACT 主导的会话上，它执行了所有作者明确标记的强操作。确定性标签不是锦上添花：它是在一条执行链中，让"作者说了什么"和"Agent 能做什么"之间保持严格因果约束的唯一机制。

## 几个问题

**目前的确定性分类是三元的。** FACT、JUDGMENT、SUGGESTION 覆盖了大部分实验记录陈述，但有少量的混合型陈述（"这个结果不太确定，但基于之前的经验我倾向于认为这是真的"）在这个简化下被归到了 JUDGMENT。实际实验推理中，确定性是一个连续谱而不是三个离散标签。

**MetaSkill 的可审计链路依赖于 Stage 2 的确定性编译。** 如果在 Stage 1 的 EDE 中，LLM 把一条 SUGGESTION 误分类为 FACT，那个错误会被 Stage 2 的确定性编译器原样继承并被放大到 Agent 的执行决策中。当前框架的可审计性确保了一个 FACT 胶囊确实来自原文中被标记为 FACT 的语句（traceability），但不能保证原文中的那段语句确实应该是 FACT（accuracy）。溯源性保证和准确性保证是两个不同的问题。

**跨记录本的迁移还在初期。** 当前实验用的是同一个记录本内的 split 评估：同一作者的实验记录被分成训练和测试。不同作者在 notebooks 中的用语习惯和语气风格差别很大：有人几乎只用 declarative 语句，有人大量使用 epistemic adverbs。确定性分类器在不同作者风格之间的泛化性能目前还没有被测量。

## 最后

Notes2Skills 最大的贡献不是一套编译管线，而是把一个问题摆到了明面上：**当 AI Agent 被引入科学工作流时，它们处理的文本不再是已经定稿的协议和论文，而是包含大型不确定性的过程记录。** 把不确定的记录当成确定性指令来执行，结果是"Agent 自信地把对的事做错了"。

在实验记录里保留确定性标签，让 FACT 驱动操作、JUDGMENT 退回人工复核、SUGGESTION 保持建议。这个分类不是修辞上的 refinement，而是这些文本从"供人看"变成"被机器执行"时，一件必须做的事。
