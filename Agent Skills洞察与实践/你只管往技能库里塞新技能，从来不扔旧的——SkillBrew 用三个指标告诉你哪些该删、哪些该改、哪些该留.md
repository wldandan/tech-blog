# 你只管往技能库里塞新技能，从来不扔旧的——SkillBrew 用三个指标告诉你哪些该删、哪些该改、哪些该留

这可能是技能管理领域最被忽视的一个事实：**每个人的技能库都是只增不减的。** 跑通一个新任务，提取一份新技能，塞进去。再跑通一个，再塞一份。从来没有人说"这份技能过时了，删了吧"或者"这两份技能说的事其实一样，合并了吧"。

这不是某个人的问题。Voyager、Trace2Skill、SkillRL，目前主流的技能管理范式全是 Append-Only。**技能库是只长不缩的。** 500 份技能、1000 份技能、5000 份技能，一路膨胀。

膨胀不是最可怕的。最可怕的是膨胀带来的三个连锁效应：**冗余**（两份技能说的是同一件事，但写法不同，Agent 检索时不知道该信谁）、**过时**（API 已经变了，技能里写的还是旧的调用方式）、**冲突**（A 技能说先验证再执行，B 技能说先执行再验证，Agent 同时检索到这两份，直接懵了）。

城大、Squirrel AI、中科大、UCSD、Griffith、华师大和上海交大的联合工作 **SkillBrew** 给了一个答案：**把技能库的管理从"逐个技能的局部决策"升级为"全库级的全局多目标优化"。** 不是看单份技能好不好，而是看在"有用、多样、覆盖全"这三个指标的约束下，整个技能库的最优形态长什么样。

---

## 一、为什么"看单份技能"不够

先把现有的技能管理分个类。

**第一类：只增不删。** Voyager 把成功执行后的新技能永久加入技能库。Trace2Skill 从轨迹池里合并出新技能。都只加不删。

**第二类：按单份技能的信号做局部决策。** ExpeL 用投票式的 ADD/EDIT 操作更新技能。EvoSkill 用验证集准确率做 Top-K 筛选。Skill-Pro 用在线评分门控来收容/淘汰技能。这类方法比只增不删进步了，至少有人开始在"管理"了。但它们都在做同一件事：用单个标量指标来评判单份技能。

**问题在于，一份技能的好坏不是独立存在的。** 一份技能单独看可能有帮助，但它跟库里已有的另一份技能高度重叠——两份都在说"如何调用 search API"。单独看都不错，放一起就是浪费 Token 还干扰 Agent 做选择。反过来，一份技能单独看效果一般，但它的覆盖范围恰好是库里其他技能都不碰的角落——**单独看它不值得留，但从整个库的覆盖度来看，删除它会留下一个盲区。**

**技能库的质量有三个互相制约的维度。** 有用（utility）：检索到的技能有没有帮 Agent 提升表现。多样（diversity）：库里有多少冗余和重复。覆盖全（coverage）：库里技能能不能覆盖到各种不同的任务。**三个维度互相拉扯：加技能扩覆盖通常会引入冗余降低多样性，删技能保多样性通常会牺牲覆盖。没有办法把它们糅成一个标量指标而不引入偏差。**

这就是为什么单技能局部决策不够——它看到的永远只是"这份技能"好不好，看不到"这份技能在这个库里"好不好。

## 二、SkillBrew 怎么把技能库管理做成多目标优化

SkillBrew 把技能库管理形式化成**一个带约束的多目标优化问题**：

**目标**：最大化效用（$J_{\text{util}}$）、多样性（$J_{\text{div}}$）、覆盖率（$J_{\text{cov}}$）。

**约束**：效用不能低于门槛 $\eta$。一个没用的技能库，再多样再覆盖全也没意义。

**不是求一个最优解，而是求帕累托前沿——找到那些"不牺牲一个维度就没法提升另一个维度"的技能库配置。**

这里面有一个关键的优先级设计：**效用是硬约束，多样性和覆盖率是结构正则化器。** 技能库必须先有用，组织得再好才有意义。但"有用"只是入场券，不是终极目标——一旦进了场，多样性和覆盖率才是真正拉开不同技能库质量差距的维度。

## 三、怎么做：双层 propose-then-verify 循环

多目标优化的挑战在于：目标不可导（技能是离散文本，没法算梯度），搜索空间是开放的（新技能不是从固定池子里选的，而是从轨迹证据中合成的）。

SkillBrew 用了三个设计来解决。

**第一，反事实信用分配。** 要评估单份技能的边际贡献，SkillBrew 用 leave-one-out 方法：**把这份技能从库里临时移除，在同一批任务上跑 Agent，看表现变了多少。变差了说明它有用，没变说明它可有可无，变好了说明它在帮倒忙。** 这种反事实评测比"这份技能被检索了多少次"准确得多——被检索不等于被用到，被用到不等于起了正面作用。

**第二，双层 propose-then-verify。** 不能让优化器在开放空间里乱搜。SkillBrew 把流程分成两层：**Propose 层**在 Support Set 上用轨迹证据提出候选编辑（新增、改写、删除）。**Verify 层**在 Query Set 上跑多目标评估，只有非支配配置才进入下一轮。**两层数据分离防止了过拟合——在"提出建议的那批数据"上表现好的技能，不意味着在"没见过的数据"上也表现好。**

**第三，帕累托感知选择。** 不是按一个标量排序然后选最高分。而是保留所有互不支配的候选配置——A 配置在有用性上更好，B 配置在覆盖率上更好，两个都留着。**让优化器在"有用 vs 多样 vs 覆盖"的三角张力中自己找到平衡点，而不是人工预设一个权重把三个维度强行糅成一个数字。**

## 四、跟之前聊过的技能管理方法有什么关系

如果你一直在追这个系列，SkillBrew 回答的问题恰好跟前面几篇形成了互补。

**SKILLPYRAMID 和 SkillDAG** 回答的是"技能之间应该是什么结构"。金字塔还是关系图，原子/抽象还是五类边。**它们把技能库从平面文件夹变成了有结构的网络。**

**SkillBrew 回答的是另一个问题：结构有了，但结构里的内容怎么管理？** 哪些该留、哪些该改、哪些该删——不是凭直觉，不是靠单份技能的表现，而是在"整个库的效用、多样性和覆盖率"三根轴的交点上找最优解。

**SkillBrew 把"技能库管理"从一门手艺变成了一个可形式化的优化问题。** 手艺人凭经验判断"这份技能好像没什么用了，要不要删呢"。SkillBrew 用 leave-one-out 反事实评测告诉你删掉之后 Agent 表现变了多少，用帕累托前沿告诉你在效用、多样性和覆盖率之间你的技能库目前落在哪个位置。

## 五、这篇论文在说什么

SkillBrew 的核心信息很简单，但触及了技能管理领域一个必须迈过去的坎。

**技能库不能只增不减。这是共识。但"怎么减、减多少、用什么标准衡量减得好不好"——这些问题的答案不能是"拍脑袋"。** SkillBrew 把这三个问题的答案形式化了：用多样化指标衡量内部冗余，用覆盖率衡量对外服务范围，用效用作为不能跌破的底线。

如果你在维护一个越来越大的技能库，这篇论文给了一条可以立刻用上的原则：**别只盯着每份技能单看。定期把整个库拿出来，用留一法测每份技能的边际贡献，用多样性和覆盖率作为结构健康度的指标。一个健康的技能库不是"每份技能都好"，是"这个库作为一个整体，在各种不同的任务上都有用、不冗余、不盲区"。**
