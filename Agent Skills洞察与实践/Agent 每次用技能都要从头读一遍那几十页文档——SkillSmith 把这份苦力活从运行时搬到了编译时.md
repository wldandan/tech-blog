# Agent 每次用技能都要从头读一遍那几十页文档——SkillSmith 把这份苦力活从运行时搬到了编译时

你的 Agent 加载了一份技能。这份技能有 SKILL.md 作为入口文档，外加十几个辅助脚本、配置模板、参考文件。Agent 把它全部读进上下文，开始推理，制定执行计划，一步步往前走。

下次任务，同一个人，同一份技能。Agent 又读了一遍全部文档，又从头推理，又重新制定了一份几乎一模一样的执行计划。

**Agent 消耗的 Token 里，超过一半花在了跟上次完全重复的事情上。** 这不是某一个技能的问题，是所有技能在运行时被反复"解读"的问题。以太之心和人大、UCSD 的联合工作 SkillSmith 给了一个朴素的解法：**把技能当成可以被编译的东西，而不是只能被阅读的东西。**

---

## 一、技能被用得很浪费

当前的技能使用模式很直接：Agent 判断当前任务跟某份技能匹配 → 加载这份技能的全部文档 → 边读边推理 → 制定执行计划 → 执行。

**这个模式里有两个肉眼可见的浪费。**

**第一，大量上下文根本不相关。** 七份 SkillsBench 技能，Agent 平均每次加载约 17.8K Token 的技能材料。其中真正跟当前任务相关的，不到一半。**9.1K Token（51.2%）是读进去但根本没用到的内容。** 这些 Token 白白占了上下文窗口、推高了推理成本，还稀释了真正有用的信息。

**第二，同样的推理在反复做。** 同一份技能，不同的任务。Agent 每次都要重新理解一遍技能说的是什么，然后重新制定执行策略。SkillSmith 测量了这种重复度：**不同任务使用同一份技能时，推理轨迹的平均相似度高到 45.5%。** 换句话说，将近一半的"思考"是在重复上一次已经思考过的东西。

**两个浪费指向同一个根源：技能被当成了一本每次都要从头读的教科书，而不是一套可以被"编译"成可执行接口的程序。** 传统编译器做的事就是把源码翻译成机器码——编译一次，运行无数次。SkillSmith 要做的就是把技能的"理解"和"推理"从每次运行时抽出来，做成只做一次的事。

---

## 二、SkillSmith 怎么做：边界优先编译

SkillSmith 的核心设计叫**边界优先编译**。不是把技能编译成统一的中间语言（IR），而是编译成一套**显式的运行时边界合约**。这个合约规定了：能做什么、怎么做、什么情况下不能做、万一不行怎么办。

### 2.1 源码形态分类：不是所有技能都长得一样

技能的内容千差万别。有的像操作检查清单（"先检查版本，再运行脚本，最后验证输出"），有的像工具箱（"这里有几个处理图片的函数，需要时调"），有的像参考手册（"表格、公式、示例，要用的时候查"）。

SkillSmith 编译前先做一件事：**判断这份技能的"源码形态"是什么。** 四类：

**workflow 型。** 有序的步骤、明确的控制流、清晰的输入/输出标注。这类技能天然适合被编译成带依赖关系的步骤级工作流图。

**dispatcher 型。** 打包了多个脚本、API 描述、可复用命令片段。这些是没有固定顺序的独立操作集合，适合被编译成一组类型化的可调用算子。

**reference 型。** 表格、示例、领域知识为主，可执行结构很弱。不适合直接编译，适合被索引化，Agent 在运行时按需检索。

**insufficient 型。** 内容太模糊、能力边界不清、缺少可靠结构。这类技能不编译，返回诊断信息，运行时退回使用原始文档。

这个分类不是靠 LLM 泛泛判断的。编译器先提取结构性特征：标题层级、有序列表、代码块、脚本清单、函数签名、验证短语。然后 LLM 基于这些有据可查的证据做最终判定。保守优先级：workflow > dispatcher > reference > insufficient。

### 2.2 边界合约：编译后的技能长什么样

分类并降级之后，编译器产出一份结构化的**边界合约**。这不是一份工作流 IR——它是一份**描述了"这个编译产物能做什么、不能做什么"的声明式 ABI**。合约包含七个字段：

**边界类型 τ。** 这份技能是纯指引（guidance）、适配器（adapter）、类型化算子（typed operator）还是接近求解器（solver-like）。

**可调用算子 O。** 编译产物对外暴露的操作。每个算子有输入/输出 schema、底层绑定和源码引用。

**输入/输出合约。** 运行时需要提供什么参数、产出什么。

**风险标记 + 验证级别。** 执行有多确定？有没有测试可以自动化验证？验证证据有多强？

**动作策略 + 选择策略。** 算子选择偏好和执行行为限制。

**无损降级胶囊 F。** 当编译产物无法处理当前请求时，**完整保留了原始技能的全部内容。** Agent 可以通过 list_skill_assets、get_skill_asset 等算子回到原始材料，不让编译产生能力损失。

**合约是编译产物的公共接口。** 工作流图、调度器、引用索引——这些都是编译器内部的降级产物，被合约封装在背后。

### 2.3 运行时：合约被解释成带护栏的状态机

Agent 使用编译后的技能时，仍然是渐进式披露。先看到的是一个紧凑的 run_{skill} 操作柄和边界摘要，详细的算子 schema、策略提示、降级材料都不占持久 Prompt。

选定技能后，运行时把合约解释为一个**带护栏的状态机**。Agent 每次调用经历三步：选算子 → 策略检查 → 执行或拒绝。

**执行路径。** 算子被直接执行（脚本、命令、类型化操作），返回结构化的执行证据或求解器输出。

**指引路径。** 策略不允许直接执行时，返回参考指引，Agent 继续自主推理。

**阻塞路径。** 合约判断当前任务不在能力范围内，返回原因和降级提示——Agent 通过降级算子加载原始技能材料，继续推理。

**编译是非破坏性的。** 任何时候，Agent 都能从合约回到原始技能的全部内容。不是"Trust the compiler"，而是"Try the compiler, if not enough, fall back to the source."

## 三、实验说了什么：57% Token 省下来，能力没掉

SkillsBench 选取七个任务（简单、中等、困难各两个到一个），三个 Agent 平台，四个模型规模。

### Token 减半，速度翻倍

**不是"10-20% 的优化"，是直接砍半。** 推理和 LLM 调用次数减少 42.99%。解决时间缩短 50.57%，**相当于 2 倍加速。** 按 Token 计费的成本同步减少 57.44%。

跟 SkVM（一套技能虚拟机方案）相比，Token 再少 46.49%，时间再缩短 47.04%，调用次数再降 18.67%。

**这篇论文的核心数字不是"我们更快"，是"快了整整一倍，而任务完成率没有统计显著的下降。"**

### 大模型编译，小模型跑——反而更好了

一个有意思的发现：**用强模型（GPT-5.5 或 Claude Opus 4.7）编译的产物，给弱模型或高效模型（DeepSeek V4 Flash 或 Qwen3.6）跑，某些任务的准确率反而提升了。** 弱模型在直接解读原始技能时可能理解偏差，但面对已经编译好的结构化的合约和算子，执行起来更准确。

这恰好印证了 SkillSmith 的核心直觉：**把"理解技能"这件事从每次运行时剥离出来，交给更强的模型在编译时做一次——然后所有运行时模型直接使用编译结果。**

## 四、局限与总结

论文坦诚了几个局限。目前的覆盖范围是七份 SkillsBench 技能，扩展到全量基准还需要更大规模验证。source-shape 分类器的四类划分在某些边界模糊的技能上可能需要更细的粒度。降级路径虽然无损，但如果编译产物的策略过度保守，可能导致过多的回退。

**但 SkillSmith 最值得记住的，是它问了一个被所有人忽略的问题：技能加载之后的事。**

每个人都在讨论怎么让 Agent 检索到更合适的技能、怎么生成更好的技能内容。但 SkillSmith 问的是：**Agent 读到了技能之后，它是怎么用的？有没有一种更好的方式让它不用每次都从头读？**

这个问题的答案来自编译原理里最古老的思想——程序不是被"读"的，是被"编译"的。技能今天的状态，就相当于每一行代码都在解释执行。编译一次，执行无数次。SkillSmith 把这个思想搬到了 Agent 技能的世界里。Token 减半、速度翻倍，只是这个思想成立的证据。
