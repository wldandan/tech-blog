# W2S：自动生成技能不是做摘要，是重构工作流

> 现有的技能大部分是手工写的。人工写成本高、跟不上工具和场景的快速变化。好消息是交互轨迹、工具调用、专家演示、执行日志里已经有大量可用的证据。坏消息是**把这些历史痕迹直接扔给 LLM 做摘要，产出的"技能"会过拟合偶然细节、漏掉关键的验证和恢复步骤、在结构上不完整，无法被 Agent 可靠复用。** W2S 做了一件事：把技能生成从文本摘要重新定义为结构化的运行时规范重构。

## 摘要丢掉的，正好是技能最需要的

把技能看作是"把历史交互压缩成一段文本"，这种直觉很自然：给 LLM 一堆轨迹，让它总结出这个任务要怎么做。

但这个思路在技能空间会系统地失败。原因在于摘要的目标函数和技能的优化方向是错位的。**摘要有语义显著性的偏好：挑重要内容、压掉细节。而技能需要的恰好是那些操作细节：触发条件、分支判断、重试规则、验证标准、输出约束。** 这些细节在语义上不重要，"失败时最多重试 3 次"没有"这个函数能做什么"重要，但在操作上是决定性的。把重试次数从 3 改成不限制，技能的执行链路直接变了。

武汉大学和南昌大学的联合团队在 **W2S（Workflow-to-Skill）** 里做了一次视角翻转：**技能生成不应该从"如何压缩历史文本"出发，而应该从"如何重构可执行的操作规范"出发。** 一篇摘要告诉你"曾经发生过什么"，一份技能规格告诉你"以后应该怎么做"。两者的信息需求从头就是不一样的。

## Skill-IR：技能应该被拆成四件东西

W2S 的第一步是定义技能的中间表示：Skill-IR。这份表示不直接面向文本，而是把技能拆成四个正交的组件。

**路由头（Routing Header）。** 技能的发现层。回答"什么条件下这个技能应该被调用"。包含技能名称、描述和适用性声明，让 Agent 在检索时能判断这个技能是否跟当前任务相关。

**工作流骨架（Workflow Backbone）。** 技能的执行控制结构。包含工作流节点和节点之间的有向跳转关系。它描述的是一份技能的"宏观流程"：先做什么、再做什么、什么条件下走分支、什么条件下回退。

**节点级语义（Node-level Semantics）。** 骨架中每个节点对应的局部操作语义。它描述的是单个节点内的决策逻辑：这一步的局部目标是什么、执行条件是什么、成功和失败的判定标准是什么。**工作流骨架告诉你"这个技能有哪些步骤"，节点语义告诉你"每个步骤具体怎么执行"。**

**运行时附件（Runtime Attachments）。** 执行时需要的工具、脚本、参考文档、模板、配置约束、输出需求和状态管理规范。这些不是技能的"文本"，而是技能执行的外围操作上下文。

这四层的总称是 WSA：Workflow + Semantics + Attachments，再加 Routing Header。把技能拆成这四层之后，技能生成就不再是"把轨迹写成一段话"，而是"从轨迹中为每一层填充该层需要的信息"。

## W2S：从多段轨迹中重建一份工作流

有了 Skill-IR 作为目标格式，W2S 的任务变成：给定一个技能的多条交互轨迹（可能来自不同场景、不同用户、不同工具版本），重构出覆盖所有场景的一份 Workflow 规范，同时过滤掉单条轨迹中的偶然步骤和冗余内容。

流程可以拆成五步。

**分段。** 将每条轨迹按操作类型切分成过程单元。一段轨迹可能横跨多个场景：同一个技能在"正常情况"下只走主路径，在"输入格式错误"场景下走入验证分支，在"API 超时"场景下走入重试路径。分段让每条轨迹的结构化信息可被单独提取。

**起草。** 对每一条路径单独诱导一份本地的 Skill-IR 草稿。这一步是"各场景分别建模"：正常路径一条草案、异常路径一条草案、边界情况一条草案。每条草案只描述自己那个路径上看到的工作流片段。

**对齐与合并。** 多条草案之间在骨架层面上互相对齐。**同一个工作流节点在正常场景和异常场景中都是"处理 API 响应"这个步骤，只是节点内部的语义不同（一个是正常解析，一个是超时处理）。** 对齐模块识别出多路径中共章的节点，合并同构的步骤，把不同路径的差异收敛到同一个工作流骨架的不同分支上。

**分支协调。** 对齐过程中发现的局部差异（某条路径在这里分了两个分支而另一条路径没有）需要通过分支协调来解决。不是"只要有就别丢弃"：偶然出现的分支可能是特定轨迹中的执行偏差而非通用的技能规则。**协调的依据是轨迹一致性和出现频率，W2S 给每个保留下来的分支标注了证据来源和置信度。**

**冗余压缩。** 多条轨迹叠加必然产生大量重复：同一步骤在十条轨迹里被走了十次，每一组起草的文本都略有不同。压缩模块在保留关键验证、审批、回滚和状态管理行为的前提下合并表述重复。最终产出一份 Skill-IR，以及一份从 Skill-IR 渲染出的可部署 Agent 技能文本。

**这跟"把十条轨迹扔给 LLM 做摘要"的本质区别是：每一步都有明确的操作对象（草稿、骨架、节点语义、附件），合并是在结构层做对齐而非在文本层做拼接。** 文本摘要可以在语义相似性基础上做很好的句子级拼接，但它无法区分"这一步在所有路径中共章"和"这一步只在一条轨迹中被偶然看到"，因而无法自然解决频率加权和异常过滤的问题。

## 实验

WSASkill 数据集，70 个技能，每个技能配多条跨场景交互轨迹。

**W2S 的行为重放一致性比 Anthropic Skill Creator 高出 10.5%。** 行为重放是一类比 pass rate 更严苛的指标：不是看"这个技能能不能完成任务"，而是看"用这个技能执行时，Agent 的行为路径和决策模式是否跟原始专家轨迹保持一致"。一个技能文本可能在结局上成功但在路径上偏了：把它用在了不该用的场景、跳过了验证步骤、回避了回退逻辑。

Anthropic Skill Creator 作为对比基线已经是对基础 LLM 摘要能力的高水准工程化封装，但它仍然以自由文本为输出格式。**10.5% 的差距主要来自结构完整性：W2S 在保留分支逻辑、恢复步骤和验证标准方面的漏项显著更少。**

## 几个问题

**场景覆盖的完整性是未观测变量。** W2S 生成的技能质量受限于输入的轨迹是否覆盖了足够多的执行场景。如果某个安全关键的回退路径从未在轨迹中出现过，W2S 不会凭空补上。这跟纯 LLM 摘要一样，区别在于 W2S 至少会在产出中标注置信度，告诉你"这部分证据充足，那部分证据薄弱"。

**WSA 四层结构目前是预定义的。** 当前版本的 Skill-IR 用四种固定组件来建模所有技能类型。对于某些过程极其简单、只有单步操作、没有分支逻辑的技能，四层结构中有两层会几乎为空。这些技能可能更适合用更轻量的表示来生成。反过来，对于多 Agent 协作、嵌套子任务等更复杂的过程，四层也许不够。WSA 的抽象粒度是否可以参数化，论文未讨论。

**轨迹分段的粒度选择是启发式的。** 当前按操作类型切分，但不同任务的"一个自然步骤"跨度差异很大。一个数据清洗技能的一步是"读 CSV 然后 check 空值"，对应约 10 行代码；一个论文搜索技能的一步是"在 Semantic Scholar 上输入查询词并解析第一页结果"，对应一整套 API 调用链和结果解析。同一套分段启发式在不同任务上可能切出尺度极不相同的"步骤"。

## 最后

W2S 指出了一个被忽视的问题：**从经验中自动生成技能，瓶颈不在生成模型的 token 预测能力，而在"文本摘要"这个任务框架本身不匹配。** 技能不是一段可以被摘要的历史，是一份必须被重构的工作流规范。

把技能生成从"把很多条轨迹写成一篇文章"重新定义为"把很多条轨迹对齐成一张有节点、有分支、有语义、有操作上下文的可执行图"，W2S 做的就是这个转换。
