# SKILLFLOW实测：11款大模型Agent Skills进化能力大比拼，大部分越学越差？

# 一、智能体能不能自己学会"进化"Skills？

搞过LLM Agent落地的朋友，估计都有过这种噩梦：Agent刚上线的时候好好的，用了俩礼拜Skills攒了一堆，反而越来越傻，犯的错越来越离谱，还找不到原因。之前一直没个靠谱的工具能测Agent的「终身学习」能力——也就是它能不能真的从经验里沉淀有用的Skills？失败后能不能修复？并在一些列任务重维护一个不断演化、一个越用越强的Skills库？SkillWeaver和MemSkill这类系统证明"从经验里学Skills"这条路能走，但没人系统性地测过"终身Skills演化"这回事。

最近中科大联合多伦多大学等团队发布的**SKILLFLOW**基准，正好补上了这个缺口，测完的结果可以说打了大半主流模型的脸。

# 二、SKILLFLOW任务构建流程

SKILLFLOW任务库评估构建流程包含四个核心步骤：

1. 收集：收集候选的种子任务，并建立一个经过筛选的外部Skills库。
2. 配对匹配：利用嵌入技术进行配对匹配，将相关的Skills与每个种子任务关联起来，形成“任务-Skills”参考对。
3. 领域扩展：该步骤在固定的领域与评价框架（DAEF）下进行任务扩展，包含一个迭代过程：
   1. 架构师智能体：首先生成一个初始的跨领域任务集。
   2. 评审家智能体：对这些任务集进行审核，决定接受或拒绝。
   3. 二次扩展：对于被接受的任务集，会进行第二轮扩展，增加更多任务并建立难度梯度。
4. 人工审查：在最终纳入基准测试前，需经过人工审查，确保任务的：鲁棒性、逻辑合理性、难度校准、无指令泄露风险。

# 三、SKILLFLOW的四大核心设计，每个都戳中落地痛点

SKILLFLOW的设计思路非常明确，就是1:1模拟真实Agent的落地运行场景，从任务生成到评估流程，全都是照着实际上线的情况做的，核心设计有四个：

## 3.1 DAEF（领域无关执行流）：从根上保证能测出真实迁移能力

这是SKILLFLOW最核心的创新，也是它和其他基准最大的区别。

简单说，**DAEF（领域无关执行流）**就是把任务里的领域相关细节全部去掉，只保留最核心的执行流程。比如「修改PPT里嵌入的表格」这个任务，不管是修改金融汇率表、化工催化剂配比表还是医院的检验数据表，核心流程都是一模一样的：读文件→提取嵌入表格→修改目标值→验证公式→保存输出，这就是一个DAEF。

SKILLFLOW一共梳理了20个这样的DAEF，每个DAEF生成8-9个不同领域的任务，难度梯度递增，最终形成20个任务家族共166个任务，覆盖金融、供应链、医疗、政务、文档处理5大领域。

这种设计的好处就是，它能真正测出来Agent是真的学会了通用的执行流程，还是只是死记硬背了特定领域的知识。如果Agent真的掌握了某个DAEF的Skills，那它应该能把这个Skills迁移到同家族的所有不同领域的任务上，而不是只会做某个特定领域的任务，从根上避免了刷分作弊。

## 3.2 双Agent任务生成流水线：任务质量比人工出的还高

为了生成高质量、无歧义、可实际运行的任务，SKILLFLOW设计了一套脑洞很大还特别实用的**双Agent任务生成流水线**，全程几乎不用人工参与，生成的任务质量极高：

1. 先从之前的成熟基准SkillsBench和GDPval里筛选了64个高质量的种子任务，都是真实的企业级工作场景，没有偏题怪题。
2. 用GPT-5.3-Codex作为**Architect Agent**，基于每个种子任务的DAEF生成跨领域的新任务，同时自动构建对应的Docker运行环境、测试数据和验证脚本，保证每个任务都能实际运行，有明确的成功/失败判断标准，不用人工写用例。
3. 用Claude Opus 4.6作为**Critic Agent**，对生成的任务做严格审核：检查任务是不是符合DAEF的流程、难度梯度是不是合理、有没有指令泄露、验证脚本是不是正确，不合格的直接打回让建筑师Agent重改。
4. 最后还要经过人工审核，过滤掉有歧义或者运行不稳定的任务，最终保留了20个家族166个高质量任务。

这套流程生成的任务，不仅没有人工偏见，而且每个都能实际运行，有明确的验证标准，测出来的结果非常客观，根本没法靠刷分蒙混过关。

## 3.3 终身学习评估协议+Skills补丁规范：完全照搬真实上线流程

SKILLFLOW的评估流程和我们真实上线Agent的过程一模一样，最大程度还原了真实场景：

- Agent一开始是空的**Skills库**，没有任何预置Skills，和我们从零开始搭的Agent完全一样。
- 按照难度从低到高依次完成同一个任务家族的8-9个任务，和我们上线后慢慢处理越来越复杂的业务流程一样。
- 每次完成一个任务后，Agent会得到执行轨迹和验证结果的反馈，然后生成**Skills补丁**来更新自己的Skills库，就像我们根据线上bug修复Skills一样。
- 更新后的Skills库会被用于下一个任务的执行，以此类推。

为了避免复杂的指令要求干扰模型的能力发挥，SKILLFLOW还专门设计了**最小化Skills补丁规范**，每个补丁只有三个字段，简单到不能再简单：

- `summary`：这次补丁的改动说明，比如「修复了Excel公式计算时的精度问题」。
- `upsert_files`：新增或者修改的Skills文件，比如新生成的通用Excel处理Skills脚本。
- `delete_paths`：要删除的旧Skills文件，比如已经被替代的过时Skills。

这种设计的好处是，不仅Skills的所有改动都可追溯、可审计，而且排除了不必要的指令干扰，真正测的是模型本身的Skills进化能力，而不是它理解复杂指令的能力。

## 3.4 多维度评估指标体系：不唯成功率论，避免纸面好看实际没用

SKILLFLOW没有像其他基准那样只看成功率，而是设计了三个维度的评估指标，全面衡量Skills进化的效果，避免出现「成功率涨了但成本涨了三倍」这种华而不实的情况：

| 指标类型 | 具体指标 | 说明 |
|----------|----------|------|
| 核心效果指标 | 任务成功率 | 最终所有任务的平均成功率，最核心的指标 |
| 效率指标 | 单任务交互轮数、成本、输出Token数 | 衡量Skills进化会不会带来效率的提升，避免捡了芝麻丢西瓜 |
| Skills质量指标 | 最终Skills库大小、Skills复用率 | 衡量Skills是不是真的通用、可复用，有没有出现Skills膨胀的问题 |

这套指标体系能帮我们更全面的判断Skills进化是不是真的有用，而不是只会刷成功率的纸面功夫。

# 四、11款大模型实测结果：差距大到超乎想象

这次SKILLFLOW一共测了11个主流大模型，包括Claude系列、GPT系列、Qwen系列、MiniMax系列、Kimi等等，结果非常扎心，我第一次看的时候都有点不敢信：

![alt text](image-2.png)

从结果里能看到几个非常有意思的结论，完全推翻了我之前的很多认知：

## 只有Claude Opus 4.6做到了「越用越好还越便宜」

它是所有模型里唯一一个成功率提升超过5%，同时成本还下降的模型，Skills复用率高达72.3%，说明它生成的Skills真的是通用的，能反复用，而且越用效率越高。

研究团队还做了个对照实验：如果把历史对话全部塞到上下文里而不用Skills库，Claude Opus 4.6的成功率只有51.04%，比不用Skills还低，说明这个提升真的是来自Skills进化的能力，而不是靠堆上下文堆出来的。

## 大部分模型要么学了没用，要么越学越傻

大部分模型的提升都在3个百分点以内，基本等于没用。尤其是Kimi K2.5，Skills复用率高达66.87%，相当于做三个任务就要用两次Skills，但成功率只涨了0.6个百分点，约等于白忙活，学了半天啥也没学会。

更惨的是GPT 5.3 Codex、Qwen系列的模型，用了Skills之后成功率反而下降了，属于典型的「越学越傻」，还不如不学，Skills库反而成了累赘。

## 最大的差距不是写Skills，而是改Skills

研究团队分析了所有模型的Skills补丁，发现几乎所有模型都能在做完任务之后写出个像模像样的Skills，但只有Claude系列的模型能准确识别之前的Skills有什么问题，并且能改对。

大部分模型要么发现不了Skills的问题，要么越改越错，甚至会把本来能用的Skills改坏，导致后续任务接连失败，这也是为什么很多模型越用越差的核心原因。

## Skills膨胀是普遍问题，越爱学的模型反而越差

表现差的模型几乎都有Skills膨胀的问题：比如Qwen系列的模型，几乎做一个任务就生成一个新Skills，Skills库大小和任务数量线性增长，但这些Skills的复用率极低，大部分只用到一次，不仅没用，还会增加检索的难度，导致后续任务更容易选错Skills，形成恶性循环。

而表现好的模型，最终的Skills库都很小，一般一个任务家族只有2-5个通用Skills，复用率很高，效果也好，真正做到了「少即是多」。

# 五、对Agent系统设计的4个反常识启示

SKILLFLOW的实测结果，完全推翻了我们之前对Skills系统的很多固有认知，我们团队甚至专门对照这个结果改了我们的Skills系统设计规范，确实解决了很多之前的痛点：

## Skills库不是越大越好，10个精品Skills胜过100个凑数的

之前我们总觉得Skills库越大，Agent能力越强，但实测结果表明，Skills库的质量远比数量重要。10个经过充分验证、复用率高的通用Skills，比100个只能用一次的零散Skills有用得多，而且不容易出问题。

以后做Skills系统的时候，不要再追求Skills的数量，而是要鼓励Skills的抽象和合并，尽量用更少的Skills覆盖更多的场景。

## Skills的迭代管理比生成重要10倍

现在几乎所有Agent框架都在强调Skills生成的能力，但实测结果告诉我们，大部分模型都能生成Skills，但很少有模型能正确修改和淘汰坏Skills。

所以在我们的系统设计里，Skills的测试、审核、版本管理、回滚、淘汰机制，远比Skills生成功能重要的多。一个没有审核机制的Skills库，早晚会被坏Skills污染，导致Agent越用越差，还查不到原因。

## Skills复用率比Skills使用率重要

很多人觉得Agent调用Skills的频率越高，说明Skills系统越有用，但Kimi的例子告诉我们，使用率高不代表有用，如果都是调用一次就扔的一次性Skills，使用率再高也没用，反而浪费资源。

真正重要的是Skills复用率，也就是一个Skills被反复调用的次数，复用率越高，说明Skills的通用性越强，价值越大。我们的系统设计应该鼓励复用现有Skills，而不是生成新Skills。

## 可审计是Skills系统的基础要求，没有就别上线

SKILLFLOW的Skills补丁设计给了我们很大的启发：Skills的所有改动必须可追溯、可审计，不然出了问题你根本不知道是哪个Skills改坏了，也不知道为什么改，想回滚都不知道回滚到哪。

就像我们写代码必须用Git做版本控制一样，Skills系统也必须有完整的版本管理、改动记录、回滚机制，这是生产级Skills系统的基础要求，没有这个能力的系统，根本不敢上线用。

# 六、未来展望：Skills生态的大门正在打开

SKILLFLOW的出现其实是一个非常明确的信号：Agent的评价体系已经从「单次任务的成功率」，转向了「长期运行的进化能力」。这意味着Agent的发展已经从Demo阶段，真正进入了生产落地的阶段。

接下来几年，我判断两个趋势会非常明显：

1. **Skills标准化协议**会很快出现。SKILLFLOW的Skills补丁规范其实已经是一个雏形，未来一定会有行业通用的Skills标准，让不同厂商开发的Skills可以在不同的Agent系统里无缝迁移，不用每个平台都重新开发一遍，现在已经有不少团队在做这个方向了。
2. **跨Agent的Skills市场**会慢慢成型。以后我们做Agent，不用每个场景都从零开始写Skills，直接从Skills市场买别人已经打磨好的通用Skills，比如Excel处理、PDF解析、合同审核这些通用能力，自己只需要开发少量业务专属的Skills就行，Agent的开发成本会下降一个数量级。

# 最后说几句

SKILLFLOW其实给整个行业浇了一盆冷水：原来我们吹了半天的Agent Skills进化能力，大部分模型根本做不到，就连GPT 5.3 Codex这种级别的模型都会越学越差。

但同时它也给我们指明了方向：现在最大的短板不是生成Skills的能力，而是管理和迭代Skills的能力。对于我们开发者来说，现在提前把**Skills生命周期管理**的能力做好，等模型能力跟上的时候，就能第一个吃到红利。

毕竟，未来的Agent竞争，拼的早就不是单个任务的效果，而是长期运行下的稳定性和进化能力。
