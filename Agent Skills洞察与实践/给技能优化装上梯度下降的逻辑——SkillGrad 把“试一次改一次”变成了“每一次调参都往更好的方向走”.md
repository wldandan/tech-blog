# 给技能优化装上梯度下降的逻辑——SkillGrad 把"试一次改一次"变成了"每一次调参都往更好的方向走"

一份技能从写好到好用，中间隔着无数轮"跑一下→发现不对→改→再跑"的循环。你有这种感觉吗？**这个过程很像神经网络训练，但缺了最关键的东西：梯度。** 神经网络的每轮更新知道该往哪个方向走、走多大步。技能的每轮修改靠的是"这轮失败了，我猜问题出在这里，把这个地方改一下试试"。

Penn State 的 SkillGrad 给技能优化装上了一套梯度下降的逻辑。不是真的在算导数（技能是离散文本）而是把梯度下降的四个核心概念全部用文本机制模拟了一遍：**参数（技能包）、损失证据（任务执行结果）、梯度信号（自动诊断）、动量（跨轮累积的修正模式）。** 然后每轮迭代像梯度下降一样系统性地更新技能。

在 SpreadsheetBench 和 WikiTableQuestions 上，SkillGrad 超过所有训练基线和静态技能方案，比最强的训练基线（EvoSkill 和 Trace2Skill）平均高出 6.7 个百分点。

---

## 一、技能优化缺了什么

现在的技能优化基本是反射式：跑一轮 → 失败了 → LLM 看看轨迹 → 写一份改进意见 → 改技能。下一轮再跑，再改。

**这个循环有两个结构性问题。** 第一，每轮修改只看最新一轮的结果，没有跨轮记忆。上一轮发现的好模式、上一轮修复过的老问题，到了下一轮完全不存在于优化器的视野里。第二，只有失败信号，没有"成功了之后哪些该保留"的正面信号。当前技能从"失败"变成了"成功"的那部分，这是最关键的信息，在纯失败驱动的优化里被丢掉了。

**SkillGrad 解决的本质上就是这两个问题：给优化加跨轮记忆，给失败信号配正面证据。**

---

## 二、四个核心机制

SkillGrad 把梯度下降的四个组件用文本机制逐一实现：

**参数 = 结构化技能包。** 一份技能不是一段平文本，而是三层结构：L1 元数据头（何时触发、技能名称）、L2 SKILL.md 正文（始终加载的核心指引）、L3 资源文件（条件加载的详细过程、边缘案例、示例）。**优化的一个关键子决策是"新增内容放在哪一层"。放在 L2 会始终占上下文但保证被读到，放在 L3 只在需要时加载但可能被遗漏。**

**损失证据 = 任务执行结果 + 完整轨迹。** 每轮当前技能在一批训练任务上执行。结果不只是通过或失败，还有完整的执行轨迹。失败的任务告诉你技能少了什么或写错了什么。**更重要的是对比成功轨迹：这轮成功了但上轮失败了。这类轨迹揭示的是当前技能到底学会了什么、哪些修改是应该保留的。**

**梯度信号 = 自动诊断。** 诊断器不是让 LLM 泛泛地"分析失败原因"，而是把每轮的失败轨迹和对比成功轨迹转换成结构化的文本诊断。每条诊断就像标量梯度，指明了技能在哪个维度上需要朝什么方向修改。诊断包含了"缺失什么""哪里有错""什么应该保留"三个维度的信息。

**动量 = 模式记忆 + 当前覆盖层。** 这是 SkillGrad 最巧妙的设计。一个动量 Agent 维护两份状态：**持久模式记忆 M**，累积跨轮反复出现的诊断模式，比如"你的技能在描述分页处理时三次出错"，这个模式会从模糊逐步变得具体和可靠。**当前覆盖层 O**，只反映本轮新出现的模式。**两份状态共同输入给更新器：老模式提供"从根上改什么"的证据，新模式提供"这轮出了什么新问题"。**

---

## 三、分层补丁：不是重写技能，是在正确的那一层做手术

拿到诊断和动量状态之后，补丁器不是"生成一份全新的技能"。而是做一个**分层级的手术**。

补丁器判断新增内容应该落在哪一层：核心流程和通用原则放 L2（SKILL.md 正文），长过程和边缘案例及代码示例放 L3（条件加载资源），需要调整触发条件则改 L1（元数据）。

**这个分层设计是关键。** 把一次性的任务细节塞进 L2 会在每次加载技能时白白消耗上下文。把核心工作流指引放在 L3 会导致 Agent 在需要时找不到。层间路由是技能优化独有的问题。Prompt 优化不需要考虑"这段指引应该放在正文还是附录"。SkillGrad 把这个决策纳入了参数更新的范畴。

每轮产生一个新的技能包 S_{t+1}，进入下一轮迭代。整个过程循环直到迭代预算耗尽或验证分数收敛。

## 四、实验说了什么

SpreadsheetBench Verified 和 WikiTableQuestions 上跑，两种 Backbone（GPT-4o 和 Claude Sonnet 4.5），两种初始技能来源（LLM 自生成和第三方下载）。对比静态技能、EvoSkill、Trace2Skill。

### 全面超过训练基线

**平均超过最强训练基线 6.7 个百分点。** 在第三方下载的初始技能上，SkillGrad 把一份"别人写的、不完全适配你任务"的技能迭代到了明显更好用的版本。在 LLM 自生成的初始技能上同样有效。

### 消融：动量和对比诊断各自贡献

去掉动量机制，分数下降。去掉正面对比诊断（只看失败，不看"本轮成功但上轮失败"的轨迹），分数下降。**两个组件加起来贡献了 SkillGrad 相对于纯失败驱动优化的大部分增益。**

### 动量代理在做什么

论文分析了动量代理累积的模式记忆。早期轮次，模式记忆模糊而分散："技能在工具调用上有问题"。后期轮次，同一个模式的描述变得具体："技能在描述如何使用 search 工具获取分页数据时，没有提到 total_count 字段的含义"。**动量代理做的事本质上就是"把一个模糊的缺陷描述逐步精确化"。**

---

## 五、这篇论文在说什么

SkillGrad 做的事情说出来很简单：**技能优化不应该是"跑一次，看到失败，凭感觉改一次"。它应该像神经网络训练一样，有方向、有记忆、有信号分解。**

这个想法本身比 SkillGrad 的具体实现更有价值。梯度下降之所以有效，不是因为梯度算得准，是因为它把优化过程分解成了明确的组件（参数、损失、梯度、动量、更新），然后每个组件各自承担一部分责任。SkillGrad 把这个分解逻辑搬到了技能优化的世界里。效果好坏还在其次，这个方向本身就是对"试一次改一次"模式的一次升级。

**如果你在用 EvoSkill、MIND-Skill 或 Trace2Skill 做技能优化，SkillGrad 提供了一套可以立刻借鉴的实践：别只看失败轨迹，把"这次成功上次失败"的对比轨迹也喂给优化器。别每轮从零开始诊断，把前几轮反复出现的模式积累起来，它们是最可靠的修正信号。别把新增内容不分层地塞进同一份文档，L2 和 L3 应该有不同的内容策略。**
