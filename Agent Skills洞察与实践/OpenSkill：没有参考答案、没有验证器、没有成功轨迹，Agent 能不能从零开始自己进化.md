# OpenSkill：没有参考答案、没有验证器、没有成功轨迹，Agent 能不能从零开始自己进化

> 现有的自进化 Agent 假设有一条可用的学习回路：要么有人手工写好的技能库，要么有成功轨迹可以参考，要么有 verifier 打分做反馈。真实开放世界部署可能一条都没有，只给你一个任务 prompt。OpenSkill 做的事就是：**从这个 prompt 出发，从公开文档、代码仓库、论文、教程里自己搜知识，自己建验证锚点，自己生成虚拟测试来打磨技能，最后零样本部署到目标 Agent。** 全程不碰任务答案，不用人工标注，不靠模型先验。

## 自进化的前提，开放世界没有

自进化 Agent 在基准上表现不错。但基准环境给 Agent 配齐了进化需要的所有要素：技能库要么是人写的要么是从成功轨迹里蒸馏的，改进信号要么来自 verifier 打分要么来自自我反思的 LLM Judge。

**现实开放世界部署只给你一个 prompt。** 没有技能库，所有领域知识要从零获取。没有验证器，帮不了判断这次改完到底是好了还是坏了。没有成功轨迹，没有参考样例告诉模型"正确的做法长什么样"。

这引出了一个比"怎么更好地进化"更基础的问题：**Agent 能不能从零开始构建自己的学习回路？** 不是用现成的技能和信号去做优化，而是自己从开放世界中找到该学什么、自己搭一套不依赖参考答案的验证机制、自己在这套机制上打磨技能。

Lehigh 大学、UIC、UBC、Vector Institute、Salesforce AI Research 和哈佛医学院的联合团队在 **OpenSkill** 里把这个场景定义成了开放世界自进化问题，并给出了一套可跑通的方案。给定一个任务 prompt、一个基座模型、工具访问权限和开放世界资源（公开文档、代码仓库、论文、教程、网页），其他什么都没有。**最终评估时才解锁参考答案，进化过程中全程隔离。**

## 三阶段：搜知识、建考场、自己考自己

### 阶段一：开放世界知识获取

直接用模型的参数化知识写技能有两个问题：模型的训练截止日期决定了它不知道最新的 API 和库，很多垂直领域的约定和规则根本没进过训练数据。

OpenSkill 主动去搜。给定任务指令 I 和环境 E，一个检索函数 D 遍历开放世界资源池 K，抓回跟任务相关的知识文档：背景概念、最佳实践、API 文档、引用来源。基于 I、E 和检索到的知识 k，生成一份结构化的技能计划 p，规定技能的架构、关键步骤和领域规则。

**与此同时，OpenSkill 还检索另一类知识：验证知识 kv。** 这不是参考答案，而是可以独立验证的锚点：官方文档里的函数输出格式、公开数据集的行数、领域标准中的交叉验证流程、预期输出格式。这些锚点不透露任务答案，但为下一阶段的虚拟测试提供了事实基础。所有被检索查询都经过信息隔离审计，排除了包含基准名称或任何可能导向任务答案的标识符。

### 阶段二：无泄漏的技能进化

手里有了知识、技能计划和验证锚点，问题变成了：**在没有 ground-truth 反馈的情况下，怎么判断一个技能好不好？**

OpenSkill 的做法是为每个任务生成一套虚拟测试。这些测试不是瞎编的，每一道断言都锚定在阶段一获取的验证知识上。例如，一个数据处理任务中，虚拟测试不检查中间计算结果（那可能泄露答案），而是检查输出的行数是否匹配公开数据集文档中的已知行数、输出的列名是否正确、输出的格式是否符合文档中的规定。

虚拟测试跑起来就是一套确定性的 pytest 套件。产出的是一个虚拟通过率 r˜，作为技能质量的一个替代指标。

**进化循环的每一步是这样的。** 用当前技能在环境中执行，跑虚拟测试。如果通过了，一轮结束。如果没通过，系统生成一份结构化的故障诊断报告，包含逐项断言结果、根因分析和修订建议。当诊断指出的是知识缺口（而非代码 bug）时，触发定向检索去补充知识，再将新知识注入修订上下文。

**整个循环最多跑三轮。** 每轮改进的评分只来自虚拟测试，从未看过任务参考答案。技能在进化过程中被原地编辑，而非在多轮之间做 best-of-N 挑快照。

### 阶段三：零样本目标评估

进化的产物是一个冻结的技能产物，不是模型权重。它可以零样本部署到任何 Agent 上，不需要重新训练。这项设计意味着用一个模型构建的技能可以直接搬到另一个模型上用。**参考答案在第三阶段才第一次被引入，而且仅用于最终评分，不影响进化过程。**

## 实验

三个基准：SkillsBench、DSBench、GAIA。两种目标模型家族，用 Claude 构建技能后部署到 GPT 上，反之亦然。

**OpenSkill 在所有基准-模型组合上拿到了最高的自动化 pass rate。** 在 SkillsBench 上，比最强封闭世界基线高出 8.9 到 8.8 个百分点。这个比较的背景是封闭世界基线可以使用多轮自我反思和外部反馈信号，而 OpenSkill 全程没有参考答案的辅助。

跨模型迁移实验的结果更说明问题。**用 GPT 构建的 OpenSkill 技能直接部署到 Claude 上，性能几乎没有损失；反过来也一样。** 技能作为显式产物而非模型权重的设计让迁移不依赖模型特定适配。

最后还有一项直接检验进化质量的分析：**OpenSkill 自建的虚拟 verifier 能覆盖 88.9% 的真实 task intent。** 也就是说，进化过程中用来给自己打分的虚拟测试，跟最终任务评估的 ground-truth 在意图上是高度吻合的。这个覆盖不是靠训练得到的，verifier 完全没有访问 ground-truth 的权限。它来自对公开知识中可独立验证锚点的系统性提取。

## 几个问题

**开放世界知识的质量和密度是天花板。** 如果某个任务领域在公开文档中没有足够密度的可验证锚点（行数、格式、标准范围），虚拟测试的覆盖就会不足，进化信号也会变弱。当前实验选择的任务都是知识覆盖面较广的基准任务，对于文档稀疏的小众领域，这个机制还没被测试。

**信息隔离的强度是审计驱动的，不是构造性保证。** 当前实现了对查询词条的过滤来防止答案泄漏，但没有提供形式化的信息泄漏证明。在某些情况下，一个看似遵循过滤规则的查询可能通过间接推理路径接近任务答案。

**三轮进化预算的合理性因任务而异。** 复杂任务的虚拟测试覆盖率更高但每个 assert 更弱，简单任务的虚拟测试更精确但覆盖率低。统一的 J = 3 预算对不同任务类型可能不是最优的：任务难度差距很大时，有些任务可能刚展开就需要额外轮次，而有些任务一轮就到位了却被迫多跑了两轮。

## 最后

OpenSkill 回答了一个比"怎么让进化更有效"更底层的问题：**在什么都没有的情况下，进化回路本身能不能被构建出来。**

答案是能。从开放世界的文档和代码中提取知识和验证锚点，用自己的虚拟测试代替 ground-truth 反馈，在隔离参考答案的前提下完成技能的迭代改进。OpenSkill 证明了开放世界中有足够多的"免费锚点"：不需要标注，不需要人工，不需要访问答案，就可以撑起一个自验证的学习循环。
