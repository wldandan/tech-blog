# SAPO：技能进化的瓶颈不是生成能力，是不验证就入库

> 现有的技能增强 RL 方法用 GPT-5.4 或 Claude Opus 来读轨迹生成技能，然后直接放进技能库。前提是"强模型生成的技能应该都是好用的"。但实验数据打了这个前提的脸：**GPT-5.4 生成的技能平均边际效用接近零。一小部分确实有帮助，更多的既不帮忙也不捣乱，还有一部分直接拉低了成功率。** 而一旦这些低效技能进入技能库，发现它们有害的信号是延迟的：等后续 rollout 反馈回来的时候，它们已经误导了多轮 Agent 决策。SAPO 的做法是把验证搬到入库之前，用一个零额外成本的机制当场测出新技能是否值得留。

## 强模型生成≠好用

技能增强 RL 的主流范式是这样的。训练中遇到失败，把轨迹发给一个更强的模型（通常是 GPT-5.4 或 Claude Opus），让强模型从失败中归纳出可复用的改进策略。产出的技能直接进库，供后续检索。

宾州州立、NTU、UCSD、犹他和哈佛的联合团队在 **SAPO** 的实验部分先做了一个关键的诊断性测量。训练期间每隔一段时间，把 GPT-5.4 生成的技能取出来测边际效用：在同样的任务和检索上下文中，对比加了该技能和没加该技能的两组 rollout 的成功率差异。

**结果：GPT-5.4 生成的技能平均边际效用全程接近零。** 有用技能的效用被大量低效和有害技能的零负效用拉平均之后，整个技能生成过程的 net contribution 几乎消失。把生成的技能分成 promoted（效用 top 20%）和 discarded（其余 80%），两组的效用差距显著。确实有一部分好技能，但它们被淹没在大量不产生边际增益的"水技能"里。

**这些低效技能不是"看起来不好"。它们恰好是读起来合理、但执行中不解决问题。** 比如技能建议"当找不到目标物品时，先检查邻近区域再扩大搜索范围"：这段文字在任何上下文中都像好建议，但在具体执行中它可能让已经是正确路径上的 Agent 产生不必要的探索性偏离。

**更重要的是，技能效用是上下文条件化的，而且会随策略进化而衰减。** 同一个技能，在当前检索上下文中没有相似技能覆盖时有用，在同类技能已经存在时完全冗余。随着在线 RL 让策略变强，一些早期有用的技能也会过时。但现有方案既不能实时感知"这个技能什么时候变冗余了"，也不能在检索时做上下文条件化的排序。

## SAPO：把一个预算切成两半，当场测出技能值不值

SAPO 的核心操作是一次预算分劈。每个任务的 rollout 预算 G 被切成两半。前一半跑基线 rollout，只用当前检索到的已有技能。后一半跑技能增强 rollout，在同样的任务和检索上下文下，多加一个候选技能 ŝ。

因为两组 rollout 共享同一个任务和同一个检索上下文、只有候选技能这一个变量，**它们的奖励差距直接估计了候选技能在当前上下文中的边际贡献。** 不需要额外分配 rollout，不需要等后续反馈回来再做滞后的效用推断。

正效用的技能被 promote 进入长期技能库 B_long。效用接近零或负的被丢弃，不进入技能库，不影响后续策略学习。

**这个 pre-storage validation 和现有的 post-storage tracking（D2Skill 等的做法）之间的区别是时序性的，但后果是质的。** 入库后的追踪在低效技能被识别出来之前，它已经被多次检索和注入 Agent 推理上下文。在这些被注入的 rollout 中，策略梯度是被一个"实际上不提供价值但系统以为提供了"的技能所扭曲的。

## 策略自己就是技能生成器

SAPO 的另一半设计是用这个效能信号来训练策略本身成为更强的技能生成器。

现有的在线 RL 方法在训练中反复调用 GPT-5.4 等外部模型来生成技能。每次调用都是 API 成本，而且如果生成的大部分技能最终被证明没有效能增益，钱就白花了。

**SAPO 把技能生成的监督信号从"模型尺寸"换成"技能效用"。** 一个技能被 promote 进入长期库，说明它的边际效用为正，那么就训练策略提高生成类似技能的概率。一个技能被 discarded，说明它的效用不够，那么就训练策略降低生成这类技能的概率。策略的 skill-generation likelihood 被持续用这个二元信号做在线更新。

策略学习到的生成概率有一个附带价值：**它可以作为检索时的上下文条件化分数。** 当一个查询来了，SAPO 不是只用语义相似度来排序候选技能，而是把策略在"给定这个查询上下文时"生成每个技能的概率作为附加排序信号。这个概率天然是上下文条件化的：策略在某个上下文中更可能生成某个技能，意味着策略"认为"这个技能对那个上下文有用。

同样的分数还被用于长期维护：**当一个技能被当前策略认为生成概率很低（即策略在最新状态下不再认为这个技能有用），它被标记为可能过时并从库中删除。** 这解决了"策略变强之后早期技能变成噪声"的问题。

## 实验

两个基准：ALFWorld（交互式具身决策）和 WebShop（网页购物）。底座模型：Qwen2.5-7B-Instruct。

**SAPO 在训练全程的验证成功率曲线上持续优于 SkillRL 和 SkillRL/S（无进化的固定技能库变体）。** 在 ALFWorld 上，SAPO 不仅最终成功率最高，而且上升速度比 SkillRL/S 更快：技能验证和选择性入库不是"多了一道工序所以更安全"，而是"多了一道工序所以更快达到好策略"。

效用分析跟 baseline 的实验形成了完整的逻辑链。**SAPO 的 promoted skills 平均效用随着训练轮次持续上升并保持正值，discarded skills 保持在零附近。** 相比之下，SkillRL 中不经验证入库的技能整体效用一直徘徊在零附近。效用的正负分离是训练动态改善的根本原因。SAPO 的策略每一步学的都是从真正有用的技能中提取的信号，SkillRL 的策略每一步都要从混合信号中做有噪声的梯度更新。

API 成本对比也很清楚。SAPO 在训练后期完全不需要外部 LLM 调用来生成技能，因为策略自己已经学会了生成高质量技能。文中的 API 成本对比表显示 SAPO 的 GPT API 调用数在训练中段就开始下降并最终归零，而 SkillRL 的调用数线性增长。

## 几个值得注意的问题

**预存储验证的精度依赖于 rollout 预算和效果估计的方差。** 用一个 split-half 设计来估算技能的边际效用，本质是用少量 rollout 的 reward gap 来推断技能的真实效能。在奖励方差大的环境（WebShop 的单次购买决策比 ALFWorld 的多步导航噪声更大），相同预算下的效能估计方差更大，可能把一些微正效能的技能错判为 discard。

**策略生成技能的质量受策略在在线 RL 中的整体学习曲线约束。** 训练早期策略本身还在稳定化，生成的技能质量大部分被 discard。只有在中后期策略足够强之后，自生成的技能才有一致的正效用。如果训练任务多样性和难度超出了单底座策略的生成能力边界，自生成技能的效果会退化。

**技能质量评分的语义粒度和检索上下文的匹配精度之间存在 gap。** SAPO 用策略在当前查询下生成一个技能的概率作为该技能的质量分数，这个分数依赖于查询和技能训练阶段接触的原始任务相似性。对于分布外查询，策略生成概率的可靠性需要有更多消融来验证。

## 最后

SAPO 把一个被忽略了的问题摆到了明面上：**技能增强 RL 的前提不是"我们有一个更强的模型所以生成的技能是好技能"，而是"我们需要在每个技能入库之前就知道它到底值不值"。**

把 rollout 预算切成两半来做 in-context 的 A/B 测试，不花钱也不增加轮次，当场给出边际效能的估计。然后把同一个效能信号喂回去训练策略，让策略学会生成高质量技能，替代外部模型调用。验证在入库之前，生成到策略之内。这两个变化加起来，让技能增强 RL 的训练动态从根本上变干净了。
