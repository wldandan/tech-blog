# LatentSkill：把技能从 prompt 文本变成模型权重，不是更短了，是消失了

> 现有的 Agent 技能都是文本。检索出来塞进 prompt，每步推理都带着。token 成本随技能库规模和交互步数线性涨，技能内容以明文暴露在上下文里。LatentSkill 走了一条不同的路：**把一条文本技能通过一个预训练的 hypernetwork 直接映射成一个 LoRA 适配器，挂载在冻结的基座模型上。技能从 prompt 中消失了，变成了权重。** 能加载、能卸载、能缩放、能组合，跟文本技能一样灵活，但 token 开销是零。

## 技能必须在 prompt 里吗

现有技能系统的设计公式很一致。技能是文本，检索出来，注入上下文，Agent 在这个条件下做决策。这套设计的模块化和简单性是其优势，但也带来了一个结构的代价。

**同样的技能文本在每一步决策中都被重复插入。** 交互步数越长、技能库越大、单次调用的技能越多，prompt 的膨胀速度就越快。prefill 成本线性增长只是一个方面。更隐蔽的问题是：长上下文让模型更难稳健地使用所有被注入的信息。技能作为可读文本还存在暴露风险，专有操作流程跟不受信任的环境观测共享同一条指令通道。

参数化替代方案（fine-tuning、curriculum learning）可以避免推理时注入技能文本，但代价是技能被不可逆地融合进模型权重。单个技能无法被单独更新、移除或替换，模块化的灵活性全部丧失。

**这个 trilemma 是这样的：现有方案最多只能同时满足两个需求，没有 prompt 膨胀、模块化可替换、推理时可用，三者不可兼得。** 避免膨胀 + 模块化，就只能把技能留在 prompt 里。避免膨胀 + 推理时可用，就只能把技能和模型融合但永久绑死。模块化 + 推理时可用，prompt 膨胀无法避免。

上海交通大学、中山大学、上海创新研究院和 OPPO 研究院的联合团队在 **LatentSkill** 里用 hypernetwork 解开了这个 trilemma：让技能从 prompt 文本变成可插拔的权重，同时保留 prompt 的三项优势。

## 文本进来，LoRA 出去

LatentSkill 的核心部件是一个技能编译器 G_φ。输入是一段技能文本 s，输出是一组 LoRA 更新值 Δ_s = G_φ(s)。Δ_s 被挂载在冻结的基座模型上，推理时 Agent 的上下文里不再包含任何技能文本。

形式化地写：p_{θ,φ}(y_t | h_t, s) = p_{θ⊕αΔ_s}(y_t | h_t)，其中 α 是控制注入强度的缩放系数。设 α = 0，回到冻结基座。设 α 更大，技能影响更强。这个单标量控制让同一个技能可以按需求调节力度，文本提示做不到这一点。

**hypernetwork 生成 LoRA 的过程只需要一次前向传播。** 技能文本过一次编译器网络，产出目标模块上全部的 A 和 B 矩阵。编译完成之后，适配器可以缓存，同一个技能在同一会话中反复使用不需要重新生成。

## 两阶段训练：先学会读文本，再学会模仿轨迹

**阶段一：技能文档预训练。**

第一批训练数据来自技能文本本身。预训练用了两个自监督任务：

**重构任务。** 编译器读完整的技能文本 s，生成适配器 Δ_s，挂上之后基座模型被要求重构出 s 本身。从权重中重建源文本是一个很强的学习信号：编译器必须把文本中包含的所有过程性知识压缩进低秩权重空间。

**补全任务。** 编译器只读技能文本的前面一部分 s˜，生成适配器，基座模型被要求补全缺失的后半部分。这个任务训练编译器从部分信息中推断一个技能的完整语义，为后续处理不完整或隐式技能描述做铺垫。

两个任务同时跑，只更新编译器参数 φ，基座模型全程冻结。目标是被训练到：将来有一份新的技能文本进来，不需要再对这个技能做任何微调，编译器直接输出可用的适配器。

**阶段二：轨迹监督微调。**

预训练教编译器理解技能文本本身。微调教编译器理解"这个技能应该产生什么样的 Agent 行为"。

训练数据来自教师 Agent 的成功执行轨迹。每条样本把一份技能文档和该文档对应的教师轨迹配对。编译器读技能文本，生成适配器，基座挂载适配器后在相同的任务输入下学习复现教师的行动序列。目标是让编译器生成的适配器不仅反映技能文本的内容，还反映"执行这个技能时实际应该怎么做"。

经过两阶段训练后，技能编译器具备了两个能力：它能把从未见过的技能文本转成有语义的适配器，并且这个适配器产生的行为跟教师轨迹一致。

## 推理时的三种操作：加载、缩放、组合

**加载。** 技能库中每个技能编译一次，存入适配器缓存。选择技能后从缓存加载挂载到基座，上下文没有任何技能文本。卸载同理，直接卸下适配器。

**缩放。** 同一个技能的 LoRA 权重可以乘以不同的 α 系数。α = 0.5 和 α = 1.5 产生不同的影响强度，不需要重新编译。这给技能的使用增加了一层精度控制，是文本注入做不到的。

**组合。** 当需要多个技能同时生效时，直接把它们的适配器权重在参数空间中相加。如果技能之间有共享的语义组件，可以进一步做更细粒度的组件级组合，只把与当前任务相关的组件相加，而非把整份技能权重加在一起，避免过度放大共享部分的行为。

## 实验

两个基准：ALFWorld（具身家务）和 Search-QA（知识检索回答）。

**ALFWorld 上，LatentSkill 在见过和未见过两个 split 上分别比 in-context 技能基线高出 21.4 和 13.4 个百分点，同时 prefill token 减少 64.1%。** 不是"性能一样但省 token"，是"省 token 同时性能显著更好"。

**Search-QA 上，LatentSkill 比 in-context 基线 exact match 高出 3.0 个百分点，技能 token 开销降低 72.2%。**

这些收益不能简单归因于"上下文短了所以模型表现好了"。论文做了进一步的结构分析。

### 技能权重的内在结构

**首先，生成的技能 LoRA 在权重空间中形成了按领域自然分离的聚类。** 不同任务的技能适配器在权重空间的嵌入距离反映了它们语义上的远近。这说明 hypernetwork 不是把文本做无结构的压缩，而是学到了一个对过程性知识的语义有感知的编码空间。

**其次，α 的调节是平滑且连续的。** 不是"0 就是没技能 >0 就是有技能"的二值开关。α 在小范围连续变化时，任务成功率平滑变化。一个技能的行为可以被精细地调到一个中间强度，这种控制在文本提示空间里没有对应操作。

**第三，组合是有条件的。** 当两个技能的文本被分解为语义对齐的组件（每个组件对应过程的一个独立维度）后，它们的组件级 LoRA 相加是有效的。如果两个技能直接整份相加而不做组件级对齐，简单加和可能不会带来增益，甚至可能出现部分抵消。这说明技能的参数空间运算需要一个合适的粒度，不是在文本层面的自由组合，而是在语义组件层面的结构化组合。

## 几个问题

**编译器的泛化能力受训练技能数据的分布约束。** 两阶段训练中接触的技能文本决定了编译器对"没见过文本"的理解边界。如果在部署中需要编译类型完全不同、结构完全不同、语言风格完全不同的技能文本，编译器目前的表现还是开放的。

**技能 LoRA 的存储开销。** 每一条文本技能编译成一个独立的 LoRA 适配器存在缓存中。对于几百条技能的小型库这不是问题。对于几万条甚至几十万条技能的大型库，适配器的存储开销和缓存管理策略需要进一步考虑：哪些适配器常驻内存、哪些在需要时才动态编译，这个调度策略论文没有展开。

**组件级组合的前提是技能可以被分解成语义上可对齐的独立维度。** 对于某些高度耦合的技能（多个维度紧密交织在一起），强制分解可能破坏过程的连贯性。什么类型的技能适合组件化、什么不适合，目前还没有一个系统的判定标准。

## 最后

LatentSkill 做了一种介质转换：**把 Agent 技能从"文本空间中检索和注入"变成了"权重空间中生成、缓存和操作"。**

这个转换带来三种文本技能做不到的能力。精细的强度控制，一个 α 系数连续调节。参数级的组合，不用在 prompt 里拼接多个文档。以及一个附带的安全优势：技能不再是上下文中的可读文本。

技能存放的位置从 context space 迁移到了 weight space，而模块化（可加载、可替换、可独立更新）完整保留了下来。这是 LatentSkill 与传统 agent fine-tuning 和传统 in-context skill prompting 的区别。
