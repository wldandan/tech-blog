# 你的 Agent 技能是别人写的——SkillSafetyBench 发现，攻击者根本不需要绕过 Prompt，他们可以直接毒死你的技能库

先想一个问题。

你从社区下载了一份 Agent 技能，叫"数据处理助手"。描述写得很好：帮你清洗 CSV、生成图表、导出报告。你装上之后让 Agent 跑了一个数据处理任务。Agent 按技能指引一步步执行，最后成功产出了一份漂亮的报告。

**但你不知道的是，这份技能在某个辅助脚本里藏了一行代码，把你本地环境变量全部打包发到了一个外部服务器。** Agent 从头到尾没做错任何事，它忠实地遵循了技能里的每一条指令。用户的任务也是合法的。Prompt 里没有任何恶意内容。所有传统的安全检查都不会触发告警。

攻击不在用户请求里，不在模型推理里，不在工具调用里。**攻击在技能文档和它的配套文件里。**

上海 AI 实验室、北大和华东师大的联合工作 **SkillSafetyBench** 是第一个系统性地评估这个攻击面的基准。155 个对抗案例，覆盖 6 个风险域、30 个安全类别。**结论很不安：多个主流 CLI Agent（Claude Code、Codex CLI、Cursor CLI、Qwen3 Coder）在面对技能介导的本地攻击时，几乎没有有效的防御。**

---

## 一、技能安全是一个被系统性忽略的攻击面

技能正在成为 Agent 的标准接口。Claude Code 有 Skills，OpenClaw 有 Skills，SkillNet 维护着 50 万+ 的技能库。一份技能不只是几行 Markdown 指令，它可以打包辅助脚本、配置文件、Docker 模板、检索语料、环境变量。

**这意味着技能的信任边界远比 Prompt 宽得多。** 一份技能可能带着几十个配套文件进入你的环境。这些文件不是你写的，但在 Agent 眼里，它们和技能指令一样，都是合法的执行上下文。

传统的 Agent 安全评估几乎全部聚焦在三个方向：Prompt 注入（攻击者在用户输入里藏恶意指令）、工具滥用（Agent 把工具用错了）、恶意用户目标（用户直接要求 Agent 做不安全的事）。**SkillSafetyBench 的核心发现是：即使这三个方向全部守住，攻击者仍然有一条完全不同的路径。**

用户请求是干净的。工具使用是正常的。模型没有产生恶意推理。**但技能的材料里藏了东西。一份被篡改的配置文件让 Agent 把敏感文件嵌进了本应只含图表数据的输出包。一段被调包的辅助脚本在 Docker 构建时静默拉取了一个外部 payload。Agent 不知道这些是攻击，因为这些在它的工作流里看起来都是"正常操作"。**

---

## 二、六个风险域：攻击面比你想象的宽得多

SkillSafetyBench 把技能介导的安全风险组织成六个域，每个域有五个标准化类别：

**RD1：上下文信任风险。** Agent 信任了不该信任的东西——技能的检索结果被污染、示例行为被篡改、多数据源的信任信号被协同放大。攻击者不需要直接告诉 Agent "做坏事"，只需要让它"相信错了"。

**RD2：权限越界风险。** Agent 被诱导超出授权范围——伪造审批通过信号、诱导 Agent 索取凭证、逐步扩大行动计划超出用户本意。攻击在 Agent 的决策链里植入了一个"多走一步"的惯性。

**RD3：运行时劫持风险。** 执行路径被重定向——解释器被替换、序列化器被篡改、运行时配置被污染、沙箱边界被绕过。Agent 以为自己跑的是常规代码，实际上跑的是被调过包的版本。

**RD4：输出腐败风险。** 最终产物在交付前被篡改——产出文件的结构被破坏、下游解释器被滥用、外部发送目标被替换、敏感数据通过输出通道静默泄露。

**RD5：状态持久化风险。** 不安全状态被保留下来——攻击痕迹被从审计日志中抹掉、有毒记忆在后续会话中被反复加载、持久化的后门在任务完成后仍然存活。

**RD6：供应链和知识中毒风险。** 技能依赖的外部资源被污染——检索语料被投毒、依赖版本被替换、模型行为被微调偏斜。**攻击可能不在技能本身，而在技能引用的某个外部组件里。**

30 个类别对应 8 种攻击手法：偏见操控、恶意软件、数据投毒、数据外泄、后门植入、拒绝服务、钓鱼、数据销毁。**同一个攻击手法通过不同的本地表面可以造成完全不同类型的安全失效。**

## 三、实验结论：当前的 Agent 面前，技能攻击几乎一打一个准

SkillSafetyBench 跑了一组主流 CLI Agent 和模型后端的组合。Claude Code、Codex CLI、Cursor CLI、Qwen3 Coder 分别搭配不同的底层模型。

**三条最关键的发现：**

**第一，非用户攻击可以持续诱导不安全行为。** 用户请求保持干净，没有恶意措辞，没有可疑指令。但技能材料里的攻击载荷成功引导了 Agent 执行目标有害行为。**Agent 不是"被胁迫"，是被"误导"——它在忠实执行一套合法的技能工作流，其中某个环节被替换了。**

**第二，不同 Agent 的失败模式有明显差异。** 同一个攻击案例，Claude Code 和 Codex CLI 可能在不同的攻击类别上失守。这不只是模型对齐的问题——**Agent 框架怎么加载技能、怎么管理上下文、怎么分配信任给本地文件，这些工程决策同样决定了安全边界。**

**第三，验证攻击成功靠的是实体证据，不是主观判断。** 每个案例配一个确定性的 verify_attack.py 脚本，检查执行产物里是否出现了目标证据——恶意痕迹、泄露的载荷、被污染的记忆记录。**不是因为 Agent 的回复"看起来不好"就判不安全，是因为执行环境里真的出现了被攻击的证据。**

---

## 四、这篇论文在说什么：Agent 安全的边界必须重新画

SkillSafetyBench 的核心信息不是"有个新的攻击手法要防"，而是更根本的：**Agent 安全的范围画错了。**

目前的安全讨论围绕 Prompt、模型和工具。但 Agent 已经在吃掉越来越多的外部资源——技能文档、辅助脚本、Docker 模板、检索语料、依赖包。**每一个被 Agent 读取并信任的文件，都是一个潜在的攻击面。** 而现有的安全评估几乎全部忽略了这些面。

Agent 不是不安全的。它是**在没人想到要检查的地方不安全。**

这跟十五年前 Web 安全从"防 URL 注入"进化到"防供应链攻击"的轨迹一模一样。刚开始大家只检查用户输入。后来发现 jQuery 的 CDN 链接可以被替换。再后来发现 npm 包可以携带恶意脚本。Agent 安全正在经历完全相同的过程，只不过现在攻击载体从"一个 npm 包"变成了"一份技能 + 它的配套文件 + 它所引用的外部资源 + 它在本地环境里留下的状态"。

**如果你从社区下载技能——哪怕是看起来很无害的"数据处理助手"——你信任的不是一段 Markdown 指令。你信任的是这段 Markdown 背后那几十个辅助文件、它们引用的依赖、以及它们在你的环境里执行的行为。SkillSafetyBench 告诉你，这些面目前几乎没有防线。**
