# LLM写的技能全是废稿？微软给它配了一个自己诊断、自己改的医生

> 技能是Agent的核心杠杆，人类专家手写的能把通过率拉高 16 个百分点。让LLM自己写呢？**零增益，甚至帮倒忙。** 微软的SKILLAXE破了这个局：不需要人工标注、不需要测试套件、不需要环境奖励，LLM从四个维度诊断自己的技能哪里不行，然后自己改。SkillsBench上相对提升 28%，跟人类技能的差距缩小了 47-67%。SpreadsheetBench上从 16.0% 飙到 52.0%，只用了 22 个技能。



# LLM写的技能，看着漂亮，一用就废

技能正在成为Agent系统的标配。一份结构化文档，告诉Agent什么时候该做什么、怎么做、注意什么。人类专家手写的技能效果立竿见影，SkillsBench上直接把通过率拉高**16.2个百分点**。

轮到LLM自己写，SkillsBench的数据很残酷：**零增益。** LLM生成的技能语法流畅、结构工整、看起来像模像样，但Agent用了跟没用一样，有时还更差。而且你完全不知道问题出在哪。触发条件设歪了？指令写得太模糊？覆盖的解法路径太窄？现有评测只给一个Pass/Fail，没有任何诊断信息。

这就是技能工程当前的核心死结：**没有可操作的诊断反馈，改进全靠瞎试。** 微软研究院的SKILLAXE就是来拆这个死结的。

# SKILLAXE怎么做：四个维度把技能解剖了

SKILLAXE是一个完全无监督的技能自改进框架。它不依赖人工标注、测试套件、环境奖励，**唯一需要的是任务描述、Agent带技能和不带技能的两次输出。** 基于这三样东西，从四个维度诊断技能出了什么问题。

## 维度一：质量影响，这个技能到底有没有用？

在诊断细节之前，先回答最基础的问题：**用了这个技能之后，Agent比不用的时候好了还是差了？**

做法是配对比较。LLM裁判同时看带技能和不带技能的两个输出（顺序随机化），先判断谁更好，再估计好多少。拆成方向和幅度有实际意义：轻微退化说明技能大方向对、局部需要修；灾难性退化说明技能里可能包含根本性错误的指导，得大改。这个分数是外环信号，告诉你"有没有问题"，后面三个维度告诉你"问题是什么"。

## 维度二：触发精度，技能是不是在正确的时刻激活？

很多技能失败不是因为内容不对，是**触发条件设歪了**。该激活的时候不激活，不该激活的时候跳出来。

SKILLAXE从技能描述中自动抽取正向触发短语和负向排除短语，用依赖解析和否定追踪来做。关键的一步是**语境化嵌入**：同一个词在不同句式里含义完全不同。"spreadsheet"在"创建新spreadsheet"中是正向触发，在"不要对不含spreadsheet的图表触发"中是负向排除。不加句子上下文，两个"spreadsheet"在嵌入空间中一模一样；加了上下文才能分开。从正负短语的嵌入分布中计算三个几何指标：**覆盖广度**（正向触发是不是太窄）、**负向特异性**（排除区域是否跟正向清晰分离）、**边界锐度**（最差情况下的正负混淆程度）。三个指标各自对应一种触发失败模式。

## 维度三：指令合规度 + 故障归因，是技能写得烂还是Agent没照做？

这是四个维度中最精巧的一个。同样观察到Agent没按技能说的做，有两种完全不同的原因：**指令太模糊，Agent不知道该怎么做；或者指令很清楚，Agent就是没执行到位。** 混在一起就会"修"掉本来正确的规则。

SKILLAXE先从技能文档中自动提取可评估的规则集，每条规则带严重程度权重。然后对每条规则同时评四个信号：是否适用于当前任务、Agent遵循得怎么样、规则本身是否精确可操作、观察到的失败该归到技能还是Agent。

关键在于**故障归因**。论文给了一个很直观的例子。规则是"使用黄色背景（#FFFF00）"，Agent用了#FFF2CC，这是Agent没照做，规则已经精确到十六进制了。但如果规则只是"使用黄色背景"，Agent用了合法的黄色 #FFF2CC，这是技能自己不够精确，怪不到Agent。同一种行为，归因不同，改进动作完全相反：前者保留规则，后者把规则写得更精确。基于归因后的分数，SKILLAXE区分出两个信号：**合规度分数**告诉你Agent有没有遵循技能，**技能质量分数**告诉你技能值不值得遵循。前者低改Agent，后者低改技能。

## 维度四：解法路径覆盖，技能是否支持多种合法解法？

前三个维度回答了技能有用吗、触发对吗、指令能照做吗。但还有一个更隐蔽的问题：**即使技能全对，它可能只覆盖了Agent可用解法中的一小部分。**

同一个任务通常有多种合法路径。表格处理可以用pandas、用openpyxl、用Excel COM自动化、或在表内直接写公式。技能不是可执行程序，它只是给Agent提供指导先验。如果Agent选了一条技能完全没覆盖到的合法路径，技能等于不存在。SKILLAXE的做法是让LLM先枚举所有合法解法路径，再计算技能内容跟每条路径的嵌入对齐程度。分数低说明技能只支持很窄的行为空间。

## 实验核心发现

### SkillsBench：提升28%，全部来自"让Agent不崩溃"

77 个任务、191 个技能、688 次Agent运行。SKILLAXE一轮迭代后，通过率从LLM自写的 32.5% 提到 **41.6%**（相对提升 28%），跟人类技能的差距缩小了 **47-67%**。

最有信息量的是分层分解。把通过率拆成**覆盖率**（Agent产出可评估结果的任务比例）和**质量**（产出后通过的比例）：

| 条件 | 覆盖率 | 质量 | 总通过率 |
|------|--------|------|---------|
| 无技能 | 46.7% | 57.1% | 26.7% |
| LLM 自写 | 49.4% | 65.8% | 32.5% |
| **SKILLAXE** | **72.7%** | 57.1% | **41.6%** |
| 人类 | 68.8% | 75.5% | 51.9% |

**SKILLAXE的全部增益来自覆盖率，质量纹丝不动。** 覆盖率从 46.7% 跳到 72.7%，涨了 26 个百分点；质量跟无技能基线一模一样，都是 57.1%。这说明技能的核心价值不在"让Agent答得更对"，而在**"让Agent不崩溃"**。技能编码的是过程性知识：用什么库、怎么处理格式、什么时候触发重算。这些东西让Agent跑通那些原本会崩掉或超时的难任务。人类技能则同时在两个维度上都有提升，说明人类专家既能写清"怎么做"，也能提升"做成什么样"。

### SpreadsheetBench：22个技能，从 16% 拉到 52%

生产场景下，SKILLAXE作为持续的技能积累引擎。200 个训练任务自动构建技能库，50 个留出测试任务评估。Agent是Excel Copilot，Claude Opus 4.5做backbone。

无技能基线只有 **16.0%**。LLM自写产出了 69 个技能，拉到 **52.0%**。SKILLAXE**只用了 22 个技能就达到了同样的 52.0%**，技能数量少了 68%。更关键的是激活率接近LLM自写的 **2 倍**（35.8% vs 20.0%）。技能更少、更精、更被Agent认可为"这个我真用得上"。SKILLAXE的构建机制天然倾向于合并相似模式，而不是每任务造一个新技能。部署时技能清单更短，注入prompt的token更少，延迟和成本都更低。


# 这到底意味着什么

SKILLAXE解决了技能工程中最堵的一环：**从"知道技能不行"到"知道怎么改"之间的鸿沟。** 它证明了一件事：不需要人类标注、测试套件、环境奖励，仅靠Agent自身的行为差异，就能产生足够丰富的诊断信号来驱动技能迭代。

这件事的意义很实际。技能工程的瓶颈不是"会不会写"，任何LLM都能产出一份像样的技能草稿。瓶颈是**"写完之后怎么办"**。SKILLAXE把"怎么办"自动化了：从Agent行为里抓信号，诊断哪里不行，生成改进简报，改写技能。整个过程不需要人插手。

局限也存在。技能内部自洽但策略根本错误的情况检测不到，多技能同时激活的交互冲突没建模，目前只有单轮迭代。但不影响核心贡献的有效性。

对在做Agent技能系统的团队，这篇工作最值得带走的就一句：**别扔掉那些跑失败的Agent轨迹。拿它们来诊断技能，比任何人工检查都快、都准。**
