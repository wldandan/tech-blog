# SKILLGEN：不再手写Skill，用成功和失败的"差集"自动生成可验证技能

![alt text](image-3.png)

给Agent写Skill是件很吃经验的事。你得知道模型在什么情况下会翻车，翻车的原因是什么，然后写一段指令去堵这个漏洞。写完了还得猜 —— 这段指令真的帮上忙了吗，还是修好了三个case却搞砸了五个原本能做对的？

SKILLGEN干的，就是把这件事自动化。它不靠人工写技能，而是从Agent已有的执行轨迹里，自动对比成功和失败的差异，生成技能，然后跑一轮对照实验来验证这个技能到底是赚了还是亏了。

# 一、核心思路：技能是一种"干预"

SKILLGEN把技能定义为一类特殊的**干预（intervention）** —— 加载技能，等于对Agent做了一次处理。既然是干预，那效果怎么衡量就很清楚了：**同一个任务，有技能和没技能，表现差多少？**

不只看修好了几个。也看搞砸了几个。

用论文里的话说：一个候选技能s的净收益是 **∆(s) = 修复数 − 回归数**。修复（repair）是原本失败的case被技能修好了，回归（regression）是原本能做对的case被技能搞错了。只有净收益为正，才值得部署。

这个视角很朴素，但在之前的自动化技能生成方法里几乎没人这么做。Trace2Skill、SkillX、EvoSkill这些方法都是从成功轨迹里提取经验，不对比失败样本，也不做因果验证。SKILLGEN的出发点就是补上这两个缺位。

# 二、三步走：采集 → 对比 → 生成验证

SKILLGEN是一个多智能体框架，分三个阶段跑。

![alt text](image.png)

## 2.1 Stage 1：基线采集

先让基础Agent在训练集上裸跑一轮，收集所有成功和失败的轨迹。同时缓存验证集上的无技能结果，供后续做对照。

这一步看着简单，但有个关键设计：**成功和失败的轨迹都要保留**。只看成功，你不知道能力缺口在哪；只看失败，你拿不到可复用的正确做法。两边的信息缺一不可。

## 2.2 Stage 2：对比归纳（Contrastive Induction）

归纳Agent把变长的轨迹压缩成一份结构化的诊断报告，包含四块内容：

- **任务摘要（a₀）**：这堆任务在干什么，不是描述某一个实例，而是描述这一"类"任务。
- **失败模式（F）**：对失败轨迹做聚类，每类提炼出根因、典型出错点、以及能避免这个错误的正确操作。
- **成功模式（S）**：对成功轨迹做聚类，提炼出可复用的操作流程和让流程稳定的检查点。
- **对比观察（C）**：对于每一个失败实例，找到嵌入空间里最近的成功实例，逐帧对比两段轨迹——"成功的那次在某个步骤做了什么，失败的那次没做"。

第四块是SKILLGEN最特别的地方。它不只看成功和失败各自的统计规律，它还做逐案例的"差集" —— **同一个基座Agent，在相似情境下，它某一步做了正确的事，另一步没做。** 这个差异信号比单纯的行为模式更有信息量，因为它锚定的是Agent自己已经展示过、但在某些场景下遗漏了的能力。

## 2.3 Stage 3：生成-验证-精炼循环

生成Agent把诊断报告转成候选技能。技能结构是一个三段式prompt：**任务上下文（u_ctx）+ 成功模式（u_succ）+ 失败规避（u_fail）**。对于工具密集型任务，还可以附带可执行脚本和参考文档。

验证Agent拿到候选技能后，在验证集上跑一轮**配对评估**：同一个输入，分别用基础Agent和技能增强Agent执行，拉出一张列联表：

| | 有技能=成功 | 有技能=失败 |
|------|------|------|
| 无技能=成功 | 保持成功 | **回归** ← 坏的 |
| 无技能=失败 | **修复** ← 好的 | 持续失败 |

净收益**Gm = n01 − n10**，就是修复数减去回归数。

验证完，Agent不是简单地用或不用 —— 它会产生结构化的精炼反馈，分四类：**保留（keep）、删除（remove）、新增（add）、强调（emphasize）**。生成Agent据此修改技能，进入下一轮。

最终，**不是取最后一轮的结果，而是从K轮候选中取Gm最大的那个**。这是SKILLGEN一个非常务实的决定。论文的数据很直白：到第8轮，最新候选的期望 ∆ 是 **−3.1 pp**，而最佳候选达到 **+8.1 pp**。差了大约 **11 个百分点**。不选最好的，等于自废武功。

最后一道闸是**验证门（Verification Gate）**：哪怕是最佳候选，如果 Gm 低于阈值（至少 ≥ **1**，且满足绝对和相对下限），技能就被标记为`deprecated`，不部署。宁可什么也不做，也不装一个没被验证过的技能。

# 三、实验结果

论文在8个基座模型（从Gemma-26B 到 GPT-5.4-Mini，开源和闭源都有）、9个benchmark上做了评估。核心结论：

![alt text](image-1.png)

**所有8个模型的平均准确率都提升了，幅度从 +3.27 到 +10.08 个百分点。** 在80组 benchmark-切分-模型 的组合里，50组提升，25组持平，只有5组退步。

**提升最大的场景是需要多步程序性操作的任务**。ALFWorld上Qwen-2.5-7B涨了 **13.33 pp**，ScienceWorld上GPT-5.4-Nano涨了**25.00 pp**。知识密集型任务（如 PubMedQA）提升小一些 —— 技能能教模型怎么做事，但补不了它不知道的事实。

![alt text](image-2.png)

和现有方法的对比：在ALFWorld和ScienceWorld的共同测试上，**SKILLGEN全面优于Trace2Skill、SkillX、EvoSkill和CoEvoSkills**，且是唯一在所有benchmark-model组合上都有正向收益的方法。

消融实验拆开了SKILLGEN的每个组件：
- 去掉精炼（no refinement）：掉
- 去掉验证门（no verification gate）：掉
- 去掉失败分析（no failure lessons）：掉
- 去掉脚本/参考文档（plain-text skill only）：在工具密集型任务上掉
- 换成简单的 3-shot ICL 替代归纳技能：掉

**每个组件都在起作用，完整系统在所有benchmark-model对上都赢了。**

跨模型迁移也很有意思。生成的技能不是只能给原模型用 —— 论文做了完整的 6×6 迁移矩阵：120组跨模型迁移中，**70%非负，42%超过 +5 pp**。一个有意思的发现：最强的技能生成者不一定是基线最强的模型。在ALFWorld上，Qwen-2.5-7B（一个中等规模的模型）是平均迁移效果最好的技能来源。

# 四、几个有意思的设计选择

**为什么做 best-of-K 而不是取最后一轮？**

因为精炼过程是不稳定的。某一轮修好了一些失败，下一轮可能又引入了回归。best-of-K把精炼当搜索而不是当收敛 —— 这更诚实，效果也更好。

**为什么验证门很重要？**

因为生成式Agent产出的候选技能质量方差很大。验证门是一个硬约束：只部署被验证过的技能。τ-Bench上，多个模型生成的候选技能被验证门直接挡掉了，避免了对基线的负向干预。

**为什么成功和失败的对比信号比单看成功更有用？**

因为"成功轨迹里Agent做了什么"和"那件事为什么决定了成败"是两回事。对比失败轨迹，才知道哪些操作不是锦上添花，而是生死线。

# 五、总结

SKILLGEN不是万能的。论文自己列了几个硬伤：

- 知识密集型任务的提升有限（PubMedQA 上大部分模型涨 0-2 pp），因为技能教的是操作流程，补不了模型的领域知识短板。
- 验证门的阈值是手工设的（g_abs，g_rel），论文没有系统性地分析阈值敏感性。
- 归纳阶段依赖嵌入聚类来发现行为模式，聚类质量直接影响诊断报告的有用性。论文用的是文本嵌入 + 标准聚类算法，在行为差异很细微的任务上，聚类可能漏掉重要的对比信号。
- 还有一个实际约束：SKILLGEN需要跑多轮rollout来做验证，token开销不小。论文没有和其他方法做token成本的横向对比。

SKILLGEN给技能自动化定了一个新标准：**技能不是从成功中总结出来的忠告，而是经过对照实验验证的干预。** 对于正在做Agent工程的团队来说，这是一个可以立刻借鉴的思路 —— **在把一段Skills提示词装进Agent之前，至少先跑一轮有技能 vs 无技能的对比测试**。
