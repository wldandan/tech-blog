# SkillJuror：技能怎么写比技能好不好用更微妙，组织方式本身会改变 Agent 行为

> 一篇技能好不好，通常测的是"有没有它"和"用它跑任务能不能过"。但 SkillJuror 问了一个更细的问题：**同样的知识，扁平地写在一个文件里、还是按渐进式展开拆成入口文件加按需加载的参考文件——这两种写法，会不会让 Agent 在执行中表现出不同的行为？** 答案是会。不只是最终通过率不同，是 Agent 在执行过程中触达的资源数量、对资源的使用深度、以及在不同任务类型上的行为模式，都变了。

## 同样的配方，不同的包装

Agent Skills 现在有明确的写法建议了。Anthropic 推荐渐进式展开（Progressive Disclosure）：一个轻量入口文件，把详细的参考文档、模板、辅助脚本拆进独立文件里，Agent 按需打开。乍看这是工程组织问题——跟代码里要不要拆模块一样，属于"最佳实践但不影响功能"。

**但这个假设之前没有被控制实验验证过。** 原因很直接：比较两个天然状态下写出来的技能，无法把"写法差异"和"内容差异"拆开。一个技能效果好，可能是因为写得好，也可能是因为覆盖的知识更多。当知识质量和组织结构同时变化时，归因是不可靠的。

同济、上交、中山和上海创新研究院的联合团队在 **SkillJuror** 里做了一套控制实验。把同一个技能的同一份知识内容，分别写成两种格式：一份扁平 Baseline——所有内容放在一个文件里，Agent 一次性读到全部信息；一份渐进式展开（PD）变体——同样内容拆成轻量入口和多个按需加载的参考文件。**82 个 SkillsBench 任务，每个任务跑 5 轮，总共 1,230 次试验。** 基座模型固定为 GPT-5.4，变动的只有技能的书写结构。

## 渐进式展开改变了 Agent 的执行方式

结果分两层看：行为层和结果层。

**行为层的差异是数量级的。** 在扁平 Baseline 下，Agent 每个轨迹平均触达 1.18 个独立技能资源（参考文件、辅助脚本等）。在 PD 下，这个数字跳到 3.85。有效资源吸收事件——Agent 不仅打开了一个资源而且确实用它做了什么（实现了一处逻辑、验证了一个步骤、修正了一个错误、或基于它做出了有效的障碍诊断）——从 1.33 跳到 3.92。

**PD 让 Agent 更多地打开参考文件，并且这些文件里的信息被更多次地转化为实际可用的操作。** 这不是因为"内容变多了"——两组的知识内容完全相同。也不是因为"Agent 被强制要求去读"——PD 只提供了导航链接，读不读是 Agent 自己的决策。变化来自结构：当信息被分发到多个文件后，Agent 在导航路径上的可选项增多了，而且每个文件内部的认知负荷降低了。

**结果层的差异是任务条件化的。** 在 410 对受控试验中，PD 比 Baseline 多过了 17 次验证器（+4.1%）。这个增益不算大，但它的分布不是均匀的。在那些支持性资源能直接指导实现、检查和修复的任务上，PD 的优势最明显。在那些成功取决于精确输出格式、数值阈值或长链产物生成的任务上，PD 的额外资源暴露反而没有转化为通过率的提升。

## 这不是 pass rate 的故事

SkillJuror 的核心贡献不是报告了一个 +4.1% 的通过率提升。而是指出了：**技能组织方式的效果在两个层面上是分离的。** 行为层面出现了一致的、幅度大的变化——资源触达和数据吸收在 PD 下系统性更高。但结果层面，这些行为变化只在一部分任务上转化为了通过率的提高。

这意味着如果把技能进化只盯在"pass rate"上，会错过一个完整的信号层：**Agent 的执行路径确实变了，即使最终结果不变。** 这对技能评估的方法论有直接影响——只测量最终通过率会让很多过程层面的真实差异被平均化后消失。

论文还提供了一个补充证据：在不使用任何技能（No Skill）的情况下，Agent 在同一任务的执行行为跟有 Baseline 技能的情况不同，但这层差异在纯结果指标中也难以被完全捕捉。

## 三层验证让控制变量可信

SkillJuror 的实验控得相当干净。

**构造控制：** Baseline 和 PD 不是独立从源技能生成的。Baseline 先把源技能中所有功能性的东西压平到一个文件里。PD 再从这个 Baseline 出发，把同样的内容按结构拆分重组。**PD 不是从原始源技能独立写出来的——它是对 Baseline 的一次约束式重组。** 这样做的好处是，如果 Baseline 和 PD 各自从源技能独立生成，措辞、覆盖范围、甚至辅助脚本恢复的差异可能混淆组织方式效应。串行生成链条让"差异来自结构"的归因更牢固。

**验证关卡：** 构造完成的变体通过三层检查才能进入执行评估。确定性门控（程序化检查目录布局、文件路径、资源可达性、行为单元 diff）。基于 Rubric 的语义审计（自动扫描变体是否保留了不变的任务范围、执行约束、输入输出契约）。人工审核覆盖（验证异常被分类处理：源技能固有的、打包过程中引入的、或可修复的结构性问题）。最终还有一个执行前筛选，把语义或规模异常样本直接排除。

**运行时控制：** 每个任务-范式组合在同一个 Harbor 沙箱中、同一套 Agent harness 上、同一组超参数下执行。非技能的执行环境——任务指令、workspace、环境和验证器——在所有条件下完全相同，唯一变化的是暴露给 Agent 的技能产物。

## 任务属性决定了组织方式什么时候有用

SkillJuror 附带了一套任务属性标签：任务的工作流类型和验证类型。在分析中，**当支持性资源可以直接转化为实现指导、验证检查和修复策略时，PD 的效用是正向的。** 当任务的成功被精确输出格式、数值精度或长流水线产物生成锁定时，额外的资源暴露要么不构成瓶颈，要么反而增加了导航成本。

这带来了一个对技能作者有直接指导价值的结论：**不是所有技能都适合用 PD 来写。** 对于那些核心障碍是"不知道怎么做"的任务，把参考知识拆分到按需加载的文件里有助于 Agent 定位关键信息。对于那些核心障碍是"做出来必须精确匹配规范"的任务，扁平文件提供的全局上下文可能更有效。

## 几个问题

**目前的控制实验只测了一种组织方式对照。** SkillJuror 的 Pipeline 支持任何书写范式作为输入，但论文只跑了 Baseline vs PD 这一组。实际上可能还有很多有意义的组织维度——按操作类型分区、按执行阶段分区、混合扁平与渐进——这些都在控制框架的射程内但尚未被实验覆盖。

**有效资源吸收（ERU）的判定目前依赖 LLM-as-Judge。** 论文用 LLM 抽取轨迹事件并判断"这个资源访问是否产生了可观察的操作影响"。虽然有事件类别标注和边界规则，但 LLM Judge 本身对"有效"的定义可能在语义模糊的边界上产生偏差。

**组织方式的效果可能依赖于基座模型的能力。** 论文固定了 GPT-5.4 作为执行模型。不同模型在自主导航文件结构的能力上可能有差异——一个不太擅长探索文件系统的模型在 PD 下可能反而因为导航失败而退化。

## 最后

SkillJuror 做的最有价值的发现不是 PD 比 Baseline 好 4.1%，而是**技能的组织方式本身就是一个独立的实验变量——它可以被隔离、被控制、被测量。** 在这之前，"技能怎么写"被当作一个最佳实践问题，靠直觉和风格指南来回答。SkillJuror 把它变成了一组可归因的实验证据。

这对正在做技能评估的团队来说，意味着需要在 pass rate 之外加入行为层的度量。Agent 有没有去读参考文件、读了之后用没用、在哪些任务上用了有效——这些信息在纯 pass/fail 指标里是不可见的。
