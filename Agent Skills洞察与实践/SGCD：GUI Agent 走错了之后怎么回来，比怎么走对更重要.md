# SGCD：GUI Agent 走错了之后怎么回来，比怎么走对更重要

> 行为克隆教会了 GUI Agent 走专家走过的路。走错一步之后呢？所有的训练数据都沉默了。

## 一个被忽视的监督真空

训练 GUI Agent 的主流方法是行为克隆：收集一批专家轨迹，让 Agent 模仿每一步操作。OSWorld 上这条路效果不错，各种 7B、8B 的专用模型越来越多。

阶跃星辰、北科大、清华和 NTU 的联合团队在 SGCD 这篇论文里指出了一个被跳过去的问题。行为克隆的监督信号只在专家状态分布上强。Agent 在闭环执行中一旦走偏——点错了按钮、跳过了步骤、被弹窗打断了——它进入的状态就掉到了专家轨迹之外。这些策略诱导的离轨状态在训练数据里完全没有覆盖。

这跟监督学习的分布漂移本质上是同一个问题，但 GUI 场景下更麻烦。离轨状态不是随机产生的，而是当前策略系统性偏好的产物。某个 Agent 就是把"另存为"点成"保存"，另一个老是在浏览器设置里找不到对的菜单项。这些错误是模式化的，反复出现。而且，要获取离轨状态下的正确操作序列，你必须在真实环境里把任务真的跑到那个状态，再从那个状态接着做完。光靠拍脑袋想不出来。

论文对失败轨迹做了统计。89.7% 的失败发生在前 40 步内，98.2% 在前 20 步内。一上来走偏几步，后面一路滑向失败。失败模式归了四类：EarlyDone（没做完就宣布完成）、Fixation（反复无效操作不换策略）、HallucinatedAffordance（找不存在的菜单或按钮）、ScopeMisjudgment（选错 UI 功能或对话框，虽然大致理解意图）。这些失败不是偶然的。同一个 Agent 在同一个控件上，会因为同样的原因反复出错。

## SGCD 的思路：让技能当救生员，但只教不陪

SGCD（Skill-Guided Continuation Distillation）是一个迭代自改进框架。逻辑很直：让普通策略先跑几步，跑到它最擅长抵达的"走偏"状态，然后把控制权交给一个有技能辅助的策略，让它从那个状态接着把任务做完。这些成功续跑轨迹被蒸馏回普通策略。**技能只在训练时用，推理时不带。**

分四个阶段。

先是采样。用不带技能提示的 plain policy 在每个训练任务上跑 M 次，按验证器分成成功和失败两组。成功轨迹提供可行的工作流和终点证据，失败轨迹暴露模式化的陷阱、目标混乱和过早终止。

然后是提取技能。用 Gemini-3-Pro 把每项任务的轨迹证据压缩成结构化的"离轨续跑技能"。技能不记录逐步操作，而是抽象成四个字段。Continuation Plan 是从当前状态完成任务的高层策略和备选方案。Critical Targets 是任务真正相关的 UI 元素、文件、对话框。Failure Traps 是具体陷阱，不要点什么、不要搜什么、不要勾选哪个选项。Success Criteria 是可观测的完成检查点，告诉 Agent 什么时候才算真做完了。

这里的设计出发点很有意思。GUI 任务没有唯一正确解。同一个操作目标可以通过不同菜单路径、不同快捷键、不同对话框状态来完成。如果技能只是某个成功路径的动作回放，太脆了——当前状态跟回放轨迹稍有不同就失效。SGCD 把技能做成轨迹抽象而不是轨迹回放：说清楚要达成什么、要避开什么、怎么判断做完了，不规定具体怎么走。就算交接时的 GUI 状态跟任何历史成功轨迹都不完全匹配，技能依然能用。

接着构建离轨续跑。从阶段一里挑出 plain policy 失败但 skill-guided policy 能成功的任务。让 plain policy 从初始状态跑 k 步，k 在 1 到 20 之间扫，到达一个真实的离轨状态。然后切换给 skill-conditioned policy 带着技能提示继续。两个筛选器把关：可执行的验证器必须过，一个 LLM judge 还要查是不是偶然成功、有没有冗余循环、操作安不安全、行为跟任务目标一致不一致。交接后的部分进训练集，交接前的 plain policy 片段只作上下文不作监督——那部分可能含错误决策。

最后混合训练。把原始专家轨迹、验证过的 plain policy 成功轨迹、验证过的续跑轨迹混在一起，用标准 SFT loss 训 plain policy。训练目标和普通 SFT 看起来一样，但数据分布变了——**监督信号被推到了 plain policy 实际会踩到的离轨状态上。**

然后迭代。新 policy 会诱导新的失败分布，之前不可恢复的任务可能变可恢复。循环继续。

## 数据

OSWorld-Verified 上，三个 backbone 的结果很一致。

Qwen3-VL-8B：base 32.7%，SGCD 后 **55.1%（+22.4pp）**。Qwen3-VL-30B-A3B：base 31.9%，SGCD 后 **58.4%（+26.5pp）**。STEP3-VL-10B：base 24.1%，SGCD 后 **53.2%（+29.1pp）**。三个不同家族、不同规模的模型，全部从低 30% 区拉到了 50% 以上。

同 backbone 比，SGCD-8B 的 55.1% 超过了 EvoCUA-8B（46.1%）、GUI-Owl-1.5-8B（52.4%）。SGCD-30B-A3B 的 58.4% 超过了 OpenCUA-72B（45.9%），30B MoE 干掉了 72B 专用模型。

Continuation Success Rate 这个指标更说明问题。专门挑 plain policy 失败的任务，先让它跑 k 步把状态搞砸，然后让评估模型从那个状态接盘——不带任何技能提示——看能不能救回来。SGCD-8B 到 39.2%，SGCD-30B-A3B 到 **50.3%**，接近 Kimi K2.5（56.3%）和 Claude Sonnet 4.6（63.3%）。Qwen3-VL-8B-Instruct 只有 18.9%。SGCD 蒸馏进模型的不仅是"会做任务"，还有**从搞砸的状态里恢复的能力。**

多轮迭代后性能持续爬升。每一轮都有新的可恢复任务和新的离轨状态覆盖。

## 消融

四个技能字段各有独立贡献。无技能 39.9%，加 Continuation Plan 到 41.3%，加 Critical Targets 到 42.7%，加 Failure Traps 到 43.2%，四个全加 45.7%。最大单次跳升来自 Critical Targets（+1.4%）和 Success Criteria（+2.5%）。Success Criteria 贡献最大——很多失败不是因为 Agent 不会操作，是它不知道什么时候该停。

交接步数 k 扫 {1,...,20} 的效果（45.7）好于任一单子区间。不同任务在不同的执行阶段暴露离轨状态，固定 k 会漏。k 越深，留给 skill-guided policy 的剩余步数越少，验证通过率越低，吻合失败集中在早期执行的现象。

还有一个设计选择：训练只监督交接后，交接前只保留上下文不监督。如果把全程都纳入监督，效果反而下降。引向离轨状态的操作本身就含噪，不该被模仿。

## 几个值得想的问题

SGCD 对困难任务的续跑获取效果有限。skill-guided policy 本身也搞不定的任务，就没续跑数据可蒸馏了。这是自改进范式的天花板。

交互开销不小。每轮迭代要在真实环境中重新跑 plain policy 实例化离轨状态。论文也提了，如果有状态缓存存中间状态直接复用，效率会大幅提升。

技能提取靠 Gemini-3-Pro。换成弱模型，抽象质量会不会显著下降？论文没测。

## 我觉得最有意思的

这篇论文最让我印象深刻的不是分数数字——虽然 30% 到 50%+ 的跳跃确实漂亮。是它对失败的分析和对应的设计。

四个失败类别（EarlyDone、Fixation、HallucinatedAffordance、ScopeMisjudgment）不是拍脑袋分的，是从真实失败轨迹里归纳出来的。每个都对应技能模式中的一个字段，不多不少。Continuation Plan 对付 Fixation，Critical Targets 对付 ScopeMisjudgment，Failure Traps 对付 HallucinatedAffordance，Success Criteria 对付 EarlyDone。**技能模式是失败分析的镜像。** 这个对称性说明团队真的理解了问题，不是在套模板。

还有一个更底层的洞见：行为克隆的局限性不在于数据少，在于数据不覆盖策略的真实运行分布。SGCD 本质上是用 skill-guided policy 当监督生成器，把监督信号推送到 plain policy 实际会踩到的状态上。skills 只是可丢弃的脚手架，推理时没有任何额外开销。

如果你在训练 GUI Agent，一个可以试试的方向：收集你的 Agent 的失败轨迹，分析它们属于哪种失败模式，用强模型从这些证据里提取结构化技能，然后让 Agent 从自己搞砸的地方重新出发。
