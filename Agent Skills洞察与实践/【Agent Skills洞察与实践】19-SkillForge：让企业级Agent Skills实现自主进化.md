## 一、背景与痛点

最近几年[大语言模型](https://zhida.zhihu.com/search?content_id=273054855&content_type=Article&match_order=1&q=%E5%A4%A7%E8%AF%AD%E8%A8%80%E6%A8%A1%E5%9E%8B&zhida_source=entity)技术发展很快，**LLM Agent** 已经开始在很多企业场景落地，云技术客服就是其中最典型的一个。和通用场景不同，企业级Agent需要掌握大量垂直领域的专业知识，遵循标准业务流程，还要能正确调用各类内部工具，这些能力被封装为**Agent Skills** —— 一种可移植、可版本管理的标准化能力包，里面包含了工作流、工具调用规则和领域知识。

不过当前企业要落地可用的Agent Skills，还得解决两个头疼的问题：

  1. **冷启动质量差** ：通用Skills生成器不了解具体业务的领域上下文，生成的Skills和实际需求匹配度很低，几乎没法直接放到生产环境用。
  2. **不会自己进化** ：Skills上线之后没有完善的反馈闭环，跑错了也不知道怎么改，时间长了质量就卡在那儿上不去。



2026年发布的**SkillsBench** 基准测试结果也印证了这个问题：人工整理的Skills能把Agent的成功率平均拉高16.2个百分点，但自动生成的Skills几乎带来不了什么效果提升。传统的Skills迭代全靠人工标注、手动更新，效率特别低，根本跟不上业务快速变化的节奏。

## 二、[SkillForge](https://zhida.zhihu.com/search?content_id=273054855&content_type=Article&match_order=1&q=SkillForge&zhida_source=entity)核心架构

针对这些痛点，阿里云团队在2026年[ACM SIGIR](https://zhida.zhihu.com/search?content_id=273054855&content_type=Article&match_order=1&q=ACM+SIGIR&zhida_source=entity)工业会议上提出了**SkillForge** ，业界首个端到端的垂直领域AgentSkills自进化框架，打通了从Skills生成、部署运行到自动优化的完整闭环。

SkillForge的设计思路很明确，围绕两个核心点展开：

  1. **完全扎根业务** ：所有Skills的知识和流程都从企业真实业务数据里挖，不依赖通用大模型的泛化能力，从根源上保证Skills和领域需求的匹配度。
  2. **全程不用人工管** ：整个迭代过程不需要人工介入，从失败分析、根因诊断到Skills重写全流程自动化，实现Skills质量的持续提升。



整个框架采用五阶段流水线设计，保证运行的稳定性和可追溯性：

  1. **初始化** ：**领域情境化Skills生成器** 基于领域数据生成初始版本Skill**Skill_v0** 。
  2. **执行与监控** ：Agent加载Skills处理业务请求，把输出和专家参考回复的差异标记为**不良案例** 。
  3. **一阶段：多维度故障分析** ，**故障分析器** 从知识、工具、澄清、风格四个维度并行处理不良案例，生成结构化故障记录。
  4. **二阶段：聚合** ，按类别聚合单条故障记录，识别系统性问题并选取典型案例。
  5. **三阶段：诊断** ，**Skills诊断器** 分析聚合后的故障数据，把故障映射到Skills的具体缺陷位置，输出诊断报告和优化方案。
  6. **四阶段：优化** ，**Skills优化器** 通过安全虚拟文件系统执行优化方案，生成新版本Skill Skill_v_n+1。



为了适配企业生产环境的安全要求，SkillForge特意做了约束型的Skills结构设计：剔除了可执行脚本目录，所有操作都只能通过预定义、已经过验证的系统工具完成，所有Agent都通过虚拟文件系统（VFS）和Skills资源交互，从设计上就避免了安全和稳定性风险。

## 三、核心技术模块详解

### 3.1 领域上下文感知的Skills生成器

这个模块主要解决Skills冷启动质量差的问题，考虑到企业数据隐私要求，整个过程都采用内部大模型（[Qwen3-Max](https://zhida.zhihu.com/search?content_id=273054855&content_type=Article&match_order=1&q=Qwen3-Max&zhida_source=entity)）处理私有数据，完全不依赖外部能力：

  * **输入** ：任务描述、历史工单数据集、企业内部技术文档。
  * **处理流程** ： 


  1. **工作流挖掘** ：从历史工单对话和操作序列中提取专家常用的解决方案模式，比如常见的”澄清问题→诊断原因→给出方案→确认解决”的标准流程。
  2. **工具挖掘** ：识别历史工单中人工专家高频使用的内部工具，提取工具的调用参数和适用场景，生成标准工具schema。
  3. **知识抽取** ：从内部知识库和历史工单引用的文档里抽取领域专属知识，比如产品规则、错误码含义、配置约束等。
  4. **Skills合成** ：把挖掘得到的工作流、工具和知识填充到预定义的云服务Skills模板中，生成完整的初始Skills包。



生成的Skills包做了安全处理，去掉了执行任意脚本的权限，所有操作都只能通过预定义的系统工具完成，避免了生产环境的安全风险。

### 3.2 智能体执行与性能监控

**智能体执行**

智能体执行流水线包含四个标准化步骤，实现Skills的按需调用与任务执行：

  1. 场景匹配：系统为当前任务场景自动匹配最适配的Skills版本；
  2. 动态加载：将Skills对应的SKILL.md文件与关联参考资料加载到当前上下文，该过程将Skills视为独立的元工具，不会对全局系统提示词造成污染；
  3. 任务执行：智能体基于加载的Skills指令与可用的系统工具执行当前任务；
  4. 结果返回：输出智能体生成的任务处理回复。



整套执行流程通过双消息机制实现：系统会在请求中注入可见的用户元数据，同时隐藏内部API调用指令，可在运行时动态调整工具调用权限与底层模型选择，无需修改Skills本身的内容。

**性能评估**

智能体加载Skills完成任务处理后，最终输出的回复会通过大语言模型评判器与历史工单中的专家参考回复进行一致性对比：如果一致性低于设定阈值则判定为执行失败，自动触发不良案例分析流水线，进入后续的故障诊断与Skills优化流程。

性能评估的数据来源覆盖两类场景：一是历史工单的离线回放测试，二是线上运行过程中的失败交互（包括一致性判定不通过、人工坐席驳回的智能体回复等）。

### 3.3 多维度失败分析框架

Skills部署上线后，LLM评审模块会把Agent的输出和历史工单中的专家参考回复做对比，识别出不一致的失败案例。这些案例会进入多维度分析框架，从四个维度并行诊断问题：

  * **知识维度** ：检查是不是存在领域知识缺失、错误或者矛盾的问题。
  * **工具维度** ：评估工具调用有没有遗漏、参数错误、或者对返回结果理解错误的情况。
  * **澄清维度** ：判断Agent有没有过度询问、问得不够，或者问了无关信息的问题。
  * **风格维度** ：检查回复语气是不是太机械、太冗长，或者不符合客户服务的要求。



四个维度的分析结果会自动聚合，确定主要故障类别和严重等级，生成结构化的JSON故障记录，给后面的根因诊断提供明确依据。

### 3.4 端到端自进化闭环

失败原因分析完之后，整个框架会自动完成从诊断到优化的全流程，全程不需要人工参与：

  1. 根因诊断 
     1. **聚合分析** ：按类别聚合所有故障记录，统计严重等级分布，选取出最能体现系统性问题的典型案例，避免头疼医头脚疼医脚的零散修改；
     2. **根因诊断** ：基于[ReAct](https://zhida.zhihu.com/search?content_id=273054855&content_type=Article&match_order=1&q=ReAct&zhida_source=entity)的Skills诊断器读取聚合报告和当前Skills内容，把故障模式映射到Skills的具体片段，比如发现是知识缺失的问题，就定位到SKILL.md里故障排查模块的缺口，输出结构化诊断报告和机器可解析的优化方案。
  2. **自动优化** ：Skills优化器按照优化方案修改SKILL.md或参考文件，遵循最小修改原则——只改必须修改的内容，保留原有正确的逻辑，修改后的内容提交为新版本Skills，保证完整的可追溯性。



这个循环会一直迭代运行，每跑一轮Skills的质量都会往上提一点。

## 四、实验效果验证

团队在阿里云五个真实的云技术支持场景（账号、域名、DNS、OSS、ECS）上做了全面测试，用的数据集包含1883个真实工单、3737个处理任务。

### 冷启动效果对比

用领域情境化Skills生成器做出来的初始Skills，和只配备了工具schema的通用Skills生成器比，**Strict一致性率平均提升4.3个百分点，Lenient一致性率平均提升3.6个百分点** ，其中在ECS场景下严格一致性率提升更是达到了7.2个百分点，确实证明了**领域专属工作流知识与知识抽取，比单纯工具更具价值** 。

### 自进化效果验证

不管初始Skills是人工写的、领域生成的还是通用生成的，跑了三轮自进化迭代之后，质量都有明显提升：

  * 人工编写的Skills：Strict一致性率涨了10.99个百分点，Lenient一致性率涨了12.21个百分点；
  * 领域生成的Skills：Strict一致性率涨了9.23个百分点，Lenient一致性率涨了8个百分点；
  * 通用生成的Skills：Strict一致性率涨了11.6个百分点，Lenient一致性率涨了4.9个百分点；



结果说明这个自进化闭环通用性很强，**不管初始Skills质量怎么样，都能通过持续迭代得到明显提升，甚至能超过人工编写的Skills水平** 。**最终版本的Skills和传统人工维护** 的决策树系统比，Strict一致性率**高了13.76个百分点** 。

### 不同故障类型优化效果

不同类型的问题优化幅度不太一样：工具和风格类的问题改善最明显，分别降了14.5%和16.4%；交互类的问题降了13.1%；知识类的问题第一轮迭代降得比较多，后面就基本稳定了，这主要是受底层知识库覆盖范围的限制。

## 五、实际价值和后续方向

SkillForge给企业级Agent的规模化落地提供了一套可行的方案，实际价值很明显：

  1. **不用从零开始搭** ：不需要大量人工标注，也不用专家花大量时间写Skills，从历史数据里就能自动生成质量不错的初始Skills。
  2. **省了维护成本** ：全自动化的自进化闭环，Skills质量会随着业务运行自己不断提高，不用人工花时间维护。
  3. **符合生产要求** ：无脚本的Skills设计加上沙箱运行环境，满足企业生产环境的安全合规要求。



后面这个框架还有很多**可以扩展** 的地方：

  * 可以支持更多企业场景，比如IT运维、内部服务台、电商客服这些。
  * 可以加更多维度的失败分析能力，比如用户情绪识别、多轮对话上下文理解等等。
  * 可以做跨领域的Skills迁移，把一个场景下验证有效的能力迁移到其他相似场景。
  * 可以和知识库的更新流程打通，实现Skills知识的自动同步。



## 六、总结

SkillForge作为首个端到端的垂直领域Agent Skills自进化框架，很好地解决了企业Agent Skills冷启动质量差和迭代效率低的两大核心痛点。实验结果充分证明，通过领域情境化的Skills生成和自动化的自进化闭环，**Skills质量可以持续提升，甚至可以超越人工整理的专家知识水平** ，给大模型Agent在企业场景的规模化落地提供了一条经过实际验证的可行技术路线。
