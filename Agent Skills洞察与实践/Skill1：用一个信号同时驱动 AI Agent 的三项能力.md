# Skill1：用一个信号同时驱动 AI Agent 的三项能力

> 论文：*Skill1: Unified Evolution of Skill-Augmented Agents via Reinforcement Learning*
> 作者：石耀瑞、陈宇昕 等（中科大 & 美团）| 代码：[github.com/AlphaLab-USTC/Skill1](https://github.com/AlphaLab-USTC/Skill1)

---

## 一、问题：Agent 缺一个能持续进化的"技能库"

用强化学习（RL）训练大语言模型（LLM）Agent，这两年进展很快。但有一个问题一直没解决：**Agent 在一次任务中学到的成功策略，只能隐式地融进模型参数里，下一次遇到类似任务，没法直接拿出来用。**

很自然的想法是给 Agent 配一个**外部技能库**（Skill Library）——把过去的成功经验存成可检索、可复用的"技能"。这不算新概念，但把这件事做完整需要同时搞定三件事：

| 阶段 | 做什么 |
|------|--------|
| **技能选择**（Selection） | 面对新任务，从库里找出最相关的那个技能 |
| **技能利用**（Utilization） | 拿着选出的技能，去环境里完成多步交互 |
| **技能蒸馏**（Distillation） | 任务做完后，把这次经验提炼成新技能存回库中 |

三个环节构成闭环。但现有方法（SkillRL、RetroAgent、EvolveR 等）的做法大多是**只优化其中一两个环节，其余冻结或用规则处理**。即便有些方法优化了多个环节，每个环节用的也是**不同来源的奖励信号**——利用阶段看任务成败，蒸馏阶段靠自评分数。信号源不一致，优化方向就会打架。

这篇论文的解法很直接：**三个环节全用同一个任务结果信号来驱动。** 本文提出的框架叫 Skill1。

## 二、Skill1 怎么做的

Skill1 把选择、利用、蒸馏全部塞进一个策略模型 $\pi_\theta$，每来一个任务 $x$，按顺序跑三个阶段：

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Stage 1    │────▶│  Stage 2    │────▶│  Stage 3    │
│  技能选择    │     │  技能利用    │     │  技能蒸馏    │
└─────────────┘     └─────────────┘     └─────────────┘
      │                    │                    │
      ▼                    ▼                    ▼
  生成查询 q          条件于技能 z         从轨迹 τ 中提炼
  检索候选，重排序      与环境交互           新技能，更新库
```

### Stage 1：技能选择——不只是"搜一下"

选择过程拆成三步，而不是简单的 top-1 相似度匹配：

1. **生成查询**：$\pi_\theta$ 根据任务 $x$ 写出一个自然语言查询 $q \sim \pi_\theta(\cdot | x)$
2. **语义检索**：用一个冻结的编码器 $E$（all-MiniLM-L6-v2）算相似度，取 Top-K：

$$B_K = \text{top-K}_{s \in B} \ \text{sim}\big(E(q), E(s.\text{desc})\big)$$

3. **重排序**：$\pi_\theta$ 对 $K$ 个候选重新排列 $\sigma \sim \pi_\theta(\cdot | x, B_K)$，选排第一的 $z$ 进入利用阶段

查询生成和重排序都是 $\pi_\theta$ 产生的——**选择过程本身就可以被梯度优化**，而不是一个冻结的检索器。

### Stage 2：技能利用——拿技能当"使用说明"

选了技能 $z$ 之后，Agent 把 $z$ 的策略描述（一段自然语言）作为条件，与环境交互最多 $T$ 步：

$$\tau \sim \pi_\theta(\cdot | x, z.\text{strat}, o_{\leq t})$$

交互结束，环境返回一个二元奖励 $r(\tau) \in \{0, 1\}$。**整个系统的所有优化信号，都从这一个值来。**

### Stage 3：技能蒸馏——把经验"压缩"成技能

每次 rollout 结束，$\pi_\theta$ 从轨迹中提炼两条信息：
- $s_{\text{new}}.\text{strat}$：这次用到的可复用方法
- $s_{\text{new}}.\text{desc}$：什么场景下这个技能适用

只有 $r(\tau) = 1$（任务成功）时，新技能才入库。库满时（上限 5000 条），按 $U(s) \cdot \log n(s)$ 的分数淘汰末尾技能——**长期好用且使用频繁的技能留下，低效用又没人用的淘汰。**

## 三、关键设计：一个信号，拆出三份"功劳"

这是整个方法最有意思的地方。系统只有一个外部奖励 $r(\tau)$，但三项能力处在不同的时间尺度上：

- **利用**关心当前 episode 的表现
- **选择**关心哪些技能**长期**有效
- **蒸馏**关心新经验是否**超过了**库的现有水平

Skill1 的做法：**把 $r(\tau)$ 拆成低频趋势和高频变化两个分量。**

### 利用的信用

最直接——任务结果就是利用的反馈：

$$R^{\text{util}}_i = r(\tau_i)$$

查询 $q$ 作为轨迹前缀，也通过策略梯度跟着一起优化。

### 选择的信用：低频趋势 → 技能效用分

每个技能 $s$ 维护一个指数移动平均（EMA）效用分，每次 rollout 后更新所有被检索到的候选技能：

$$U(s) \leftarrow (1 - \alpha) \cdot U(s) + \alpha \cdot r(\tau_i), \quad \forall s \in B_K^i$$

$\alpha = 0.05$，平滑掉单次结果的随机波动，反映的是技能的**长期贡献**。然后用 **NDCG**（归一化折损累积增益）来衡量重排序的质量：

$$R^{\text{rerank}}_i = \text{NDCG}\big(\sigma_i, \ \text{argsort}(-U(B_K^i))\big)$$

排得越接近真实效用顺序，奖励越高。

### 蒸馏的信用：高频变化 → 超越已有水平多少

设 $\hat{U}_i = \max_{s \in B_K^i} U(s)$ 为当前候选技能中的最高效用——代表库能提供的最好方案。蒸馏奖励简单做差：

$$R^{\text{distill}}_i = r(\tau_i) - \hat{U}_i$$

- $R^{\text{distill}} > 0$：这次经验比库里的都好，提炼出的技能有增量价值，正梯度
- $R^{\text{distill}} < 0$：不如已有的，属于冗余技能，负梯度

**这个设计逼着策略去探索库还没覆盖到的更好策略**，而不是反复制造相似技能。

### 联合优化

三项损失加权求和，一次梯度更新搞定：

$$\mathcal{J}(\theta) = \mathcal{J}^{\text{util}}(\theta) + \lambda_1 \mathcal{J}^{\text{rerank}}(\theta) + \lambda_2 \mathcal{J}^{\text{distill}}(\theta)$$

$\lambda_1 = \lambda_2 = 0.3$。利用和蒸馏用 GRPO 优化，重排序用 REINFORCE（因为不同 rollout 的候选集不同，组内对比没意义）。

### 整个流程的伪代码

```
对于每批 N 个任务，每个任务 G 个 rollout：
    对每个样本 i：
        q_i ← π_θ(x_i)                              # 生成查询
        B_K^i ← Top-K 语义检索
        σ_i ← π_θ(x_i, B_K^i)                       # 重排序
        z_i ← B_K^i[σ_i(1)]                         # 选最佳技能
        τ_i ∼ π_θ(·| x_i, z_i.strat)                # 条件化交互
        s_new,i ← π_θ(x_i, τ_i)                     # 蒸馏新技能
        
    R_util_i    ← r(τ_i)
    Û_i         ← max_{s∈B_K^i} U(s)
    R_distill_i ← r(τ_i) − Û_i                      # 变化 → 蒸馏信用
    R_rerank_i  ← NDCG(σ_i, argsort(−U(B_K^i)))     # 趋势 → 选择信用
    
    U(s) ← (1−α)·U(s) + α·r(τ_i), ∀s∈B_K^i         # EMA 更新效用分
    若 r(τ_i)=1，将 s_new,i 加入技能库
    θ ← θ + ∇_θ[J_util + λ₁J_rerank + λ₂J_distill]
```

## 四、实验结果

### 主结果

| 方法 | ALFWorld 成功率 | WebShop Score | WebShop Success |
|------|:--------------:|:-------------:|:---------------:|
| ReAct（无训练） | 31.2% | 46.2 | 19.5% |
| Reflexion（无训练）| 42.7% | 58.1 | 28.8% |
| PPO（RL，无技能）| 80.4% | 81.4 | 68.7% |
| GiGPO（RL，无技能）| 90.8% | 84.4 | 72.8% |
| SkillRL（RL+技能）| 89.9% | 85.2 | 72.7% |
| RetroAgent（RL+技能）| 94.9% | 88.9 | 82.3% |
| **Skill1** | **97.5%** | **89.7** | **82.9%** |

ALFWorld 上比此前最好的 RetroAgent 高出 **2.6 个百分点**，6 个子任务里 5 个排第一。WebShop 上同样最优。

额外值得一提的是 GiGPO，这是纯 RL（无技能库）中最强的，拿了 90.8%。Skill1 比它高了 6.5 个百分点。差距最大的子任务集中在 Look 和 Pick2——**恰好是需要组合多个子步骤的任务，最能体现可复用技能的价值。**

### 消融实验

| 变体 | ALFWorld 成功率 | 跌幅 |
|------|:--------------:|:----:|
| 完整 Skill1 | **97.5%** | — |
| 移除技能库 | 80.9% | −16.6 |
| 移除蒸馏阶段 | 92.4% | −5.1 |
| 移除选择阶段 | 91.8% | −5.7 |
| 仅关闭选择信号（λ₁=0）| 94.0% | −3.5 |
| 仅关闭蒸馏信号（λ₂=0）| 94.9% | −2.6 |
| 两个辅助信号都关掉 | 90.2% | −7.3 |

两个值得注意的点：

1. **技能库是地基**——拆了之后 Heat 和 Pick2 各自跌了超过 28 个点，基本退回到纯 RL 水平
2. **关掉选择或蒸馏信号，利用能力也跟着跌**。利用本身还有直接的任务奖励，但依然被拖累了。三项能力之间确实存在正向反馈

### 训练曲线说明了什么

论文追踪了三条曲线：选择精度（技能效用分）、利用成功率、蒸馏正向率（新 rollout 超过库中最佳的比例）。

完整 Skill1 的选择精度最先收敛，大约第 20 步就到 0.95。有了高质量技能供给之后，利用和蒸馏也跟着加速，第 60 步左右都到了 0.8。**关掉任何一个信号，三条曲线全变慢**——协同进化不是一句口号。

### 库的多样性没有坍缩

t-SNE 可视化对比了有/无辅助信号时技能库的分布。完整 Skill1 的高频技能覆盖了明显更广的策略空间，且分布更均匀。变化驱动的蒸馏在起作用：**生成跟已有技能差不多的东西会得负分，策略被迫去覆盖还没被填上的场景。**

## 五、两个具体案例

论文附录里给了 ALFWorld 上的两个例子，直观地展示了技能库怎么帮 Agent 少走弯路。

### 案例 1：避开"坏炉子"

任务："加热盘子放进柜子"。房里同时有炉灶和微波炉，但炉灶对盘子不 work——任务描述里不会告诉你这件事。

Skill1 检索到的技能（效用 0.951，被用过 593 次）里明确写了"炉灶加热盘子失败过，用微波炉"。Agent 看到这段直接跳过炉灶，**6 步完成最优路径**。

### 案例 2：找对闹钟

任务："用台灯查看闹钟"。房里不止一个闹钟，选错对象任务就失败。

检索到的技能记录了"必须先去台灯下找到闹钟，再用台灯"。Agent 第一步就精准定位，**4 步搞定**。

## 六、局限性

论文自己列了几个问题，坦率程度值得肯定：

- **环境覆盖窄**：只在 ALFWorld 和 WebShop 两个纯文本环境上测过。视觉输入、深度搜索等更复杂场景能不能迁移，没验证
- **库容量瓶颈**：5000 条上限在当前任务上够用，任务种类再多就悬了。淘汰策略和层级化组织还有改进空间
- **计算开销**：比不带技能库的 GRPO 慢 1.3~1.7 倍，主要是库上下文越来越长。蒸馏起了压缩作用——不开蒸馏的话，原始轨迹直接入库，库膨胀速度是 Skill1 的 2.4 倍

---

## 七、总结

Skill1 做了一件事，但做得很干净：**把一个任务结果信号按时间尺度拆开，低频的归选择，高频的归蒸馏，原始的归利用。** 没有额外的 reward model，没有 auxiliary loss，没有 teacher model。就是对着 $r(\tau)$ 做了一次分解。

结果上，**ALFWorld 97.5%，WebShop 双项最优**。消融实验把三条路都走了一遍，每一项删减都导致全局退化。

如果你在做 LLM Agent + RL，这篇论文值得花时间读。它回答了一个实际的问题：**别急着给每项能力单独设计奖励函数——先想清楚，你手里已经有的那个信号，是不是比你以为的包含了更多信息。**
