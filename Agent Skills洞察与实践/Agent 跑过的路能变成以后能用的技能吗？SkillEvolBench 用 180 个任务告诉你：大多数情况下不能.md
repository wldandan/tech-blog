# Agent 跑过的路能变成以后能用的技能吗？SkillEvolBench 用 180 个任务告诉你：大多数情况下不能

Agent 每天在跑任务。跑一条轨迹，留下一段经验。这段经验里藏着什么东西能用在下一次。可能是某个 API 的正确调用顺序，可能是一个边界 case 的处理方式，可能是一个工具组合的最佳实践。

**但如果把"这一段经验"变成"一份以后可以加载、可以复用的技能文档"，Agent 能不能做到？**

SkillsBench 已经给出了一个不太乐观的答案：**零样本生成的技能装上之后平均没用。** 但 SkillsBench 测的是"动手前先写一份技能"。Agent 在开始跑任务之前，凭对任务的理解先写一份操作指南。这不是最自然的技能形成方式。**真正的技能形成是：先跑一遍，跑完之后回头看，从这次经验里提炼出能用于下一次的东西。**

OSU、芝加哥大学、UCL、密歇根大学、港中文和 Amazon 联合推出的 **SkillEvolBench** 就是来测这件事的。180 个任务，六个真实 Agent 环境，十个模型配置，三个 Agent 平台。核心问题只有一个：**Agent 能不能把一次性的执行经验，变成可复用的程序知识？**

答案很扎心：**大多数情况下不能。而且直接复用原始轨迹，经常比蒸馏出来的技能更管用。**

---

## 一、SkillEvolBench 怎么测

SkillEvolBench 的设计围绕一条完整的技能进化弧线。六个环境（代码修改、API 编排、数据处理、文档转换、研究合成、沟通操作），每个环境五个任务家族，每个家族六个角色任务。

**三个学习任务：** 先给一个标准任务让 Agent 跑，采集轨迹和验证反馈；然后一个"升级版"暴露标准任务中缺失的子能力；再来一个"变体版"改变表面形式但保持相同的底层逻辑。每一步跑完，Agent 都可以选择写新技能、改已有技能、或者不改。

**三个冻结评估任务：** 上下文漂移（技能需要被正确识别和调用，但任务描述变了）、对抗性捷径（存在绕开正确流程的浅层方案）、组合（需要被评估的技能跟其他技能协同工作）。**评估时技能库完全冻结，Agent 能用但不能改。** 所以评估分数反映的完全是"之前学到的东西能不能迁移"，而不是"现场再学一遍"。

四个控制条件：**裸跑**（没有任何技能）、**零样本技能**（任务前生成，不更新）、**自主生成**（从空技能库开始，从经验中自我归纳）、**精炼种子**（从一个有盲区的 human-curated 技能开始，通过经验来修补）。

## 二、实验说了什么：学到的东西出不了原任务

十个模型配置，三个 Agent 框架（Claude Code、Codex CLI、Gemini CLI），跑完全部 180 个任务。

### 技能在练习任务上能涨分，在未见任务上涨不了

**Claude Opus 4.6 + Curated-Start with Revision：练习任务成功率从 38.9% 涨到 44.4%（+5.5%），回放成功率涨到 56.7%（+17.8%）。** Agent 确实从经验中学到了东西，而且能把这些东西用在回头重跑原来的任务上。

**但切到冻结部署任务（上下文漂移、对抗、组合），分数没有统计上一致的提升。** CSSR 在 Opus 4.6 上是 +1.1%，ARSR 是 -6.7%，CompSR 是 +3.3%。这些涨幅太分散、太不一致，无法说明技能在真正需要它的地方帮上了忙。

### 精炼种子比凭空生成好，但也好不到哪去

**从 curated seed 出发 + 允许修订，比从零开始自主生成好不少。** GPT-5.4 在 curated-start + revision 条件下，四个部署子指标都比 no-skill baseline 有明显提升。但 Opus 4.5 在同样条件下，LSR 反而比 no-skill 还低 3.3 个百分点。

**增益强依赖于具体模型和具体条件，没有一种设置在全部配置下稳定有效。** 同样的 curated-start + revision，GPT-5.4 在 ESR 上 +5.6 分，GPT-5.3-Codex 在 ESR 上是 0.0。

### 最扎心的发现：直接用原始轨迹比用技能更管用

论文把 raw-trajectory reuse（直接把之前任务的经验轨迹作为上下文注入后续任务）跟 skill-based 条件做了对比。

**在部署任务上，原始轨迹复用经常超过蒸馏出来的技能。** 蒸馏的过程中丢失了上下文信息和程序线索。这些在技能生成时被"压缩"掉了，但在面对未见任务时恰好是最需要的东西。

**技能生成的本质是一次有损压缩。** 你从一段丰富的轨迹中抽取"最精华"的部分写成一份整洁的技能文档。但你丢掉了那些看起来"不重要"的细节：具体的错误信息、尝试过的失败路径、环境状态的变化轨迹。而这些细节，恰好是面对一个稍有变化的新任务时最需要的东西。

## 三、诊断分析：问题不在写得多，在于提炼得不准

论文还做了两个诊断实验来定位瓶颈。

### 写更多技能不能解决问题

一些模型在"always-update"条件下（每次任务跑完都触发技能更新，不管是否需要），技能库膨胀得更快。但成功率没有跟着涨。**更多的技能更新带来了更高的覆盖率，但也引入了话题漂移和程序噪声。** 每一次更新都在偏离最初学到的结构，最后系统陷入"改得越多，越不准确"的循环。

### 更强的模型≠更好的技能形成

十个模型配置跑下来，没有谁能稳定战胜所有其他配置。Gemini 2.5 Pro 是表现最差的之一。不是因为它完成任务的能力弱，而是**它在"从经验中提炼技能"这一步衰减得最厉害。** 更大的模型、更好的推理能力，跟"能不能把经验变成技能"之间没有简单的线性关系。

---

## 四、这篇论文在说什么

SkillEvolBench 做的事情很简单，但触及的问题很深。它把技能形成的整个链条——从经验采集到撰写到冻结部署——做成了一套标准化的测试协议。**不是测"技能有没有用"，不是测"Agent 能不能完成任务"，是测"Agent 从一次经验到一份可复用技能之间的这道抽象鸿沟，到底能不能跨过去"。**

当前的答案是：**能跨，但跨不稳。** Agent 能从经验中提炼出一些东西，这些东西在原任务上确实有帮助。但当任务稍微变化（上下文不一样了、陷阱出现了、需要跟其他技能协同了），这些提炼出来的东西就不够用了。

**如果你在做技能生成或技能管理，这篇论文给了一个非常具体的行动建议：别只看技能在原始任务上的回放效果。** 那是在测"记住了吗"，不是在测"学懂了吗"。真正的测试是把技能冻结，扔到它没见过但有相同底层逻辑的任务上去。如果帮不上忙，那这份技能不是"会了"，是"背了"。
