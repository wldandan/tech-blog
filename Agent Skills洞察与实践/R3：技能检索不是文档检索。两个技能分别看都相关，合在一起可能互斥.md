# R3：技能检索不是文档检索。两个技能分别看都相关，合在一起可能互斥

> 把 21.2M token 的技能库全塞进 prompt 让模型自己挑？主流推理服务的单次调用上下文窗口根本装不下。技能检索是 Agent 的前置环节，但它一直被当作文档检索来做，评估每个技能跟查询是否相关就够了。R3-Skill 提了一个反例：**两个技能分别看都跟查询高度相关，但合在一起不能协作。** 文档检索没有这个概念，技能检索必须有。

## 技能检索跟文档检索，差在哪

文档检索默认一个前提：两个文档各自跟查询相关，同时返回就没有问题。这个前提在技能空间不成立。

技能不是被"读"的，是被"组合调用"的。你给 Agent 返回一组技能，Agent 会把它们都激活，按流程串起来执行。如果返回的集合里有两个技能功能重叠、生态不兼容、或者一起用会产生冲突，Agent 的行为会直接受影响。**返回技能集合的质量不取决于单个技能的相关性分数之和，而取决于这组技能能否在一起协作。**

举个实际例子。A 是 Terraform IaC 技能，B 是 AUR Package Publish 技能。

- 查询 "打包一个内部工具发布到 Linux 软件仓库并配一套 IaC 管线"：A 和 B 应该同时返回。
- 查询 "给团队配一套基础测试基础设施"：只需要 A，B 跟这个意图不沾边。

同一对技能，同一个相关性分数，换了一个查询，合在一起该不该返回答案就变了。**技能兼容性不是两个技能之间的二元全局关系，而是查询条件化的局部属性。** 文档检索可以假设"兼容性因子 C ≡ 1"，技能检索不行。

腾讯 IMA 产品中心和优图实验室的联合团队在 **R3（Reject-as-Resource Retriever）** 里做了三件事：把"两个技能能否合在一起用"变成可训练的信号，建了一个包含 32,828 条兼容性标注的中英双语技能检索基准，以及训练一套在召回和重排两个阶段各司其职的两阶段检索系统。

## 被扔掉的数据是最有价值的信号

现有技能检索数据集的做法是让 LLM 判断一组技能能否自然组合。能，写一条查询，收入训练集。不能？**直接丢弃。** 但丢弃这部分恰是兼容性监督信号最密集的地方。"看起来像但实际不能一起用"的样本比"完全无关所以不能一起用"的样本有价值得多。

R3-Skill 把 LLM 的一次判断分成两根分支：WRITE 和 SKIP，两根分支平行保留。WRITE 分支产出查询和对应的目标技能集。**SKIP 分支保留技能组合和拒绝原因，构成兼容性的负监督信号。** 数据集一共保留 41,592 条 WRITE 查询和 32,828 条 SKIP 标注。

光是"保留被拒绝样本"还不够。随机抽两对技能大概率完全不相关，这些 SKIP 没有训练价值。**R3-Skill 做了一个近邻硬约束采样：用 BGE-M3 做嵌入聚类，同一个子簇内抽技能配对。** 聚类参数 K = 40 时合并成 8 个超级领域。k = 2 的技能必须在同一个子簇内，k = 3 必须在同一个超级领域内。这种约束下，LLM 只判断 56.3% 的 2 技能组合和 36.5% 的 3 技能组合为"可共同执行"。**超过六成的近邻三技能组合是不兼容的。** 接近但不可合，这才是兼容性信号的有效样本。

SKIP 标注还被进一步细化成 8 类拒绝原因：功能重复、API 格式冲突、管理平台互斥、生态不兼容、风格不一致、执行前提冲突、人设错位、强制堆积。中英方向的拒绝模式结构性地不同。英文方向以领域错配为主，中文方向以"强制堆积"的判定为主。跨语言方向的 k = 3 SKIP 率达到 80.9%，在跨语场景下 LLM 更难接受技能组合。

测试集查询也做了处理。不是直接用 LLM 生成的结构化查询，而是经过改写贴近真实用户请求的语言风格，避免检索器靠关键词匹配刷分。

## 两阶段：召回只管相关性，重排再管兼容性

R3 是一个两阶段检索系统。R3-Embedding（bi-encoder）做粗召回，R3-Reranker（cross-encoder）做精排。

**兼容性信号放在哪一层？** 论文做了一个梯度层面的分析。在 bi-encoder 里，每个技能的表示是同时被多个查询联合决定的。兼容性信号注入的"推远"梯度（两个不兼容的技能应该被推离彼此），在这种双边平衡机制下被高度稀释，对召回阶段的贡献微乎其微。**所以 R3-Embedding 只用标准 InfoNCE 训练，不做兼容性约束。**

在 cross-encoder 里，每一对查询-技能是独立打分的，不存在表示层面的互相干扰。**兼容性信号在这里可以作为排序的梯度监督直接使用。** R3-Reranker 用 graded ListNet 目标，把 SKIP 搭档当作有序负标签引入重排。比完全无关的候选差，比目标技能肯定更差，但具体差多少由它在候选列表中的位置反映。

消融实验证实了这个设计：在 bi-encoder 阶段塞兼容性信号几乎不涨点，在 cross-encoder 阶段塞就是涨。

整个 R3-Embedding + R3-Reranker 管线在 R3-Skill 上达到 **Hit@1 = 0.7714，NDCG@10 = 0.8327，Set-Compat = 0.3525。**

## 数据和模型都开源了

R3-Skill 数据集的构造质量经过了多层检查：训练集和测试集使用不同的底层 LLM 生成（避免共享生成器偏差），全量多技能 ground-truth 的意图召回经人工复核，25% 的查询经人工抽查自然度，多专家独立标注并交叉验证，技能池层面切分确保测试技能不出现在训练中。

数据集一共 10,246 个技能（中英双语），41,592 条已接受查询，32,828 条 SKIP 标注，覆盖四个语言方向（en→en / en→zh / zh→en / zh→zh）和六种查询风格。测试集查询靠近真实用户的表达方式。

训练代码、模型权重和数据集全部开源。

## 最后

R3 的核心主张是：**技能检索和文档检索的根本区别在于兼容性。** 文档检索不需要想"两个相关文档一起返回有没有冲突"，技能检索必须想。

这份兼容性信号其实一直存在于数据管线里。LLM 每拒绝一个技能组合，就在说"它们不能一起用"。R3 的做法是把这些被丢掉的数据捡回来，放进一个两阶段检索系统的正确位置。召回层不管兼容性，重排层用它做梯度监督。

对正在做 Agent 技能路由的团队来说，R3-Skill 是目前唯一包含兼容性标注的中英双语基准，也是唯一把 LLM 拒绝信号系统化保留和利用的数据集。
