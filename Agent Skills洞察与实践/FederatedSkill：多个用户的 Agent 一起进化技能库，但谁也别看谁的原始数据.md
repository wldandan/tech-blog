# FederatedSkill：多个用户的 Agent 一起进化技能库，但谁也别看谁的原始数据

> 一个用户的任务流只能覆盖技能空间的一小片。Agent 碰到新任务时还没攒够经验，反复试也缺少成功的参考。把多个用户的数据合在一起显然能破这个困局，但如果靠上传原始轨迹来做，截图、表格、私人对话全暴露了。FederatedSkill 的做法是：**每条轨迹在本地完成反思和提炼，上传的只有语义技能 patch，服务器端再按每个用户的能力边界做个性化进化。**

## 单用户数据瓶颈

Agent 技能库的质量取决于喂给它的经验有多广。一个用户的任务流天然是窄的。他可能天天做网页导航，但很少碰数据分析；或者大量做代码调试，几乎没做过表格操作。当 Agent 遇到用户在训练数据里没覆盖的任务类型时，技能库帮不上忙。

把多个用户的数据集中到一起，协同进化技能库，能覆盖更长的尾部和更难的任务。但现有的跨用户协同方案走了轨迹共享的路，用户把原始执行轨迹上传到中心服务器，服务器汇总后生成一份共享技能库发回来。

这条路有两个死结。**第一，原始轨迹里嵌着太多私密信息。** 网页截图里的银行余额、表格里的客户数据、命令行里的文件路径，这些不应该也没必要离开用户的机器，哪怕目标服务器是可信的。**第二，一份全局共享的技能库对每个人都是次优的。** 不同的基座模型、不同的 Agent 框架、不同的实际任务分布，要求不同的技能表达方式。一份给所有人的平均化技能库在谁身上都打折扣。

UCSB、MIT-IBM Watson AI Lab 和 Cisco Research 的联合团队在 **FederatedSkill** 里同时解了这两个死结。框架的思路借了一个共识：像联邦学习保护训练数据那样，用技能 patch 替代原始轨迹在用户之间传递经验。

## 客户端：轨迹不出门，只出 skill patch

每个客户端跑两件事。先用当前技能库执行一批任务，生成轨迹。然后一个本地 patcher 把轨迹压缩和反思后，转换成一个技能 patch δ。patch 不包含任何原始交互文本。它只包含三类语义操作：**ADD（加一条新技能）、EDIT（改一条已有技能）、DELETE（删一条过时的）。**

patcher 跟执行 Agent 共享同一个本地模型。它被提示要产出可跨任务泛化的可复用经验，task-specific 的具体值、ID、一次性输出全都不准写进 patch。所有文件路径被严格验证，拒绝绝对路径和目录穿越。**原始轨迹、验证器输出、中间文件、任务输入，全部留在本地，从未离开过客户端。**

这就是联邦学习的思路搬到了技能空间：不传原始训练数据，只传本地更新。但 FederatedSkill 比参数级联邦学习多了一层优势：**它的传输单元是语义技能 patch，不是模型权重。** 语义级 patch 天然跨异质模型、跨异质框架，一个用 Qwen 产生的技能经验可以被 Claude 的 Agent 理解和复用，这在参数聚合里是不可能的。

## 服务器端：不看轨迹，靠 patch 推断每个用户的能力边界

每轮结束时，服务器收到所有客户端上传的 patch 和它们的静态档案（基座模型、Agent 框架）。**服务器从没见过任何一条轨迹。** 它只能从 patch 的语义内容中推断每个用户的实际能力边界：哪些任务频繁出 ADD，对应能力空白；哪些技能常被 EDIT，对应表述偏差；哪些技能正在被 DELETE，对应不再需要的过时操作。

一个演化 Agent M 负责这层推断。它把聚合建模为一个 POMDP：客户端真实的任务环境是隐藏状态，M 只能通过每轮收到的 patch 来动态更新对各个客户端能力边界的隐式模型。基于这个不断更新的理解，M 为每个客户端生成本轮个性化的技能更新，不是一份全局平均库，而是面向该客户端当前能力边界的定向补全。

**服务器输出的是为每个用户定制的建议：你应该加这些技能，修这些表述，删这些冗余。** 集体经验的理解和对特定用户短板的推断，同时存在于同一个演化 Agent 中。

## 实验

SkillFlow 基准，20 个离散 Agent 任务族。局域网中部署 4 个客户端跑 5 轮联邦进化。

**FederatedSkill 相比纯自进化基线，任务成功率最高提升 44.4%，计算成本节省 37.5%。** 消融实验把个性化模块拆掉，用一份统一全局库替代每个用户的个性化库，平均性能下降了 12.2 个百分点。**"给所有人同样的技能库"比"给每个人不一样的"差了超过 12%。**

## 几个问题

**客户端模型的反思能力是 patch 质量的上限。** patcher 跟执行 Agent 共享同一个本地模型。如果模型本身的反思能力弱，它产出的 patch 可能漏掉关键的经验模式，或者把偶然的成功提炼成不靠谱的"通用策略"。而服务器端看不到轨迹，没法交叉验证 patch 是否正确，只能信任客户端的自我提炼。这本质上是一层信任委托。

**安全路径验证目前是规则驱动的。** 客户端对绝对路径和目录穿越做了拒绝，但攻击面不止这些。隐式信息泄露，比如专利文档的文件名、组织架构的目录结构，可能通过比绝对路径更隐蔽的方式流入 patch 内容。当前框架对这类语义侧信道没有防护。

**冷启动问题没被讨论。** 论文假设每个客户端已经有一个初始技能库。如果用户是完全的新用户，没有技能、没有轨迹、也就没有 patch 可以上传，第一轮从哪来？服务器端需要给冷启动的客户端一个合理的初始种子，这个机制在论文中未覆盖。

## 最后

FederatedSkill 在两条线上同时推了边界。**一是用语义技能 patch 替代原始轨迹共享，把隐私保护从"承诺"变成了"只剩 patch 能离开本地"的构造性约束。** 二是用演化 Agent 的隐式能力建模替代全局平均库，让每个人拿到的技能集合是面向自己的短板定制的。

联邦技能进化走下去，技能生态会从"每个人在自己的数据岛上攒经验"变成"经验可以流通，但谁都不需要暴露自己的数据"。这需要把隐私的承诺从信任转移到协议层面。FederatedSkill 的 patch-only 设计是朝这个方向迈的一步。
