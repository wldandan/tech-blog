# 每个人都在卷"怎么生成技能"，但没人认真评过"谁生成的技能更好"——SkillGenBench 把这件事做成了标准化基准

先问一个问题。

你有一份代码仓库，里面有 README、配置文件、脚本、依赖声明。你让 Agent 从中提炼一份可复用的技能文档。Agent 交上来一份 SKILL.md。你怎么判断这份技能好不好？

现在的答案基本是：**装上跑一下，看下游任务有没有变好。** 这个方法有用，但它把两件完全不同的事搅在了一起：技能生成的能力，和执行的能力。下游变好可能是因为技能写得好，也可能是因为执行 Agent 本身够强。反过来也一样。

SkillsBench 告诉我们零样本生成的技能平均没用。SWE-Skills-Bench 告诉我们低质量技能会拖后腿。但有一个问题没人回答过：**不同的技能生成方法之间，到底谁更厉害？如果把技能执行环境统一、把下游测试统一、只比较"生成"这一步，差距有多大？**

SkillGenBench 就是来回答这个问题的。一个由 SJTU、NUS、THU、PKU 等联合推出的基准，187 个任务，统一协议，只测一件事：**给你一份原始材料（代码仓库或长文档），你生成的技能到底能不能用。**

---

## 一、已有的基准为什么不测技能生成

技能相关的测评分两类。**技能效用类**（SkillsBench、SWE-Skills-Bench）测的是"一份给定的技能装上后有没有帮到 Agent"。**任务能力类**（GAIA、SWE-bench）测的是"Agent 从原始材料出发能不能解决任务"。**没有一类在测"从原始材料到技能这一步的质量"。**

更麻烦的是，已有基准把技能生成和技能执行绑在一起。生成器用的 Backbone、执行器用的 Backbone、检索策略、路由策略、环境配置，全是变量。一个技能生成方法拿到高分，到底是生成得好，还是执行那一套刚好适配？反过来，低分到底是生成得烂，还是执行器没读懂它？

SkillGenBench 的做法很简单：**把生成和执行拆开。** 生成器负责从原始材料产出标准化技能包。执行器是固定的，不参与比较。评测也是统一协议，确定性执行验证为主，辅助信号做诊断。

**187 个任务，两种生成模式，两类知识来源，六种生成 Backbone，五种生成方法，只测一件事。**

---

## 二、两种生成模式：知道任务再写 vs 不知道任务就要写

SkillGenBench 设了两个子赛道。

**任务已知模式（Task-Conditioned）。** 生成器拿到原始材料 + 任务描述，从中提取最相关的程序知识，合成一份针对该任务的技能。测的是"知道要干什么之后，你能不能精准提炼"。

**任务未知模式（Task-Agnostic）。** 生成器只拿到原始材料，不知道下游会来什么任务。必须把材料里所有有跨任务复用价值的程序知识蒸馏成一套技能库。等下游任务揭晓之后，从库里检索技能去执行。**生成只做一次，不改。** 测的是"不看考题，你能不能写出一本好教材"。

**任务未知模式目前一片空白。** 现有的技能基准全部是任务已知：给你任务，你生成技能，马上用。任务未知模式下，你生成的技能需要在一组完全没见过的下游任务上发挥作用。难度高了一个量级。实验数据也证明了这一点：任务未知生成的技能在某些情况下**甚至不如没有技能**——这说明不成熟的抽象不仅帮不上忙，还可能产生负面迁移。

## 三、两类知识来源：代码仓库 vs 长文档

原始材料分两种，难度完全不同。

**代码仓库类（Repository-Grounded）。** 程序知识不写在任何一段说明文字里。它们隐藏在目录结构、调用关系、入口脚本、配置文件和环境约定中。你需要从"这个 API 函数的参数签名暗示了什么"和"这个 Makefile 的依赖顺序意味着什么"中拼出整个工作流。**六个 Backbone 在这类任务上的平均 pass@3 只有 10.8%–14.4%。**

**长文档类（Document-Grounded）。** 程序知识是显式写出来的：系统手册、API 文档、技术报告。但它们散落在不同章节里，中间夹着条件分支、参数规则、前置约束、多版本兼容说明。你需要把这些散布的约束整合成一份可执行的技能。**这类任务的 pass@3 在 21.4%–25.0%。** 比代码仓库类高一倍，但绝对值仍然很低。

**两类来源的难度差距很能说明问题。** 写出来的东西好歹有迹可循。代码里的隐含知识，目前没有任何生成方法能稳定提取。

## 四、基准怎么建的：五阶段流水线保证题目质量

SkillGenBench 的 187 个任务不是手写的。一套五阶段流水线保证每个任务都在"有难度但不离谱"的区间内。

**阶段一：知识图谱构建。** 原始材料（代码仓库或文档）被抽象成实体-关系三元组、程序证据群落和上下文摘要。

**阶段二：场景生成。** 从知识图谱推导候选任务场景：代码开发、工作流执行、规则推理等常见任务形式。

**阶段三：任务和测试用例生成。** 每个场景产出一套任务描述和测试用例，覆盖正常、边界和对抗输入。一轮自反思精炼。

**阶段四：无技能验证。** 用一个强 Baseline 裸跑任务。如果不用技能就能轻松搞定（通过率 ≥20%），说明这任务太简单，打回重写。如果加上原始材料全量灌入就能搞定（通过率 ≥50%），也打回。这说明不需要技能提炼，直接查材料就行。

**阶段五：参考技能验证。** 生成一份参考技能，反复用测试反馈迭代优化。如果连参考技能都跑不通这个任务，说明任务难度设置不合理，打回重写。

**整个循环直到任务落入目标难度区间才停止。** 最后再加一轮人工审核。

这套流程保证了每个实例都在"参数知识不够用、需要从材料里真正提炼程序逻辑、提炼出来的技能可以稳定验证"的甜点上。

## 五、实验说了什么：方法有差距，但都不大，而且很不稳定

五个生成方法：Naive Prompt、EvoSkill、SkillNet、SkillCreator、SkillSeekers。六种 Backbone：Claude Sonnet 4.5、GPT-5、Kimi K2.5、GLM-5、MiniMax-M2.7、Qwen3.6-Plus。

### 普遍低于无技能 Baseline？

**有几种方法在某些 Backbone 上甚至跑不过什么都不给的 Baseline。** Naive Prompt 在代码仓库类上的 pass@3 只有 7.3%（Sonnet）到 16.3%（GLM-5），而 No Skill Baseline 是 13.8%。生成的技能比不生成还差：接口不一致、过程不完整、假设写错了，直接把执行器带沟里了。

### 最高分也不高

代码仓库类最高均分 **14.4%**（SkillSeekers），长文档类最高均分 **25.0%**（SkillSeekers）。**最好的方法 + 最强的 Backbone，在代码仓库类上也只拿到了 17.1%（GPT-5 + SkillSeekers）。**

### 提升不稳定，方法和 Backbone 强相关

更复杂的方法（如 SkillCreator）在很多 Backbone 上的表现并不比 Naive Prompt 强。在 MiniMax-M2.7 上，SkillNet 的代码仓库类 pass@3 甚至低到 5.7%。

**技能生成这个赛道还没出现一个"碾压级"的方法。** 目前所有方法都在差不多的低水平区间，差距主要来自 Backbone 能力而非生成方法本身的设计。

### 静态分析：不同方法产出的技能结构完全不同

论文还分析了生成出来技能包本身的静态质量。六个维度：接口合约完整性、环境依赖覆盖、程序接地程度、约束条件覆盖、安全检查、总体 N 分。**不同方法在这些维度上的得分差异巨大。** 有的方法擅长写干净接口但不接地气（SkillCreator），有的方法接地气但约束和安全意识差（SkillSeekers）。**目前没有一种方法六个维度全优。**

## 六、这篇论文在说什么

SkillGenBench 做了一件事，而且只做了一件事。它把"技能生成"从一个被淹没在端到端评测里的子步骤，单独拎出来，用统一协议、统一执行环境和统一评测标准去比。

**实验结果有两个同样重要的结论。** 第一，当前的技能生成方法，在代码仓库这种"程序知识不在字面上"的场景里，能力严重不足。最好的方法也只对了不到 20%。第二，任务未知模式——不看考题先写教材，目前的生成方法基本做不到。生成的技能不仅没用，甚至可能有害。

如果你用 SKILLPYRAMID、MIND-Skill 或 SkillCreator 在生成技能，SkillGenBench 给了你一个标准化的跑道来比较不同生成策略、不同 Backbone、不同方法之间的真实差距。**不是"我的 Agent 变强了"那种模糊的证据。是"这份生成的技能在 187 个任务上跑通了多少"这种硬的数字。**
