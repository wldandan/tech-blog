# 以前的 Agent 只会用技能，SKILLMASTER 教会了它自己写技能、自己改技能、自己决定用哪个

技能对 Agent 的重要性已经没人质疑了。Voyager、Claude Code、OpenClaw，这些最成功的 Agent 系统全都在靠技能驱动。但仔细看你会发现一个微妙的事实：**技能的管理者从来不是 Agent 自己。**

技能怎么生成？外部 teacher model 从 Agent 的轨迹里蒸馏。技能什么时候更新？固定的周期到了才触发。技能怎么选？检索模块根据相似度挑几个塞进 Prompt。Agent 做的唯一一件事就是读技能、用技能。至于这份技能好不好、什么时候该改、该改成什么样，**Agent 完全被排除在这些决策之外。**

山东大学、清华、中关村实验室、东南大学和中科大的联合工作 SKILLMASTER 要改变的就是这件事。它把技能管理（创建、修改、选择）全部教给了 Agent 自己。不是"给 Agent 配一套技能管理系统"，而是"让技能管理成为 Agent 本身的能力"。

---

## 一、以前的技能管理长什么样，问题在哪

Voyager 靠一个外部模块不断往库里加技能。SKILLRL 靠 teacher model 定期从轨迹中蒸馏新技能。Agent 只负责执行：跑完一条轨迹，teacher 来收作业，看完总结成技能，塞进库里。

**这些方案共享三个结构性缺陷。**

**第一，技能是外部管理的，不是 Agent 自己掌握的。** Agent 学会了"怎么用技能"，但从没学会"怎么判断一份技能好不好、什么时候该改"。技能库对它来说是一个只能读不能写的黑盒子。

**第二，技能质量没法评估。** 任务成功或失败是一个太粗糙的信号。Agent 这次成功了，是因为技能好还是因为运气好？失败了，是技能的问题还是 Agent 执行的问题？**二元奖励把技能质量和执行质量搅在了一起，分不清。**

**第三，执行和管理的优化目标互相干扰。** 执行要的是"每一步选对动作"，技能管理要的是"改完之后长期收益更大"。两个目标有完全不同的奖励尺度和时间尺度。把它们混在一起训练，谁也学不好。

## 二、SKILLMASTER 怎么做的：让技能管理变成 Agent 学得会的事

SKILLMASTER 的核心理念是一条：**技能管理不是什么神秘的外部能力，它是 Agent 自己可以通过 RL 学会的一组工具调用。** 把"创建技能""修改技能""保留技能"做成三个工具，Agent 在每条轨迹结束后自己决定调用哪个。

### 2.1 两阶段轨迹设计：先干活，再反思

每条训练 episode 分两段：

**执行阶段。** 跟标准 Agent 一样。从技能库里检索相关技能，注入 Prompt，Agent 一步步执行，收到环境奖励。

**技能掌握阶段。** 执行阶段结束后，系统构建一个技能回顾 Prompt，包含：任务描述、检索到的技能、完整执行轨迹、最终环境反馈。Agent 看完这些，必须输出**恰好一个工具调用**：propose_skill（新建技能）、update_skill（修改已有技能）、keep_skill（不改）。

**Agent 不是被动的技能使用者，它在每一次执行后都在主动审视："我该从这次经验里提炼什么？已有的技能需要改吗？"** 而且这个审视过程被建模为可优化的 RL 动作，不是固定的启发式规则。

### 2.2 反事实技能效用奖励：这份技能改得好不好，拿探针任务测一下

这是 SKILLMASTER 最巧妙的设计。如何判断一份技能修改好不好？最直接的办法：**拿它去跑几个相关的探针任务，看表现变了没有。**

当 Agent 调用 propose_skill 或 update_skill 时，系统选 K 个跟当前任务语义相关的探针任务（比如同属"清洁物品"这个任务家族）。每个探针任务跑两遍：一遍用原技能库，一遍用本次修改后的临时技能库。比较两次表现。

评分公式考虑了两种效果：**以前失败的任务现在能成功了**（好的技能让失败变成功）、**以前就成功的任务现在步数更少了**（好的技能让效率更高）。最终效用奖励综合了平均改进幅度和方向一致性。只有那些在多个探针任务上效果一致正向的修改才能拿到高分。

**这个设计的核心洞察是：技能质量不再是模糊的主观判断，而是可量化的下游效果。** 一份好技能能用两组数据说话：它让多少之前失败的任务变成功了，让多少任务变快了。

### 2.3 DualAdv-GRPO：执行和技能管理，分开算优势，一起训

执行阶段收到的是二元环境奖励，技能掌握阶段收到的是连续的效用奖励。两者的尺度和语义完全不同。标准 GRPO 把它们混在一起做归一化会出问题：执行的好坏和技能改的好坏纠缠在一起。

SKILLMASTER 的做法很直接：**两个阶段的奖励分别归一化。** 执行阶段的优势只跟同一批次其他执行阶段的奖励比，技能掌握阶段的优势只跟同批次其他技能掌握阶段的奖励比。然后在 Token 级别用类型条件控制：执行 Token 走执行优势流，技能管理 Token 走技能优势流，两者之间用可调权重 γ 控制比例。

这样执行学习和技能管理学习就能同时进行，互不干扰。**一个策略同时学会了怎么执行任务、怎么管理技能。**

## 三、实验说了什么

ALFWorld（具身家居）和 WebShop（网络购物），用 Qwen2.5-7B-Instruct 做基座，对比 ReAct、Reflexion、GRPO、RLOO、SkillRL 等。

### 主要结果：全面领先，而且优势很大

**ALFWorld 总成功率 98.7%，超过 SkillRL（89.9%）8.8 个百分点。WebShop 成功率 82.0%，超过 SkillRL（72.7%）9.3 个百分点。** 六个任务类型全部拿下，其中 Pick、Look、Pick2 三个类型直接 100%。

**连闭源模型都超了。** GPT-4o 在 ALFWorld 上只有 48.0%，Gemini-2.5-Pro 60.3%。不是模型大小的问题。是 7B 的小模型学会了技能管理之后，碾压了大得多的闭源模型。

### Agent 把技能内化了

一个漂亮的验证：**训练结束后关掉技能检索，让 Agent 裸跑。** SKILLMASTER 的 Agent 在 ALFWorld 上从 98.7% 掉到 93.9%，只掉了不到 5 个点。这说明 Agent 不是靠"查技能"拿高分：**它在训练中已经把很多技能内化成了自己的行为模式，不需要显式检索也能做对。**

### 消融实验：每个组件都在出力

去掉反事实效用奖励（只用二元环境奖励训练技能修改），ALFWorld 从 98.7 掉到 92.8。去掉 DualAdv-GRPO（混合归一化），掉到 94.5。去掉技能掌握阶段（只做标准 GRPO），掉到基准水平。

### 技能质量在训练中持续提升

论文追踪了训练过程中技能库的质量变化。**随着训练推进，技能编辑的成功率（探针任务上表现提升）稳步上升，无效编辑的占比持续下降。** Agent 不是学会了"乱改技能"，它学会了"挑着改、改得准"。

### 只改少量技能就能大幅提升

SKILLMASTER 在 ALFWorld 整个训练过程中总共只做了约 20 次有效的技能编辑（新建+修改）。**用不到 20 次修改，把一个小模型送到了 98.7%。** 技能管理不是越多越好，是改得对。

## 四、局限与总结

技能库初始化需要初始种子技能（论文用了 SKILLRL 蒸馏的技能）。反事实探针评估需要额外 rollout 预算，虽然论文论证了 K 可以很小（默认 3）。探针任务的选取依赖任务家族定义，更通用的选取策略是开放问题。

但 SKILLMASTER 真正值得记住的不是它的具体数字。是它的认知起点：**技能管理不是 Infrastructure 的问题，是 Agent 能力的问题。** 不应该在 Agent 外面再搭一套系统来管技能。应该让 Agent 自己学会创建、修改、选择，然后把这件事做成训练目标的一部分。

当一个小到只有 7B 的模型，学会了管理自己的技能库之后，能把闭源巨模型和手写技能的 Agent 一起碾压。这不是在说"小模型更强"，而是在说**"会管理技能的 Agent"和"只会用技能的 Agent"之间，有一道根本不是模型尺寸能填平的鸿沟。**
