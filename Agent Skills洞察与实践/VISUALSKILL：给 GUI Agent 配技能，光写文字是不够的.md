# VISUALSKILL：给 GUI Agent 配技能，光写文字是不够的

> 截图能顶一千个字。但现在的 Agent 技能库，把截图都丢了。

## GUI Agent 碰到了一个尴尬的问题

计算机使用 Agent（CUA）这几年进展很快。Claude、GPT、Gemini 这些模型在 OSWorld 等基准上的分数一直在爬，接近人类水平了。

但两个老毛病还在。长链路任务做不好，碰到没见过的软件界面就懵。

原因不复杂。每个软件有一套自己的隐性知识——哪个菜单藏着哪个命令，点了某个按钮之后弹什么对话框，多步操作怎么走。这些东西预训练数据里没有。模型不能靠猜。

技能（Skill）正好能补这个缺口。提前把应用的操作知识写成文件，Agent 推理时查阅就行。CUA-Skill、OSExpert、XSkill 都在这个方向上做尝试。

但有个问题被所有人跳过去了：**这些技能全是纯文本。**

对 GUI 操作来说，这是个奇怪的限制。界面上很多东西用文字很难讲清楚。工具栏上有十几个长得差不多的图标，你写"点击格式刷按钮"——Agent 怎么知道哪个是格式刷？对话框里一排按钮，你写"点右下角的保存"——Agent 可能点到标题栏上，直接把程序关了。

UCSB、MIT CSAIL 和 MIT-IBM Watson AI Lab 的 VISUALSKILL 论文只做了一件事：把截图还给技能。

## VISUALSKILL 的设计

每个应用配一套技能，两层结构。

顶层一个 `SKILL.md` 索引，列出所有话题和一句话"什么时候用"。底层是每个话题的文件夹，一篇 `guide.md` 操作指南，一组 PNG 截图。LibreOffice Writer 的技能有 19 个话题类别，"格式化文本/字符格式化"这个话题下，指南教你字体、大小、加粗、上下标，截图展示字符面板的实际样子和字符对话框六个标签页的布局。

Agent 推理时不一口气把所有技能读进上下文。系统 prompt 里只有索引。每做一步之前，Agent 扫一下索引，当前子目标和某个话题的说明对上了，就调 `load_topic` 这个 MCP 工具，把文字和图一起拉过来。

加载机制里有一个设计细节值得注意：文字和图片按顺序交错推送。文章里写"见 fig01.png"的时候，这张图就紧跟在那句话后面出现。描述和图像在同一个上下文窗口里，Agent 不用自己去匹配"这段话对应哪张图"。纯文本对照版里，同样的视觉信息被改写成文字，不附带图片。

两种变体共享同样的操作内容和话题结构，唯一区别就是**视觉信息以图还是文字来呈现**。这意味着性能差异可以干净地归因到模态上，不是内容本身的问题。

## 造技能：两步走

Stage 1，挖文档。成熟软件通常有官方手册。LibreOffice 有 PDF 版 Getting Started，GIMP 有 HTML 版用户手册。把目录结构解析出来复用为话题集合，提取每个话题对应的页面，抽出文字和官方配图，原样保留。手册里的截图是 vendor 画的，干净标准，但可能跟实际运行的 UI 有出入。

Stage 2，跑真实应用截图。用 LLM 驱动的探索器去实际操作应用，把 Stage 1 漏掉的 UI 知识补上。分两种模式。

Free Explorer。Opus 级 planner 放到空闲应用窗口前，把界面分成若干区域（论文里分 8 个）。每个区域派一个 Sonnet 级 worker 去操作：点击每个控件、切换每个状态、截图、记录行为。比如"字符位置对话框"区域，worker 点开"上标"按钮，截图；点"下标"，再截图；记录自动勾选的复选框、灰掉的 spinner。这些都是手册不会写的状态转换细节。

Trajectory-Targeted Explorer。训练任务的 rollout 里有些失败了，一个 reviewer agent 去读这些失败轨迹，找出哪些 UI 区域让 Agent 反复踩坑、系统性犯错。然后针对性派 worker 去那个区域截图、标注、写操作说明。

论文里有个例子很能说明问题。LibreOffice Writer 的段落样式下拉框。Stage 1 的文字描述只说"用下拉框选样式"。实际上下拉框是两块拼起来的组合框：左边一个文本字段，右边一个极小的 ▼ 箭头。要点箭头才出样式菜单。`exam_paper_formatting` 这个训练任务上，Agent 没截图参考，点了左边的文本字段，用"打字加回车"花了 8 步才把"标题 2"应用上。三个标题，每个浪费 6 步，总共 18 步全废了，最后跑完 80 步预算也没来得及保存文件。

Reviewer 发现同一个控件在 4 个不同训练任务上都导致失败，判定这是系统性缺口不是偶然。Targeted worker 被派过去专门操作这个下拉框：点 ▼、截图打开的菜单、截图应用后的状态。两张图加一段说明（"两个点击目标：文本字段在 ~(75,118)，▼ 箭头在 ~(196,156)，两步搞定"）写回了话题文件。

后面跑测试的 Agent 有了这张图，直接点对 ▼，两步完成。

## 效果

两个基准，177 个任务。CUA-World（Writer / Calc / Impress、QGIS、OpenToonz）和 OSExpert-Eval（LibreOffice、GIMP、Tableau）。Agent 是 Claude Code CLI + Opus 4.6。

不加技能，八个领域未加权平均 0.303。Stage 1 VISUALSKILL（纯文档提取）到 0.363（+6.0%）。Stage 2 VISUALSKILL（文档 + UI 探索）到 **0.456（+15.3% over baseline）**。

跟纯文字对照版比。Stage 1 阶段，多模态版只高 1.9%（0.363 vs 0.344），差距很小。手册配图多是官方示意图，文字够用了。但 Stage 2 阶段，多模态版比纯文字版高 **8.3 个百分点（0.456 vs 0.373）**。UI 探索补上去的截图，对视觉密集型的创意工具拉升最大：GIMP +16.6%，OSExpert Tableau +15.0%，OpenToonz +8.9%。

## 截图到底帮了什么

论文对 VISUALSKILL 比纯文字版好的案例做了定性分析。优势集中在两类情况。

找对 UI 元素。三种反复出现的情景。一是图标类控件没文字标签，工具栏上的画笔、调色板里的色块、绘图字形，写"点击格式刷"听起来简单，但 Agent 很容易点偏到旁边按钮。二是动态出现的界面，点两层菜单才出的子对话框、右键上下文菜单、下拉选项面板，这些东西在空闲截图里看不到。三是复合控件里的子区域，组合框箭头旁边的文本字段、高对话框底部的一排按钮，文字说"点底部保存"，Agent 可能点到标题栏。

论文附录里真有一个翻车案例。文字描述是"点击对话框底部的保存按钮"，Agent 点了 (1011, 44)，正好是标题栏，LibreOffice 直接关了。多模态版的截图把保存按钮钉在了实际位置，没了歧义。

验证中间 UI 状态。多步操作每执行完一步，Agent 要确认画面是否符合预期才能继续。用文字做这个检查很间接：先根据文字想象画面长什么样，再跟截图对比。有了参考截图，比对是直接像素级的。最麻烦的是沉默失败场景：Agent 往表单字段里打了字，字段看起来接受了，但提交时被拒绝了；多步对话框的进度只体现在微小布局变化上。除了画面本身，Agent 没有其他信号。

## 加载方式也关键

论文做了一个消融：把 MCP 工具 `load_topic` 换成 Agent 直接 `Read` 技能文件。

差异大到出乎意料。MCP 方式，Agent 加载了 100% 话题，平均每任务 7.9 张图，最后一次查阅技能在 10.4 步——边做边查。Read 方式，加载率 92.6% 看起来还行，但平均只读了 0.8 张图，最后一次查阅在 1.5 步。Agent 开头翻了一下技能，就再也没回去。

原因很直接。MCP 工具把文字和配图打包成一次调用返回，一个 round trip。直接 Read，每张图要单独调一次，Agent 不会为了一张不确定有没有用的图额外打一次工具调用。

准确率差距也明显。MCP 比 Read 在三个领域上稳定高 3-4%。视觉信息在截图里，不用 MCP 推过去的话，大部分知识留在硬盘上了。

## Stage 2 两个子阶段的贡献

LibreOffice 三个应用的消融数据。Stage 1 VISUALSKILL 把 LibreOffice 平均分从 0.331 拉到 0.436（+10.5%），成熟应用的官方文档确实覆盖了大量操作知识。但在 Stage 1 内部，多模态 vs 纯文字差距很小（+1.6%），手册配图多是干净示意图，文字够用了。

加上 Stage 2(a) free explorer，到 0.462。加上 Stage 2(b) trajectory-targeted explorer，到 0.453。两者都加，**0.504**，超过任何一个单独子阶段。覆盖的是互补的表面：free explorer 摸到了空闲窗口里所有静态可见的区域（工具栏、侧边栏、菜单面板），targeted explorer 摸到了只有在任务交互中才会出现的动态表面（两层深的模态对话框、状态转换行为、提交后确认画面）。

## 局限

论文也提到了，当前模型不太擅长在一个话题的多张截图中判断哪些和当前决策相关。所以 `load_topic` 的设计是把话题下所有截图一口气返回。这对大模型没问题，但小模型上会造成上下文压力。Qwen3.5-397B 在 OSWorld LibreOffice 上加了多模态技能后反而退化，轨迹里出现格式错误的工具调用，很可能是上下文被截图撑爆了。

怎么在保留视觉信息和控制上下文开销之间找平衡，还没解决。

## 我觉得 VISUALSKILL 做对的事

VISUALSKILL 给 CUA 技能领域带来的启发其实很朴素：**GUI 操作的知识不能只靠文字。** 这不是文字表达能力的局限，是视觉界面本身的属性。图标的意义在于外观，对话框的状态在于像素排列，点击位置在于布局坐标。这些信息天然属于图像模态，硬翻成文字就丢了东西。

工程上也有一个直接能用上的教训。技能加载的方式决定了内容能不能被 Agent 真正用上。把图放文件夹里等 Agent 自己读，Agent 大概率不读。打个包一次推过去，效果差一个量级。如果你在搭 CUA 系统，这可能是今天就能改的一件事。
