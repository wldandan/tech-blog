# 技能在进化，但进化策略从来没变过——HiSME 让"怎么进化"这件事本身也学会进化

Agent 技能进化的标准叙事是：Agent 跑任务 → 留下轨迹 → 从轨迹里提炼经验 → 写成技能 → 下次用。这个循环跑起来之后，技能库会越来越丰富，Agent 的表现会越来越好。

但你有没有想过一个问题：**那个负责"从轨迹里提炼技能、判断技能好不好、决定改不改"的进化策略本身，它进步了吗？**

没有。它是一套固定的流程。解析失败模式的方式是固定的，判断一份技能该保留还是该淘汰的规则是固定的，从经验中提取什么、丢弃什么的策略是固定的。**技能在长，但管技能的那个"管家"从来没长过。**

清华和华为的联合工作 **HiSME（Hierarchical Skill Meta-Evolving）** 要改变的就是这件事。它把技能进化拆成了两层：第一层，技能本身在进化（从轨迹中学习和更新）；第二层，**进化策略本身也在进化**（从管理技能的经验中学习如何更好地管理技能）。第二层产出的东西叫 **Meta-Skill**——不是用来指导 Agent 怎么做任务的，是用来指导进化策略怎么更好地提炼、评估、维护技能的。

在多个 Agent 基准上，HiSME 产出的技能库质量稳定超过静态进化基线。

---

## 一、静态进化策略的问题

先理解为什么进化策略需要进化。

现在的技能进化管线通常长这样：一个固定的算法接收轨迹 → 提取经验 → 生成或修改技能 → 评估质量 → 决定保留还是淘汰。**这个算法的每一步都依赖人工预设的启发式规则。** 比如"只从成功轨迹中提取""技能描述不能超过 500 字""相似度超过 0.8 就合并"。

这些规则在第一次被写下来之后就不再变了。但 Agent 面对的任务在变、环境在变、技能库的规模在变、可用的轨迹风格也在变。**一套固定的进化策略不可能在所有场景下都最优。** 数据处理任务可能受益于宽松的函数式技能提取，安全敏感任务可能需要更严格的 checklist 式技能。同一个策略不可能两头讨好。

更隐蔽的问题是：**如果进化策略本身有系统性缺陷，每一次进化都在重复同样的错误。** 比如策略倾向于把技能写得太具体，每一份新技能都过拟合到一条特定轨迹。Agent 花大量维护成本不断修补同一类问题，但策略从来不知道自己有这个癖好。

## 二、HiSME 的两层优化

HiSME 把技能进化重新定义成**一个多层次的残差优化问题**。

**第一层：技能优化。** Executor（执行 Agent）本身是固定的，参数不能更新。技能进化算法 EVO 通过从执行轨迹中学习，不断更新技能库 S，让 Executor 在 S 的帮助下表现得更好。**这跟传统技能进化做的事一样——从零开始构建一个技能库，让 Agent 越来越强。**

**第二层：Meta 优化。** 但 EVO 本身也是一个"有缺陷的先验"——它是由人类工程师按照自己的理解设计的。它在技能提取、评估、维护的每个环节都可能做出次优决策。**HiSME 在 EVO 之外再引入一个 Meta-EVO 层，让它从 EVO 的操作历史和执行反馈中学习，产出 Meta-Skill 来修正 EVO 的行为。**

**Meta-Skill 不是"教 Agent 做任务的技能"，是"教进化策略怎么做进化"的技能。** 它回答的问题不是"这个任务怎么完成"，而是"这类轨迹该怎么提炼""这份技能什么时候该淘汰""什么样的评估策略在这个场景下最有效"。

两层的更新都在测试时、在文本空间中完成，不碰底层模型参数。轻量。

## 三、技能怎么生成和评估

HiSME 的技能生成用了两种策略。

**提取器（Extractor）。** 接收单独一条轨迹，从中识别可复用的片段或经验。**这个提取器依赖的是 LLM 的通用能力，不是针对某个领域的专有规则。**

**重构器（Refactorer）。** 接收一组相似的轨迹或技能，抽象出共享的结构或知识。**重构不只看一条轨迹——它寻找跨轨迹的重复模式。** 用 Embedding 和 N-gram 统计构建相似性图，从中提取团（clique）来发现潜在的可合并模式。

技能的评估分两个时间尺度：

**短期效用。** 一个 Bundle Tester 为每份技能用 LLM 构建一套测试用例（单元测试 + 集成测试），检查技能是否忠实地履行了它所声明的"合约"。

**长期效用。** 一个 Credit Assigner 从真实的执行器轨迹中回溯：这份技能在实际任务中到底是帮了忙、添了乱、还是打了个酱油。另外还有使用统计记录每份技能的被检索次数和被执行次数。

**四个信号汇在一起，构成每份技能的完整评估档案。** 这份档案会被 Meta-Skill 读取——下一层会讲。

## 四、Meta-Skill 怎么让进化策略变聪明

Meta-Skill 是 HiSME 最核心的概念。它是第二层优化的产物——**从"进化策略自身的操作历史"中学到的改进经验。**

具体来说：

**Meta-Skill 不是手工编写的。** 它和普通技能一样，是从执行反馈中自动提炼的。区别在于，普通技能的输入是 Agent 的任务执行轨迹，Meta-Skill 的输入是**进化策略的操作轨迹**——"在这次进化中，提取器提取了什么""重构器合并了什么""Bundle Tester 给出了什么评估""Credit Assigner 判定这份技能在一组任务上是有害的"。

当同样的失败模式反复出现——比如提取器总是生成过拟合的技能、Bundle Tester 总是漏掉某类边界 case——Meta-EVO 检测到这种模式，生成一条 Meta-Skill，指导进化策略在下一次遇到类似情况时怎么调整。

**HiSME 把人类工程师对进化策略的"调参"自动化了。** 人类工程师不需要去分析"为什么进化策略在这个领域表现不好"——Meta-EVO 自己从操作历史中找到规律，自己写改进方案，自己把方案固化成 Meta-Skill。

## 五、实验说了什么

多个 Agentic 基准上测试。

**Meta-Evolving 产出的技能库质量稳定超过静态进化基线。** 不是某个领域的偶然优势，跨多个基准一致。

**学习到的进化策略超过了框架的初始能力。** 这意味着 Meta-EVO 不仅在修正缺陷，它在真正地让进化策略变得比最初人手设计的版本更强。

**Meta-Skill 的多样性本身就是一个验证。** 论文展示了不同场景下产出的 Meta-Skill 风格截然不同——有些偏向更激进的提取策略，有些偏向更保守的维护策略。**这说明 HiSME 确实学会了根据目标领域调整进化策略，而不是在所有场景下用同一套逻辑。**

## 六、这篇论文在说什么

HiSME 触及了一个比"技能怎么进化"更深一层的问题：**谁能保证进化策略本身是好的？**

目前的技能进化生态里，每个人都在设计自己的进化策略——什么时候提取、怎么评估、如何维护。这些策略被写进代码之后就不再变了。HiSME 的核心论证是：**进化策略也是一种可以被优化的对象，而且优化它的方法论和优化技能本身的方法论是同构的——从执行反馈中学习，提炼成可复用的经验，固化到文本空间中。**

如果你在维护一套技能进化管线，这篇论文给了一条值得认真对待的建议：**别只盯着技能的评估数据。把进化策略自身的操作也记录下来——它提取了什么、合并了什么、淘汰了什么、这些决策后来被验证为对还是错。这些东西是最宝贵的训练数据，训练的不是 Agent，是那个管 Agent 技能进化的"管家"。**
