# 你的技能库越大，检索越瞎——SkillDAG 把 500 份孤立技能变成一张会自己进化的关系图

![alt text](image-4.png)

你的技能库里有500份Skill。Agent接到一个任务，用Embedding算相似度，召回Top-5，塞进Prompt。你觉得没问题。

但有一份技能叫"JSON解析器"，它从来不会出现在Top-5里。因为它不描述自己做什么，只描述怎么做："将API返回的嵌套JSON扁平化为key-value对"。没有任务描述会跟这句话相似。**但它是一份"冷却物品"技能的前提。不先加载JSON解析器，冷却流程就会在第三步报错。**

**相似度检索看不到结构。它只能看到两段文字有多像，看不到一份技能需要另一份才能工作、两份技能不能同时用、A是B的特化版本应该优先选。**

复旦、NUS和A*STAR联合发了一篇论文，叫**SkillDAG**。它做的就是把技能之间那些"相似度看不见的关系"显式建出来，然后把这个结构直接交到Agent手里。Agent不只是在检索，它是在一张有类型、有方向、能自己进化的关系图里做决策。


# 一、平铺检索的隐性代价

当前的技能检索标准做法是：把所有技能Embedding存进向量库，每次用任务描述算余弦相似度，取Top-K拼进Prompt。

这个做法有三个盲区，而且每个都随着技能库变大越来越严重：

**第一，前提技能被静默吃掉。** 一个"冷却物品"技能依赖一个"拿起物品"技能作为前置。但在Embedding空间里，"冷却物品"离"拿起物品"很远，它们描述的是完全不同的动作。Top-K检索把"拿起物品"排在30名以外，Agent永远看不到。**相似度衡量的是"描述像不像"，不是"需不需要"。**

**第二，冲突和冗余混在一起，都是高分邻居。** Embedding世界里，一份跟当前任务高度相似的特化技能和一份跟当前任务高度相似但互斥的替代方案，它们的余弦分数可能一模一样。Agent看到两份"差不多"的技能，选了不能同时用的那两份，然后互相干扰，全挂了。

**第三，技能库越大，检索越瞎。** 500份技能的时候Top-5还能碰对。5000份的时候，前五名全是"描述看起来相关但其实没用"的干扰项。**库越大，召回越差。这不是工程问题，是方法论问题。** 余弦相似度只能做语义近似，它不知道什么叫"需要"，什么叫"冲突"，什么叫"这个问题不应该用这个技能"。


# 二、SkillDAG的核心设计：把技能关系图变成Agent的工具

SkillDAG做了一件跟之前所有方法都反着来的事。

之前的做法（包括最强的基线 GoS，Graph of Skills）是：**先离线建一张技能关系图，跑PageRank扩散出固定技能包，塞进Prompt。Agent从头到尾不知道图长什么样，也不知道为什么选了这些技能。**

SkillDAG反过来：**图不是检索算法的内部数据结构。图是Agent的工具。** Agent在执行中随时查图、随时展开邻居、随时加边、随时注册新发现的关系。LLM自己决定结构证据怎么影响执行，而不是一个固定的图排序策略替它决定。

## 2.1 五种类型的边：不止是"有关"

SkillDAG用五类有向边来建模技能间的关系。每一类边在Agent执行时有明确的语义和行为后果：

| 边类型 | 含义 | Agent 怎么用 |
|--------|------|-------------|
| **depends_on(A, B)** | A需要B作为前提 | 选了A就必须拉上B |
| **specializes(D, A)** | D是A的特化版本 | 任务落到D的领域内时优先选D |
| **composes_with** | 两个技能组合使用效果好于单独用 | 信号提示可以一起加载 |
| **similar_to** | 功能重复，互换 | 选一个就行，别都选 |
| **conflicts_with** | 一起用会导致失败 | 选了A就必须排除B |

同一对技能可以有零条、一条或多种关系。图的有向性让 Agent 能区分"我需要你"和"你跟我很像"。这两个在 Embedding 空间里长一个样，在图里是完全不同的边。

## 2.2 Agent可调用的图检索接口

SkillDAG暴露给Agent的不是一个排序好的技能列表，而是三个独立的信号通道：

**search(query, K, D)** 一次调用返回三个字段：**matches**，Top-K余弦相似技能；**neighbors**，从匹配技能出发沿四类正向边BFS遍历深度D找到的邻居技能，每条邻居带着边类型标注；**conflicts**，匹配技能的一跳conflicts_with边。

这三个字段是分开返回的，不融合成一个排序列表。这样Agent可以根据自己的判断取舍：上下文紧张时扔掉neighbors，需要全局视野时展开，还可以在推理Trace里记录为什么选了或没选每个技能。

**show(skill)** 按需加载技能的完整内容。**不会提前灌满上下文。**

## 2.3 冷启动：两视图建图

第一张图怎么来？最直觉的做法，用每份技能的自我描述做Embedding然后连最近邻，有一个具体的失效模式：**互补技能通常描述完全不同的事情。冷却技能和拿起技能在自我描述上毫无交集，但前者需要后者产出的"物品在手中"。**

SkillDAG用了一个两视图策略来补这个洞：

**第一个视图e_self**：技能自我描述（名称、描述、正文预览）的Embedding。这张视图能抓到话题相近的技能。

**第二个视图e_needs**：让LLM想象几个需要调用这份技能的场景，总结出共享的前置条件，比如"需要物品在手中""需要已认证的session""需要已解析的JSON 对象"，然后把这个总结Embedding。这张视图能抓到功能互补的技能：一份技能需要的东西，恰好是另一份技能的产出。

两张视图各自拉候选，过自适应阈值，然后用LLM分类器分配边类型。**conflicts_with被有意留空。冲突在用之前是看不出来的，只有执行之后才知道。这是后面在线演化的任务。**

## 2.4 在线演化：Agent在执行中发现新关系，当场写进图

冷启动只是开始。Agent在执行中会发现三种冷启动永远发现不了的关系：**一个没被预判到的依赖、一对同用才发现互相干扰的技能、一条需要修正的旧边。**

SkillDAG把图编辑做成Agent的标准动作，跟调工具平级。两步走：

**propose-edge**：干跑。预览如果提交会发生什么，展示同对技能上所有的历史边和编辑记录。Agent看到"上次有人在这对技能上标了depends_on，现在你想标conflicts_with——你确定吗？"

**edit-edge**：提交。附带自然语言理由和执行Trace证据。支持添加、删除、修改边类型。

有三条结构性约束在提交时强校验：**无环性**（depends_on/specializes不能构成环，循环前提会形成死锁）、**非矛盾性**（同一对不能同时有正向边和conflicts_with）、**可回滚性**（追加式日志按时间或任务ID回滚，一条烂边不会永久污染图）。

**关键设计决策：提交只由Agent的判断门控，不需要投票、不需要人类curator。** 一次执行就够了，图以Agent的速度演化。

# 三、实验：在ALFWorld和SkillsBench上的结果

对比三种基线：Vanilla Skills（全量灌入 Prompt，召回上界但Token爆炸）、Vector Skills（Embedding Top-K，无结构）、GoS（Graph of Skills，离线建图 + PageRank 扩散 + 重排序）。

![alt text](image-2.png)

### Agent越弱，SkillDAG帮的越多

**MiniMax-M2.7上，SkillDAG把ALFWorld成功率从 54.3% 拉到 67.1%（+12.8），SkillsBench奖励从 18.7 拉到 27.3（+8.6）。** GoS在这两个任务上分别是 54.3% 和 18.7%。

**gpt-5.2-codex上，ALFWorld打平 93.6%，SkillsBench从 34.4 到 36.8（+2.4）。** 这个模式有清晰的解释：当主干模型足够强时，它自己能推理出"那个技能可能需要那个前提"。即使检索给了一个烂包，它能靠推理补。主干弱时，检索包的质量直接决定生死，SkillDAG的结构信息成了救命的东西。

**SkillDAG的价值定位不是"替代强模型"，是让弱模型能打、让强模型更省。**

### 不只是Agent更强了，检索本身也更准了

论文把两套检索器（GoS vs SkillDAG）放在完全相同的 87 条查询上跑，Embedding是同款模型，Pool是同款1000技能。**SkillDAG的纯余弦matches全面超越GoS的HNSW+PageRank 扩散：Ret@5 65.5→78.2，Ret@1 50.6→66.7，MRR 57.3→71.3。** 还没算neighbors和conflicts的作用。这只是最基础的那层matches。

### 技能池膨胀10倍，排序质量不降

![alt text](image-3.png)

把技能池从200扩到 2000（10×），SkillDAG的Ret@5只掉了3.5 个点（78.2→74.7）。GoS在2000技能上各项指标都比 SkillDAG在2000技能上低 13-16 个点。**固定 Embedding 骨架不变，SkillDAG的typed-graph node embedding 排序天然比扩散更抗规模膨胀。**

# 四、总结与局限

**技能不是独立的。把它们当独立的检索单元，就是主动放弃了技能之间最有价值的那层信息。**

如果你在维护一个越来越大的技能库，SkillDAG给了三个马上能用的启示：**别只用相似度做检索。相似度高不代表需要，相似度低不代表不需要。让Agent能看到技能之间的关系，而不只是名字和描述。让每次执行都是一次"测绘"。发现新关系就记下来，下次就有了。**

同时也要说清楚局限。论文坦诚列了一个核心问题：**单次观察驱动的自演化。** propose-then-commit允许Agent基于一次执行的证据就提交边编辑。结构性约束（无环、非矛盾、可回滚）提供了护栏，但长期积累后的质量退化、噪声边的累积效应、跨模型迁移后的行为变化，这些都还是开放问题。此外冷启动配对分类依赖gpt-5-nano，弱模型下分类稳定性未测；评测局限在文本交互任务；五类边对更复杂的语义关系也无能为力。
