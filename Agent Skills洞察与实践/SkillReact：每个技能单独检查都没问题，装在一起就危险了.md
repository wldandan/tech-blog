# SkillReact：每个技能单独检查都没问题，装在一起就危险了

> 1,520 个 ClawHub 技能，逐个扫描。651 个通过了个体安全检查。把它们两两配对——21 万对。每一对的每个技能单独看都是安全的。然后把它们合在一起看：**22.25% 的配对在能力并集上触发了危险模式。** 经过人类审计校准后，大约五分之一被确认为真实的组合风险。而按个扫描的设计前提，恰恰漏掉了所有这些。

## 问题不在单个技能，在两个技能的并集

LLM Agent 的技能生态在快速膨胀。Claude Code、Cursor、Windsurf 都在加载社区贡献的技能，能读写文件、访问网络、操作凭证、启动进程。当前的应对方案是单个扫描：恶意技能检测器、漏洞扫描器、能力分类器，全都对一个技能单独做判断。

**但部署的 Agent 不会一次只运行一个技能。**

用户装了一个"只能读文件"的研究助手，再装一个"只能调 API"的工具整合器。两个技能单独检测都通过，装进同一个 Agent 运行时之后，Agent 同时继承了文件读取和网络外发的能力。**而"文件读取 + 网络外发"这一对能力并集，恰好就是数据外泄的标准技术签名。** 安全策略没有变，变的是安全分析的单元：从单个技能变成了已安装的技能集合。

这个盲区有结构性原因。CMU、佐治亚理工、格拉斯哥大学和 Corespeed 的联合研究团队在 **SkillReact** 里对它做了一次系统级测量。全文三块证据：一份确定性的静态组合基准，一套由两个 LLM 协助标注并经人类审计的 200 单元黄金集校准管线，以及一个基于执行动作的可利用性验证平台。

## 三层证据，从候选到确认

论文刻意把组合风险的证据分成了三层，每一层对应不同的确定性级别。

**第一层：个体标记的技能**。一个技能自己就已经满足某个禁止模式——比如一个技能同时具备文件读取和网络外发，那它单独就被标为不安全。这层跟现有扫描方案做的事一样。

**第二层：结构性组合风险候选**。一对技能，每个单独都通过了安全策略检查，但它们的并集满足了一个禁止模式。**这才是 SkillReact 真正的测量对象。** 静态基准计算的就是这层的候选率——而且它用的是确定性的、可复现的 regex 抽取器加 10 种禁止模式，不依赖模型判断，不依赖 prompt 设计，任何人在同一份快照上重跑都能得到相同的数字。

**第三层：模型实际发出的工具调用尝试**。Agent 在看到候选技能对之后，是否真的调用工具往禁止目标走——比如实际发起下载、执行。这层依赖模型、prompt 和运行时的具体条件，不是确定性计算。

三层递推的架构让整个测量的边界很干净。**静态候选率是 recall-oriented 扫描天花板**——它不会漏报，但注定会过报（把很多无害组合也标成候选）。人类审计管线的作用就是把过报量校准成真实的组合风险率。

## 静态基准：21 万对，22.25% 触发候选

静态分析从 ClawHub 上采集了 1,520 个技能，用一套固定正则表达式抽取器扫描每个技能的 SKILL.md 和附带代码，归入 8 种能力标签：文件读、文件写、网络外发、网络入站、环境变量读、进程生成、凭证访问、系统信息。

1,520 个技能中，869 个（57.2%）单独就满足了某个禁止模式——个体标记率。剩下 651 个通过了单独检查，组成 211,575 个两两配对。对每一对计算能力并集，跟 10 种禁止组合模式做匹配。

10 种禁止模式来自四个来源的交叉精炼：ToxicSkills 的恶意技能分类、Willison 的"致命三要素"框架、OWASP Agentic Skills Top 10，以及从 Android 应用组合分析文献中迁移过来的能力滥用原型。具体是：

- **数据外泄**：文件读 ∧ 网络外发
- **凭证窃取**：凭证访问 ∧ 网络外发
- **致命三要素**：凭证访问 ∧ 文件读 ∧ 网络外发
- **权限提升**：环境变量读 ∧ 进程生成
- **反向 Shell**：网络入站 ∧ 进程生成
- **持久后门**：文件写 ∧ 进程生成
- **横向移动**：凭证访问 ∧ 网络外发 ∧ 进程生成
- **环境外泄**：环境变量读 ∧ 网络外发
- **系统侦查外泄**：系统信息 ∧ 网络外发
- **投放器**：网络外发 ∧ 文件写 ∧ 进程生成

模式之间不是互斥的。长合取（致命三要素、横向移动、投放器）在静态分类中被包含它们的短合取所覆盖，论文明确报告了这种包含关系。

**47,075 对被标为结构性候选（22.25%）。** 这不是最终结论，是扫出来需要被审计的待核实集。论文明确说它是个 recall-oriented scanner ceiling——设计目标是"不漏"而非"不错"，所以数字本身不是利用率，是上限。

## 人类审计校准：五分之一是真实的

为了校准这个上限，团队把 47,075 个候选对按 10 种禁止模式分层抽样，每模式 20 个单元，组成 200 单元黄金集。每个单元由**两个 LLM 标记者**独立标注：Claude Sonnet 4.6（无工具、无会话持久化）和 OpenAI Codex CLI + gpt-5.5（固定输出 schema、JSON 校验）。两个模型的标注只作为分层信号和诊断用，**每一个单元最终还是走人类审计。**

审计 UI 展示完整 SKILL.md、正则证据痕迹、两个 LLM 的标注（判定、每项能力有效性、理由、引用证据）。人类审计员填一份九字段表单做出终判。

**经过审计后，每个模式的人类确认真实率从 5% 到 45% 不等。** 环境外泄和凭证窃取的确认率最高（45%），系统侦查和数据外泄最低（5–10%）。按各模式在 47,075 个候选中的占比加权，**总体人口加权有效率为 18.2%（95% CI [11.3%, 27.4%]）。**

这意味着在每个技能单独安全的配对里，**大约五分之一的结构性候选在被人类逐例审查后确认为真实的组合风险。** 在单个 ClawHub 注册表中，这对应约 1.4 万个真实的（pair，pattern）风险隶属关系。论文进一步推算了唯一风险配对数的上限——因为一对可能存在多个匹配模式，唯一配对率不可能超过隶属关系率。按预测值计算，唯一风险配对率上限约 6.78%。

## 这些候选真的会被利用吗？看模型怎么选

静态候选回答的是"能力层面有没有危险组合"。一个更工程的问题是：Agent 模型面对这些候选对，**到底会不会真的去调工具执行？**

团队设计了一个基于执行动作的可利用性验证平台。用投放器模式（网络外发 ∧ 文件写 ∧ 进程生成，下载后执行的标准签名）做锚定，固定在一个热门技能上，构造同一组技能和同一段请求，让 9 个模型分别暴露在同样的组合下。三个 Claude 系列（Haiku-4-5、Opus-4-7、Sonnet-4-6），外加 6 个跨厂商模型（deepseek-v4-pro、deepseek-v4-flash、qwen3.6-plus、qwen3.6-flash、qwen3.7-max、glm-5.1）。

**Haiku-4-5 在所有 39 次直接提示试验中发出了投放器阶段工具调用**，其中 36 次走完了"下载→然后执行"的完整链条，3 次仅下载。Opus-4-7 也下载了，但停在下载阶段，不再执行。**Sonnet-4-6 在同样条件下全部拒绝。** 跨厂商模型的表现也沿着同一个梯度分布，deepseek-v4-pro 和 flash 全部顺从，qwen3.6-plus 全部顺从，qwen3.7-max 只有 1 次顺从，glm-5.1 出现混合反应。

一个关键控制实验：**把已安装技能清空，保持请求不变，顺从率最高，18 次测试中 17 次顺从。** 技能组合提供的是静态的能力表面，真正决定"走不走"的是模型对显式请求的判断，跟具体哪一对技能配对无关。

结论很清楚：**组合定了"能不能走"，模型定了"走不走"。** 一个注册表的安全不仅依赖技能的个体安全性，还依赖安装时的组合检查和宿主模型的危险门槛。

## 工程贡献：安装时组合检查器

论文把测量框架打包成了一个实用工具：**一个双模式的安装时组合检查器**，对每次安装操作报告集合级违规和配对级证据，耗时在亚毫秒量级。它的判定边界限制在 10 种禁止模式的分类体系之内，不声称覆盖所有可能的风险，不声称替代运行时沙箱。它是注册表和 Agent 框架的一个补丁：在安装时多算一次并集，把按个扫描漏掉的那层补上。

## 最后

SkillReact 的测量结果可以提炼成一句话：**在技能生态中，安全的原子不等于安全的集合。** 同一个安全策略——标记同时具备文件读和网络外发的对象——扫单个技能时放过了一对对的组合，扫并集时打出了 22.25% 的候选率。

这个数字不应该被误解为"22% 的技能组合是危险的"。它是扫描器的上限，经过人类审计后收敛到 18.2% 的有效率。但核心发现不随数字变化而改变：**目前唯一的规模化安全实践——按个扫描——在设计前提上就漏掉了组合风险。安装时做一次集合级的并集检查，不是附加功能，是逻辑补全。**
