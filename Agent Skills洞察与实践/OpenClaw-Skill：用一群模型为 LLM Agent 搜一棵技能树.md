# OpenClaw-Skill：用一群模型为 LLM Agent 搜一棵技能树

> 单个模型产出的技能自带偏见、碎片化、换个模型就不好用。如果把多个模型组织起来，像下围棋那样联合搜索一棵技能树呢？

## 技能的三个坑

LLM Agent 领域的技能构建正在从手工 prompt 转向自动化。SkillRL 从交互历史里学技能库，Trace2Skill 从执行轨迹里蒸馏操作模式，CoEvoSkills 用代理验证来迭代优化。这些工作解决的是"技能怎么自动生成"。

但生成出来的技能本身有没有毛病？香港理工、NTU、清华、RMIT 和北航的联合团队在 OpenClaw-Skill 这篇论文里问了一个更前置的问题。他们归纳了三种症状。

**技能碎片化。** 现有方法产出的技能本质上是孤立子任务的局部操作序列。各技能之间没有编排机制，处理不了跨步骤的长程依赖，也没有验证逻辑。你知道怎么拧一颗螺丝，不知道整台发动机怎么装。OpenClaw 这种需要协调文件、工具、网页、执行反馈和中间产物的真实环境，碎片化技能碰两下就卡住了。

**技能多样性不足。** 主流做法是让单一模型跑批轨迹，然后蒸馏技能。问题在于，单一模型有自己的解题偏好。它探索的路径天然偏向某些推理风格，漏掉的东西不少。从这样一批轨迹里提炼技能，覆盖面天然受限。

**技能迁移性差。** 这可能是最让人头疼的一个。A 模型蒸馏的技能换到 B 模型上执行，效果往往大打折扣。论文里直接展示了同一条技能在不同 backbone LLM 上的表现落差。你给 Qwen 蒸馏的技能库装到 DeepSeek 上，可能就不太灵了。

这三个问题还会互相放大。碎片化加剧同质化（你看到的永远是局部模式），同质化削弱迁移性（技能被过拟合到某个模型的解题风格上）。

## 核心方法：CSTS

OpenClaw-Skill 的核心方法叫 **Collective Skill Tree Search（CSTS）**。思路不复杂：用集体智能联合搜索、识别和组合有效技能，迭代构建一棵结构化的技能树。

两个灵感来源。AlphaGo 时代的树搜索——用搜索探索解空间而非依赖单次推理——这解决了"怎么覆盖更多可能性"。集体智能——多个模型有不同的解题风格，组织起来联合探索，能覆盖比任何单模型都广的技能空间——这解决了"怎么打破单一模型的偏见"。

CSTS 怎么运作的。

第一步，**把复杂任务拆成有序的子任务序列**。比如"用提供的 GitHub 仓库复现论文 Table 2"，可以拆成：定位目标实验 → 环境与仓库配置 → 构建执行命令 → 运行与验证。这一步定义了技能树的深度——每个子任务是树的一层。

第二步，Collective Skill Node Generation（CSN-Gen）。对于某个子任务，多个异构模型各自独立去解，产出一批执行轨迹。每条轨迹被一个共享技能合成器压缩成候选技能节点，里面包含适用范围、所需输入、推荐动作、预期输出、验证标准和恢复策略。不同模型会走不同的解题路线、碰不同的失败模式、触发不同的验证机会，所以这批复候选节点天然多样化。

第三步，Collective Skill Node Assessment（CSN-Assess）。有了一堆候选，怎么挑？两把尺子。

第一把尺子是**集体质量评分**。多个评判模型各自给候选技能打分——是否清晰、可执行、完整、相关——然后取平均。比单评判者稳健。

第二把尺子更有意思：**集体可迁移性评分**。逻辑很简单：从模型 A 轨迹里蒸馏出来的技能 s，对 B、C、D 有没有用？做法是把 s 作为额外上下文喂给其他模型，让它们去解同一个子任务，看执行得分。如果 A 产出的技能能让 B 也做好，说明这个技能捕捉到的是超越特定模型的通用操作模式，不是 A 的解题癖好。

最终得分 = 质量分 + 可迁移性分。每个子任务层选得分最高的节点，所有层的胜出节点串成一条完整技能路径。

CSTS 产出的是一棵分层的技能树，不是一条孤立的技能。每层对应一个子任务，每个节点是一个候选，一条从根到叶的路径就是一套完整的技能组合方案。这个设计让技能之间有结构、有依赖关系。Agent 推理时可以根据当前状态在树中选择不同分支，不用盲从单一路径。

论文用这棵技能树生成了约 2000 条 SFT 训练数据。每条样本包含三个部分：复杂任务描述、选中的技能路径、完整执行轨迹。先用这批数据做监督微调，让模型学会基本操作流程。

## CSRL：让不同技能在同一个子任务上互相 PK

SFT 教模型"这样做是对的"。但它不教"多种可行做法里哪个更好"。CSTS 为每个子任务产生了多个候选技能，评估阶段只选了一个，但次优的在某些场景下可能更合适。模型需要学会选择。

CSRL（Collective Skill Reinforcement Learning）干的就是这个。

对同一个子任务，CSRL 让模型在多个候选技能条件下各采样一批 rollout，形成一个集体技能条件组。计算 advantage 时，**不在同一条技能内部做归一化，而是在整个集体组上跨技能归一化**。用 GRPO 风格的 clipped objective，同一子任务下、不同技能产生的 rollout 彼此竞争。

效果很直接：某条技能条件下的 rollout 表现好，action 概率就升；不好就降。模型学会的不是背诵某个技能，而是从多个候选里找到当前最合适的。

## 实验数据

两个基准：QwenClawBench 和 PinchBench，覆盖工具使用、文件操作、代码执行、网页交互、多步决策。

QwenClawBench 上，四个 backbone 全部提升。Qwen3-4B：7.0 → 12.8（+5.8）。Qwen3-8B：11.5 → 15.8（+4.3）。Qwen3.5-4B：31.5 → 41.2（+9.7）。Qwen3.5-9B：34.5 → 44.9（+10.4）。提升幅度最大的两类是 SVM（33.2 → 70.9）和 CS（30.2 → 78.4），考察的正好是持续多步工具协调和验证反馈——单一技能最容易碎片化的地方。

PinchBench 上，23 任务版本，OpenClaw-Skill 9B 的最好成功率 67.5% → 72.8%，平均 53.8% → 58.9%。123 任务版本，最好 61.1% → 68.2%，平均 47.1% → 53.6%。平均提升幅度大于最好提升幅度——**CSRL 提升的是执行鲁棒性，不只是峰值**。

消融实验也很清楚。Qwen3.5-9B：base 34.5 → +CSN-Gen 39.8（+5.3）→ +CSN-Assess 42.8（+3.0）→ +CSRL 44.9（+2.1）。每个组件都有独立贡献。

## 值得琢磨的几个点

CSTS 的计算开销不小。每个子任务要跑 N 个模型生成轨迹，J 个评判模型打分，N-1 个模型做可迁移性验证。论文在 2K 条 SFT 数据的规模下完全可控，但如果想扩展到更细粒度的子任务分解或者数量级更大的任务集，开销是线性增长的。论文没有谈这个。

技能树的更新也是个问题。树是在训练阶段构建的，相对静态。真实系统里任务分布、工具接口、环境配置都在变，技能树怎么感知这些变化然后更新自己？论文没碰这个方向。

还有一个：CSTS 目前用的是 Qwen 家族内部不同版本，算是族内多样性。如果换成跨家族异构模型集合——Qwen + DeepSeek + Llama + Gemma——候选技能的多样性会不会有质的提升？这还是个开放问题。

## 我觉得有意思的地方

这篇论文做了一件挺聪明的事。没有设计更复杂的技能表示，没有堆更大的训练集。它把多个模型的集体探索组织成一棵可搜索的树，然后靠跨模型验证来筛出最具泛化性的技能。**Tree Search + Collective Intelligence + Cross-model Transferability**，这三样凑在一起，技能构建从单打独斗变成了群策群力。

对于在搭 Agent 系统的工程师，这篇论文最直接的启发可能是：构建技能库的时候别只盯一个模型。让不同模型互相验证、互相补充——产出的技能大概率比你想象的好。
