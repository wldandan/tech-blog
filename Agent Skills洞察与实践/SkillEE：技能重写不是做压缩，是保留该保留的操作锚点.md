# SkillEE：技能重写不是做压缩，是保留该保留的操作锚点

> 技能太长，直觉做法是压缩：把文本变短、token 变少。中科大和华为的联合团队在技能重写上做了一组受控实验，发现了一个反直觉的结果：**把技能文档简短化之后，Agent 执行任务的总 token 消耗反而涨了 14%。** 理由很简单：技能里那些看起来像"废话"的解释性文字，很多实际上嵌着 Agent 在探索、调试和错误恢复中需要的操作锚点。删了锚点之后，技能本身变短了，但 Agent 在执行中要多做额外的 API 探索、代码修复和工具重跑才能补上这些缺失的信息。

## 压缩技能的直觉为什么是错的

技能是有结构的操作文档。里面不只有纯粹的过程描述，还有 API 构造函数、命令行标志、验证阈值、文件格式约定、公式和恢复规则。这些内容在语义上不是最重要的：一个技能的核心内容可以用三句话概括。但在操作上，它们是 Agent 在执行时用来定位、校验、纠错的锚点。

**把一个技能从 2000 token 压缩到 800 token，直接省了 1200 token 的技能注入成本。但如果压缩过程中删掉了一个 API 构造函数，Agent 在运行时花了额外的 3000 token 去尝试不同的 API 调用、回滚错误的参数、最终才定位到正确的用法。** 节省的技能 token 远小于新增的执行 token。

SkillsBench 上的实验中，一个工作流导向的重写策略将技能文档缩短了，但下游 Agent token 使用量飙升到原始基线技能的 1.14 倍。技能更短了，总成本更高了。

**这个现象的核心原因是：技能里大量内容不是为"直接告诉 Agent 答案"存在的，而是为"减少 Agent 在执行中的不确定性"存在的。** prompt 压缩的优化目标是减少输入 token 数。技能重写的优化目标应该是减少总执行成本，而总成本 = 技能注入 token + Agent 执行过程中的全部 token 消耗。两个目标函数在操作锚点上的决策方向经常是相反的。

## 什么样的锚点对什么样的任务重要

论文设计了三类信息保留策略来分别测试不同类型锚点的作用。

**API/代码锚点策略。** 保留 API 签名、导入语句、对象构造模式、命令示例和代码片段。适用于实现密集型技能：数据工程、代码转换、API 集成类的任务。**这类技能的文本中，代码 token 的比例天然高，一个构造函数的差异可能相差数轮 execute-debug 循环。**

**工作流守卫策略。** 保留有序的验证步骤、检查点、失败恢复提示和边界条件。适用于多步验证流程：安全审计、合规检查、多阶段数据处理。**这类技能的锚点不在代码里，在"先做什么后做什么、哪一步失败了该回退到哪"的顺序逻辑里。**

**规则/公式锚点策略。** 保留公式、阈值、schema 定义和打破平局的规则约定。适用于规则密集型任务：数学计算、评分排序、配置生成。**这类技能的文本看似很短，但每一个阈值和公式都可能在执行中触发截然不同的决策分支。**

三种策略分别在三类任务上取得了各自的性能-成本最优，但没有任何一种策略在所有任务上统一最优。**技能重写不是一个"找到最好的压缩模板"的问题，而是一个"根据任务结构和技能内容选择保留哪种锚点"的问题。**

## 从策略到选择器：让策略学会"这个任务该用哪种重写"

基于这个发现，论文训练了一个轻量级的任务条件化策略选择器。输入是任务-技能结构特征（技能数量、总 token 数、代码 token 比例、API 使用情况、规则公式密度、主导技能 archetype、任务领域），输出是在候选重写策略中选择哪一个。

**在主要的留出评估中，策略选择器在维持验证器质量的前提下，将总执行成本降低了 7.0%，下游 Agent token 成本降低了 6.0%。** 将同一个冻结的策略选择器在跨模型 Stack 上迁移，总成本降低平均达到 14.7%，Agent token 成本降低 13.7%。

选择器的迁移能力说明了一个关键点：**"哪种锚点对哪类任务重要"这个规律本身是跨模型成立的。** 它不依赖于具体推理模型的表现差异，而是根植在任务的操作结构里。一个代码密集型的技能对 GPT 和对 Claude 都需要保留 API 锚点。

## 这个框架定义了什么

论文把整个方法称为 SkillEE（Skill Economic Engineering）。不只是一套重写策略，而是一个包括了技能画像、锚点保留策略、审计和修复、受控评估和策略学习的完整框架。

技能画像 ϕ_τ 抓取每个任务-技能对的关键结构特征。锚点保留策略 a ∈ A 各自定义了一类操作的保留规则，不是统一的压缩模板。审计和修复在重写后做轻量检查（token 比例、代码保留率、锚点遗漏检测），误删的锚点被从源文档中恢复。

**受控评估把任务指令、环境和验证器固定住**，变动的只有技能文档本身。这使得所有观察到的质量-成本变化可以被唯一归因到技能的重写方式上。

经济指标设计也突破了单纯的"技能 token 数"。**总执行成本 C^total = C^skill（技能注入 token）+ C^agent（Agent 整个执行过程中消耗的 token）。** 另外还有执行超限率 ρ 来衡量因技能信息缺失导致的失败重试。只盯着 C^skill 看，会奖励一个让 Agent 跑崩了的短技能；只盯着 task quality 看，会奖励一个 5000 token 的冗余技能。

## 几个值得注意的点

**当前的重写策略是预定义的，策略选择器的动作空间也限定在预定义策略中。** 新的任务类型或技能结构可能需要新的人工定义策略来覆盖。SkillEE 的框架让发现性研究（"这个任务适合哪种锚点策略"）变成了搜索和学习的问题，但策略本身的生成目前不在自动化范围内。

**重写检查器目前是轻量级的规则驱动审计。** 对锚点遗漏的检测依赖预定义的 token 匹配模式：API 签名、公式模式、工作流关键词。如果某个任务的锚点类型不在当前检查器的匹配模式中，误删的锚点会漏过审计直接进入 Agent 上下文。

**跨模型的迁移实验目前测试了 GPT 家族和 Qwen 家族的直接迁移。** 两个推理模型家族在 API 使用模式和执行行为上可能存在结构性差异（比如一个倾向于更长的探索式 debug、另一个倾向于更短的工具调用链），这会使同一个锚点保留策略在不同模型上的效用产生偏移。选择器的跨模型迁移性能是否能在更多样的模型家族上保持，需要更系统的实验。

## 最后

SkillEE 把技能重写从"怎么把技能变短"掰成了"怎么让技能在执行中花最少的钱"。

压缩可以降低前端成本，但错误压缩会将成本从技能 token 转移到执行 token，而且转移系数不是 1:1：删掉一个 50 token 的 API 锚点可能导致几千 token 的额外探索和修复。**技能重写的优化目标不是技能长度，而是从技能文本到任务完成的全程经济账单。**
