# SkillAudit：别光看 star 和 README，让技能先跑一遍审计再决定装不装

> 技能市场几千个包。你决定装不装，看的是什么？star 数、下载量、README 文笔？SkillAudit 换了个思路：自动生成审计，跑完看报告。

## 技能安装决策缺了一个环节

Agent Skill 生态不小了。Anthropic 发布格式之后，GitHub 和社区市场上的技能包从几十个涨到了几十万个。每套包里有 SKILL.md 操作指南、脚本、依赖和元数据。Agent 加载了，理论上就能处理某一类任务。

但你在技能页面上决定装不装的时候，手上没有任何基于实证的判断依据。主要看 README 怎么写、star 有多少、下载量多大。没跑过，不知道到底有没有用。

两个原因。一是现有的评测基准全是固定任务集。AgentBench、τ-bench、The Agent Company，都是把 Agent 扔进预定义任务里跑分。SkillsBench 虽然评的是技能，任务也是预先搭好的。你手里一个冷门技能，这些基准根本覆盖不到。二是任务成功把技能的真实贡献和 backbone 的能力搅在了一起。用了强模型本来就能做，技能加了多少分说不清楚。

东北大学、港城大、西北大学、复旦、中科大、NUS 等一堆机构的联合团队在 SkillAudit 里做了一套端到端的审计框架。思路很简单：给定任意一个技能包，自动生成与之能力对齐的评测任务，在隔离沙箱里跑，出报告。覆盖效用、效率/成本和安全性三个维度。

## 怎么做的

分五步。

先收集和规范化。扫外部技能包，提取 SKILL.md 和配套文件，归一化。有根目录下的 SKILL.md 就接收，不管什么语言或布局。

再生成评测方案。分两类。效用方案：LLM 读 SKILL.md，提取核心能力和操作流程，为每个技能生成三个有代表性的使用场景。安全方案：静态语义扫描器扫整个技能包，覆盖 21 种风险模式，5 大类（prompt 注入、凭据访问、不安全文件/网络行为、隐藏指令、供应链），分高/中/低严重级别。零风险就跳过动态测试，有发现就为每个风险生成一个安全探针。

然后构建任务。每个效用场景编译成两个任务，一个不带技能，一个带。**共享相同指令、输入、环境和评分标准，唯一区别是技能是否可用。** 配对的，性能差异只能归因到技能本身。安全探针各自编译成单独的带技能风险任务。

接着沙箱执行。隔离沙箱中运行，内置日志抓 Agent 每一步的工具调用和结果、执行时间和 token、文件变更、外部网络请求。每次试验产出一个证据包。

最后自动评分加报告。效用维度用 pass-rate gain（PRG）：同样的检查清单，带技能和不带技能各算通过率，PRG 是差值。效率/成本记录时间和 token 相对节省。安全维度综合静态扫描的存在性判定和动态执行的利用性判定，加权惩罚后出 10-100 的安全分。≥100 是 Pass，80-99 是 Caution，低于 80 是 Risky。

还有一个浏览器扩展，在 GitHub 技能页面上直接展示审计报告。把评测从离线基准变成了发现时的决策工具。

## 226 个真实技能跑完，几个发现

公开技能仓库里收了 226 个排名靠前的真实技能包，23 个职业类别，6 个 Agent-模型配置。

效用的天花板效应非常明显。六个配置的 mean PRG 和 no-skill baseline pass rate 之间 r = -0.90。no-skill 基线越强，技能带来的额外通过率越低。Codex/GPT-5.1 的 baseline 只有 0.630，PRG 高达 0.248；Claude Code/Sonnet 4.6 的 baseline 到了 0.799，PRG 掉到 0.164。**强模型已经把坑填了，技能能额外解锁的分数不多。**

同 harness 内部也一样。Codex 配 GPT-5.1 的 PRG 高于 GPT-5.4，Claude Code 配 Sonnet 4 的 PRG 高于 Sonnet 4.6。骨架越强，技能边际价值越被压。

但职业类别之间差异很大。教育辅导和社区社会服务在所有配置下 PRG 一直高位，建筑清洁和农林渔业始终垫底。中等类别如办公室行政管理，同一个 backbone 在 Claude Code 和 OpenCode 下 PRG 差了 0.31——技能效用不光看技能本身，还跟 harness 有交互依赖。

整体效率/成本增益（ECG）是 -0.186。23 个类别里只有办公室行政管理（+0.085）和农林渔业（≈0）不是负的。绝大部分技能让 Agent 多花时间和 token。这不意外。技能多了上下文，操作指南、示例、注意事项，Agent 要消化。多读多想多检查，换来了更高的通过率（overall PRG = 0.183），但时间多了 13%，token 多了 24%。

安全方面，17 个技能（7.5%）被标为 Risky。128 个满分，76 个 Caution。静态扫描器高/中严重级别召回率 97.0% 和 96.0%，宁可多报不漏报。动态执行结果因骨架而异：Claude Sonnet 4.6 的 confirmed rate 只有 23%，agent_refused 率远高于其他——Claude 更主动拒绝风险操作。GPT-5.4 的 confirmed rate 高达 53%，但低严重级别的触发率（62%）甚至超了高严重级别（46%），说明 GPT-5.4 的高触发反映的是泛化任务完成能力，不是对高风险操作的针对性敏感。

效用和安全几乎不相关（Pearson r = 0.111）。一个技能可以很有用又可以很不安全。这两个维度得分开看。

## 局限

226 个技能，23 个类别，相对几十万技能的生态还是冰山角。

评测任务自动生成的质量直接依赖 LLM 读 SKILL.md 的能力。写得模糊、不规范、信息不全的技能，跑出来的场景未必反映真实能力。

效率/成本只报告时间和 token 相对节省，没含沙箱基础设施开销和技能包下载/安装成本。要全面 TCO 还需要更多维度。

## 我觉得做得最对的一件事

SkillAudit 把技能评估从"在固定基准上跑分"转向了"针对每个技能生成定制化审计"。这个转向很像软件工程里从单测到 fuzzing——不是一套预设 test case 测所有代码，而是根据代码的结构和声明自动生成针对性输入。

PRG 这个指标也干净。传统评估就看"带技能的 Agent 得多少分"。PRG 算差值，减基线。骨架强度的影响被剥掉了，换更强的模型技能看起来作用更小，不是技能变差了，是基线高了。

浏览器扩展把审计报告嵌进技能发现页面这一点也实用。评估从离线研究活动变成了 adoption-time 决策辅助。你在下载之前就能看到效用、效率和安全评级。这种从"给研究者看"到"给开发者决策用"的位移，是技能生态在往前走的一个信号。
