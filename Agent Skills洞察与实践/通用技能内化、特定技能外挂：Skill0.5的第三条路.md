# 通用技能内化、特定技能外挂：Skill0.5的第三条路

![alt text](image-3.png)

# 一、扩张的技能库，和一条死胡同

给LLM Agent配技能已经是标配了。你把"遇到死胡同就回溯"、"加热完直接送到目的地别冷却"这些程序性知识写成自然语言，需要时检索注入prompt，Agent照着执行。这套玩法一开始挺好用。

问题出在后面。技能库不会停在原地 —— 用户不断贡献新技能，规模持续膨胀。更麻烦的是，新技能对应的常常是Agent训练时**完全没见过的任务场景**。此时Agent面临的是典型的 **OOD（Out-of-Distribution，训练没见过的）** 问题：技能是新的，任务也是新的，你必须立刻上手。

现有的技能训练方法在这件事上走了两个极端。

- **Skill 1.0，全外化**：把通用技能和任务特定技能一股脑塞进上下文窗口。刚开始信息充分，但随着技能越堆越多，**prompt篇幅爆炸**。LLM在几千token的技能海洋里开始迷路，尤其长horizon任务中，不相关的技能噪音淹没了真正有用的那几条。
- **Skill 0.0，全内化**：把全部技能吃进模型参数，推理时零上下文开销。听着很美，参数容量有天花板，更致命的是**新旧技能冲突**。模型内化了一个"冰箱→冷却"的**ID（In-Distribution，训练见过的，如ALFWorld的Pick、Cool、Clean等常规家务场景）** 经验，结果遇到OOD指令"加热后直接放进冰箱"，它心里读懂了技能文本，手上做的还是冷却—— **执行幻觉**。

认知科学里有个经典发现（Sweller, 1988）：**人掌握技能是先建基础schema，再在此基础上处理具体规则**。道理其实一样，Agent得先把"怎么探索环境"、"怎么检查目标是否完成"这些通用逻辑吃透，才能灵活驾驭"这件任务具体怎么做"的临时指南。

但现有方法把**两类技能统一处理**了。全外化让通用技能的文字噪音遮蔽了特定技能的信号；全内化让易变的特定技能固化成参数，遇新必冲突。

**Skill0.5**做的事就是**打破这个死胡同：通用技能内化进参数打底，任务特定技能外挂灵活调用**。名字本身就在宣告立场 —— 不选 0，不选 1，走第三条路。

# 二、为什么不能一视同仁

先看清楚两类技能到底差在哪。

**通用技能**跨域适用，像"系统化探索环境"、"遇到死胡同回溯到上一个分叉点"、"检查所有目标是否完成后再停止"。它们的特点是**稳**（更新频率低，覆盖范围广）和**长**（每条几百token，十多条就是几千）。ALFWorld的通用技能集有12条，WebShop有15条。

**任务特定技能**是另一个物种。比如"加热后直接送到目的地，不要冷却"、"用一次USE命令就能打开台灯检查物品"。**细粒度、短小精悍**，每条可能就几十token。但**更新频繁**，不同任务域之间完全独立。

问题出在"统一处理"上。全外化把十几条通用技能 + 几条特定技能全塞进上下文 —— ALFWorld上SkillRL的prompt里**通用技能占了约88%，特定技能只剩12%的空间**。全内化把频繁更新的特定技能固化进参数 —— **ID和OOD的技能规则互相抵触**，模型从参数层面就精神分裂了。

这其实不是"取个折中"的问题。两类技能服务的对象不同。通用技能管"**怎么思考**"，这是不该变的底层逻辑；特定技能管"**这次做什么**"，随任务切换。就像操作系统的内核不会每次打开新应用就重装一遍，但也没人会把每个app的图标位置写死在ROM里。

所以**Skill0.5的主张**是功能解耦：**通用技能该内化**，蒸馏进参数，变成模型的"本能"；**特定技能该外化**，需要时检索注入，保持对变化的即时响应。

# 三、Skill0.5怎么做的

框架分成两阶段，整个流程像个自适应流水线。

![alt text](image.png)

## 3.1 第一步：测一测你有多菜，难度感知路由

每个训练step，Agent先用自己的标准配置（只带任务特定技能，不带通用技能）对batch里每个任务跑**G条轨迹**。算出每个任务的经验通过率$p_i$。然后分桶：

- $p_i = 0$：**困难任务**。G条轨迹全失败，连一丁点reward都没摸到。
- $0 < p_i ≤ η_t$：**中等任务**。有点成功率，但还没达到近期平均水平。
- $p_i > η_t$：**简单任务**。超额完成任务，在当前batch里算优等生。

门槛$η_t$不是拍脑袋定的，而是**过去W个step的batch级通过率的滑动均值**。所以"中等"和"简单"的边界会随Agent整体水平变化 —— 初期大量困难，后期简单占比逐渐上升，路由自己在适应。

## 2.2 第二步：对症下药，三层分级优化

路由标好标签后，同一batch里的任务各走各的优化路径。**三种损失互斥** —— 一个任务只触发一个。

- **困难任务 → 特权蒸馏**：这时候Agent连基础推理逻辑都不够。常规RL碰到**全零reward直接哑火 —— 方差没了，梯度也没了**。Skill0.5 的策略是"开卷考"：给Agent额外加上通用技能，让它重新执行任务。收集到成功轨迹后，用**token级JSD散度**让学生策略（没通用技能的）去模仿教师策略（有通用技能的）。**教师不动（stop-gradient），学生往前追**。本质上是用成功经验给模型补课 —— 补的不是"怎么做这道题"，而是"怎么想这类问题"。
- **中等任务 → 标准 GRPO**：这层是RL的**sweet spot**。Agent有点基础了但还不稳，通过率有上升空间。直接复用Phase 1采到的轨迹做GRPO策略梯度优化，把成功率往上顶。
- **简单任务 → 反捷径诊断**：成功率高了不一定是好事，Agent可能走了**捷径**：它学会了**任务指令→动作的直接映射**，根本没看你给它的技能。ID上数据好看，换到OOD马上现原形。

怎么检测？对每个简单任务，额外跑一组**拆掉技能的轨迹（无技能提示）**，算出无技能通过率。然后看 **利用增益 = 有技能通过率 − 无技能通过率**。这个差值衡量了Agent到底在不在用技能：差值大说明技能确实在起作用，差值小甚至为负说明Agent根本没把技能当回事。

把这个利用增益做成**任务级调制器**，叠到标准GRPO的轨迹级优势上。走捷径的任务得到负调制，整个batch里它的优化信号被压制。真正依赖技能的任务得到正调制，正常步调优化。用论文的原话："**轨迹级优势衡量这条rollout在组内的相对质量，利用优势衡量这个任务在全局中对技能的依赖程度**。"


# 四、两个最值得细看的设计

## 4.1 设计一：特权蒸馏打破零梯度

困难任务是Agent RL的阿喀琉斯之踵。**GRPO的组内归一化依赖reward方差**，全零reward意味着全零方差，优势信号消失，训练卡死。SkillRL、SKILL0都只能靠**随机探索硬扛这段冷启动**，慢慢撞出第一条成功轨迹。

Skill0.5换了个思路：**既然学生答不出，就让老师先答**。给Agent加上通用技能$S_G$（特权提示），在"开卷"条件下收集成功轨迹，然后用**token级JSD散度**让学生策略（无通用技能）模仿教师策略（有通用技能）的每一步推理，**stop-gradient保证教师不动，学生单向对齐**。结果：通用技能被蒸馏进参数，不是死记硬背，而是作为**推理能力内化**的。

## 4.2 设计二：反事实诊断防捷径

简单任务的通过率高不一定是好事，Agent可能完全没看技能，直接背下了"任务→动作"的映射。这在ID上查不出来，效果甚至更好（省去了读技能的开销），**一换OOD就崩**。

Skill0.5的做法很直接：**拆掉技能再跑一组对照实验**，计算**利用增益 = 有技能通过率 − 无技能通过率**。增益大 → Agent 确实在依赖技能；增益小或为负 → Agent 在走捷径。利用增益的滑动均值$ū_t$作为动态锚点（与路由门槛$η_t$共用$W=5$窗口），低于锚点的任务，其GRPO优势被负调制分量拉低 —— **这个任务在batch里被降权了，不是因为轨迹差，而是因为它没在真正用技能**。

核心矛盾由此解开：**在ID上逼Agent走"读技能→理解→执行"的长链，短期可能牺牲效率，换来的是OOD上面对全新技能时真正读懂并执行**，而非激活错误的ID记忆。


# 五、实验结果

### 分数说话

ALFWorld的ID/OOD划分方式为：{Pick, Cool, Clean} ∈ ID，{Look, Heat, Pick2} ∈ OOD。skill-augmented RL方法的对比如下：

![alt text](image-1.png)

几个值得注意的点：

- OOD差距拉得很大。Skill0.5比最强的skill-based baseline（SkillRL）高出 **13.2个百分点**。
- ID上也有提升（**+2.3pp**），解耦策略没牺牲分布内性能。
- SLIM代表了另一种探索方向，**动态管理技能生命周期**，决定保留还是退役每一条技能。但它的统一退役策略丢掉了一些还需要的通用技能，导致OOD表现（35.8）远低于Skill0.5。
- WebShop上呈现了一样的模式：Skill0.5 ID 40.4、OOD 40.6，**OOD反超ID**。全外化和全内化都没做到这一点。

prompt-based方法（Zero-shot、Few-shot、ReAct等）在ALFWorld上平均落后Skill0.5超过 **45个百分点**，在WebShop上落后约 **28个百分点**。光靠往prompt里塞技能，不够。

### 拆开看：消融实验

![alt text](image-2.png)

"只利用"**直接崩了**。没有内化打底，Agent连基本能力都没有 —— 给不给技能都失败，**利用增益恒为零**，反事实诊断形同虚设。

"只内化"没那么惨但上限被卡住。内化提供了推理基础，缺少反捷径机制导致OOD适应性打折扣。**两个组件分开能用，合在一起才有明显跃升**。

# 六、总结

Skill0.5解决的问题很具体：**技能库在动态膨胀，OOD不是意外是日常**。解法在思想上简单，**通用技能和任务特定技能性质不同，就该不同对待**，但在RL训练框架里让内化和利用协同运作，需要**难度路由、特权蒸馏、反事实诊断**三层机制精密咬合。

效果摆在那。OOD上**13 个百分点的提升**，WebShop 上**OOD反超ID**。说明这不是修修补补的边际改进。

当然也有局限。目前只在文本交互仿真环境上验证（ALFWorld和WebShop）。代码生成、多模态、开放网页导航这些更复杂的场景还没覆盖。技能库需要预先手工定义，自动化技能发现和演进是另一回事。基座模型是7B规模，更大模型上的scaling行为未知。

Agentic RL领域过去一年的进展主要集中在reward工程和优化稳定性上。Skill0.5给的线索是另一个方向的：**下一个突破也许不在于怎么给更强的reward信号，而在于重新想清楚"技能"这件事本身**。
