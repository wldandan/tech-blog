# Skill Coverage：技能文档写了一大堆，Agent 到底练到了几条

> 任务完成不代表技能被充分测过。代码覆盖率问的不是"测试过没过"，是"哪些行跑到了、哪些没跑到"。技能也需要一个这样的指标。

## 任务成功 ≠ 技能被验过了

Agent Skill 正在变成标准的能力交付物。技能文档里写操作规范、工具用法、常见错误、交付标准，Agent 推理时加载，按图索骥。

评估技能的主流方式很直接：配一个技能，跑一批任务，看成功率。SkillsBench 就这么做的。任务做完，技能就算有用。

MBZUAI 和利物浦大学的团队在 Skill Coverage 这篇论文里提了一个反问：任务完成了，但技能文档里写的那么多条行为规范，到底哪几条被真正执行到了？哪几条从来没被触发过？

举个例子。一个 Excel 技能文档里写了"始终用 Excel 公式而不是 Python 计算""交付的模型必须零公式错误""修改已有文件时不要强加标准化格式"。Agent 完成了"分析基金持仓"任务。成功。但它有没有用到 Excel 公式？交付的模型有没有公式错误？如果任务压根没涉及公式计算，那两条规范就完全没被测到。

这就是技能覆盖率要解决的问题。它借鉴了软件测试里代码覆盖率的概念——**不问你"测过没有"，问你"哪些行为约束被实际执行到了，哪些还是空白"。**

## 从技能文档里提取"可测试的行为约束"

技能文档是自然语言写的，操作指南、背景说明、元数据、示例、规范全混在一起。不能把整篇文档当分母，很多内容在执行中不会也不该被触发。

论文的做法是从技能文档中提取 Skill Behavior Constraint（SBC），一条可溯源、可观测、EARS 风格归一化的条件-行为规则。每条 SBC 必须满足四个条件：有明确的源文本支撑，施加了义务/禁止/工作流/交付标准，能从执行证据中判断，表达单一独立的行为约束。

比如原始技能文本是"Always use Excel formulas instead of calculating values in Python and hardcoding them"，提取出来变成：**When adding calculated values to a spreadsheet, the agent shall use Excel formulas instead of calculating values in Python and hardcoding them.** 条件清晰，行为明确，可验证。

文档里的背景介绍、作者信息、维护记录不产生 SBC。SBC 的目标不是复现技能文档的完整语义，是把技能里可测试的部分投影到一个有限、可审计的分母上。

提取分两步。LLM 从技能全文提取候选 SBC，人类审计员逐条审核——检查源文本有没有支撑、是否约束了行为、是否可观测、是否是原子性的。归一化为 EARS 风格。最终产出的约束集就是覆盖率分母。

## 覆盖的定义：两道门都要过

一条 SBC 被"覆盖"需要同时满足两个条件。第一，执行轨迹实例化了 SBC 的适用条件。比如一条关于公式保留的约束，只有在轨迹确实编辑了包含公式的电子表格时才适用。第二，轨迹中有足够的可观测证据让评判者能够判断这个约束是被满足还是被违反。

两个都是二值。都通过标记 Covered，任一不通过标记 Uncovered。关键点是：**Covered 不要求 Agent 满足了这个约束。** 违反了也算 Covered，只要证据充分到可以判断对错。没被覆盖意味着这次执行根本没有提供判断这个行为的信息。

Uncovered 不代表 Agent 做错了。它只代表这条约束在这次执行中没机会被检验。这一点很重要。

评判器用了 DeepSeek V4 Flash。先用 Claude Code + Opus 4.6 的轨迹集采样 350 组约束-轨迹对做人类标注验证，LLM 评判和人类标注的一致率 **93.71%，Cohen's κ = 0.874**。验证通过后才上全量。

## 结果：SkillsBench 只覆盖了不到 44%

78 个源任务，7 个 Agent-模型配置（Gemini CLI + Gemini 3 Flash/Pro、Codex + GPT-5.2、Claude Code + Opus 4.6/4.5/Sonnet 4.5/Haiku 4.5），每个配置三条轨迹。

平均覆盖率最高 Gemini 3 Flash（43.98%），最低 Opus 4.6（39.90%）。所有配置都在 39.90-43.98% 之间。基准执行覆盖的技能行为约束不到一半。

按技能粒度看更直观。162 个技能中，只有 8-22 个达到 90-100% 覆盖率，41-52 个落在 0-30%。不同配置略有差异，量级一致。

按任务成功/失败分组，成功的执行多覆盖了平均 8.96 个百分点。但即使在成功执行中，覆盖率也只有 52.86-61.04%。任务成功了，仍然有大约四到五成的技能行为约束完全没被检验到。反过来，失败的执行也能覆盖 44.82-50.62%——任务可能失败，但执行中仍然触发了相当一部分技能规范。

## 和代码覆盖率一样，这是个可操作的信号

覆盖率的价值不在数字本身。在 Uncovered 的部分可以精确定位。

如果一条约束没被覆盖是因为没有任务触发它的适用条件，加一个任务就行。如果适用条件被触发了但轨迹里没足够证据判断，加强轨迹捕获、加文件对照检查、加断言。每条 Uncovered 的 SBC 都带着源文本位置和缺失的证据类型。基准设计者补任务，技能作者修订含糊的规范。

论文把这个叫"evidence map"——不是给技能打分，是告诉你当前基准下，技能的哪些部分还没被光照到。

## 局限

只在 SkillsBench 上做了验证。技能来源、风格、复杂度都影响 SBC 提取质量和覆盖率结果。扩展到其他技能生态怎么样，还不知道。

SBC 提取依赖人类审计。这一步把分母控在了"经得起审计"的范围内，但也意味着流程还不能全自动。怎么在保持审计质量的同时降人工成本，是落地到工程实践的关键。

## 我觉得最有意思的

这篇论文把技能评估从"好不好用"往前推了一步，到了"**你测到的那个'好用'，到底覆盖了技能文档里的多少内容**"。这和软件工程里从"测试过没过"到"哪些代码行没被跑到"是同一方向的进步。

一个很直接的工程启发：维护 Agent 技能仓库时，试着从每条技能文档里提取行为约束清单，对照执行轨迹逐条标注覆盖情况。这张表能告诉你两件事：技能文档里哪些内容写了但从来没被验证过（可能是冗余），你的测试任务集在哪些行为维度上留了空白。
