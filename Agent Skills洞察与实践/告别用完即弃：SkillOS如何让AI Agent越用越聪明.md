# 告别"用完即弃"：SkillOS如何让AI Agent越用越聪明


> 你有没有想过一件事：为什么你用的AI Agent，每次换个新任务就要从头摸索？上周刚学会的操作，这周又忘了。人和动物能从经验里学东西，Agent为什么不行？
>
> Google Cloud AI Research联合UIUC和 MIT甩出了一篇SkillOS，回答的就是这个问题。他们的思路很有意思 —— 不折腾大模型本身，而是训练一个**8B参数的小模型来当"技能管家"**。结果呢？**Agent性能提升近10%，操作步数还少了6%**。最绝的是，这个8B的管家在管理技能这件事上，直接**碾压了Gemini-2.5-Pro**。
>
> 我觉得这是近期Agent方向最值得读的论文之一。下面拆开聊聊。

# 一、问题出在哪：为什么Agent学了就忘？

现在的LLM Agent有个比较尴尬的问题：**单次任务很能打，但换任务就重置。** 你今天让它帮你订机票，它完成得漂漂亮亮；明天再让它订，还得从头走一遍相同的探索流程。

根因是什么？**Agent没有一套机制把过去的经验变成可复用的"技能"。**

学术界管这个叫**程序性记忆（Procedural Memory）**。就像你学会了骑自行车就一辈子不会忘，Agent也应该能把成功经验存下来、在需要时调出来。而**技能（Skill）** 作为程序性记忆的载体，模块化、可编辑、好组合，自然成了这个方向的研究热点。

Anthropic率先搞了SKILL.md格式——每个技能就是一个文件夹，里面有YAML元信息和Markdown指令。社区基本都接受了这个设计。但格式只是皮囊，真正难的是“**策展”（Curation）** —— 什么时候新建技能？什么时候更新旧的？什么时候删掉过时的？

现在业界的做法分两派：

- **人工策展**：让专家手写技能（比如Anthropic官方的Skills仓库）。问题是没法规模化，任务种类一多就崩。
- **启发式规则**：写死一套提示词控制记忆操作。问题是收不到下游反馈，技能质量没法根据实际效果来调。

SkillOS问了一个很好的问题：**能不能用强化学习（RL）训练一个专门的技能策展人，让它根据Agent在后续任务中的真实表现，自己学会怎么管技能库？**

# 二、SkillOS怎么做的：把策展变成一个RL问题

SkillOS的架构本质上就两个角色，分工很清晰：

**1. Agent Executor（冻结的执行器）**：只管干活。从一个叫**SkillRepo**的外部技能库里检索相关技能，然后照着技能的指引执行任务。整个过程中参数不变。

**2. Skill Curator（可训练的策展人）**：只管管技能。执行器每完成一个任务，它就观察完整的执行轨迹，然后决定做三种操作之一：**插入新技能（insert_skill）、更新已有技能（update_skill）、删除过时技能（delete_skill）**。

这两个角色跑起来就是一个闭环：

![alt text](image.png)

执行器干活 → 策展人看轨迹提炼经验 → 更新技能库 → 技能库变了，执行器下次干活的方式也跟着变。

但要让策展人学会"管好"技能库，要解决两个核心难题：

## 2.1 难题一：反馈太慢且绕了太多层

策展人在第3个任务时更新了一个技能，这个操作好不好，得等第10个相关任务的结果出来才知道。**延迟 + 间接反馈**，传统监督学习根本没法用。

**SkillOS的解法：分组任务流（Grouped Task Streams）**

他们不是随机排列训练数据，而是先让Gemini-2.5-Pro给每个任务标上技能相关属性（话题、所需能力、常见陷阱等），然后用软Jaccard相似度把相关任务聚成"任务组"。每个训练实例不是一个孤立任务，**而是一组语义相关的任务序列**。

这样一来就很妙了：策展人在组内前面任务里动了技能库，后面相关任务的表现直接就反映了策展决策的质量。**延迟反馈被转化成了可优化的学习信号。**

## 2.2 难题二：奖励函数怎么设计

光有任务分组不够，还得有好的奖励信号引导策展人。SkillOS设计了**复合奖励函数**：

$$r = r^{\text{task}} + \lambda_f r^{\text{fc}} + \lambda_u r^{\text{cnt}} + \lambda_c r^{\text{comp}}$$

四个部分各司其职：

- **$r^{\text{task}}$（任务结果奖励）**：除第一个任务外（这时技能库还是空的），其余任务的平均成功率。这是最核心的信号，**直接让执行器的表现来"投票"策展人的操作好不好**。
- **$r^{\text{fc}}$（函数调用奖励）**：策展人产出的操作得是有效的、能执行的，不能格式错误或者调了个不存在的东西。
- **$r^{\text{cnt}}$（内容质量奖励）**：拉来Qwen3-32B做外部评判器，从抽象性、可复用性、可操作性、忠实性四个维度给技能打分。
- **$r^{\text{comp}}$（压缩奖励）**：**鼓励精炼，惩罚照抄。** 策展人如果直接把原始轨迹塞进技能里，会被扣分 —— 做减法才是本事。

策展人用**GRPO**（Grouped Reward Policy Optimization）来优化，每个任务组跑8条独立rollout再算优势函数。16张H100，ALFWorld上训3天，推理任务训2.5天 —— 不算便宜，但换来的是一个真的会管技能的策展人。

# 三、实验结果：8B策展人凭啥赢了大模型？

论文在两套任务上做了系统评测：**多轮交互式Agent任务**（ALFWorld、WebShop）和**单轮推理任务**（AIME24/25、GPQA-Diamond）。

### 核心数据

ALFWorld基准上，以Qwen3-8B为执行器：

![alt text](image-1.png)

几个看点：

**第一，性能提升是实打实的**。相比无记忆基线 **+13.3个百分点（相对 +27.8%）**，相比最强记忆基线ReasoningBank **+5.5 个百分点（相对 +9.8%）**。

**第二，快了不是一点。** 成功率更高的同时，平均交互步数还少了2.2步。这说明技能库给的**不是堆更多上下文，而是精准的程序性捷径** —— 让执行器少走弯路。

**第三，8B策展人 > Gemini-2.5-Pro策展人。** 这是我个人觉得最反直觉的结果。直接拿Gemini-2.5-Pro当策展人（SkillOS-gemini，成功率53.1%），竟然不如训过的8B小模型（61.2%）。论文的解释是：**大模型推理强，不代表它产出的技能就能匹配执行器的实际能力。** 经过RL训练的策展人学的不是"看起来聪明的技能"，而是**执行器真正能用得上的技能**。这个mismatch问题，值得所有在做Agent + 大模型组合的人注意。

**第四，跨任务泛化也稳**。策展人在训练时只见过Qwen3-8B当执行器。测试时换上Qwen3-32B甚至Gemini-2.5-Pro，性能照样往上走。用Gemini-2.5-Pro执行时，ALFWorld 成功率从**66.4%干到80.2%**。推理任务上学出来的策展人（富含分解、验证这类抽象策略），迁移到ALFWorld和WebShop上的效果尤其好。反过来，从WebShop学出来的技能因为太绑环境，迁移就差一些——这其实符合直觉，环境特异的知识天然就不如抽象策略好转移。

### WebShop上的表现

Gemini-2.5-Pro当执行器时，SkillOS把WebShop得分从48.6拉到 **56.0**，成功率从 38.4%拉到**41.3%**。注意**这些提升是在减少交互步数的同时实现的** —— 策展人确实帮执行器找到了更高效的搜索和决策路径，不是靠多试几次硬怼出来的。

# 四、最好看的部分：技能库怎么"进化"的

如果说实验结果是"what"，那分析部分就是"how"和"why"。论文仔细拆解了技能库在训练过程中的变化，这些变化不是量变，是**质变**。

### 从"只管往里塞"到"会整理"

训练早期，策展人基本只会**insert**，不停地往仓库里塞新技能 —— 就像一个从不整理的人，东西越堆越多。

但训着训着，**update_skill的占比持续往上走**，insert慢慢降下来。策展人开始合并重复技能、提取共性模式、修正不够通用的描述。delete虽然整体占比不高，也在缓慢增长 —— 压缩奖励起作用了。策展人学会了**精炼，而不是堆砌**。

### 技能内容越来越"有料"

早期技能加的都是些泛泛的章节：Tips、Guidance、Recommendations……看着多，其实不太顶用。

训久了，技能演化出了真正有操作性的结构：
- **失败处理逻辑**：搞砸了怎么补救
- **条件分支**：什么时候可以不走默认流程
- **状态验证**：当前进度对不对
- **系统化搜索**：有策略地遍历，不是瞎试
- **回退规划**：Plan B怎么写

**RL不止教会了策展人"什么时候"操作技能，更教会了它"怎么写"一个好技能。**

### 从"怎么处理水杯"到"怎么处理问题"

这是我觉得全篇最精彩的发现。

训练初期，技能库里超过一半是**任务-物体专有技能**，比如"如何检查抽屉里的物品"、"如何处理水杯" —— 高度具体，换个场景就没用。

训练后期，**元策略技能（meta-strategy skills）成为主导** —— 验证策略、备选方案生成、系统搜索模式、策略调整时机……这些都是跨任务通用的控制性知识。

换句话说，SkillOS学出来的不是一堆技巧，而是一套**可组合的元认知框架**。从"怎么做某件事"到"怎么思考做某类事"——这个转变是根本性的。

### 用得不多，但招招到位

测试时，SkillOS的技能调用覆盖率达到**100%** —— 每个样本至少用了一个技能。但每个样本平均使用的技能数量反而**更少了**。

这说明什么？进化后的技能库不是给执行器"灌更多上下文"，而是让执行器**更精准地找到那个真正需要的技能**。一招制敌，不用三板斧。

# 五、总结：我为什么觉得这篇论文重要

SkillOS没说"我们发明了更强的模型" —— 它说的是：**技能策展本身就是一个可学习的能力，而且训过的小模型在这件事上能干翻未经训练的大模型。** 这个结论在我看来比任何一个具体数字都值钱。

具体贡献拆开来看：

1. **分离式架构**：冻结执行器 + 可训练策展人。这意味着你不需要重训底层的Agent模型，策展能力可以独立升级，也可以跨模型、跨任务迁移。
2. **分组任务流 + 复合奖励**：用任务分组把延迟反馈变成即时信号，用四维奖励引导策展人产出高质量技能。这两招合在一起，解决的是"怎么让RL驯化一个没有明确正确答案的问题"。
3. **数据说话**：多个基准、多种执行器，效果和效率双杀。不是cherry-pick的结果。
4. **进化路径的分析**：从"盲目堆积"到"精炼整合"，从"任务专有"到"元策略" —— 论文把技能库的演变过程拆得明明白白，这给后来的人提供了很有价值的洞察。

往后看，我觉得有几个方向会很有意思：
- **更长的演化周期**：现在的任务组还比较小，如果把时间拉长到数万次交互，策展人能进化出什么？
- **多模态技能**：纯文本技能扩展到视觉、代码、工具调用，会不会更实用？
- **多Agent技能共享**：不同Agent之间的技能库能不能互通、怎么互通？
- **安全边界**：技能库自主演化时，怎么保证产出的技能是安全、可审计的？

SkillOS这篇论文给我的最大感受是：**未来的Agent拼的可能不是模型大小，而是谁会学习。** 而学习的钥匙，握在一个训练有素的技能管家手里。
