# LifeSkill：终身学习的 Agent 不只该"记住"经验，而该把经验内化成参数

> 现有的终身学习 Agent 靠外部记忆来复用经验。轨迹存下来、摘要存下来、反思存下来，下次遇到类似任务检索出来用。但记忆会膨胀、检索会延迟、权重始终不变。LifeSkill 换了条路：**把技能当作训练时的中间脚手架，先让技能引导 Agent 从失败中探索出更好的策略，再把成功策略内化成模型参数。内化完成后，技能从推理上下文中消失。** 参数变了，prompt 不变。这就是 test-time co-evolution。

## 记下来，但没学会

终身学习对 LLM Agent 不是加分项，是基础需求。任务分布在变、有效策略靠反复交互才能浮现、Agent 必须在部署过程中持续积累能力。但现有的终身学习 Agent 方案卡在了同一个模式上：**模型参数在测试时就冻结了，所有的改进都压在外部记忆上。**

记忆驱动的方法（Synapse、AWM、AgentKB、CER）把过去的轨迹或摘要的工作流存下来，下次检索出来引导决策。测试时扩展的方法（Reflexion、Self-Refine、多路径搜索）在当前 episode 里分配更多推理计算做修正和探索。

这两条线的共同问题是：**积累的经验留在了记忆里，没有留进模型。** 记忆持续膨胀推高了检索延迟和上下文长度，跨任务的组合泛化被离散的文本条目限制住，而测试时的改进是一次性的。这次发现了更好的策略，下次遇到类似任务还要重新发现一遍。

华东师范大学和上海 AI 实验室的联合团队在 **LifeSkill** 里做了一种不同的结构：把技能重新定位为"训练时的中间变量"而非"推理时的外挂"。Agent 在部署中持续在线更新参数。技能不再是为了让 Agent 在推理时多读一段文本，而是为了让 Agent 在失败后多探索一条更好的路径，然后把那条路径上的成功行为吸收进模型自身。

## Verifier-Guided Skill Learning：技能好不好，跑一遍就知道了

LifeSkill 维护两个同架构但不同参数量的模型。**Policy Model π_θ 负责执行任务。Skill Extraction Model π_φ 负责从失败的轨迹中蒸馏可复用的技能。** 两个模型通过两阶段 RL 耦合在一起。

第一个阶段叫 VGSL（Verifier-Guided Skill Learning），解决的是"什么样的技能值得被提取"的问题。

现有技能提取的做法是让模型根据失败轨迹自己写一段反思，反思的质量由一个 LLM Judge 打分。**问题是 Judge 打出来的分数跟这条反思能否真正帮助 Agent 在下一次尝试中成功，是弱相关的。** 一段读起来很合理的反思，注入上下文后可能根本没用，甚至可能因为误导而降低成功概率。

VGSL 的解法是：把技能提取的监督信号从 Judge 换成 Verifier。流程是这样的。Policy Model 先跑一次，失败。Extraction Model 根据这次失败轨迹提出一组候选技能。然后，Policy Model 在注入每条候选技能的条件下分别重跑同一任务多次。**每条技能的分数不是"它读起来有多合理"，而是"注入它之后，Policy Model 在这个任务上的多次滚动的平均 Verifier 成功率"。** 技能提取模型的 RL 奖励信号就是它所产出的技能在下游策略中的实际执行成功率。

这个设计的核心价值在于：**它把技能质量从"主观文本质量"掰到了"客观执行质量"。** 一个长得不好看但能切实帮 Agent 找到正确路径的技能，得分高于一个措辞完美但在执行中无法转化为正确行为的技能。Extraction Model 被训练去最大化"它写的技能能让 Agent 在下一次尝试中成功的概率"，而不是"它写的技能看起来像一段好建议"。

## Online Skill Internalization：技能是梯子，爬上去就拆掉

第二个阶段叫 OSI（Online Skill Internalization），解决的是"技能增强探索和实际部署之间的 gap"。

探路的时候有技能加持，推理路径更长、上下文更多、但也因为有了明确的行为先验所以更可能成功。部署的时候不能每步都查技能库、都加载技能文本进入上下文。**这两者之间的 mismatch 如果不解决，技能增强探索中发现的更好策略永远无法在部署中被复用。**

OSI 的做法很直接。在技能增强滚动成功后，把那轮轨迹中的技能文本从输入中删掉，让 Policy Model 在原始任务输入的条件下重新学习这轮轨迹中的成功行为。**本质上是用技能找到了更好的行动序列，然后逼迫 Policy Model 在不看技能的情况下也能产出同样的行为。** 学习信号只来自成功轨迹中那些技能引导产生的行动 token，且只有来自技能增强轨迹且被 Verifier 确认为成功的样本才参与更新。

技能在这个过程中扮演的角色变成了一次性脚手架。先帮 Agent 在失败后走出来一条更好的路，然后默默退出，让 Agent 自己学会走这条路。

## 两阶段耦合：技能越好，探索越有效；策略越好，下一步的技能提取越高质

VGSL 和 OSI 不是先后串行跑一遍就结束了。它们构成一个持续的 **test-time co-evolution** 循环。

一轮任务流跑下来：Policy Model 先尝试，失败。Extraction Model 产技能，技能经过 Verifier 验证筛选。通过验证的技能引导 Policy Model 在重试中成功。OSI 把成功的行为吸收进 Policy Model 的参数。

Policy Model 变强之后，下一次遇到新任务时自己成功的概率就更高，失败的样本更少。同时，那些依然失败的样本因为 Policy Model 基准更高，暴露的是更精细的缺陷，Extraction Model 产出的技能质量也因此更高。

**迭代滚动：更强的策略→更少的失败→但每次失败暴露的是更深的策略缺口→更精准的技能→进一步强化策略。** 两个模型在同一个任务流上同步进化，不需要额外的外部监督信号，只靠 Verifier 的 binary 判定。

## 实验

LifelongAgentBench 基准，覆盖长交互的数据库和操作系统两大领域，持续任务流评估。

**LifeSkill 在所有基准上一致优于所有基线，平均性能高出 7 个绝对百分点。** 基线覆盖两类：训练自由型（外部记忆检索，包括 Synapse、AWM、Self-Refine 等）和训练型（SkillRL、DAPO 等）。LifeSkill 在两类上都有显著增益。

进一步分析确认了两个机制各自的贡献：去掉 VGSL 用纯 Judge 打分替代 Verifier 验证，技能质量显著下降。去掉 OSI 保留技能文本在推理时继续使用，性能不如 OSI 内化后的纯策略模型。**说明把技能留在上下文里不如把技能吸收进参数。** 持续性验证显示 LifeSkill 在连续任务流上的累积成功率曲线持续高于对比方法，没有出现典型的记忆膨胀导致的后段退化。

## 几个问题

**Verifier 必须是确定性的。** VGSL 的核心假设是任务有明确的 binary verifier：数据库查询结果对或错、操作系统操作成功或失败。对于开放式、主观性的任务（写一篇文章好不好），这套机制需要替换成更细粒度的 reward model，但论文未覆盖。

**两个模型的参数独立是对训练开销的妥协。** Policy Model 和 Extraction Model 用同样的 backbone 架构但独立参数，意味着需要同时维护两份模型权重。对于计算预算有限的部署场景，一个可能的优化方向是让同一个模型通过不同的 adaptor 或 prompt 在不同角色间切换，论文未探索。

**任务流的分布偏移没有显式建模。** 在线终身学习的最坏情况是任务分布突然大幅偏移（从数据库查询突然全部变成操作系统任务），当前框架中的 Policy Model 更新频率和 Extraction Model 的技能生成策略没有针对这种分布突变做特殊处理。

## 最后

LifeSkill 把终身学习 Agent 的范式从"记下来，下次翻出来用"推到了"练进去，下次参数本身就更好"。

记下来有价值，但它有膨胀和检索的天花板。练进去意味着 Agent 在部署中真的在变好，不是多了一个记忆条目，而是参数本身承载了可泛化、可组合、不需要每次加载的持久能力。**技能是训练时的梯子，不是推理时的拐杖。** 这个方向如果持续走下去，终身学习 Agent 的重心会从"更好的记忆"逐渐转向"更好的在线参数更新策略"。
