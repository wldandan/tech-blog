# SkillHone：技能进化不能只留最后一个版本，而要留下每一次改动的决策记录

> 现有的技能优化方案产出的是一个更好的技能文件。上一轮怎么诊断的、哪些替代方案被拒绝过、当初为什么采纳了这次修改，这些信息在优化结束后就消失了。下一轮优化只能看到一个最新快照，不知道之前的决策上下文。SkillHone 的做法是：**把每次技能修订的诊断、候选方案、评估证据和决定结果都作为结构化记录持久保留。** 未来的 Agent 继承的不只是当前的技能版本，还有一份完整的决策链。知道什么已经试过了、什么被否决了、上次为什么改这行。

## 技能优化丢掉的上下文

现在有两种主流方式来做技能改进。Skill-Creator 式的合成构造：把任务描述、示例和参考文档变成一份可加载的技能，产物交给后面的会话直接用。GEPA 式的反思优化：跑多轮变异、评估和选择，在 prompt、工具描述和代码上做迭代改进。

两种方式的共同点是**把技能改进视为一个有界运行：运行结束，输出一份更好的产物。** 中间产生的诊断、曾考虑过但被放弃的替代方案、评估时的证据、采纳或拒绝的理由，这些在每一次运行结束时都被丢弃了。

问题在后续会话中暴露。一个新的 Agent 拿到技能的最新快照，但它不知道这个技能的某个步骤为什么长这样：是从哪次诊断推导出来的，改动是为了修哪个失效模式，有没有接近的方案被否决过。**当环境变化导致上次的修改变成问题本身时，新的 Agent 无法判断是该回退、该重新探索、还是走一条之前没尝试过的路。** 它会重复旧诊断，或者在缺少上下文的情况下做出与之前冲突的修改。

微信 AI 团队在 **SkillHone** 里引入了一个概念：**决策历史（persistent decision history）应该和技能本身一起被持久化。** 不是要一个版本 diff，而是要一份链接了诊断、修改、评估证据和结果的完整记录链。

## 两条线分开跑，证据只给该看的人

SkillHone 不是一个新的 Agent 框架。它是在现有 Agent 运行时（Claude Code、Codex、Hermes 等）上的一层 harness，用角色隔离和路由来组织技能的持续进化。

两条线在运行时被隔开。

**优化线维护技能仓库。** 优化子 Agent 可以检查当前技能、读取先前的决策记录和失败摘要、形成诊断、提出类型化的修订、审查候选变更并记录接受的结果。

**评估线维护技能评估仓库。** 评估仓库里存着练习探针、验证套件、回归集、执行 trace 和评估报告。评估子 Agent 可以检查 Oracle targets 和 traces，在当前技能上跑探针项，分析失败，产出**经过脱敏处理的报告。** 什么被脱敏了？探针的真实目标、验证器的内部状态、完整的执行轨迹。评估线能看这些，但优化线不能。优化线只看脱敏后的报告。

**两条线之间的路由不是固定的多 Agent 系统，而是动态分派的角色。** 运行时每一步根据当前状态决定分派哪种角色的子 Agent，把脱敏后的证据路由到对应角色，记录接受的结果。采纳决策各自在权限允许的证据范围内做出。这个设计保证评估反馈不会被优化侧直接当训练标签来用。探针的精确答案不会泄漏进技能修订中。

## 决策记录不只是一份 diff

SkillHone 把每一次开发步骤表示成一个决策记录：

```
h_t = (q_t, r_t, e_t, o_t)
```

q_t 是诊断：这条技能在这一轮暴露了什么失效模式。r_t 是候选修订：针对这个诊断提出的修改方案。e_t 是脱敏后的评估证据：跑完评估后返回的反馈。o_t 是结果：接受修订、拒绝、要求进一步修改、还是推迟诊断。

**一份 diff 只能告诉你文件怎么变了。一份决策记录告诉你这次改变是冲着什么问题去的、用什么证据评估的、最终为什么被接受（或者被拒绝）。** 这个区别在持续进化中是关键性的。当环境变化时，后续的优化子 Agent 可以翻 H 来判断：这次的失败是新的还是已知的？类似的修改是不是已经试过并且被否决了？上次拒绝那个候选修订的理由在新的证据下是否还成立？

**新的脱敏报告 E˜ 进来之后，优化子 Agent 从 H 中检索相关的历史记录。** 它可能重新审视一个旧诊断、在新证据下重提一个之前被否决的修订、或者避免一次和已失败尝试重复的无用修改。决策历史充当的是进化过程本身的记忆。

## 实验：深度搜索，裸跑 vs 有了技能之后

两个开放域基准：GAIA 和 WebWalkerQA-EN。比较设置在两种环境里。

**原始开放 Web 环境：不给任何预集成的搜索工具。** Agent 只能通过通用运行时和公共 Web 访问来组织搜索、解析、提取和恢复流程。这些流程被组织成可移植的技能包。对比在一个共享的初始技能池上运行：Existing-Skills 直接加载不用改，Skill-Creator 用内置迭代构造做优化，Hermes-SE 用反思式迭代做优化，SkillHone 用完整的角色隔离、脱敏反馈和持久决策历史。

另一条是对比线：**一个有商业检索服务加持的深度搜索 Agent。**

结果清楚。**SkillHone 在 GAIA 上拿到 64.6%，比深度搜索 Agent 高出 15.8 个百分点。它用的只是公共 Web 访问和可移植技能包，没有预集成的商业检索。** 在 WebWalkerQA-EN 上拿到 66.4%，比深度搜索 Agent 高 3.2 个百分点。增益在更难的任务分层上最明显，那些需要 Agent 自己决定什么时候该广泛搜索、什么时候该深入某个来源、什么时候该验证中间发现、什么时候该从失败路径中恢复。

在原始 Web 环境内的横向对比中，**SkillHone 比 Skill-Creator 在 GAIA 上高 20.5 个百分点，比 Hermes-SE 高 14.2 个百分点。** Existing-Skills（直接用不改）和 SkillHone 之间的差距说明：把相关技能暴露给 Agent 只是起点，真正的提升来自让技能变成一个持续优化的过程：什么时候广泛搜索、什么时候追踪一个来源、什么时候验证答案、什么时候从失败的路径中撤回。

## 几个值得注意的点

**决策历史的价值是一个长期函数。** 在早期的几轮进化中，历史还很短，决策记录的检索和复用频率不高，SkillHone 相对 Hermes-SE 和 Skill-Creator 的优势没有后期明显。当决策链积累到一定长度后，避免重复诊断和重复失败修改的收益才开始显著拉开差距。对于只需要短期优化的场景（三轮以内），决策历史的边际价值可能被高估。

**脱敏反馈的强度与进化效率之间存在 trade-off。** 优化线看不到探针的真实目标，只能从脱敏报告中间接推断。如果某个探针的失败报告写得太模糊（比如只说了"步骤 3 的输出不符合预期"但没有说明预期是什么），优化线可能花多轮次去猜测真正的修复方向。脱敏力度是一个可调参数，目前论文用的是固定策略。

**Harness 层面不规定底层模型。** SkillHone 的实验用 Claude Opus 4.6 做开发时的控制器（读反馈、做诊断、改技能），用 Qwen3.6-35B-A3B 做执行和评估时的骨架。角色分离和权限路由都是在 Agent 运行时层面实现的，不依赖特定 LLM 的能力。但不同模型在诊断和修订生成上表现差异会直接影响进化质量。

## 最后

SkillHone 往技能进化的链条里加了一环之前被普遍跳过的部件：**为什么上次改了这个。** 把一份 diff 扩展成一份诊断-修订-证据-结果的完整记录，让技能进化从"连续覆盖"变成"连续继承"。

这个部件对短跑不重要，对长跑是必需。当一个技能需要跨很多个会话、面对不断变化的环境、不断被新 Agent 接手继续改进时，最后一个版本远不足以承载之前每一次决策的价值。SkillHone 补上的就是这块。
