# 经验压缩光谱：统一LLM Agent的记忆、技能与规则

# 一、引言：LLM Agent的经验管理困境

搞过生产级LLM Agent部署的朋友，肯定都踩过经验管理的坑。Agent从demo跑通到真正上线接长期多会话任务的时候，交互经验越攒越多，怎么管这些东西就成了卡住性能的核心瓶颈。

就说日均处理上千任务的企业级Agent吧，每天产生的对话日志、执行轨迹堆得像山一样，用不了多久上下文窗口就装不下了，检索历史数据的成本也会跟着数据量疯涨。现在学界有两个方向都在啃这个硬骨头：

- **Agent记忆领域**：做从交互痕迹里提取和检索经验知识的系统。
- **Agent技能发现领域**：建从执行轨迹里发现和重用流程能力的框架。

但离谱的是，这两个明明在解决同一个问题的社区，几乎完全各干各的。有人统计了22篇核心论文的1136条引用，跨社区引用率居然不到1%：做记忆的论文里只有0.7%提了技能相关的工作，反过来做技能的也只有1.2%会提到记忆研究。这种各自为战的状态，其实严重拖慢了可扩展Agent系统的迭代速度。

# 二、核心框架：经验压缩光谱的四个压缩层级

这篇论文最牛的地方，就是提出了**经验压缩光谱**这个统一框架，把记忆、技能、规则全都放到了同一个压缩程度递增的轴上。说白了就是：压得越狠，占的上下文越少，检索越快，算起来也越省资源。

## 2.1 四个压缩层级

现有智能体学习系统分布于从原始轨迹到抽象规则的单一轴线上。记忆系统集中于第1层级，技能系统集中于第2层级，而第3层级基本空白。少数跨层级系统（虚线）连通第1—2层级，但无一支持自适应层级选择。压缩比均为近似值。

| 层级 | 类型 | 压缩比 | 特点 | 实例 |
|------|------|--------|------|------|
| Level 0 | 原始痕迹 | 1:1 | 未压缩的交互记录，完全上下文绑定，可重用性极低。 | 完整对话日志、执行轨迹。 |
| Level 1 | **情景记忆** | 5-20倍 | 结构化提取发生的事件，保留关键上下文细节，丢弃冗余交互信息。 | 键值对记录、带时间戳的事件摘要："[2026-03-15] 用户要求通过SQL进行Q3收入分析，偏好表格格式"。 |
| Level 2 | **程序技能** | 50-500倍 | 提取在一类场景下如何行动的模式，跨实例抽象为可重用的行为范式。 | 结构化流程、代码片段、工作流模板："数据分析流程：(1) 确认数据源 (2) 选择工具 (3) 按偏好格式呈现 (4) 验证结果"。 |
| Level 3 | **声明规则** | 1000倍+ | 提取决策遵循的原则，领域无关的知识，可重用性最高但可能缺乏可操作的具体性。 | 自然语言原则、约束、策略："在呈现计算结果前，始终对照源数据进行验证"。 |

## 2.2 不同层级的权衡

- **通用性 vs 特异性**：压得越狠，知识适用的场景越广，但丢的上下文细节也越多。
- **压缩比 vs 信息保留**：高层级压缩本质就是做语义抽象，得能分清哪些模式是真通用，哪些只是偶然出现的特例。
- **获取成本 vs 维护成本**：L1的**情景记忆**单条交互痕迹就能生成，但攒多了维护起来巨贵；L3的**声明规则**得攒一大堆数据才能归纳出来，但一旦形成，结构紧凑，基本不用怎么管。

# 三、关键发现：论文揭示的四个结构性洞见

作者把20多个现有Agent系统往这个光谱上一套，一下就揪出了四个之前单独在记忆或者技能领域研究时根本发现不了的问题：

## 3.1 洞见1：分开搞根本不行

之前大家觉得两个社区分工挺合理的：做记忆的管个性化，做技能的管能力提升。但实际上两边都在重复造轮子：知识库检索、冲突检测、过时内容识别、下游效果评估，这些问题两边都在独立解决，从来没想过共享方案。真到生产环境你就知道，Agent这俩能力缺一不可：一个能抽象出通用工作流但记不住用户偏好的Agent根本没法用；一个能把所有上下文都记下来但不会总结重复模式的Agent，用不了多久检索预算就直接爆了。

## 3.2 洞见2：评估指标和压缩层级绑得太死

现在不同层级的系统用的评估指标完全不搭：做L1记忆的看QA的F1值、精确匹配率，做L2**程序技能**的看任务成功率，到L3**声明规则**甚至连成熟的评估方法都没有。这种绑定就导致大家都在对着各自的指标优化，但优化出来的结果放到真实场景里可能根本不好用：比如一个MemSkill系统QA的F1值特别高，但放到多步骤任务里可能对完成率一点提升都没有；EvoSkill的任务成功率看着好看，但它完全抓不住用户的个性化偏好。

## 3.3 洞见3：压缩程度越高，知识越容易迁移

实验数据也验证了这个规律：L1的记忆能跨基础模型用（MemSkill在LLaMA和Qwen上效果都不错）；L2的技能既能跨任务迁（EvoSkill从SealQA用到BrowseComp还能提5.3%的效果），也能跨模型大小迁（Trace2Skill从35B模型用到122B模型直接涨了57.7个百分点）。整体看这个关系是条凹曲线，L2层级刚好平衡了可迁移性和特异性，性价比最高。

## 3.4 洞见4：没人在乎知识的生命周期管理

现在的系统都只顾着怎么把知识提出来，维护这件事根本没人当回事。虽然有些系统也做了点相关功能：Mem0用LLM做记忆操作，Memory-R1用RL更新记忆，ExpeL给记忆算重要性，AutoSkill做了版本管理，但都是零散的点。没有一个系统能搞定完整的生命周期：跨知识类型的冲突检测、过时内容自动识别、合理的废弃机制、跨层级的一致性保证，这些全是空白。

# 四、未来方向：开放问题与设计原则

有了这个统一框架的视角，一下就能看出来现在做可扩展Agent学习系统最急需解决的三个问题，以及对应的设计思路：

## 4.1 三个核心开放问题

1. **自适应层级选择**：设计一个元控制器，拿到新的交互痕迹时自动判断压到哪个层级最合适。背后得有一套信息价值的评估框架：得算清楚生成和维护这条知识的成本，和它未来能带来的收益比到底值不值。

2. **跨层级一致性**：同一份知识同时存在于多个层级的时候怎么办？比如有一条L1的记忆说用户喜欢表格输出，又有一条L3的规则说数据结果优先可视化，这时候系统怎么检测和解决这种冲突？知识是语义化的不是关系型数据，数据库那套规范化的思路只能参考，不能直接照搬。

3. **原则性生命周期管理**：直接抄软件工程的作业就行：给知识工件做版本控制，跟踪不同层级知识之间的依赖关系，建立合理的废弃机制，还有冲突解决流程。最近有个工作流优化的调研提了个"最小可塑性"原则，放到知识管理这里也适用：只有证据足够充分的时候再把知识往更高层级提，这样既能保持灵活性，又能降低维护成本。

## 4.2 三个设计原则

1. **层级无关的压缩核心**：压缩引擎别写死输出层级，最好做成参数化的。DSPy已经证明了声明式规范可以编译成指定层级的优化流水线，我们可以把这个思路扩展成全层级支持。

2. **双向提升/降级**：知识要能上下流动：当同一个模式出现次数够多的时候，就从记忆往上提成技能，再攒够了就提成规则；反过来，如果太抽象的规则在具体场景不好用，就往下拆成更具体的技能甚至记忆。

3. **持续生命周期治理**：每条知识都要带元数据：来源是啥，置信度多少，被用过多少次，上次验证有效是什么时候，这样才能做合理的维护。

## 4.3 一个极具潜力的思路：空闲时间整合

还有个特别有意思的思路：可以在系统空闲的时候做向上压缩，就像人睡觉的时候海马体把短期记忆整理成长期记忆一样。现在所有系统都是数据进来的时候当场压缩，还没人做这个闲时整理的功能。

# 五、结论：打破社区壁垒，构建更高效的Agent系统

这篇论文最大的贡献，就是第一次把Agent记忆、技能发现、规则学习这三个方向放到了同一个压缩轴上，告诉大家这三个领域其实是在解决同一个根本问题，只是侧重点不同。把20多个现有系统映射到光谱上之后，作者发现了明显的**缺失的对角线**：现在没有一个系统能支持自适应的跨层级压缩，全都是固定在某个层级上的。

这个框架的价值主要有三点：

- 能让两个社区不用再重复造轮子，检索、冲突解决、生命周期管理这些基础能力完全可以共享
- 给知识评估提供了统一标准：不用管是哪个层级的知识，就看每存一个token的知识，能给Agent的长期性能带来多少提升
- 给未来的Agent系统设计指了条明路：得支持**自适应层级选择**，能处理跨层级一致性，还要有完整的生命周期管理能力

仅2026年第一季度就有10多篇相关论文发表，加上现在生产环境对持久化经验管理的需求越来越强，这个框架出来的正是时候。接下来最值得做的方向，就是搞能自适应选择压缩粒度的系统，根据每条经验的价值和通用性决定怎么压缩，这样Agent才能在长期部署里真正高效地扩展。
