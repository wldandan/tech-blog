# SKILLGRAPH：用演化技能图让LLM Agent从经验中持续成长

> 来自USTC、阿里巴巴和NUS的最新工作。他们给Agent的技能库加了一张图，然后用强化学习让这张图和策略一起进化。效果是：**一个7B模型在多个基准上把GPT-4o甩在了后面**。

# 一、LLM Agent 的经验困局：技能有了，关系丢了

LLM Agent 在网页导航、具身操作、搜索问答这些场景里已经能做很多事了。但有个问题一直没解决好：**大多数 Agent 把每个任务当成独立回合来处理**，刚解决过一个几乎一样的任务，遇到新变体照样从零开始摸索——哪怕实际任务里到处是重复的子问题和固定的动作组合。一个正常的 Agent 应该能从已完成的交互里**抽出可复用的知识**，下次直接调用。

技能库（Skill Library）就是干这个的：把可复用的策略存起来，遇到新任务时检索相关技能作为参考。方向没问题，但现有方案在**怎么组织**这件事上卡住了。

主流做法把技能库当成一个**平铺的列表**——每个技能单独存，检索纯靠语义相似度。问题在于，技能之间天然有关系：

- **A是B的前置条件**：在 ALFWorld 里，"定位物体"必须在"拾取物体"之前。
- **A能增强B的效果**："每次行动前检查子目标是否完成"这个通用策略，能直接拉高后续操作技能的成功率。
- **A和B经常一起出现**：两个技能在成功轨迹里反复共现，说明它们有隐含协作。

忽略这些关系，导致了两个具体的问题：

1. **检索不关心组合性**：复杂任务需要的是一条有序的技能链，Top-K 检索只吐出一堆"相关"技能，不告诉你先执行谁后执行谁。Clean 任务需要"定位→拾取→加热→放置"的顺序，扁平检索只能给四个技能名，Agent 自己猜排列组合。
2. **技能更新没有依据**：两个技能是不是冗余？某个技能是不是太泛了？某条关系是不是该砍掉？光盯着技能本身看，判断不了。

换句话说，**瓶颈不在"怎么提取技能"，而在拿到一堆技能之后，怎么组织、怎么检索、怎么让它们持续进化**。

# 二、SKILLGRAPH：把技能变成一张有向图

SKILLGRAPH 的做法很直接：**每个技能是图里的一个节点，技能之间的关系用带类型的边表示，然后让这张图和 Agent 的策略通过强化学习一起进化**。

技能从哪里来？从轨迹里蒸馏。怎么用？通过图遍历检索出有序的技能序列喂给策略模型。怎么更新？在每个验证步骤根据轨迹反馈同时调整节点和边。这三个环节串成一个闭环，下面逐个拆开看。

## 2.1 技能图的骨架：节点、边与初始化

先看节点。每个技能是一个紧凑的记录：**标题**（3-5个词）、**核心原则**（1-2句话描述策略）、**适用条件**（什么时候用）、**类别标签**（通用还是某类任务特定）、**唯一标识符**。此外还挂了一套**运行时统计** —— 使用次数、成功次数、经验成功率、创建步数、废弃状态——这些数字是后续图演化决策的全部依据：该合并还是拆分还是废弃，全看它们。

技能分两类：**通用技能**是跨任务的推理策略（"执行动作前验证每个子目标是否已完成"），**任务特定技能**绑定某种任务类型（"检查微波炉里有没有已加热的物体"只在 Heat 类任务中出现）。

再看边。SKILLGRAPH用三种类型化边连接节点：

- **Prerequisite（前置依赖）边** `A → B`：A 必须在 B 之前执行，有方向的，保证检索出的技能序列先后正确。
- **Enhancement（增强）边** `A → B`：通用技能 A 提高任务特定技能 B 的执行质量。方向从通用指向具体——"验证子目标"增强"在微波炉里加热物体"，反过来不成立。
- **Co-occurrence（共现）边** `A ↔ B`：A 和 B 反复在成功片段里同时出现，对称的，捕获协作模式。

每条边带一个**权重**（0 到 1），训练中动态调整——成功路径被强化，长期不用的衰减剪枝。基于前置和增强边（共现边不参与方向性计算），每个节点有一个**拓扑层级**：0 级是地基技能（无前置依赖），高级技能站在低级技能的肩膀上。层级在每次图演化后重新计算。

初始化时有些结构先验：**共现边（权重 0.3）**连接同类别的任务特定技能，**增强边（权重 0.2）**从通用技能拉向所有任务特定技能。**前置边初始化时不创建**——顺序依赖没法预判，得靠 Agent 在训练中自己发现。这个初始图相当于一个"有根据的猜测"：弱连接当骨架，后续靠轨迹反馈不断修正。

## 2.2 三阶段闭环

SKILLGRAPH的三个阶段互相喂料：

### 1. 图构建（Graph Construction）

构建流程分两步：

先用基座策略在环境里跑一批rollout，收集成功轨迹和失败轨迹。然后教师模型上场 —— 它分析这些轨迹，提取可复用的策略模式，输出结构化的技能记录（标题 + 原则 + 适用条件 + 类别）。

这里的一个关键设计是**同时利用成功和失败轨迹**：成功的告诉Agent怎么做，失败的告诉Agent别怎么做。论文中，"错误避免"类型的知识也作为技能的一部分注入prompt（"Don't: X。Instead: Y"）。

构建完成后，按上述初始化规则连边、计算层级，得到一个初始技能图。

### 2. 图感知检索（Graph-Aware Retrieval）

这是SKILLGRAPH跟扁平检索拉开差距的地方。给定一个任务描述和任务类型，检索分三步：

**Step 1: 选择种子**：从当前已激活的技能集中，挑出所有通用技能和匹配任务类型的任务特定技能作为入口点。这一步等价于"找出跟这个任务直接相关的东西"。

**Step 2: 双向图扩展**：光有种子不够 —— 种子依赖的前置技能、种子通向的后续技能，都需要补进来。扩展分两路：

   - **后向扩展（Backward BFS）**：沿入边方向做广度优先搜索，最大深度 **D=2**，找回种子依赖但可能属于其他任务类别的基础技能。比如一个Clean任务的种子技能，可能依赖一个Pick类别里的"捡起物体"技能，后向扩展会把这条依赖链补上。
   - **前向扩展（Forward Beam Search）**：沿出边方向做束搜索，束宽 **B=3**。每个候选节点获得一个**扩展分数**，由它的前驱节点传播过来：分数等于前驱节点分数乘以边权重的最大值。种子节点的分数初始化为 1。直觉是：一条路径有多可靠，取决于路径上每条边的权重乘积。被反复验证过的强关系路径会得到更高的扩展分数。

**Step 3: 拓扑排序**：三步汇聚的节点取并集——种子 + 后向扩展 + 前向扩展——然后按图里的依赖边做拓扑排序。结果是**一个从基础到高级、尊重前置条件的技能序列**，而不是一堆毫无顺序的"相关技能"。

这个有序序列（上限**K_max=8**个技能）被渲染成带依赖标注的文本块，注入到任务prompt的最前面。

Agent看到的是一个**有先后顺序的执行参考**，而不是一堆技能名堆在一起——这对于多步子任务的拆解帮助巨大。

### 3. 图演化（Graph Evolution）

策略在变强，技能图不动的话就会过时。SKILLGRAPH在**每个验证步骤**（论文中每 5 步做一次验证），根据新一轮轨迹的反馈，对节点和边同时做演化：

#### **节点级：自适应粒度控制（四种操作）**

这四种操作各自有明确的触发条件：

| 操作 | 触发条件 | 动作 |
|------|----------|------|
| **插入（Insert）** | Agent在现有技能覆盖不到的任务上持续失败 | 教师模型分析一批失败轨迹，生成最多**m=3**个针对性新技能。新技能初始为孤立节点，后续通过共现发现机制自动连边。 |
| **合并（Merge）** | 两个技能的图邻居Jaccard相似度 ≥ **τ_merge=0.85** | 合成一个统一技能，新节点继承两个原节点的所有边（重复边保留较高权重）。 |
| **拆分（Split）** | 技能使用次数 ≥ 10 但成功率卡在 **15%-40%** 之间 | 高使用率 + 中等成功率 = 这个技能太泛了。教师模型将其拆为 2-3 个更专注的子技能，子技能之间用前置边连接。 |
| **废弃（Deprecate）** | 使用次数 ≥ **20** 且成功率 < **15%** | 标记为废弃，从活跃技能集中移除（但保留在图中用于审计）。|

这四种操作形成了一个**技能质量的自循环** —— 差的被淘汰，泛的被细分，重复的被压缩，新的失败模式催生新技能。

#### **边级：拓扑演化（三种操作）**

- **路径强化（Path Reinforcement）**：成功轨迹中走过的每条边，权重 +**α=0.05**（上限 1.0）。这相当于在说："这条依赖路径被验证有效，下次多走走"。
- **共现发现（Co-occurrence Discovery）**：两个技能在成功片段中频繁同时出现（最少 **c_min=2** 次）但图中尚无边 → 自动加共现边。这避免了单次偶然共现产生噪声边。
- **衰减与剪枝（Decay & Prune）**：每个checkpoint，所有边权重按 **γ=0.99** 衰减。低于 **w_min=0.05** 的被移除。这保证了长期不被验证的关系自然消失，图保持精简。

每次边级更新后，节点层级重新计算以反映新的拓扑。节点统计也在每个checkpoint做增量更新：新的成功率基于累计的成功次数除以累计的使用次数 —— 一个技能被检索进prompt就算一次"使用"，对应的rollout成功了就算一次"成功使用"。

#### **渐进式解锁（Progressive Unlocking）**

这是个很实用的设计。如果一上来就把所有层级的技能都开放给Agent，高级技能会干扰基础技能的学习——就像一个还没学会走路的人被塞了一本跑步技巧指南。

初始只开放 0 级基础技能。前 5 步是热身期，不做解锁。之后每个验证 checkpoint 检查当前最高活跃层级的平均成功率，超过 **θ_unlock=0.6** 就解锁下一层。解锁可以跨级——如果新解锁的层级在同一个 checkpoint 里也达标，就继续往下开，让策略能快速追上已经具备的能力。

为了防止小样本误判，成功率用 Beta(1,1) 先验做了平滑——不会因为某个技能只用过 2 次成功了 2 次就认定它已经稳定。

## 2.3 闭环训练：三者互相推动

策略优化用的是GRPO（Group Relative Policy Optimization），训练循环长这样：

1. 对每个任务，通过图感知检索拿到有序技能序列。
2. 采样 **G=8** 个rollout，每个rollout收到二元奖励（成功/失败）
3. 组内归一化估计优势值。
4. 梯度更新策略，使用带裁剪的目标函数，附加KL惩罚防止偏离太远。

然后到了验证步骤，图演化流程触发：节点调整、边调整、渐进解锁检查。更新后的图在下个epoch的检索中生效。

整个训练形成**正向反馈环**：

> 策略变强 → 生成更丰富的轨迹 → 轨迹精炼技能图（节点+边的演化）→ 更好的图提供更精准的检索 → 更精准的检索加速策略学习 → 策略继续变强

这就是SKILLGRAPH 里"co-evolution"的真正含义 —— **图和策略一起进化，每一轮迭代都让双方受益**。

# 三、实验结果：7B模型，碾压GPT-4o

实验在三种环境上跑：

| 环境 | 特点 |
|------|------|
| **ALFWorld** | 文本版家居操作，6类多步任务（Pick、Look、Clean、Heat、Cool、Pick2） |
| **WebShop** | 网页导航 + 商品购买，需要搜索-浏览-比较-决策 |
| **搜索增强 QA** | 7个基准（3单跳 + 4多跳），信息检索 + 推理 |

基座模型是 **Qwen2.5-7B-Instruct**，教师模型 **OpenAI o3**。

### 主要数据

**ALFWorld和WebShop：**

![alt text](image-2.png)

- ALFWorld总成功率：**90.6%**，比 GPT-4o（48.0%）高出**42.6个百分点**，比Gemini-2.5-Pro（60.3%）高出 **30.3个百分点**。
- WebShop任务得分：**91.5**，成功率：**84.4%**，比GPT-4o分别高出**59.7分**和**60.7个百分点**。
- ALFWorld的Clean和Heat子任务：**100%成功率**（ExpeL只有55.0%和56.2%）。这两个任务**严格要求按固定顺序执行前置动作** —— 图感知检索直接解决了这个需求。

和之前最强的SkillRL比：ALFWorld略优（90.6 vs 89.9），WebShop则领先了**11.7 个百分点**。差距来自图演化 —— WebShop涉及搜索-筛选-比较的链条，演化的图结构持续发现和强化技能间的组合关系，而扁平技能库做不到这一点。

**搜索增强QA：**

![alt text](image-3.png)

只在NQ和HotpotQA上训练，7个基准平均**48.9 分**，**5个未见过数据集全零样本泛化**。多跳任务上，前置依赖排序的检索帮助模型把链式查询拆成子问题依次解决。

### 消融实验

| 移除的组件 | 影响 |
|-----------|------|
| 图感知检索 | ALFWorld跌 **31.2点** —— 顺序依赖型任务最吃排序。 |
| 图演化 | WebShop跌 **14.1 点** —— 灵活导航场景更依赖技能库质量而非顺序。 |
| 冷启动SFT | 两个基准合计跌 **17.2 点** —— RL在复杂环境里需要好的起点。 |

### 图演化过程

训练中节点从约20个涨到约140 个，但**活跃节点数提前止涨** —— 废弃机制形成了自动刹车，不会无限膨胀。共现边增长最快，前置和增强边通过路径强化稳步增加，说明图在初始构建之外一直在发现新的关系结构。平均节点成功率全程爬升，演化在过滤垃圾技能的同时强化了有用的。

### 收敛和效率

SKILLGRAPH大约在**50 步**后就超过SkillRL，之后一直领先。更快收敛有两个原因：**依赖排序检索减少了早期瞎探索**，**渐进式解锁自动规划了学习节奏**。另外，图引导检索在训练全程保持**更短的提示长度**（平均省约**500 token**），因为图遍历只拿拓扑相关的技能，不像扁平检索拿所有语义相似条目。

# 四、为什么图结构起作用，以及它的局限

回头想一下扁平技能库的两个硬伤 —— 检索不关心组合性，更新没有依据。SKILLGRAPH用图结构逐个击破了这两个问题，为什么图起了作用？

- **组合性规划有了骨架**：扁平检索吐出来的是无序集合，Agent得自己猜执行顺序。SKILLGRAPH通过拓扑排序把依赖关系显式化了，Clean、Heat这种严格顺序任务收益最大。
- **技能有了生命周期管理**：插入-合并-拆分-废弃四个操作组合起来，形成了一个闭环：失败的技能被自然淘汰，冗余的被合并，太泛的被拆分，新的失败催生新技能。技能库不会越用越烂。
- **边权重沉淀了集体经验**：每条边的权重来自跨轨迹的反复验证 —— 被多次确认的路径权重高，长期没用过的自动衰减。比起静态规则或单次启发式推断，这种编码方式健壮得多。

当然，这张图也不是没有代价。

- 最大的成本在对强教师模型的依赖。技能蒸馏和图演化操作 —— 插入新技能、合并冗余技能、拆分泛化技能 —— 全靠o3来做。图演化期间的推理开销不低。论文作者也承认，这条路如果能换成自蒸馏或者用critic模型直接生成技能，成本会友好很多。
- 另一个限制是目前技能图在单一环境内闭环。在ALFWorld里训出来的图，能不能直接给WebShop用？论文没有做这个实验。但方向很清楚 —— **如果技能图能跨环境迁移，那想象空间会大得多**。一个环境里积累的结构化经验，直接引导另一个环境的学习，这比零样本泛化又进一步了。
- 最后，扩展到更大的基座模型（比如 70B 级别）和更多样的任务分布，论文也没碰。7B上能跑通，更大的规模会怎样？这是个留给后续工作的开放问题。

回头看SKILLGRAPH的核心贡献，其实就一句话：**把技能库从"扁平列表"升级成"结构化依赖图"，然后让图跟策略同步进化**。这一步解决了三个关键问题：**组合性规划弱、粒度控制差、技能间的关系信号无法累积**。

但最打动人的是这个事实：**只用了一个7B的开源模型**，在多个基准上把GPT-4o和Gemini-2.5-Pro都超了 —— 不是比肩，是大幅超越。这背后的信息量不小：**知识的组织方式，有时比知识的存量本身更关键**。参数量的差距，至少在一定程度上，可以用更好的结构来弥补。
