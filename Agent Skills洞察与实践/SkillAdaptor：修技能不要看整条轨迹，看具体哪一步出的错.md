# SkillAdaptor：修技能不要看整条轨迹，看具体哪一步出的错

> 一条失败的执行轨迹可能有几十步交互。最后结果错了，但中间很多步骤其实是正确的。如果把整条轨迹当作反思的基本单元，失败的信号会被扩散到无关步骤上，导致技能修改太广、太飘。SkillAdaptor 的做法是**定位到第一个可操作的实际故障步骤，把责任链接到具体技能，只改该改的那一点**，其他都保持原样不动。

## 整条轨迹反思太粗糙了

LLM Agent 越来越依赖可复用的外部技能来解决长交互任务。技能在执行前被检索出来注入上下文，执行完了根据结果做调整。

目前的免训练技能适应方案大多用同一个思路：**拿完整轨迹做反思单元。** 从轨迹级结果或会话级摘要出发，提炼经验、修改策略、更新技能库。

这种做法的麻烦在**信度分配。** 一条失败的轨迹里有很多步骤，每步都涉及不同技能的检索和调用。最终失败可能只由某一步的某个技能缺陷引起，但轨迹级反馈把这个信号均匀抹到了所有参与步骤上。改完之后，本来没问题的技能反而被引入了新的偏差。如果一个早期错误使后续大量步骤失效——比如参数解析出错导致整条链跑偏——修改不应该波及那些本身没有问题的技能。

浙江大学和蚂蚁集团的联合团队在 **SkillAdaptor** 里把适应单元从轨迹级换成了步骤级。不拿完整轨迹做反思，而是定位到第一个可操作的故障步骤，把责任链接到候选技能，做定向修改，再通过明确的资格审核决定是否保留。免训练，可插拔接入 OpenClaw 类 Agent 框架。

## 三步闭环：归因、修改、审核

SkillAdaptor 的核心逻辑围绕一个动态技能集合 K。初始技能从第一轮成功的执行轨迹中蒸馏出来。之后每轮迭代：用当前的 K 执行任务，把失败的轨迹送进适应阶段。

适应阶段分三步。

### 归因：谁是罪魁祸首

给定一条失败轨迹，**Localizer 直接预测最早的可操作故障步骤 t\***，同时输出一段故障行为描述和改进建议。它不是简单地找"哪一步报了 error"，而是在推理最早到哪一步时事情开始偏离正确路径，并且这一步可以被技能修改覆盖。

然后 **Linker** 把责任分配给候选技能。输入是任务、轨迹、故障步骤 t\* 和当前检索到的技能集，输出是一组加权的嫌疑技能 {(s_j, w_j)}，权重越大表示对该故障的估计责任越重。Linker 还会额外输出一个二元决策 â：是 **REVISE**（修改现有技能的表述不当或误导性内容）还是 **GENERATE**（生成一个新技能覆盖未被现有库覆盖的能力缺口）。**如果故障被归因于某个检索技能的内容不当，走修订。如果是能力盲区，走新生成。**

### 修改：只改该改的

如果 â = REVISE，Modifier 拿权重最高的那个嫌疑技能，只改它，替换回 K。如果 â = GENERATE，Modifier 从故障步骤的上下文中合成一个新技能，追加到 K。

为了控制膨胀，生成的新技能会用同一个编码器跟已有技能做语义相似度检测，超过阈值 θ_dup 的就丢弃。

### 审核：改完得跑一遍验证

候选更新在正式并入技能集合之前必须通过资格审核。系统用 K 和 K+ 分别重跑任务集，比较两者的执行反馈指标。**只有 ∆ ≥ 0（新集合不比旧集合差）时，候选更新才被接受。** 否则丢弃，K 保持不变。

这个审核门槛的作用是**压住有害更新。** 消融实验中把 Qualifier 拿掉之后，PinchBench 的标准差从 ±5.2 跳涨到 ±8.1，WebShop 成功率从 33.0% 掉到 26.3%。审核不是锦上添花，是稳定性的核心机制。

## 实验

三个基准：WebShop（在线购物）、PinchBench（多领域 Agent 任务）、Claw-Eval（通用能力评估）。三个底座模型：Kimi-K2.5、GLM-5、GPT-5.2。

### 主结果

**三个基准、三个底座上 SkillAdaptor 全面覆盖所有对比基线。** 对比包括 BaseModel（无技能）、OpenSpace、A-Mem、AWM、ExpeL 和 EvoSkill。最大单项收益：PinchBench GLM-5 上 +1.5 点 Avg Score%，Claw-Eval Kimi-K2.5 上 +1.8 点 Avg Score，WebShop GLM-5 上 +2.3 点 Score、Kimi-K2.5 上 +1.7 个百分点成功率。

收益幅度不算巨大，但在三个差异化极大的基准上跨三个底座都保持正向。步骤级归因带来的稳定性提升比绝对分数的涨幅更有信息量。

### 消融：每一步都在扛分

把 SkillAdaptor 拆开来看。

**去掉 Localizer 和 Linker**——直接对完整轨迹做反思修改，不做步骤级归因和技能链接。WebShop 成功率从 33.0% 跌到 28.6%，Claw-Eval AvgScore 从 75.8 跌到 74.5。步骤级归因是有效修订的前提。

**去掉 Qualifier**——不审核直接入库。WebShop 成功率跌到 26.3%，PinchBench 标准差膨胀到 ±8.1。没有审核门槛的不稳定技能污染了检索池。

**全拆掉，只保留第一轮从成功轨迹中提取的初始技能**——WebShop 成功率 25.3%，score 32.5，跟裸跑的 32.1% 几乎持平。**仅从成功轨迹中蒸馏的技能不会跨任务泛化，有效的适应依赖精确的故障归因和审核，而不是技能数量的单纯积累。**

### 适应轮次

虽然允许跑十轮，但**大部分被接受的更新集中在前两轮。** WebShop 约 75% 的接纳写入发生在第 1-2 轮，Claw-Eval 和 PinchBench 分别是 64% 和 65%。三到四轮之后接纳写入基本归零。随着轮次推进，剩余的性能瓶颈越来越来自底座模型的内在能力上限，而非技能覆盖的缺失。

### 案例：CSV + Excel 数据汇总

PinchBench 上一个典型案例。任务要求读两个数据文件做汇总报告。原始技能在验证步骤只做"输出看起来合理就过"，执行到第 6 步时被浅层检查骗过去了。SkillAdaptor 把故障定位到第 6 步交互，责任链接到第 4 条技能的验证步骤，做定向修订：将验证条件从"输出大致合理"改为"必须重新计算后验证所有统计指标、实体完整性和跨表一致性都通过才能终止"。**改的只是一个技能的中间一条步骤，其他全部不变，任务从失败变成功。**

## 什么场景最有效

SkillAdaptor 的效果不是均匀分布的。**在故障可被定位到显式中间决策的场景中收益最大**——解析错误、无效工具参数、缺失执行步骤。这类条件在数据和代码类任务中最常见，因为执行轨迹暴露了清晰的过程结构，单个故障的技能片段可以精准修订并跨相关问题复用。在生产力类任务中收益中等，定位性修订能改善执行顺序和参数选择，但环境侧规则和外部验证需求会限制修订的天花板。在研究、记忆和安全类任务中最弱，这些场景下的失败往往依赖持久化状态、外部知识或跨会话上下文，超出了单次技能修订能覆盖的范围。

## 最后

SkillAdaptor 的核心结论：**训练自由的技能适应，瓶颈不是技能不够多，也不是反思不够深，而是反思发生在哪个粒度上。** 把适应单元从完整轨迹压缩到具体步骤，配合归因-修改-审核的三步闭环，不改模型、不改框架，技能集合就能稳定地自我改进。

改动越小越稳定，这是这个方向最直白也最容易被忽略的原则。
