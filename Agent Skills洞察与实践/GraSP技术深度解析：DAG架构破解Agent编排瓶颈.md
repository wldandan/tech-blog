
做Agent落地的开发者几乎都踩过同一个坑：给Agent加了十几个相关技能覆盖全流程，效果反而不如只给2-3个核心技能的版本，步骤一多就逻辑混乱，一步错全流程崩，token烧了不少结果却没用。

腾讯AILab最新论文《GraSP: Graph-Structured Skill Compositions for LLM Agents》提出的**GraSP**架构直接点破了行业核心误区：当前Agent的瓶颈早已不是技能数量不足，而是编排能力跟不上。GraSP在技能检索和执行之间新增编译层，将扁平技能列表转换为带显式依赖的**DAG（有向无环图）**执行结构，配合**本地修复算子**，把重规划成本从$O(N)$降到了$O(d^h)$，实测在四大Agent基准上奖励最高提升19%，环境执行步骤最多减少41%。

# 一、核心痛点：技能越多越差的悖论

GraSP论文首先给出了一个反直觉的实验结论：在ALFWorld基准测试中，**扁平化的技能执行**模式下，3个技能时任务成功率最高，增加到8个技能时成功率反而下降6个点——技能越多，效果越差。

当前主流的平执行模式存在两个致命缺陷：

1. **认知过载**：大量技能描述挤占上下文，无关技能干扰LLM判断，决策空间指数级上升导致幻觉、参数错误概率飙升。
2. **因果结构丢失**：扁平化执行会导致技能的前置条件、效果和依赖关系被丢弃，没有显式依赖跟踪，单步失败后无法判断影响范围，只能全量重规划，长链路任务几乎必然崩溃。

过去一年行业集体陷入"卷技能数量"的误区，但SkillsBench数据显示：绝大多数任务仅需2-3个精准匹配的技能就能完成，超过5个技能就会出现边际效益递减，超过10个甚至会导致效果下滑。本质上是默认LLM能自行处理所有依赖关系，而实际上即便是GPT-4o，处理5步以上的复杂任务也容易出现逻辑混乱。

# 二、GraSP核心架构：给技能加个编译层，从平列表到DAG图

GraSP的设计特别符合工程直觉，没有搞什么花里胡哨的新概念，就是把编译原理、结构化编程、错误恢复这些软件工程里用了几十年的思路，搬到了Agent的技能编排上。拆开来看，四个模块设计都挺实在：

## 2.1 记忆条件检索：不搜最相关的，只搜最有用的

普通的技能检索只看和当前任务的语义相似度，很容易搜出来"看起来相关但实际上用不上"的技能。GraSP的检索做了两个很实用的优化：

- **双维度加权**：把语义匹配得分和历史成功经验加权融合，优先选过去类似任务里跑通过的技能，不是只看语义相关度。
- **置信度计算**：根据检索结果的匹配度、历史成功率、覆盖度算出整体置信度，如果低于阈值，直接回退到ReAct模式，不会硬上结构化执行。

这个设计说实话特别务实，没有吹什么"全场景覆盖"，而是做了个动态开关，合适就用DAG，不合适就退回老模式，至少不会比原来差。

## 2.2 DAG编译：最精髓的设计，把技能变成可执行图

这一步是**GraSP**最核心的创新，把平的技能列表变成了带类型的**DAG（有向无环图）**，节点是实例化的技能调用，边分成三种类型，直接把依赖关系拍死，不用LLM自己猜：

- **状态边**：前一个技能的执行效果满足后一个的前置条件。比如"找到土豆"的效果是"土豆在视野内"，满足"拿起土豆"的前置条件，这俩节点之间就用状态边连接。
- **数据边**：前一个技能的输出作为后一个的输入。比如"拿到土豆"的输出是"持有土豆"，作为"加热土豆"的输入参数，这俩就用数据边连接。
- **顺序边**：软的优先级约束，没有直接的状态或数据依赖，但实际执行中必须按顺序来。比如要先关微波炉门再开加热，虽然没有直接的数据依赖，但顺序不能乱，这种就用顺序边连接。

编译过程也不复杂：先让LLM根据任务和检索到的技能提出技能调用节点，验证参数绑定是否正确，然后根据前置后置条件匹配、输出输入匹配、历史经验优先级这三个规则生成边，解决循环依赖，最后给每个节点加上验证器和修复预算，输出合法的可执行DAG。

这里有个非常重要的设计原则：**最小执行集**。编译出来的DAG只会包含完成任务必须的技能，所有无关的、多余的技能都会被过滤掉，从根源上避免了技能过多带来的认知过载。论文里的数据显示，即使给GraSP输入8个技能，编译出来的DAG一般也只会用到3-4个，所以完全不会出现性能下降的问题。

## 2.3 验证执行+本地修复：错了只修局部，不用全量重来

DAG编译出来之后，执行阶段就按拓扑顺序跑，每个节点执行前查前置条件，执行后查后置条件，一旦失败就触发**本地修复算子**，一共五个，覆盖了绝大多数常见错误场景：

- **REBIND**：技能本身合适，但参数绑错了，直接改参数就行。比如加热的是苹果不是土豆，把参数里的苹果改成土豆就行，不用改整个流程。
- **INSERTPREREQ**：缺少前置步骤，在失败节点前面插需要的步骤就行。比如加热土豆发现微波炉门没开，加个"打开微波炉门"的步骤就行。
- **SUBSTITUTE**：当前技能执行失败，换个能达到同样效果的替代技能。比如微波炉坏了，换成用烤箱加热土豆。
- **REWIRE**：依赖关系错了，修改边的连接就行。比如本来应该先切土豆再洗土豆，把边的方向反过来就行。
- **BYPASS**：当前节点的后置条件已经满足了，直接跳过就行。比如土豆已经是热的，不用再执行加热步骤。

论文里举了个很直观的例子：任务是"加热土豆然后放到台面上"，执行加热步骤的时候发现微波炉门没关，触发前置条件失败。GraSP直接用INSERTPREREQ插了个开微波炉门的步骤，然后继续执行后面的步骤，整个修复只动了相关部分，已经做好的找土豆、拿土豆这些步骤完全不受影响。

**划重点**：修复只在出错节点的h跳邻居范围内做（默认$h=2$），不会动已经执行成功的节点，重规划成本直接从$O(N)$降到了$O(d^h)$，$d$是平均出度，$h$是修复半径，一般都是个位数，成本比全量重规划低太多了。而且因为DAG里有显式的依赖关系，修复的时候只会改真正受影响的部分，不会出现"牵一发而动全身"的问题。

## 2.4 置信度路由：不行就认怂，保证下限

要是检索的置信度太低，或者DAG编译失败，**GraSP**不会硬跑，直接退回到普通的**ReAct**模式，保证不会崩得太难看。这个设计太适合工业落地了，相当于给系统加了个安全阀，在不擅长的任务上不会比基线差，在擅长的任务上能拿到大幅提升。

团队做了消融实验，每个组件贡献都实打实：去掉经验记忆掉1.6个点，去掉DAG编译掉1.9个点，去掉本地修复掉1.3个点，去掉路由掉0.9个点。其中DAG编译和本地修复是贡献最大的两个模块，完美验证了核心设计思路的正确性。

# 四、落地建议：不用全抄，核心思路直接可用

GraSP的设计思路非常适合工业落地，不用全量重构也能借鉴核心逻辑：

1. 不用追求完美的自动DAG编译，先从半结构化模板做起。人工给高频任务定义结构化执行模板，比如"用户要查数据"的标准流程是"理解需求→选数据源→查数据→分析结果→生成报告"，然后把每个节点映射到对应的技能，执行的时候按模板走，错了按节点修复，效果也会比平执行好很多。
2. 本地修复的思路非常值得借鉴。不用全套五个算子都做，可以先从最常见的错误场景开始，比如参数错误就重试一次，前置条件不满足就加个补全步骤，这两个就能解决80%的常见错误，成本很低收益很高。
3. 技能必须加标准化前置/后置条件描述，这是结构化编排的基础。
4. 不要盲目堆技能，先把现有技能的编排和错误处理做完善，效果提升可能比加10个技能还明显。
5. 置信度路由是工业落地必备。不要指望你的编排系统能搞定所有任务，一定要有降级方案，编排不了的就回退到普通ReAct或者人工干预，保证系统的可用性。

# 五、总结：Agent的下一个战场是编排能力

GraSP这篇论文最大的价值，不是提出了一个新的架构，而是给整个行业指了一个新的方向：**卷技能数量的时代已经过去了，下一个阶段的核心竞争力是编排能力**。

过去一年多，我们见证了LLM从"能理解语言"到"能调用工具"再到"能做复杂任务"的进步，但现在的Agent还是太"脆"了，稍微复杂一点的任务就容易崩，根本没法落地到工业场景。GraSP给我们提供了一个非常有前景的思路：用结构化的方法把LLM的能力框住，让它在给定的骨架里发挥能力，而不是让它自由发挥，这样才能得到稳定、可预测、可维护的Agent系统。

我相信，未来的Agent架构，一定会是"大模型+结构化编排+领域技能库"的组合：大模型负责理解、推理、生成，编排层负责任务拆解、流程控制、错误恢复，技能库负责具体的原子操作。这三者结合起来，才能真正做出能落地到工业场景的、稳定可用的Agent系统。
