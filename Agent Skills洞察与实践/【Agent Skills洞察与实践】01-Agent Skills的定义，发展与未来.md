做Agent开发的从业者，想必没人会对“Skill（技能）”这个词感到陌生。从[Claude Code](https://zhida.zhihu.com/search?content_id=273295345&content_type=Article&match_order=1&q=Claude+Code&zhida_source=entity)的Skill市场到Copilot的各类扩展，从自定义[GPT](https://zhida.zhihu.com/search?content_id=273295345&content_type=Article&match_order=1&q=GPT&zhida_source=entity)到市面上各类Agent框架的插件系统，如今Skill基本已经成了Agent能力扩展的标配方式。

## 一、什么是Agent Skills

### 1.1 Skills的核心定义

**Agent Skills（智能体技能）** 是给大语言模型智能体扩展能力的标准化模块，简单说就是把各种领域知识、工作流程、工具能力打包成可以被自动调用的”能力原子”。和我们平时用的插件、工具最大的不同是，Agent Skill不用用户手动找、手动触发，智能体自己会根据需求自动识别、调用甚至组合多个技能来完成复杂任务。

**Agent Skills** 本质上是一套”开箱即用”的能力封装规范，把底层技术细节全屏蔽了：普通用户不用懂编程，就能用到复杂的专业能力；开发者不用重复造轮子，可以直接复用成熟的领域解决方案。现在主流的Agent Skill标准已经支持自然语言定义、多语言脚本、资源打包等能力，最小的技能甚至只有一个几百字的说明文件，就能实现很复杂的业务功能。

### 1.2 Skills核心架构

一个Skill文件夹通常包含这几部分：
    
    
    my-skill/
    ├── SKILL.md              # 必须：主文件，包含元数据和核心指令
    ├── scripts/              # 可选：可执行脚本，.py、.sh等
    ├── references/           # 可选：参考文档，规范、指南、知识库
    └── examples/             # 可选：使用案例

**元数据层：技能的”身份说明书”**

在SKILL.md文件中使用YAML写元数据，包含名字、描述信息、版本号等，描述这个技能是什么。
    
    
    name: PDF处理
    description: 从PDF文件中提取文本和表格、填写表单、合并文档。在处理PDF文件，或者用户提到PDF、表单、文档提取时使用

在使用时，Agent可快速扫描目录，判断用户意图是否匹配某个Skill，避免加载完整内容造成的资源浪费。

**指令层：结构化工作流定义”**

在SKILL.md文件中使用Markdown写详细的指令，如定义输入输出契约、工作流程、执行步骤、错误处理和使用边界，描述这个技能该怎么做。

**资源层：技能的”素材仓库”**

资源层包含技术文档、可执行脚本和模板资源，智能体在执行任务时直接使用的文件,比如：

  * 模板文件：报告模板、Excel模板、PPT模板、邮件模板；
  * 配置文件：技能的默认参数、行业规则、分类标准等；
  * 其他资源：图标、示例数据、说明文档等辅助资源。



### 1.3 渐进式披露

最核心的技术突破是**渐进式披露（Progressive Disclosure）** 架构：模型不会一次性加载所有Skill的全部内容，而是在推理过程中根据用户意图动态挂载需要的规则和数据。Skills加载分为三个阶段：

  1. **阶段1：元数据（始终加载）  
** 在智体启动时，只加载所有Skill的YAML元数据，用来快速判断“用户的问题可能用到哪些Skill”。
  2. **阶段2：指令（触发时加载）  
** SKILL.md主体包含说明和指导，用于告诉智能体完成任务的具体步骤、应该遵循的规则、以及如何使用文件夹里的其他资源。仅在Agent确定匹配到对应的Skill后，再加载这个Skill的完整`SKILL.md`指令。
  3. **阶段3：资源和代码（按需加载）  
** 附加资源（脚本、代码、参考文件）等，只有当SKILL.md里的指令明确要求，或者智能体在执行中判断需要时，才通过文件系统导航和执行调用按需访问。



这个设计的好处很直观：Agent启动更快，每次对话的Token成本更低，更重要的是工具选择的准确率明显提升——模型不用在几百个不相关的工具里猜，只需要在少量强相关的能力里选就行。

## 二、Agent Skills的发展历程

Agent的能力扩展机制已经走过了三代，每一代都比上一代更灵活、更好用，是实打实的迭代升级：

### 2023年：API插件时代

最早是ChatGPT带火的**Plugins插件系统** ，也就是第一代能力扩展机制，设计上偏保守：

  * 每个插件对应一个远程API服务，通过OpenAPI schema定义接口；
  * 所有代码都跑在服务商的服务器上，用户本地不执行任何代码；
  * 上架需要经过中心化人工审核，每次调用都要用户确认权限；
  * 安全性倒是有保障，但太死板，只能做简单的API调用，封装不了复杂工作流和领域知识，最后也没真正普及开，现在基本没人用了。



### 2024年底：[MCP协议](https://zhida.zhihu.com/search?content_id=273295345&content_type=Article&match_order=1&q=MCP%E5%8D%8F%E8%AE%AE&zhida_source=entity)时代

**Model Context Protocol（MCP）协议** 的推出，标志着第二代能力扩展机制走向成熟：

  * 统一了工具调用协议，解决了之前不同平台和工具之间M×N的集成难题，不用再为每个平台单独做适配；
  * 支持本地运行和远程部署两种模式，灵活性提升了不少；
  * 有类型化接口，安全性有基本保障，不用中心化审核，开发者可以自由发布；
  * 但还是需要写服务代码、处理协议逻辑，非技术人员想封装复杂业务流程还是搞不定，门槛还是有点高。



### 2025年至今：Skill时代

Anthropic推出的**Agent Skills标准** 直接把开发门槛压到了最低，开启了第三代能力扩展的新时代：

  * 一个技能就是一个标准文件夹，核心只有一个SKILL.md定义文件，不用复杂配置；
  * 不用写服务代码，用自然语言写清楚指令就能定义技能能力，非技术人员也能上手；
  * 支持脚本、资源打包，能封装任何复杂的工作流和领域知识；
  * 装一次永久有效，智能体会根据用户需求自动识别和调用，不用用户手动管理；
  * 生态爆发式增长，半年内主流平台就冒出了11.8万个公开技能，覆盖办公、开发、数据分析等几乎所有场景，Claude、Cursor、Copilot、Gemini这些主流平台现在全支持这个标准。



## 三 Agent Skills的未来发展趋势

### 3.1 技术演进方向

**Skill自生成、自进化**

Agent自己写Skill。 这是最令人兴奋的前沿方向之一。2026年3月Anthropic发布的[Skill Creator 2.0](https://zhida.zhihu.com/search?content_id=273295345&content_type=Article&match_order=1&q=Skill+Creator+2.0&zhida_source=entity)已经迈出了第一步——实现人机协作持续优化，使Agent能像装App般加载专业技能。但这只是开始。未来的目标是让Agent能够自主发现新的任务模式、自动生成对应的Skill、通过自我测试验证效果，最终形成一个“Skill递归进化”的闭环。

**[跨平台Skill可移植性](https://zhida.zhihu.com/search?content_id=273295345&content_type=Article&match_order=1&q=%E8%B7%A8%E5%B9%B3%E5%8F%B0Skill%E5%8F%AF%E7%A7%BB%E6%A4%8D%E6%80%A7&zhida_source=entity)**

当前Agent Skills虽然已是开放标准，但不同平台之间仍存在实现差异。未来的关键挑战是实现真正的“一次编写、到处运行”。浙江大学的研究综述将“跨平台Skill可移植性”列为七大开放挑战之一。

**Skill组合与编排**

单个Skill解决单个问题，但现实世界的复杂任务往往需要多个Skill协同工作。未来的Skill系统需要支持自动化的Skill发现、组合与编排，让Agent能够根据任务自动选择和编排多个Skill，形成动态的能力组合。

### 3.2 生态发展方向

未来，**Agent Skills** 生态将会往更成熟、更规范的方向走，从目前行业的进展来看，未来几年大概率会朝这几个方向发展：

**标准化进程加速**

行业会逐步形成统一的**Agent Skills** 开发、测试、发布、运行标准，解决现在不同平台之间技能不兼容的问题，真正做到”一次开发，多平台运行”。安全标准也会逐步完善，建立起技能的安全审计、漏洞检测、权限管控等机制，从根本上减少安全风险。

**全生命周期管理体系成熟**

未来会形成覆盖技能开发、测试、上架、运行、迭代、下线的全生命周期管理体系：

  * 开发阶段会提供更完善的低代码/无代码开发工具，进一步降低开发门槛
  * 测试阶段会建立自动化测试和安全扫描机制，从源头保障技能质量
  * 运行阶段实现精细化的权限管控、调用审计和性能监控
  * 建立完善的版本管理和下线机制，及时淘汰有问题的旧版本。



**技能召回与组合能力大幅提升**

随着大模型能力的提升和专用召回模型的发展，技能召回准确率会提升到95%以上，真正做到”用户不用明确提，智能体自动选最优技能”的理想状态。同时，技能的自动组合能力也会取得突破，智能体能够根据复杂任务需求，自动组合多个技能完成任务，不需要人工编排。

**专业化、垂直化技能成为主流**

现在通用技能占比过高的局面会改变，面向特定行业、特定场景的垂直化、专业化技能会成为主流。这些垂直技能深度融合了领域知识和行业最佳实践，能给用户提供更精准、更高效的服务，大幅提升AI在各行业的落地价值。

## 四、挑战与思考——不要神话Skill，也不要低估它的影响

当然Skill也不是银弹，有自己的适用边界，落地的时候也有不少坑要踩。

首先是适用边界：Skill最适合常规、重复、有明确标准的任务：如代码Review、合同审查、财务报账、数据分析这些。对于需要高度创造性的任务，比如艺术创作、基础研究、突破性产品设计，Skill反而会限制思路，不适合用。

其次是维护成本：Skill不是写完就一劳永逸的，行业规范更新了要更参考文件，业务流程变了要更指令，模型升级了要适配新的指令遵循特性。企业如果有1000个Skill，每年的维护成本也不是小数目，需要配套的Skill治理体系。

最后是安全风险：Skill可以调用脚本、访问外部工具，没做好权限控制的话，恶意Skill可能执行危险操作、泄露敏感数据。未来的Skill生态必须配套安全审查、权限隔离机制，才能保证运行安全。

## 五、总结

Agent Skill不是什么高大上的黑科技，本质是AI工程化走到今天的必然产物。它解决了大模型输出不稳定、开发门槛高、难以规模化落地的核心痛点，让大模型的能力能真正落地到千行百业的具体业务场景里。

**Agent Skills** 作为AI应用的核心基础设施，现在还处在发展的早期阶段，未来还有巨大的创新和发展空间。随着标准统一和生态成熟，Skill体系会成为AI时代的核心基础设施之一，推动整个AI产业进入真正的普惠化阶段。
