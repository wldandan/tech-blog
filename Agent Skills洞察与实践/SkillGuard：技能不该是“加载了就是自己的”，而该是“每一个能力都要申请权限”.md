# SkillGuard：技能不该是"加载了就是自己的"，而该是"每一个能力都要申请权限"

> Agent 技能生态在膨胀，ClawHub 上超过 140 万个技能可被加载。但技能的管理方式还停留在信任模式：用户看一眼 SKILL.md，觉得没问题就装。SkillGuard 把 Android 的权限模型搬到了技能生态上。**每条技能必须声明它需要哪些能力，运行中每次敏感操作都要过一遍策略引擎，默认拒绝，按需授权。**

## 技能不是静态文本，是活的执行体

Agent 技能正在从一个"说明书加工具绑定"变成一种新的编程范式。一个技能包里装了自然语言指令、代码脚本、工具绑定、上下文依赖和委托逻辑。它不只是一个工具包装，而是能系统性地影响 Agent 看到什么、怎么推理、最后做什么。

但当前技能生态的安全管理基本停留在两个手段上：**信任式加载和静态检查。** 开发者看一眼 SKILL.md，扫一遍脚本，觉得没问题就装进 Agent 运行时。安全边界在加载之后就消失了。技能可以自由注入指令、读文件、发网络请求、调用其他技能，没有任何运行时关卡。

更根本的问题是，**技能的威胁横跨两个平面。** 在上下文平面上，一个技能可以在 prompt 中嵌入隐式指令、加载外部内容、引入记忆片段，在 Agent 执行任何工具调用之前就影响了它的推理轨迹。在动作平面上，它可以触发工具调用、访问文件系统、执行代码、联网通信。现有的安全方案通常只管一个平面。工具级策略能拦截危险调用但管不到上下文污染，上下文防御能标记可疑输入但管不到执行副作用。

CSIRO、澳大利亚国立大学和新南威尔士大学（UNSW）的联合团队在 **SkillGuard** 里提出了一个技能权限框架。核心思路很直接：**把技能当作携带权限声明的可执行产物，而不是可以被默认信任的文本文件。** 权限在加载时声明，在运行时每次操作都做检查。默认拒绝。

## 从 Android 得到的灵感

SkillGuard 的设计原则来自 1975 年 Saltzer 和 Schroeder 的经典安全原则，以及 Android 权限系统的实践：最小权限、完整中介、故障安全默认值。

Android 把每个 app 视为权限载体。app 必须在 manifest 里声明自己要哪些权限，运行时还要动态申请，用户可以拒绝。Android 生态能从"每个 app 随意访问通讯录和定位"进化到有可控的权限边界，这套模型是核心。SkillGuard 对技能生态提了同样的诉求：**技能不该是隐式信任的扩展，而该是携带权限声明的产物。** 它的能力必须被声明、授权、中介和审计。

## 五层架构

### Skill Manifest：技能必须说清楚"我能干什么"

第一个挑战是表达技能权限。要精确到运行时能强制执行，同时对作者和审核者保持可读。

SkillGuard 用一个紧凑的 JSON DSL 来定义技能 Manifest。这套 DSL 的核心概念是 capability。它把被保护的每一类行为抽象成独立于具体工具名称的能力标签。文件读、文件写、网络外发、凭证访问、进程生成、技能间委托，在运行时 SkillGuard 把每个宿主工具调用映射到对应的规范能力标签，同样的策略词汇可以跨平台复用。

**DSL 分了三个层级。** 最底层是 workspace baseline policy，为新会话设定默认权限种子。中间层是每个技能的 manifest，列出该技能需要的能力、每个权限的执行效果（允许/拒绝/询问用户）、冲突解决优先级和约束条件。最顶层是 session-state document，记录当前会话中活跃的运行时策略状态。

### Permission Access Control：每次操作都查一次

技能被加载之后，Agent 执行过程中每次尝试访问受保护资源，SkillGuard 的访问控制模块都会做一次权限检查：**当前技能是否拥有执行该操作的权限？是否在允许的上下文范围内？如果该操作标记为敏感，是否已经通过用户授权？** 只要任何一个条件不满足，请求被拒绝。

这个设计对应 Android 的"完整中介"原则，权限不是加载时验一次就永远有效，每次实际调用都是独立的授权点。

### User Consent：用户在场，不是旁观者

对于高敏感权限（如网络外发、文件写入、进程生成），SkillGuard 引入了一个用户交互机制。用户可以给一次授权（仅本次）、给会话级授权（同一次会话中同样约束条件下的同类操作自动放行）、或者拒绝。**用户不是被排除在授权循环之外的旁观者，而是权限决策的法定参与方。**

### Policy Enforcement：默认拒绝，除非有明确授权

运行时策略引擎把 workspace 默认、skill manifest、持久授权和会话审批合并成一条统一的策略管线。**默认行为是拒绝。只有当上述四层中至少有一层提供了正面授权时，操作才能放行。** 对于 shell 类执行，SkillGuard 还额外启动一个权限生成 mini-agent 来分析命令文本和引用的脚本，推断出需要的底层能力。这一步在命令被执行之前完成。

### Runtime Monitoring：所有操作都可追溯

SkillGuard 维护审计日志，记录每一次被中介的操作：哪个技能发起的、对应哪种能力标签、策略来源是什么、决策结果是什么、拒绝的原因是什么。在排查安全事故时，日志是一个可追溯、可归因的完整记录，不是事后猜。

## 实验

评估分五个维度。315 个从 SkillsMP 收集的真实技能做权限覆盖测试，SkillInject 基准做攻击防御评估。

### 权限覆盖：几乎全覆盖

SkillGuard 的权限分类体系对 315 个真实技能中观察到的受保护对象实现了 **99.76% 的预定义对象覆盖率和 100% 的组级覆盖率**。这套能力标签体系在真实技能生态中几乎没有盲区。

### 自动 Manifest 生成

手写 Manifest 对技能作者是一个负担。SkillGuard 的自动 manifest 生成在 SkillInject 基准上达到了 **91.0% 的 F1 分数**，召回率尤其高，需要的权限很少被漏报。自动化降低了权限声明的人为成本，同时保持了准确性。

### 攻击防御

SkillGuard 部署后，**上下文注入攻击的成功率从 32.37% 降至 23.02%，显式注入攻击从 25.56% 降至 16.67%。** 正常任务的完成率没有显著下降，权限框架没有以牺牲可用性为代价换取安全性。

## 几个值得注意的点

**Shell 命令的权限推断是一个开放问题。** SkillGuard 引入了一个 mini-agent 来分析 shell 命令文本在运行前判断需要哪些底层能力，这个静态推断环节是整条管线的精度瓶颈。复杂管道命令、动态变量替换、脚本间依赖，这些场景下 mini-agent 的推断可能出错。漏报会放行不该放行的操作，误报会阻断正常的任务执行。

**技能到技能的委托链暂未完全覆盖。** SkillGuard 的威胁模型涵盖了传递性依赖投毒和传递性信息泄露，但当前实现主要在一级技能加载和执行层面做控制。一个技能委托给另一个技能、再继续委托下去的长链，权限的传递和衰减规则目前还没有被完整形式化。

**自动化 Manifest 生成的 91% F1 是好消息，但剩余的 9% 在实际部署中是最危险的盲区。** 自动化系统倾向于在清晰、常规的场景中表现好，失败往往集中在边缘情况。而安全攻击恰恰会选择边缘情况入手。

## 最后

SkillGuard 做了一件事：**把 Agent 技能从被默认信任的文本文件，升级为携带声明、受运行时监控、默认拒绝的可执行权限产物。**

当前技能生态的安全基线是"相信作者"。超过 140 万个技能在 ClawHub 上可供加载，靠人工审查每一份 SKILL.md 来保证安全是明显不可持续的。SkillGuard 给出的替代方案是：不再靠相信，而是靠权限声明和运行时中介来约束。这对正在把 Agent 技能引入生产环境的团队来说，不是加分项，是必需的底层设计。
