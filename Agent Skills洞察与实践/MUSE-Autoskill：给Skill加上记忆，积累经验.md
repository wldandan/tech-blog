# 87.94%超越人类天花板！MUSE-Autoskill刷新SkillsBench纪录

> 字节跳动与罗切斯特理工学院提出**MUSE-Autoskill**，一个让Agent技能在**创建、记忆、管理、评估、精炼**五阶段统一生命周期内自我进化的框架。其在SkillsBench上达到**68.40%**，自生成技能在35个任务上反超人类天花板**87.94%**，且生成的Skill包能**跨Agent迁移**（Hermes +10.51pp）。这套框架已经在字节跳动的**SkillMarket、ArkClaw、SkillHub**三个生产系统中落地。

![alt text](image.png)

# 一、问题的根源：Skill不应该是一次性产物

做Agent的人越来越同意一件事：**Skill是Agent系统规模化的核心抽象**。

任务越来越复杂，光靠模型裸推理根本扛不住。Agent需要可复用的能力单元：把流程、可执行代码和领域指令封装成 Skill，像搭积木一样组合出解决方案。Skill把能力从模型权重里拆出来，让Agent能积累结构化的领域知识，不用每次都从零开始。

但眼下这些Skill系统都有同一个毛病：**Skill的"生产"和"维护"是割裂的**。AutoSkill、EvoSkill、SkillGen能自动生成Skill，Anthropic的Agent Skills标准化了打包格式，但它们都只碰了Skill生命周期的一部分。四个工程痛点就冒出来了：

- **创建与执行的脱节**：Skill在脱离Agent运行时上下文的情况下生成，生成时不知道实际执行环境长什么样。
- **没有结构化的Skill级别记忆**：跨任务的自由形式经验无法沉淀到单个Skill上。
- **静态的、未验证的Skill**：没有单元测试驱动的评估和精炼机制，生成完就放那了。
- **上下文管理粗放**：平坦的对话历史在长周期任务上要么截断要么溢出。

MUSE的主意很直接：**Skill应该是有生命周期的、持续进化的资产，不是一次性的生成输出**。围绕这个想法，它搭了一套五阶段统一生命周期。

# 二、五阶段生命周期：Skill从出生到持续进化的完整路径

MUSE-Autoskill把Skill的生命周期拆成五个阶段：**创建（Creation）、记忆（Memory）、管理（Management）、评估（Evaluation）、精炼（Refinement）**。这五个阶段不是各自独立的模块，而是在同一个Agent循环里相互咬合运转。

![alt text](image-2.png)

## 2.1 创建：从运行时中来，到运行时中去

MUSE通过内置的`skill_create`工具发起创建。这个工具在Agent的ReAct循环内部被调用，不是外部离线执行 —— "创建-执行脱节"问题就这么解决的。

现有Skill不够用时，Agent给一个高层意图描述（目的、输入、预期输出），系统按结构化流水线自动生成完整Skill包：先生成 `SKILL.md` 定义接口，再规划 `scripts/`、`resources/`、`tests/`这些子目录，最后生成对应文件。

核心机制是**create → evaluate → register循环**：Skill生成后必须跑通`tests/`目录里的单元测试，才能注册进Skill Bank。测试挂了，Agent检查错误日志，调 `update_skill`打补丁，再跑测试，直到通过。**只有通过了测试关的Skill才会进入Skill Bank，被未来的任务检索和复用**。

## 2.2 记忆：Skill级别的经验累积

MUSE的多级记忆体系是它跟现有方案最不一样的地方。三个层级：

- **短期记忆**：当前任务的上下文，包含中间推理步骤、观察结果、临时执行结果。上下文涨了会被自适应压缩。
- **长期记忆**：跨会话持久化的笔记，可复用的结论、环境特性、通用经验（比如"批量 I/O 比逐行操作快"）。
- **Skill级别记忆**：每个Skill自带一个`.memory.md`文件，Agent把跨任务累积的笔记、教训和使用观察往里面追加，如已知的失败模式、输入格式的坑、性能上要注意的点。

这个设计的聪明之处在于：`.memory.md`被刻意放在Skill目录的"可见面"之外（`.` 开头的隐藏文件，且不打包进`.tar`导出）。也就是说，**传递Skill不会带着经验一起走 —— 经验是每个Agent自己攒的**。Skill可以跨Agent共享，但经验不会互相污染。

同一个Skill再次被加载时，它的`.memory.md`跟`SKILL.md`一起被注入Agent上下文。之前学到的教训不用重新推导，Agent直接"记得"。

## 2.3 管理：检索、合并、裁剪

Skill Bank不能是个只进不出的垃圾桶。MUSE用三层机制维护质量：

- **渐进式检索**：任务开始时，系统提示只注入Skill目录（每个Skill的name + description），Agent决定要用哪个Skill后才通过`read_skill`加载完整内容。100个Skill的 Bank只产生5~10K token的目录开销，而不是500K token的全量加载。
- **合并**：新建Skill跟已有Skill高度重叠时，Agent把它们合并成更通用的版本，干掉冗余。
- **裁剪**：持续失败或长期没人用的Skill直接踢掉，保持Bank紧凑。

## 2.4 评估：测试驱动的质量守门

Skill的评估主要通过`tests/`目录里的单元测试来跑。测试不只是创建时的注册关卡，后续复用中一旦Skill产出错误结果或挂了，测试失败信号会自动触发精炼流程。

这个设计的价值一句话：**Skill的质量不靠人工审查，靠一套自动跑的测试**。跟软件工程的CI/CD一个思路。

## 2.5 精炼：闭环自愈

单元测试挂了或者运行时产出不对？Agent基于错误反馈修订或重新生成Skill。精炼不是独立模块，跟评估咬合在一起：**评估发现问题，精炼执行修复**，自动纠错闭环就转起来了。


# 三、自适应上下文压缩：两层压缩策略

长任务中上下文膨胀是所有Agent系统都头疼的问题。MUSE把对话建模成 **DAG结构的会话节点图**，每个节点是一个ReAct回合（plan、action、observation三元组），带着自己的token用量。

压缩分两级触发：

1. **Level-1（单节点压缩）**：当某个节点的token数超过阈值（默认15K）时，系统将其内容替换为紧凑摘要，但保持节点在链中的位置。这一级操作"损害最小"：只重写出问题的那一个节点，回合边界和plan/action/observation结构完整保留。
2. **Level-2（跨度合并）**：Level-1之后总上下文还是超预算（默认180K，模型硬限制200K），系统就把中间一段连续的节点合并成一个合成摘要节点。这一步比较暴力，好几个回合的结构被折成一个了。

两级压缩的共同前提是**头尾固定**：最前面KEEP_FIRST（5）个回合和最后KEEP_LAST（5）个回合一直原样保留，只有中间的回合能被压缩。动机来自"Lost in the Middle"研究：**模型对长上下文中间部分的信息利用很差**，任务框架的锚定信息和当前工作上下文必须原封不动。

而且所有压缩只作用于**active chain**，原始节点通过不可变的`history_prev/history_next`指针保留在完整历史里。任何时候想回放或者跨会话恢复任意历史状态，都能做到。


# 四、实验数据：Skill让Agent 更好，自生成Skill让Agent更强

### SkillsBench基准表现

MUSE-Autoskill在SkillsBench的51个任务上，使用GPT-5.5作为骨干模型，与Codex和Hermes两个Agent对比（三者共享同一模型，差异仅在于Agent系统设计）：

![alt text](image-3.png)

**MUSE在有Skill和无Skill两种条件下都是最优的**。Skill机制本身的有效性在三个Agent上都验证了，灌入人类Skill后全部涨了13-15个百分点。

### 自生成Skill反超人类天花板

这是论文最引人注目的结果：

![alt text](image-4.png)

- 在51个任务中，MUSE成功为**35个任务**（68.6%）生成了Skill。
- 在这35个任务上，Phase 2准确率达到**87.94%**，超过了人类Skill的**68.40%**天花板。
- 全量51任务上得分60.35%，较低是因为16个Phase 1完全失败的任务贡献了0%。

**瓶颈在覆盖度（Phase 1基线能力），不在生成质量**。只要生成了Skill的任务，质量全碾压人类手写。

为什么？从真实成功轨迹里蒸馏出来的Skill，带着高度任务相关的知识：具体的输入输出模式、失败模式、逐步操作流程，这些细节人类作者写通用Skill时往往觉得"这还用写吗"就跳过了。实际上MUSE生成的SKILL.md中位数**326行（15.8KB）**，是人类Skill（146行/6.6KB）的 **2.2倍**。

### 跨Agent迁移

将MUSE生成的35个Skill原封不动注入Hermes（一个完全不同设计的Agent），结果：

![alt text](image-5.png)

Hermes用MUSE生成Skill后提升了 **+10.51pp**，把跟人类Skill的差距弥合了 **79%**。两个Agent用同一套生成Skill的差距只剩1.95pp，**这些Skill是真正外部化的知识资产，不是绑在某个Agent实现上的内部行为模式**。

## 成本效率：Skill是降本，不是增本

一个反直觉的数据点：MUSE生成的SKILL.md比人类Skill长2.2倍，但用起来反而更便宜：

![alt text](image-6.png)

用自生成Skill比用人类Skill省 **20% token**和**37%延迟**。原因也说得通：**更详细的流程指引替代了更长的噪声推理轨迹**——过程紧凑了，Agent少兜圈子。生成一个Skill 的一次性成本约383K token / 164s，按每次使用省122K token 算，**大概3次复用就回本了**。


# 五、生产落地：从研究到工程系统

MUSE-Autoskill已经不只是实验室里的东西了。字节跳动内部有三个系统在跑它：

- **SkillMarket**：把创建流水线暴露给终端用户，成功的执行轨迹一键蒸馏成可复用、自测试的Skill包。
- **ArkClaw**：集成Skill检索，Agent在造新Skill之前先搜最相关的已有Skill。
- **SkillHub**：完整生命周期（创建、评估、记忆、管理、精炼）的托管服务，团队在同一个地方存Skill、评估Skill、治理Skill，连带它们的累积经验。

这些落地说明生命周期这个抽象不是绑死在某个基准上的：**检索还是创建的选择、捆绑测试、按Skill累积经验，这些套路拿到不同团队构建的系统里一样成立**。共享Skill的一处改进，会传到每个依赖它的Agent和产品。


# 六、局限性：四个需要正视的问题

第一，**覆盖率瓶颈**。51个任务只有35个成功生成了Skill，16个失败几乎全集中在科学计算和系统运维，MUSE Phase 1基线最弱的两个领域。框架目前是"强者恒强"的：你擅长的领域，自生成Skill质量爆表；你不擅长的，连生成机会都拿不到。

第二，**单轨迹蒸馏的泛化风险**。每个Skill只从一条成功轨迹里蒸馏，这条轨迹未必是最通用的解路径。hvac-control从80%跌到20%就是活生生的教训。

第三，**跨Agent迁移只验证了单向**。MUSE → Hermes 成功了（+10.51pp），但反过来行不行、换别的Agent行不行，一概没测。

第四，**评估面不够宽**。SkillsBench 94个任务只跑了51个，被排除的任务Docker环境更复杂、可能也难做，报告的数字也许高估了系统整体水平。

## 七、总结

MUSE-Autoskill做了一件明确的事：**把Skill从一次性产物定位为有生命周期、持续进化的Agent资产**。五阶段骨架 + Skill 级记忆 + 测试驱动评估，这套方法论给后续Skill系统设计树了一根可参考的标杆。其几个设计值得单独说：

- **Skill级别记忆**（`.memory.md`）：把经验累积从模糊的全局记忆下沉到每项Skill一本使用笔记，传Skill不带经验，共享和隔离都照顾到了。
- **两级自适应压缩**：有Lost in the Middle作理论支撑，工程上做到了可回放可恢复。

在AutoSkill、EvoSkill、SkillOS这一批同期方案里，MUSE的独特牌是**训练无关的全生命周期覆盖**，可移植性和部署成本天然占优。**跨Agent迁移**（Hermes +10.51pp）则给出了最硬的论据：Skill是外部化知识资产，不是绑在某个Agent实现上的行为。
