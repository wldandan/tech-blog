# SKILLHARNESS：Agent 学会了一个技能，这个技能安全吗？

> 从成功轨迹里学技能是主流做法。但成功不等于安全。Agent 可能在完成任务的同时，顺手把不该做的事也编码进了技能。

## 成功 ≠ 安全

计算机使用 Agent（CUA）越来越依赖技能。Voyager 从 Minecraft 探索中提炼代码技能，ASI 从成功轨迹中提取可复用的程序化操作，CUA-Skill 为 Windows 应用编码交互流程。这些方法的逻辑都一样：跑通一条轨迹，抽出操作模式，存成技能，下次复用。

但有个假设一直没人问：任务成功了，轨迹就安全吗？

浙大团队在 SKILLHARNESS 里直接把这个假设推翻了。他们指出两个系统性风险。

监督偏置。现有方法把任务成功当作技能学习的充分信号。但实际上，成功的执行可能依赖了临时的、不安全的环境状态。Agent 绕过了某个权限检查，恰好没碰到恶意注入弹窗，或者利用了一个马上要被修掉的 UI 漏洞。这些在当前轨迹里看起来没问题，一旦编码成技能反复调用，危险就来了。

僵硬的交互流。现有技能被编码为固定操作模板，不会根据环境变化调整执行粒度。界面改了、DOM 结构变了、交互流程调了——技能不知道，照样按老模板跑。直接点向错误的按钮。

这两个问题的根子是同一个：**现有方法只学了 know-what（做什么能成功），没学 know-how（在什么条件下做才安全）。**

## SKILLHARNESS 的思路

出发点很朴素。人类不是只从成功中学习的。人类从成功、失败和危险情境三方面综合提取经验，逐渐建立对技能边界的直觉——知道什么时候该用、什么时候不该用、什么时候该停下来问。

SKILLHARNESS 把这个思路工程化成两个阶段。

技能学习。Agent 通过自主探索产生交互轨迹，演化策略模块从三个来源提取监督信号：成功的子任务段提供可行的操作流程，失败的子任务段暴露不稳定的行为模式，检测到的风险信号标出安全策略被触碰的地方。三个信号共同定义技能边界——**技能不光包含"怎么做"，还包含"在什么条件下不能做"。**

技能使用。Agent 先把任务目标和高层策略（macro skills）匹配，planner 逐个检查每个 macro skill 附带的风险约束在当前环境下是否满足。满足的激活关联的 micro skill，不满足的抑制该技能，退回 LLM 灵活规划。

这套设计里有个实用的切换机制。环境稳定、模板能匹配当前 UI，直接走代码执行，确定且高效。绑定失败时退回到语义模式，用自然语言描述的操作意图引导 LLM 生成当前状态下的操作。模板反复失败到阈值后该技能被 bypass，完全交给 LLM。这种渐进退化意味着环境变了就慢一点、灵活一点，不死守一个已经失效的模板。

## Macro + Micro 两层解耦

SKILLHARNESS 把技能拆成两层。

Macro Skill 编码高层策略：宏观意图描述（"在 GitLab 创建内部项目并邀请成员"），一组成功模式（do + done_when），一组从失败中蒸馏的经验教训，一组风险约束（执行不可逆操作前必须确认用户授权）。

Micro Skill 提供参数化的代码片段，绑定当前 UI 状态的元素坐标或标签。界面变了、坐标对不上、元素找不到，micro skill 绑定失败，系统退回到 macro skill 的语义指引。

两层的好处很实在。同一个"创建项目"的宏观意图下，可以对应多个成功路径和多种界面形态。UI 变了换套 micro skill 就行，macro skill 的策略和约束不受影响。

## 数据

四个基准。ST-WebAgentBench 测政策合规（GitLab 和 SuiteCRM），WASP 测对抗注入攻击（GitLab 和 Reddit），OS-Harm 测 OS 环境下的注入安全，OpenApps 测 UI 扰动下的技能执行稳定性。

技能层面的安全指标最直接。Unsafe Skill Rate：SKILLHARNESS 仅 **2.2%（46 个技能中只有 1 个违规）**，SkillWeaver 高达 43.6%，ASI 高达 75.0%。多源监督信号在技能学习阶段就把不安全行为压住了，不是事后过滤。

任务层面，SKILLHARNESS 在 ST-WebAgentBench 上相比无技能 baseline 成功率提升 21.4%，政策合规完成率提升 17.1%。WASP 上 Success Rate 85.0%（+28.6%），Attack Success Rate 仅 2.5%（下降 17.5%）。

OpenApps 上的技能完成率（SCR）更能看出抗扰动能力。默认环境下两边差不多（0.8-0.9）。加上弹窗、对抗性描述、误导性标签、混合扰动后，SkillWeaver 的 SCR 从 0.91 跌到 0.50，SKILLHARNESS 从 0.83 跌到 0.69，跌得明显更少。Macro-micro 解耦让局部的界面变化不至于掀翻整个技能。

跨模型实验也有意思。把学到的技能在不同规模的 CUA 模型上测试，成功率随模型能力变化很大，但 **Attack Success Rate 在不同模型上持续保持低位**。安全性和任务成功率在框架层面被拆开了。弱模型做不完任务，但倾向于"安全地失败"。

消融数据。去掉 Skill Boundary 后攻击成功率涨了 9.6 个百分点，最关键的组件。去掉 micro skill 几乎没有影响，说明决定安全的不是代码模板本身，是 Planner 层面的技能选择和安全约束检查。

## 一个有意思的发现

论文提到，自提议探索阶段 Agent 自己生成的探索目标往往太窄太复杂，产生的技能在实际评估时根本用不上。技能粒度和可复用性之间有一种内在的张力：分解太细产出刚性技能，跨界面泛化差；分解太粗又漏掉关键的上下文约束。

SKILLHARNESS 用能力簇引导探索方向缓解了一部分，但预定义的能力簇也圈死了可发现技能的范围。这是自提议范式的天花板。

## 我觉得最重要的一点

SKILLHARNESS 做对了一件很基础的事：**它重新定义了"学好一个技能"是什么意思。**

不是"成功执行过一次"。那是 know-what。真正的学会是在成功、失败和危险三类经验里，逐渐摸清这个技能在什么条件下可靠、什么条件下该停。这是 know-how。

这个区分直接落地成一个设计选择：每条技能带一个 boundary，包含成功模式、失败教训、风险约束三个组件。不是装饰，消融数据说明去掉 boundary 后攻击成功率翻了几倍。

搭 CUA 系统的话，一个直接能用的实践：技能库里每条技能不光记"怎么做"，还要记"什么时候不该做"和"出了什么信号就要停"。后面这两条比"怎么做"更容易被忽视，也更能防住线上的意外。
