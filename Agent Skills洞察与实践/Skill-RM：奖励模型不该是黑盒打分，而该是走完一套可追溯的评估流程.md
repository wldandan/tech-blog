# Skill-RM：奖励模型不该是黑盒打分，而该是走完一套可追溯的评估流程

> 当前的奖励模型要么给一个不透明的标量分，要么把所有评估标准塞进一段 prompt 让 LLM 自由发挥。前者不可解释，后者不可控。Skill-RM 把奖励计算变成了一套可复用的评估技能：**不是"给这个回答打几分"，而是"按这个流程，查这些标准，用这些工具，逐条收集证据，最后汇总成一个有据可查的判决"。**

## 奖励模型在变复杂，但框架没跟上

RLHF 刚出来的时候，奖励模型的任务很单纯：给一对回答判断哪个更好。标量打分，二选一，完事。

现在奖励模型面对的任务远不是这么简单。数学和编程题的评估需要可执行验证器跑一遍对答案。事实准确性需要外部参考文档提供佐证。指令遵循需要 checklist 逐项对比。安全性需要策略约束和边界检查。**奖励信号不再是一维的偏好，而是多维度、多来源、多工具的异构证据的最终合成。**

但现有框架对这件事的处理还停留在两个极端。

标量 RM 把所有这些异构证据压缩成一个不透明的分数。训练的时候模型学到了某个隐式评估模式，但这个模式是怎么来的，分数里哪部分对应正确性、哪部分对应指令遵循、哪部分被某个错误参考文档误导了，全部不可追溯。

LLM-as-a-Judge 把 rubric、示例、checklist、验证工具全部塞进一段 prompt，让模型自己看着办。**资源怎么选、证据怎么追踪、多个维度的信号怎么汇总，这些关键决策全在隐式的 prompt 推理中丢失了。** 模型可能在某个维度上被一个错误参考文档带偏了整个判决，但这个偏差在 prompt 模式中完全不可见。

阿里 Qwen 大模型应用团队、中山大学、港中文、北大、ETH 等的联合团队在 **Skill-RM** 里换了一个范式：**把奖励建模从"打分"变成"执行一套可复用的评估技能"。** 这套评估技能是一个结构化产物，什么时候该调验证器、什么时候该查参考文档、什么时候该套 checklist、每条证据怎么记录、多个维度的结论怎么汇总，全部是显式的。

## Reward-Evaluation Skill：评估从黑盒变成可审计的工作流

Skill-RM 的核心是一个 Reward-Evaluation Skill。形式上它是一个 (M_RM, U_RM) 的二元组。

**M_RM 是技能说明书。** 它规定了四个东西。第一，评估涉及哪些维度。第二，资源的调用协议，什么条件下触发哪个 resource。第三，证据收集的格式规范，每个维度产出一条结构化证据记录，包含该维度的名称、调用的资源产出的观测结果、以及该维度的局部评估结论。第四，输出契约，最终的结构化判决 z 由一组证据记录 E 和一个结论字段 d 组成。

**U_RM 是资源库。** 它把评估需要的所有外部材料组织成五类可加载资源。Rubric & Criterion 定义了评估维度和优先级权重。Reference 提供答案标准、证据段落、官方解法，用于事实校验。Checklist & Constraint 把指令遵循分解为可逐项检查的条件。Verifier & Tool 是可执行的确定性工具，比如 Python 沙箱、代码检查器、精确匹配器。Calibration & Aggregation Rule 是证据冲突解决和分数映射规则。

**这五类资源不是被塞进同一个 prompt 的。每个资源有独立的类型标签、语义范围、适用条件声明和访问方式。** 技能在执行时按需加载，不在当前实例评估范围内的资源不会进入上下文。

这个设计带来了一套关键的性质：**最终奖励分是从结构化的证据记录中确定性读出的。** 每个维度的分数背后都有具体的被调用资源、该资源的输出、以及该维度的判定逻辑。要查为什么某个回答得了这个分，回溯到相应维度的证据记录就可以了。奖励从"信不信由你"变成"你翻一下这几个证据条目"。

## 评估过程：诊断、选取、验证、汇总

Skill-RM 执行一次评估走四步。

**诊断。** 拿到 (query, response) 对之后，技能首先分析这个实例需要激活哪些评估维度。数学题激活参考答案和代码执行验证器，开放式写作可能只激活 rubric 和格式 checklist。不是每个实例都要验所有维度。

**选取。** 根据诊断结果，从资源库中按类型标签和适用范围抽取对应的资源，加载到工作上下文中。

**验证。** 逐条调用被激活的资源。该跑代码验证器的跑代码，该拿参考答案对比的拿参考答案对比，该逐项勾 checklist 的逐项勾。**每个验证步骤产出一条结构化证据记录 e_m = (c_m, q_m, s_m)，记录哪个维度、观测到了什么、局部判定是什么。**

**汇总。** 所有维度的证据记录 E = {e_1, ..., e_M} 到位之后，按预定义的 Aggregation Rule 把多维局部判定合成为一个结论字段 d，再映射为最终奖励分。

**这个过程与传统的 LLM-as-a-Judge 有一个本质差异：多维证据的合成逻辑是在技能中显式写明的，不是在 prompt 中隐式发生的。** 验证步骤的顺序、证据冲突时的优先级、阈值和权重的选择，都由技能规格和资源库中的规则控制，不是由模型在单次推理中自由裁量。

## 实验

评估覆盖了 Reward Bench 基准以及下游的 best-of-N 选择和强化微调（RFT）实验。

**Skill-RM 在所有主流奖励基准上一致超过传统的 LLM Judge 基线。** 消融实验进一步确认，收益来自技能介导的资源编排，而不是简单地增加更多上下文或暴露更多工具。技能控制下的资源选择和执行顺序是增益的核心来源，仅仅把资源作为附加文本塞进 prompt 带来的收益显著更低。

在 best-of-N 选择和 RFT 的下游应用中，Skill-RM 的输出作为训练信号，训练出的策略在最终任务指标上也优于使用传统 Judge 信号的对照组。

## 几个值得注意的点

**技能本身需要构建和维护。** Skill-RM 的评估技能包含一套人工设计的资源库和调用协议。对于全新的评估领域（比如特定行业的合规审查），需要对应的专业知识来构建 rubric、checklist 和参考答案资源。资源库的质量决定了评估的天花板。

**证据记录的可审计性目前是面向开发者的，不是面向最终用户的。** 每条证据记录包含维度名称、观测结果和局部判定，但观测结果的表达形式（一段自然语言描述或一个结构化键值对）因资源类型而异。把证据记录自动转成人类可读的审计摘要需要额外一层翻译，论文未涉及。

**多维合成中的冲突解决规则的完备性没有穷举测试。** 当 checklisk 判定某个维度不满足而 reference 判定同一维度满足时，Aggregation Rule 的处理方式（取最小、取加权平均、还是触发人工仲裁）是预设的。规则的边界情况可能在特定实例中被触发但未被覆盖。

## 最后

Skill-RM 做了一种范式转移：**把奖励建模从"模型参数里藏的偏好"或"一段 prompt 里自由发挥的判断"拉到了"一套可检查、可复用、有调用协议的评估技能"。**

这对 LLM 训练流水线的实际影响可能比基准分数更重要。当奖励信号的结构化证据可以被审计、被追溯、被修正时，训练数据的质量控制和训练策略的迭代速度就不再受限于"这个分数到底对不对"的猜谜。因为每一分背后，都可以翻出对应的证据记录。
