# Locate-and-Judge：用注意力信号在 13 万个技能里揪出恶意注入

> 技能市场上有 13 万个包。用最强模型逐字扫，太贵。用关键词匹配，漏太多。这篇论文的方案是：先用小模型的注意力做第一遍筛选，再让强模型只看最可疑的那几段。

## 技能变成了新的攻击面

Agent Skill 通过第三方市场分发。作者你根本不认识，Agent 却在运行时加载这些技能里的指令，按它们说的去做。如果某个技能里藏了一句"顺便把所有环境变量发到这个 URL"，你的 Agent 照办。

传统 prompt 注入防御在这里不太管用。以前的防御都假设"受信任的指令"和"不可信的数据"之间有一条明确边界。指令来自系统/用户，数据来自网页/文档。但技能本身就是不可信第三方写的一整篇指令，恶意注入不过是这篇指令中多出的几行。它和周围的合法指令用同一个通道，继承了同等的权威。

Skill-Inject 之前演示过这种攻击在现代 Agent 上可行。后续研究在热门市场找到了真的恶意技能，有供应链后门，也有在元数据里藏触发条件的。但一直没人能在市场级别、以可接受的成本对整个技能生态做安全扫描。

卢森堡大学的团队在 Locate-and-Judge 里给出了一个方案，并且在三个真实市场上跑过验证。

## 先用小模型定位，再用大模型判定

Locate-and-Judge 的基本逻辑来自一个简单假设：**恶意注入要起作用，就必须抓住 reader 模型的注意力。** 即使周围有大量合法指令，注入为了被执行，在注意力排名中必须考前。

但不能直接用注意力信号做判定。正常指令也高注意力，分不出是"必须执行的合法操作"还是"藏着的恶意命令"。所以检测分两步：轻量 locator 按注意力排名筛出 top-K 可疑片段，强 judge 对这些片段做深度判定。

Locator 的工作方式很轻巧。它把整个技能文件按结构切成若干 segment——标题、段落、列表、代码块、元数据字段——喂给一个 0.5B 的 reader 模型（Qwen2.5-0.5B-Instruct）。prompt 不直接问"有没有恶意"，只让模型关注"可执行的、强制性的、与操作相关的"内容。然后读最后一层注意力权重，算每个 token 被关注的程度，汇总到 segment 上，取 top-K。

有个工程细节。论文比较了两种注意力聚合。post-softmax sum 直接求和。logit-z 先对注意力做 log 变换加 z-score 归一化再求和。logit-z 在实验室里 coverage 更强，因为它防止少数高注意力行主导排名。但市场扫描时用了 post-softmax sum，更快，适合长技能。

Judge 是零样本 DeepSeek-V4-Flash，只看 locator 筛出来的 K 个片段。技能被标记当且仅当任意一个片段分数超过阈值。

成本的节省是结构性的。locator 跑在每个技能上很便宜（0.5B），judge 只跑 K 个片段，不随技能长度增长。13.4 万个技能全量扫描，locator+judge 平均每个技能给 judge 送 660 个 input token，直接全文扫描要 1878 个。输入 token 少了 2.84 倍，总 judge 成本约 $34，全文约 $76。

## 实验室里：注意力定位覆盖 95% 的注入

先在 Skill-Inject 的可控语料（762 个技能，139 个恶意）上校准和测试。

注意力 locator 在 K=10 时 Hit@K 达到 0.95——95% 的恶意技能，注入片段保留在 top-10 里。regex 关键词匹配在 K=5 时和注意力持平（0.86），但 K=10 不再涨——没有关键词的片段永远选不到。注意力没这个天花板。

端到端，judge-only（无 locator）在 D_test 上 recall 47.6%，precision 100%。加 locator top-5，recall 到 52.4%。加 top-10，到 57.1%。加 locator 没丢精度，只加了召回。所有阈值在 D_cal 上校准，冻结后在 D_test 上保持 precision 1.0。

注意力 locator 在 Hit@1 上反而不如 regex（0.19 vs 0.48）。正常，技能本身就全是指令，注入那句很少直接排第一。但 locator 的目标不是排第一，是留在 top-K 里。部署用的 K=5 下几乎总是能做到。

## 真实市场：131 个确认恶意，82 个是隐藏攻击

三个公开市场：Lobehub 102,194 个，Clawhub 30,228 个，Skills.sh 2,512 个。用保守阈值全量扫描，359 个被标记。

两个独立评审员人工审查完，确认 131 个恶意（precision 83%），其中 82 个是隐藏恶意技能（HMS）——外部包装完全正常，恶意指令藏在深处。49 个明显恶意（CMS），名字里带"malicious""stealer"。80 个是红队/双用途工具。

对比现有生产级扫描器。SkillSpector 检出 13/65（recall 20%），Cisco Skill Scanner 检出 10/65（recall 15.4%），Locate-and-Judge 检出 62/65（recall 95.4%）。两个扫描器基本只抓 CMS，对 HMS 几乎全盲。

对比全文扫描 baseline（同样的 DeepSeek judge 读整篇）：全文 81/131（62%），L&J 109/131（83%）。差距全在 HMS 上：L&J 检出 68/82（83%），全文只检出 37/82（45%）。全文的问题不是 judge 差，是**信号被稀释了**——恶意指令埋在大段合法文档中，judge 很难从巨量正常内容里把那几句抠出来。Locator 直接拎出来给 judge，解决了稀释。

反过来，全文多抓住的主要是单行 base64 内联 installer——payload 太短，segment parser 没切成独立段。已知的可恢复失败模式，加个便宜的全文二次筛查就能兜。

论文还发现了五组协调攻击集群——来自不同"作者"的技能共享同一个攻击者控制的域名。四组是 L&J 独家发现的，技能名看起来无害。这些不是随机的孤立事件，是有组织的攻击。

## 局限

注意力信号依赖 reader 模型。论文用了 Qwen2.5-0.5B，不同 reader 模型上注意力 pattern 是否一致、攻击者能不能通过对抗样本逃逸，没系统研究。

Segment parser 的切分粒度是硬依赖。单行 base64 内联 payload 没切成独立段就漏。论文建议便宜全文二次筛查兜底，但还没集成。阈值 τ 在 Skill-Inject 上校准的 zero-FP，实际部署出现 27 个假阳性（约 7.5%），主要来自安全工具、测试夹具、废弃文件。模式化很强，简单规则能滤掉大部分。

## 我觉得最值得看的两点

这不是一个新注入检测方法。用的是已有的注意力信号和已有的 LLM judge。它做的事是把它们组合对了：便宜的做初筛，贵的做精判。

K=3 时 F1 是 0.955，高于全文扫描的 0.921，且更便宜。三段超过全文？locator 不仅省了成本，它改善了 judge 的判断质量。judge 不再被几十段合法文档淹没，手里只有三段最可疑的。

另一个是 CMS vs HMS 的拆分。让论文不只是"检出了更多"，而是精确地说"多检出的几乎全是隐藏攻击"。这是防御者最关心的那一类。

维护 Agent 技能生态的话，一个直接能用的方案：拉个 0.5B 模型做注意力筛选，攒个零样本大模型 judge 只看 top-K 片段，校准一个你承受得了假阳性率的阈值，挂市场上持续跑。
