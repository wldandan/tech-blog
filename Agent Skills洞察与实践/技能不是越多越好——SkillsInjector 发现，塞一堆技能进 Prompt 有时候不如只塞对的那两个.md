# 技能不是越多越好——SkillsInjector 发现，塞一堆技能进 Prompt 有时候不如只塞对的那两个

先看一组数据。τ2-bench（航空公司客服场景）上，Agent 的技能库里有 82 份策略参考文档。

全部塞进 Prompt：Agent 直接懵了。只塞最语义匹配的 Top-1：效果还凑合。塞 Top-4：到峰值。继续扩，8、16、32、64，**Agent 的表现开始持续下滑。塞越多越差。**

这就是技能注入的矛盾：不是技能没用，是**注入方式不对，把有用的技能也变成了噪音。** 南京大学和上海 AI 实验室的 **SkillsInjector** 就是来解决这个问题的。核心思路很简单：**技能注入不应该是"从库里检索然后塞进 Prompt"，而应该是一次动态的、任务自适应的上下文构建。**

在 τ2-bench、SkillsBench 和 ALFWorld 上，SkillsInjector 分别超过最强基线 3.9、6.1 和 7.3 个百分点。

---

## 一、当前技能注入的三个盲区

现有技能注入走的是检索逻辑：Embedding 匹配 → 取 Top-K → 拼进 Prompt。**三个盲区每个都在拖后腿。**

**盲区一：只靠语义相似度做选择。** 语义匹配告诉你"这段文字跟任务描述有多像"，但它不告诉你"这段文字真的帮到 Agent 了吗"。τ2-bench 上有一份叫"修改航班保险政策"的技能，语义上跟"我要取消航班"高度相关——关键字"保险"直接命中了。但它 $U(t, s)$ 的执行效用是 **-0.20**。它让 Agent 做错了。而另外四份策略参考技能，语义匹配程度中等，执行效用却是 **+0.40**。**语义相似和执行效用之间，根本没有稳定的相关性。**

**盲区二：预算一刀切。** Top-K 是多少？5、10、20。这个数字对所有任务、所有 Agent 都一样。但不同任务需要的技能数量是完全不同的。一个"取消航班"任务可能需要检查 4 条策略规则（取消资格、退款政策、保险覆盖范围、修改选项），一个简单的"查询订单状态"可能只需要 1 条。"一刀切 5 条"意味着有些任务被塞了多余的噪音，有些任务没拿到足够的信息。

**盲区三：每条技能的描述是固定不变的。** Retrieval 返回什么描述，就原封不动注入什么描述。但多条技能一起注入时，它们之间是有关系的——A 技能和 B 技能说了同一件事，但表述不同。**Agent 看到两份"差不多但又不一样"的描述，不知道该听谁的。**

## 二、SkillsInjector 的两个组件

### 2.1 Context Planner：学会挑技能，而且知道什么时候该停手

SkillsInjector 不靠语义相似度挑技能。它训练了一个打分器 $f_\theta(t, s)$，预测的不是"这段文字跟任务多像"，而是**"这段文字让 Agent 做对的概率提升了多少"**。监督信号来自真实的执行效用 $\Delta(t, s)$——在同一个任务上有技能和没技能表现差异有多少。

训练数据怎么来？**从训练任务池里采样 Agent 的实际执行结果。** 对每个训练任务，用所有候选技能的 leave-one-out 评测来标注执行效用的真实值。打分器学到了哪些技能真正帮 Agent、哪些在帮倒忙、哪些可有可无。

有了打分器之后，预算不再是一刀切。**打分器对每个任务的每份候选技能给出一个效用分。通过一个在验证集上调出来的阈值来截断——超过阈值的全要，低于的全部不要。** 预算 $B_t$ 是自然涌现的，不预设。对于那个取消航班的例子，打分器给四份有用的技能分了 0.81-0.94，给那个"保险"干扰项 0.39。阈值卡在中间，四份入选，一份不入选。

### 2.2 Set-Aware Renderer：把"各说各的"变成"各司其职"

Planner 选好了技能集。Renderer 做的事很细腻：**把每一条技能描述在"当前这组一起注入的技能"的上下文中重新呈现。** 只有当选中技能超过 1 条时才工作——单独一条技能没有邻居需要跟它区分。

Renderer 读取每条技能的原始描述 $d_i$、当前任务 $t$、以及本次一起注入的其他技能描述 $\{d_j : j \neq i\}$，然后产出改写后的 $\tilde{d}_i$。**改写不是凭空创作，是在原描述基础上做两件事：区分边界（"这条技能管的是这个，那条管的是那个"）和减少重叠（"技能 2 和技能 3 都提到了退款，但它们的权限范围不同"）。**

一个关键设计是 **Renderer 被蒸馏成一个小模型**，在每次 Agent 调用前只推理一次，没有在线大模型的推理开销。选择的技能集小于等于 1 时直接跳过。

## 三、实验说了什么

τ2-bench（航空公司、零售、电信三个客服场景）、SkillsBench（187 任务）、ALFWorld（具身家居），对比检索基线、技能路由方法、上下文优化方法。

### 三项全部最高

**τ2-bench 上超出最强基线 3.9 个百分点。SkillsBench 上超出 6.1 个百分点。ALFWorld 超出 7.3 个百分点。**

### 消融：三个组件各自贡献

去掉自适应预算（固定 K=5），掉分。去掉 set-aware rendering（注入原始描述不做改写），掉分。把 Planner 换成纯语义匹配，掉分。**三个组件加起来，贡献大于各自独立之和——它们互相配合。** Planner 选得越准，Renderer 要改写的素材越干净。Renderer 改写越好，Planner 的价值越能被 Agent 最大化利用。

### 执行效用分布揭示了一个残酷的事实

论文在 τ2-bench 上画了所有候选技能的执行效用分布。**大量技能的执行效用是零或负值。** 它们被检索到的概率不低（语义匹配判定为"相关"），但实际对 Agent 的帮助为零，甚至有害。**静态 Top-K 检索的方式不可避免地会把这些零或负值的技能放进 Agent 的 Prompt 里。**

## 四、这篇论文在说什么

SkillsInjector 触及了一个被大部分人当成已解决的问题：**"技能检索之后怎么注入"不是一句"拼进 Prompt"就能打发的。**

选择（哪些该进、哪些不该进）、预算（进多少）、呈现（怎么描述它们之间的关系），这三个决策每个都在影响 Agent 最终能不能用好技能。目前的技能管理生态里，绝大多数精力集中在"怎么生成更好的技能"和"怎么把技能组织得更结构化"。SkillsInjector 问的是最后一个环节：**东西都有了，也排好了，怎么递到 Agent 手里？怎么递它才接得住？**

如果你在用技能增强 Agent，这条建议非常具体：**别设死 K。不同任务需要不同数量的技能。别只看语义匹配。真正帮 Agent 的技能跟表面上"看起来像"的技能经常不是同一份。别让技能描述互相打架。当多条技能同时注入时，它们之间的关系本身就应该被编码进描述里。**
