## 1 概述

最近，Claude Agent Skills爆火，[OpenAI](https://zhida.zhihu.com/search?content_id=269597937&content_type=Article&match_order=1&q=OpenAI&zhida_source=entity)、Google等企业纷纷跟进，推出自己的Skills。那么，Skills是什么呢？Agent Skills本质是一个包含指令、脚本和资源的文件夹，用于扩展智能体的能力和专业知识。

Agent Skills设计的精妙之处在于”[渐进式披露](https://zhida.zhihu.com/search?content_id=269597937&content_type=Article&match_order=1&q=%E6%B8%90%E8%BF%9B%E5%BC%8F%E6%8A%AB%E9%9C%B2&zhida_source=entity)”机制，目录中的文件仅在任务需要时才被加载。该机制共分为三个阶段：阶段1，元数据（始终加载）；阶段2，指令（触发时加载）；阶段3，资源和代码（按需加载）。

本次将分享关于Agent Skills的2篇最新研究论文《When Single-Agent with Skills Replace Multi-Agent Systems and When They Fail》和《Agent Skills in the Wild: An Empirical Study of Security Vulnerabilities at Scale》，两篇前沿研究从性能优化与安全实证双重视角，揭示了Agent Skill的核心价值与发展瓶颈，为这一新兴生态提供了关键洞见。其**核心观点** 如下：

  * 对于小型技能库，**单智能体技能系统** （[SAS](https://zhida.zhihu.com/search?content_id=269597937&content_type=Article&match_order=1&q=SAS&zhida_source=entity)，Single-Agent with Skills）可降低54%的Token消耗和50%的延迟，能够有效消除多智能体协作的通信开销。但是，技能数量和技能语义相似性对单智能体的性能影响较大：  

    * **单智能体** 性能会随着**技能数量** 进行**系统相变** ，而非渐进退化：当技能数超过**50-100** 个时，性能急剧下降，准确率从**95%** 暴跌至**20%** 。
    * 如果技能**语义高度相似** ，即使**20** 个技能，准确率也会从**100%** 降至**37-70%** 。
  * 借鉴“分块处理”原理，**层级化技能** 组织可有效**突破容量限制** ：通过 “先选领域分类，再选具体技能” 的两阶段路由，将大规模扁平库转化为多个小规模子库。
  * 技能安全方面，构建了多阶段检测框架[SkillScan](https://zhida.zhihu.com/search?content_id=269597937&content_type=Article&match_order=1&q=SkillScan&zhida_source=entity)，发现**26.1%的技能存在安全漏洞** ，数据泄露最普遍，其次是权限提升；本次共收集**42,447** 个技能，扫描**31,132** 个技能，**8,126** 个包含至少一种漏洞。



## 2 SAS 何时可以取代 MAS

[多智能体系统](https://zhida.zhihu.com/search?content_id=269597937&content_type=Article&match_order=1&q=%E5%A4%9A%E6%99%BA%E8%83%BD%E4%BD%93%E7%B3%BB%E7%BB%9F&zhida_source=entity)是解决复杂推理任务的利器。它们通常由多个专业化智能体构成，通过通信协作完成任务。然而，由于重复的上下文交换、多轮协调以及冗余的自然语言交互，这些系统会产生巨大的计算开销。那么，能否在保留模块化推理优势的同时，降低显式多智能体协调的成本？

研究者提出一个设想：通过“**单智能体 + 技能库** ”的方式，**将多智能体系统中的每个角色编译为对应的一项技能，单智能体通过切换不同的技能完成任务** ，如此，可实现单智能体在统一上下文内完成协作任务，无需跨代理通信。

图1 基于技能的智能体：效率提升与规模限制。（a）将多智能体系统编译为单智能体技能库可减少通信开销。（b）随着技能库规模扩大，技能选择准确率呈非线性下降，在容量阈值处呈现相变特征。

### 2.1 核心思想：从”多人协作”到”一人多能”

将多智能体能力内化于单个智能体的替代范式定义，如下所示：

**技能**

每个技能是一个三元组：s=(δ,π,ξ)

  * 技能描述符（δ）：用于技能选择的语义描述。
  * 执行策略（π）：指定技能执行方式的指令。
  * 执行后端（ξ）：内部推理或调用外部工具。



**SAS（Single-Agent with Skills）**

每个单智能体技能系统（SAS）是一个元组：S=<a,S,σ>，通过选择器从技能库中匹配任务所需技能，替代MAS的多智能体协作。

  * a：基础语言模型。
  * S：技能库。
  * σ：技能选择器。



**编译（Compilation）**

编译将MAS转化为SAS的过程，包括三步：分解智能体能力为原子技能、分配执行后端（内部/外部）、内化智能体通信拓扑为技能输入输出约束。

其中，多智能体间的通信图中的显式通信边转化为技能间隐式输入输出约束，即智能体先后调用技能A和技能B时，技能A的输出格式要求，必须符合技能B的输入格式要求。

### 2.2 实验结果

**2\. 2.1 编译的实验验证**

**实验设计**

研究团队在三个基准测试上评估编译效率，每个基准测试对应一种可编译的多智能体系统架构，包括：GSM8K、HumanEval和[HotpotQA](https://zhida.zhihu.com/search?content_id=269597937&content_type=Article&match_order=1&q=HotpotQA&zhida_source=entity)。

**实验结果**

如下表，展示了MAS与编译后的SAS在所有基准测试中的主要对比结果。

关键数据如下：

  * **准确率** ：两者基本相当，平均提升+0.7%，在HotpotQA上，甚至提升4.0%。
  * **Token消耗** ：SAS的Token消耗平均减少了**53.7%** ，其中HotpotQA（58.4%）和GSM8K（56.2%）的减少幅度最大。
  * **延迟** ：SAS的延迟平均减少了**49.5%** ，其中HotpotQA（60.9%）和HumanEval（58.9%）的提升最为显著。
  * **API调用次数** ：从3-4次减少到1次。



### 2.3 扩展瓶颈：技能选择的认知容量限制

基于前文结果表明，对于可编译架构，SAS是可行且高效的。但是，随着技能库规模扩大，或者技能之间存在高度相似时，发生了非线性相变现象。

在该部分，研究者们基于认知心理学和决策科学的原理，对技能的认知容量进行了分析。四大认知基础：

  * [希克定律](https://zhida.zhihu.com/search?content_id=269597937&content_type=Article&match_order=1&q=%E5%B8%8C%E5%85%8B%E5%AE%9A%E5%BE%8B&zhida_source=entity)与决策复杂度：选择反应时间随可选数量呈对数增长，当选项超过约8个时，该关系不再成立。
  * [认知负荷理论](https://zhida.zhihu.com/search?content_id=269597937&content_type=Article&match_order=1&q=%E8%AE%A4%E7%9F%A5%E8%B4%9F%E8%8D%B7%E7%90%86%E8%AE%BA&zhida_source=entity)（工作记忆容量限制）：当总认知负荷超出工作记忆容量时，表现会急剧下降。
  * 基于相似性的干扰：[Shepard的泛化通用定律](https://zhida.zhihu.com/search?content_id=269597937&content_type=Article&match_order=1&q=Shepard%E7%9A%84%E6%B3%9B%E5%8C%96%E9%80%9A%E7%94%A8%E5%AE%9A%E5%BE%8B&zhida_source=entity)，混淆概率随心理距离呈指数衰减。
  * 层级化处理与组块化：Chase & Simon的组块化理论表明，专家通过层级化组织应对复杂性。

**2.3.1 非线性相变**

**实验设计**

  * 基于GPT-4o-mini和GPT-4o上评估技能选择性能。
  * 技能库规模：范围为{5,10,20,35,50,75,100,150,200}。



**实验结果**

图2 展示了两种评估模型的选择准确率随技能库规模变化的情况。

  * |S| ≤ 20：准确率保持在90%以上。
  * |S| = 50：开始急剧下降。
  * |S| > 100：准确率跌至20%左右。



**2.3.2 语义混淆驱动的错误**

为了进一步确认是相似度还是库规模引起的问题，进一步进行了实验。

结果如图3所示，不同混淆性条件下的选择准确率。

  * 无竞争技能的情况下，即使20个技能也能保持100%的准确率。
  * 每个技能添加**1个竞争技能** 后，准确率下降**7-30%** ；添加**2个** 竞后，准确率暴跌**7-63%** 。



由上所得，**语义重叠** 和其**数量本身** 大小无关，都会对性能产生影响。

**2.3.3 解决方案：层级化处理**

基于三种策略，对技能技能进行分层处理。

  * 扁平选择：直接从所有技能中选择（基准）。
  * 朴素领域层级：先选择领域类别，再在该领域内选择具体技能。
  * 混淆感知层级：将语义相近的技能（即易混淆项）进行分组，先在异化集群间进行选择，再在选定的小型相似技能集群内选择。



结果如图5所示，所有三种方法在不同库规模下的选择准确率。

当|S|>60（超过阈值），层级路由显著恢复了准确率，准确率绝对提升37-40%，从约45%提升到约83-85%（GPT-4o-mini）。

关键点，每个决策阶段涉及的选项数应小于κ。

### 2.4 技能库设计建议

  * **监控库规模** ：关注**模型对于|S|的有效阈值大小**（测试的GPT模型的κ≈50–100）。超过该阈值后，性能会急剧下降。
  * **降低混淆性** ：添加技能前，**检查语义重叠度** 。对相似度超过阈值的技能，应进行合并或差异化处理，避免积累近似重复项。
  * **规模化时采用分层结构** ：当|S|≫ κ时，应采用**分层路由机制** ，每个决策阶段涉及的**选项数应小于κ**。
  * **优化技能描述** ：技能选择依赖描述信息，因此需精心设计具有**区分度** 且**表述明确** 的**技能描述** 。
  * **模型与任务匹配** ：更强的模型通常具有更高的κ值和更好的抗混淆能力。对于技能库规模大或固有混淆性高的应用，**提升模型能力** 可直接带来准确率增益。
  * **考虑替代架构** ：针对**大规模技能库** ，采用配备专用路由器的**多智能体** 架构可能**优于单智能体** 方案。



## 3 Agent Skills的安全分析

技能（Skills）通过其模块化、按需加载的设计理念，正在成为构建智能Agent的新标准。模块化架构虽提升了Agent的可扩展性，但也引入了严重的安全风险。技能可包含任意可执行代码，且智能体对其信任度高、审查不足，可能导致数据泄露、提示注入、权限提升等攻击。

### 3.1 研究方法

为了系统性对技能的安全进行分析，研究者们开发了一套包含数据收集、技能分类、漏洞检测和验证的系统性方法，并构建了多阶段检测框架SkillScan，整合静态分析、LLM语义分析，覆盖四大漏洞维度，如下图所示：

  * **数据收集**
    * 爬取skills.rest和[http://skillsmp.com](https://link.zhihu.com/?target=http%3A//skillsmp.com)两大公开市场，共**42447** 个技能。
    * 筛选去重、去无效等技能，对**31132** 个技能（含3574个关联脚本）进行分析。
  * **检测框架**
    * 构建了多阶段检测框架SkillScan，发现8126个技能至少包含一个漏洞。
    * 静态分析，基于规则的扫描器，采用正则表达式、关键词匹配识别危险模式。
    * LLM语义分析，采用开源产品LLM-Guard，并基于手动标注数据进行调优。
    * 对静态分析、LLM语义分析后的技能，使用Claude 3.5 Sonnet进行分类、聚合，并输出最终判定。
  * **验证**
    * 通过人工标注200个分层抽样技能构建基准数据集，对框架进行验证（n=200个技能，63个存在漏洞），类别计数总和 > 63，因技能可能含多个漏洞。最终精度86.7%、召回率82.5%、F1分数84.6%。



### 3.2 核心发现

**3.2.1 漏洞分类**

4大类14种漏洞模式，提示注入（4 种）、数据泄露（4 种）、权限提升（3 种）、供应链风险（3 种）。

**3.2.2 漏洞分布**

26.1%的技能含潜在危险模式，安全类技能的漏洞概率是文档编制类技能的3.5倍，5.2%含强烈暗示恶意意图的高严重度指标。

**总体漏洞发生率**

共**26.1%（8126个）** 的**技能至少** 含有**一个漏洞** ，数据泄露（13.3%）和权限提升（11.8%）占比最高，提示注入（0.7%）占比最低。

**高风险技能** 占比**5.2%** ：这类技能包含混淆代码、凭证窃取等明确恶意特征，可能直接导致系统入侵或数据泄露。

严重程度| 占比| 子模式  
---|---|---  
高严重度（可能恶意）| 5.2%| 混淆代码、凭证窃取、隐藏指令、外部脚本获等  
中严重度（意图模糊）| 8.1%| 外部数据传输（E1）、文件枚举（E3）、sudo使用（PE2）等  
低严重度（可能疏忽| 12.8%| 未固定依赖版本（SC1）、过度授权（PE1）等  
  
**按类别的漏洞发生率**

不同类别的漏洞发生率差异显著，安全/Red-team技能漏洞率最高（67.4%，需排除合法工具功能，调整后约21.4%），文档类技能最低（19.4%）。

**3.2.3 高风险特征**

漏洞可能性与含脚本和技能大小强相关。

  * **脚本关联** ：带可执行**脚本** 的技能漏洞概率是**纯指令** 技能的**2.12** 倍（OR=2.12，p<0.001）。
  * **结构关联** ：技能**规模超 500行** 、**外部依赖 ≥5 个** 的漏洞风险显著升高。
