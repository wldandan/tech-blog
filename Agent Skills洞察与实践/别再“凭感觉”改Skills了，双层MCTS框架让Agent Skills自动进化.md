# 调Prompt调到头秃？双层MCTS框架让Agent Skills自动进化，性能反超手调

如果你曾经花一下午调**LLM 智能体**的Skills包，肯定懂这种痛：你吭哧吭哧写了好几个小时的清晰指令，加了一堆有用的参考文档，把辅助脚本的bug都改完了，结果一上生产环境，表现还是忽上忽下。改一行prompt吧，智能体突然就开始犯你以为早就修好了的低级错误。加个实用的参考文件吧，结果一半时候它要用的时候都忘了加载。很长时间以来，唯一的办法就是没完没了地试错。不过现在终于有更好的法子了。

新加坡国立大学、UC Berkeley和港中文的团队最近发了篇论文《BILEVEL OPTIMIZATION OF AGENT SKILLS VIA MONTE CARLO
TREE SEARCH》，提出了一个系统性的框架，用**双层优化**和**蒙特卡洛树搜索（MCTS）**来优化**智能体Skills**，在真实基准测试上的表现比人工设计的Skills还要好。

今天这篇文章，我就给大家拆解一下他们要解决的问题、这个框架是怎么工作的、测试出来的关键结果，以及对我们这些做智能体系统的人来说，实际工作里能用得上的启示有哪些。

# 一、Agent Skills优化到底难在哪？

首先咱们先统一下概念，什么是**Agent Skills**？用过 Anthropic Skills系统的同学都知道，就是一套结构化的工具包，帮智能体搞定特定领域的任务。一个标准的Skills一般包含这几部分：

- 一个**SKILL.md**文件：里面有**YAML头信息**（元数据、路由信息、工具白名单），还有给智能体的markdown格式指令。
- 可选的子目录：`scripts/`放可执行代码，`references/`放参考文档，`assets/`放模板、查询表这类静态资源。

好的Skills能精准给智能体需要的指导、工具和上下文，大幅提升它的表现。但优化这些Skills真的比看起来难太多了，主要是三个问题卡脖子：

1. **设计空间是异构的**：你要在**异构设计空间**里做优化。Skills里啥类型的东西都有：自然语言写的指令、可执行的代码、参考文档、结构化的资源，完全不是一回事。没法像传统机器学习那样，直接建模成数值优化问题来解。
2. **组件之间依赖极强**；各个组件之间是**强依赖**的，耦合得特别紧。你在 **SKILL.md** 里加一段长指令吧，一不小心就超了**token 预算**，不得不把一些支持性的细节移到单独的参考文件里。改一个地方，通常得连着调好几个其他地方才能适配。
3. **搜索空间是离散组合的**：你面对的是**离散组合搜索空间**。Skills的结构和内容有多少种可能的组合？指数级的。暴力枚举肯定不可能，手动改呢，速度慢，结果还不稳定，还受限于你自己的直觉，想不到的地方就改不到。

现在现成的工具，比如 prompt 工程框架或者代码生成器，都只能解决部分问题。要让Skills在生产环境里真的好用，得同时优化结构和内容，也就是**联合优化**，这些工具都做不到。

# 二、解决方案：双层MCTS框架

这篇论文最聪明的地方，就是把Skills优化建模成了**双层优化问题**，明明白白把结构搜索和内容优化拆成两件事来做。这么一拆，原来根本解不了的问题，就变得好办多了，每个子问题用最适合的工具来解决就行。

## 2.1 工作原理

他们把Skills抽象成一个简单的元组 `S = (θ, φ)`：

- `θ` 是**结构配置**：就是Skills里有哪些组件，这些组件是怎么组织的——有哪些文件、**SKILL.md**里有什么章节、内容怎么在不同文件之间分配，都归它管。
- `φ` 是**内容实例**：就是固定结构下的具体内容——**SKILL.md**里的指令写了啥、脚本里的代码是啥、参考文档里有什么内容，都是它的事。

优化目标也很明确：找到最好的 `(θ, φ)` 组合，既要满足**结构有效性**的规则，不超 token预算，还要让任务表现最好。

整个框架用两个嵌套的循环，分别处理结构和内容的优化：

## 2.2 外层循环：基于MCTS的结构搜索

外层循环用**蒙特卡洛树搜索（MCTS）**来找最优的Skills结构。MCTS这种算法特别适合那种有延迟反馈的离散顺序优化问题，刚好就是我们现在的场景：你改了结构好不好，得把内容调整适配之后，在真实任务上跑了才知道，没法立刻拿到结果。

MCTS做起来就是四步，反复循环：

1. **选择**：用**上置信界（UCB）**策略，从搜索树里挑看起来最有希望的结构节点，平衡「试试新东西」（探索）和「用好已经验证有效的」（利用）。
2. **扩展**：让LLM来提有效的结构修改建议——比如加/删/重排章节、把内容在不同文件之间移、改元数据之类的。LLM会参考之前的性能数据和这个Skills本来是用来干啥的，尽量提影响大的修改。
3. **评估**：把提出来的新结构传给内层循环，优化好内容之后，拿到下游任务上去测效果。
4. **反向传播**：把测试结果更新到搜索树里，指导后面的选择。

## 2.3 内层循环：内容优化

外层循环每次提一个新结构，内层循环就在固定这个结构的前提下，把内容优化到最好。分三个阶段：

1. **对齐**：把原来的内容往新结构里搬，能复用的尽量复用，同时给新增、删掉或者重排的组件留出位置。
2. **分类型优化**：根据外层改的是什么结构，用对应的优化策略。比如：
   - 要是改了章节结构，就重点优化指令文本。
   - 要是改了脚本相关的，就重点优化代码，跑测试。
   - 要是只改了元数据，就只要更新下路由和描述字段就行。
   
   每次优化会生成好几个不同的内容版本，都拿到下游任务上去测。
3. **保守选择**：不是直接选测试得分最高的那个版本——因为测试结果有噪声，很可能只是运气好。框架会用**下置信界（LCB）**来选最稳定、提升最可靠的版本。这种保守的选法能大幅降低选到「假提升」的风险，对生产环境来说特别重要，毕竟出了回归问题要修很麻烦。

# 三、ORQA基准测试的结果怎么样？

研究人员用**运筹学问答（ORQA）**基准来测试这个框架。ORQA是个选择题任务，智能体要读优化问题的描述，回答相关的问题——比如哪个是决策变量、哪个是约束、目标函数是什么这类的。

## 实验是怎么设置的？

实验设置很简单：

- **初始Skills**：用`skill-creator`生成的最基础的两文件Skills，一个主**SKILL.md**，一个单独的问题类型参考文档。这个作为要超过的基线。
- **数据集**：ORQA数据集一共20个领域，1513个样本，分成三份：搜索集用来跑优化，验证集用来选模型，测试集留到最后做最终评估，保证结果公平。
- **搜索配置**：测了两套MCTS参数：一套比较保守，搜索轮数少，用UCB选择；一套更敢探索，搜索轮数多，用混合概率选择。

## 性能结果如何？

结果很明确：

- 初始的基线Skills，在留出测试集上的精确匹配准确率是90.625%。
- 用探索性配置优化出来的Skills，准确率到了93.75%，比基线高了**3.125个百分点**。

可能你觉得3%不算多，但做过基准测试的都知道，现在SOTA的提升往往都是零点几个百分点，所以这个提升其实很可观了——而且这还是自动优化出来的，不是人工一点点调出来的。

## 优化后的Skills到底改了啥？

我觉得最有意思的是他们最终选出来的最优Skills的改动，都是些你手动调的时候可能根本想不到的点：

1. **结构上的改动**：把原来单独的参考文件直接合并到主**SKILL.md**里了，这样智能体就不用再去加载外部文件了。还加了一个专门的「问题类型分类检查清单」章节。
2. **内容上的改动**：把工作流重写成了更清晰的一步步的流程，加了更严格的校验规则，比如答案类型要匹配、要一步步排除选项、最后只能输出单个token之类的。还在 **YAML头信息**里把Skills的用途和输入规范写得特别清楚。

最妙的是，这个提升根本不是靠什么花里胡哨的prompt技巧，就是把原来的内容重新组织了一下，让智能体更容易拿到需要的信息而已。

提升其实是两个改动协同的结果：外层循环把关键信息集中到一起，让智能体更容易拿到；内层循环把信息写得更清楚、更明确，智能体更容易理解和遵循。

# 四、对我们做AI的实际有啥用？

这可不是啥没用的学术研究，对现在正在做智能体系统的人来说，有很多可以直接用到工作里的启发：

**结构和内容一样重要**

你指令写得再好有啥用？如果都埋在智能体根本想不起来加载的参考文件里，等于白写。这个实验就证明了，把关键的指导信息直接放到主指令文件里，效果提升是实实在在能测出来的。

**系统优化比瞎试强太多**

别再随机改Skills，指望瞎猫碰上死耗子刚好改出更好的版本了。用这种框架，你可以系统地去探索各种改动，测出来实际效果怎么样，然后在有效的基础上继续迭代，比瞎试效率高多了。

**MCTS可不止能用来优化推理链**

我们之前见MCTS用得最多的是优化LLM推理，比如思维树之类的方法。但这个研究说明，它还能用来优化更复杂的东西，比如Skills包、智能体的工作流，甚至代码库，用处大着呢。

**选改进的时候保守点，能少踩很多坑**

测Skills改动的时候，别光看哪个版本得分最高就选哪个。用**下置信界**这种统计方法，选那些提升更稳定、更可靠的，别选那些可能只是运气好的。这样能大幅减少上线之后出性能回归的概率，省掉很多线上查问题的麻烦，对工程团队来说太香了。

# 五、最后总结几句

智能体Skills是让**LLM 智能体**专门做特定任务最有效的方法之一，但之前优化Skills基本靠感觉，更像艺术不是科学。这个双层MCTS框架就解决了这个问题，给了我们一个系统的优化方法，不用没完没了手动调，就能得到可衡量、稳定的提升。

如果这篇文章你只能记住四点，就记这几个：

- **结构和内容要一起优化**：别把它们当成两回事，改了结构通常就得改内容才能发挥出最大效果。
- **每个问题用对工具**：MCTS适合做有延迟反馈的结构搜索，LLM优化适合固定结构下的内容调整，搭着用效果最好。
- **评估的时候保守点不吃亏**：测试结果有噪声很正常，选改进的时候稳一点，比选看起来提升大但不稳定的强得多。
- **小的结构改动可能带来大提升**：别瞧不起合并文件、加个清单这种小改动，很多时候带来的提升比你调半天prompt还大，而且你手动调还不一定想得到。

以后智能体系统会越来越复杂，越来越多会大规模上生产，这种系统性的优化框架肯定会变成刚需。说实话，靠人手动调prompt本来就没法规模化，这种自动优化才是未来的方向。
