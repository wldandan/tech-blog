# Agent 读技能文档就像你在读一本没有目录的说明书——Skill-as-Pseudocode 把它重写成了清晰的结构化接口

Agent 最尴尬的死循环不是它做不对，是它每次都要从一大段散文中重新推导"这个技能到底要我调哪个工具、传什么参数"。然后调错了，环境返回"没事发生"，Agent 重新检索同一份文档，重新推导，再错一次。

NTU、复旦和上海 AI 实验室的 **Skill-as-Pseudocode（SaP）** 观察到了一个具体的失败模式：Agent 检索到一段技能散文，读到"拿起物品然后加热"——它发出一个 `put mug in microwave`。但环境只接受 `heat mug with microwave`。环境返回 "Nothing happens."。Agent 把这段反馈理解为"信息不够"，重新检索同一份技能，又读到同样的散文，再推导，再错。**不是 Agent 笨，是技能散文让 Agent 每次都要自己做"翻译"——从自由文本翻译成环境能接受的具体调用格式。**

SaP 做的事很简单：**把技能散文自动转换成带类型的伪代码，配上确定性质量检查。** 转换后的技能有两个信号：一份类型签名（告诉你这个技能干嘛、输入什么、输出什么、前置条件是什么），一份具体动作模板（告诉你环境到底接受哪个动词、哪个参数格式）。Agent 不再需要从散文中推导——一眼看到"怎么做"。

在 ALFWorld 134 局中，**SaP 以 82/402 的配对比分赢了 GoS（Graph of Skills）基线的 47/402，同时输入 Token 减少 22.8%，LLM 调用次数减少 14.5%。**

---

## 一、技能散文的问题：不是在帮 Agent，是在考 Agent

当前的技能格式本质上是给人读的散文。SKILL.md 里、MCP 描述里、GoS（Graph of Skills）库里，全是自然语言段落。**自然语言里混着三样东西：这个技能干什么（输出 schema）、怎么调（具体的工具调用格式）、注意事项和示例。** 这三样东西交织在一起，Agent 每次检索都要从同一段文字里重新提取"我需要调什么、传什么参数"。

问题不在于技能内容错了。问题是**散文格式强迫 Agent 在每一步都重新做一道阅读理解题。** 而阅读理解是有失误率的。一次失误，动作格式偏了一点，环境不接受，Agent 不知道是"参数错了"还是"理解错了"，于是重新检索同一份散文，再做一次阅读理解。同样的输入，同样的模型，大概率同样的失误。

**这个循环不是靠更强的模型能彻底解决的。因为问题不在推理能力，在信息格式。**

---

## 二、SaP 怎么做的：五阶段流水线 + 四重确定性校验

SaP 的核心思路是：**把技能文档从散文表达转换成带类型的伪代码合约 + 具体动作模板。** 转换过程用确定性校验把关——不是"让 LLM 生成然后信了"，而是"LLM 生成候选，四条规则逐条检查，不通过就驳回"。

### 2.1 阶段一：解析——把散文拆成段落单元

Parser 把每一份技能文档按 Markdown 标题拆成程序单元。**每个单元是一个独立的"子过程描述"段落**——比如一段讲"如何在房间里定位物体"，另一段讲"如何拿起物体"。这些单元是后面聚类和合约提取的基本单位。

### 2.2 阶段二：候选提名人——哪些段落说的是同一件事

对每个程序单元提取一个**框架元组**：动词、宾语、涉及的代码语言和脚本。然后把单元文本 Embedding，按框架和相似度做单链聚类。

**重点是——这一阶段故意保持高召回。宁可多聚类，不可漏掉。** 因为后面有验证器兜底，假阳性会被过滤掉。一个候选聚类 c 就是一组跨不同技能文档的段落，假设它们描述的是同一套底层逻辑（比如"物体转移"这个动作，在十几份技能文档里被写了十几遍）。

### 2.3 阶段三：合约提取——让 LLM 把散文写成类型签名

对每个候选聚类 c，发一次 LLM 调用，产出一个**类型化合约 κ**。合约包含：触发条件、输入 schema、输出 schema、前置条件、后置条件、副作用。

**这个 κ 就是散文的结构化替身。** 同样的信息——"技能要做的事、需要什么输入、产出什么"——原来散落在几段散文里，现在被压缩成了一份带类型的伪代码声明。Agent 一眼就能看完，不需要再推导。

### 2.4 阶段四：四重校验——不通过的合约不发牌

这是 SaP 最核心的设计。四条确定性校验，全部基于规则，不靠 LLM 打分：

**Coverage（覆盖）。** 合约的触发条件和 I/O 字符串覆盖了原段落中的多少 Token。防止合约写了跟原文对不上的东西。

**Binding（绑定）。** 合约要求的每个输入参数，在多少个源段落里能找到对应的文本绑定。防止合约提取了原文中没有的输入。

**Replacement（可替换）。** 源段落在结构上是否允许被 `invoke(κ, args)` 占位符替换。防止合约跟父技能的 Markdown 结构不兼容。

**Risk（风险）。** AST 扫描合约附带的脚本或资源，检查是否存在危险调用——例如裸 `rm -rf`、未声明的网络出口。加权风险分 ≥ 0.80 直接驳回。

四条检查通过后，合约进入 auto_promote（自动提升），写回技能库。**四条检查各有各的过滤重点，而且在 rejection 上是近乎互斥的——没有单一一条检查占了大部分驳回，说明四条都在出实际的力。** 论文用 30 个合成的负样本标定了决策阈值，确保零假阳性。

### 2.5 阶段五：重构——把合约嵌回父技能

通过验证的合约成为"子技能"。父技能中对应段落被替换为 `invoke(κ, {arg=value})` 占位符。然后两步后处理：

**Binding Extraction（BE）。** LLM 在 pass 一个具体调用点判断——"这里的 invoke 占位符是货真价实的吗？每个必需输入的绑定子串是什么？"确定性后置过滤掉绑定为空或 Token 不重叠的调用点。30% 的候选调用点在这一步被剔除。

**Rewrite Cleanup（RC）。** 清理残留的散文冲突——比如父技能的 example 部分提供了一个用不同动词的动作示例，跟合约不一致。一次 LLM 调用将这个矛盾点也重写为 `invoke(κ)` 调用。

## 三、运行时 Agent 拿到了什么

Agent 检索到技能后，SaP 给它的是**一份三合一套餐**：

1. **具体动作模板（先看）。** `goto {appliance}`、`heat {obj} with {appliance}`。这是环境直接接受的动词和参数格式。配上当前父技能的绑定值：`obj=mug, appliance=microwave`。

2. **重写的父技能骨架（再看）。** 原来的段落被替换为 `invoke(κ_heat, bindings)`。Agent 知道"这件事在技能文档的哪个位置、什么顺序"。

3. **内联的子合约（需要时看）。** 类型签名给出抽象保障：触发条件、输入输出、前后置条件。

**三样信息，三个层级，Agent 按需使用。** 不用再从一段散文里三样一起推导。论文的消融实验证明：如果子合约作为独立检索结果浮出而不是以内联 invoke 方式嵌入父技能，得分反而掉 27%——说明 Agent 真正需要的是"在父技能上下文中被替换好的类型化信息"，不是多一个检索条目。

## 四、实验说了什么

ALFWorld unseen 134 局，用 `skills_500` 技能库，Backbone 是 gpt-4o-mini，对比 GoS baseline。

### 同步赢分又省钱

配对比分：**SaP 82/402 vs GoS 47/402**（pooled McNemar $p = 8.2 \times 10^{-5}$）。**同时做到：输入 Token 减少 22.8%，输出 Token 减少 17.3%，LLM 调用次数减少 14.5%。** 不是"牺牲准确率换效率"——是准确率更高，效率也更好了。

### 增益根源于打破了"散文→动作"的死循环

论文做了详细的 case study。GoS 基线下的 Agent，每局平均落入"困惑→重新检索→重新推导→再次困惑"的循环 2.7 次。SaP 替换后降到 0.8 次。**类型签名的"what"和动作模板的"how"提供了两个原本被淹没在散文中的信号——Agent 不再需要一次次地从散文中提取这两个信息，死循环被剪断了。**

### Hierarchical-only 设计是关键

子合约不参与检索——Agent 只有检索到父技能时，才能通过 invoke 看到它们。如果把 children 作为独立检索池浮出，得分掉 27%。**因子化的好处在于"在需要的时候出现在需要的地方"，而不是"多给 Agent 一份看起来相关的检索结果"。**

## 五、这篇论文在说什么

SaP 跟之前聊过的 SkillSmith 在同一个方向上的不同位置。SkillSmith 在解决"技能读完后怎么执行得更快"。SaP 在解决"技能读的时候能不能不给 Agent 一段散文让它自己推导"。

**两个工作共享一个认知：技能的表达格式本身就是一个可优化变量。** 不是"agent 会不会用技能"的问题，是"技能的格式是否让 agent 能用得更少犯错"的问题。

如果你在维护技能库或者写代理技能，SaP 给了一条非常具体的建议：**别把技能写成散文。把"做什么"（类型签名）和"怎么做"（具体动作模板）分开。前者让 Agent 知道技能能给它什么，后者让 Agent 直接拿去执行。两个信号分开给，比混在一段散文里让 Agent 自己推导，效率高得多。**
