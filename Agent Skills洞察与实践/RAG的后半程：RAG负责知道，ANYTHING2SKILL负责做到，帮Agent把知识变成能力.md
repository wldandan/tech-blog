#   RAG的后半程：RAG负责知道，ANYTHING2SKILL负责做到，帮Agent把知识变成能力

> RAG让Agent能检索外部知识了，但搜回来的是一堆陈述性片段。怎么执行，Agent还得自己从头琢磨。华东师大和上海AI Lab的ANYTHING2SKILL换了个思路：从文档、手册、日志里**提前把"怎么做"编译成结构化技能**，推理时跟RAG一起用。qsv上成功率 98.85%，GitHub-CLI上 94.10%。RAG告诉你什么是对的，技能告诉你每一步该怎么走。

# RAG给了拼图，没给拼法

RAG已经是Agent获取外部知识的标准姿势。搜相关文档，把内容拼进prompt，让模型基于这些证据作答。知识密集型任务上，这条路走了无数遍，确实有效。

但Agent跟问答系统有一个本质区别。**Agent需要的不光是"什么是对的"，更是"怎么做才对"。**

打个比方。你要用 `gh` 命令行工具在GitHub上创建一个release。RAG能搜到 `gh release create` 的文档页，参数列表列得清清楚楚。但这份文档不会直接告诉你：先确认本地tag推上去了没有，用 `--generate-notes` 自动生成changelog，发布完记得验证assets上传。这些**过程性知识**散布在文档角落，有些甚至没明写出来，藏在示例和踩坑经验里。Agent每回都要现场从零拼，拼错了就挂。

这就是RAG的隐性负担。**Agent每次都在现场重建过程知识，而不是复用已经编好的技能。** 像你每次临时翻菜谱做菜，而熟练的厨师脑子里有固化的操作流程。RAG给了菜谱，没把它变成肌肉记忆。

# ANYTHING2SKILL：把"怎么做"从知识里编译出来

ANYTHING2SKILL的核心问题很简单：**外部知识里的"怎么做"，能不能提前提炼成Agent可以检索和复用的技能？**

整个框架是一条从异构知识到持久技能库的编译流水线。

![alt text](image.png)

## 一份结构化的技能合约

首先要定义技能长什么样。ANYTHING2SKILL里每个技能不是一段自由文本，而是一份结构化的**技能合约**。它包含：名称和描述、资产类型和分类节点、**触发条件和禁忌场景**（什么情况下该用、什么情况下绝对不要用）、**具体步骤和动作序列**、约束和注意事项、期望输出、支撑证据的引用、置信度分数。

这个定义背后有一个清晰的逻辑：**一个可执行技能必须同时说清楚四件事：什么时候触发、怎么执行、注意什么、产出什么。** 有步骤没触发条件，Agent不知道该在哪个时机调用。有步骤没禁忌条件，Agent可能在危险场景下照做不误。RAG 搜回来的文档片段里有所有这些信息，但它们从来没有被组织成这个结构。

## 五种资产类型：一个约束性分类体系

从零散文档里提取技能，最大的风险是**提取出来的东西不像技能**。LLM很容易把"qsv是一个CSV处理工具"这种事实陈述也当成技能，或者把太宽泛的概述和太细碎的参数说明混成一锅粥。

ANYTHING2SKILL用一个紧凑的分类体系来约束提取方向。所有技能归入五种类型：

- **宏协议**定义跨任务的全局流程；
- **会话技能**管理多步交互模式；
- **微技能**封装具体的原子操作；
- **安全规则**标注危险动作和禁忌；
- **知识参考**存事实性信息。每种类型在分类树里有自己的层级位置。

提取时每个候选必须能归入某个节点，否则直接丢弃。这个约束让提取器产出的是**过程性技能而非自由摘要**，也给后续的合并冲突检查提供了安全保障。

## 先规划、再展开

直接把长文档扔给LLM让它提取所有技能，结果是灾难性的：全是冗余、欠具体、或者直接总结成了摘要。

ANYTHING2SKILL的做法是两步走。先把每份知识记录切分成证据窗口，每个窗口对应文档中一个结构单元，附带标题路径、位置锚点等元数据。这一步确保每个提取出的技能都能**追溯到原始证据的具体位置**。

规划阶段，LLM扫窗口、识别潜在的技能候选，但只产出轻量级的提纲。展开阶段，每个候选被独立扩充为完整草案。这里有个防幻觉的关键设计：**窗口内容如果不直接支撑规划中列出的步骤，展开器返回空草案而不是硬编。** 草案还要过一道过滤器——必须包含至少一个过程性组件（步骤、动作、约束），否则丢弃。

## 三层管理：从草案到可用的技能库

单份文档产出的草案是局部的。不同文档可能描述了同一个技能的不同侧面，同一文档里相邻的窗口也可能在说同一件事。草案直接堆进库里，Agent没法用。

ANYTHING2SKILL入库前做了三层处理。

**文档级编译。** 同文档内，用分类感知的标识键对草案分组。同来源、同类型、同节点、同名称的草案被视为同一技能的局部描述。一个编译器把每组合并为一份规范技能，只准引用组内证据，不准外推。

**注册表级协调。** 新候选入库时，系统用混合检索（稠密语义 + 稀疏词汇 + 分类兼容）找到库里最相关的已有技能，由管理模型判断执行哪种生命周期动作。七种动作：CREATE新建、STRENGTHEN强化证据不改主体、REVISE修订内容、MERGE合并重复并废弃旧记录、SPLIT拆出子技能、UNCHANGED只记录来源、DISCARD丢弃低质量候选。**关键的安全约束是，更新类动作只在两个技能的资产类型和粒度级别一致时才被允许**，防止把安全规则合进可执行操作这种跨类型事故。

**可视技能树投影。** 注册表是真实数据源，但Agent检索需要清晰的导航。活跃技能按分类体系组织成树，同级重复合并。Agent看到的是可浏览、可检索的技能树，底层有完整的版本历史、证据链和生命周期日志。


# 推理时：RAG + SkillBank，双通道检索

Agent推理时不再只搜文档片段。它同时跑两条检索管线：**传统RAG拉取任务相关的源文档片段，SkillBank拉取相关的可复用技能。**

两者互补。RAG解决"这是什么"，SkillBank解决"这怎么走"。

![alt text](image-1.png)

实验结果很直白。qsv上：Base Agent 81.60%，RAG 单独 95.41%，SkillBank 单独 91.95%，两者组合 **98.85%**。GitHub-CLI 上更说明问题：Base 64.70%，RAG 单独 76.50%，SkillBank 单独 **82.30%（已经超过 RAG 单独）**，组合 **94.10%**。

GitHub-CLI上SkillBank 单独就超过了RAG单独，这件事很有信息量。GitHub-CLI的任务涉及多样化的仓库管理工作流，Agent需要识别正确的命令族、满足命令特有的约束、在不同场景下组合多个选项。**检索到文档片段只是第一步，从片段里推出正确的执行流程才是硬骨头。** ANYTHING2SKILL把这件事提前做完了，推理时的负担降了一大截。

组合使用时，效果叠加不是简单的加法。从 76.50% 到 94.10%，涨了 17.6 个百分点。过程性技能和声明性证据解决的是不同层面的问题，合在一起才覆盖了Agent执行所需的全部信息类型。


# 这到底意味着什么

ANYTHING2SKILL的洞察一句话就能说清楚：**知识库里有大量的过程性知识，它们现在是隐性的、分散的，Agent每次都要从零推导。把这些隐性经验提前编译进SkillBank，Agent就能从"每次从头读菜谱"变成"带着肌肉记忆进厨房"。**

这件事的适用范围远不止命令行工具。文档、手册、操作指南、故障日志、对话记录，任何包含"怎么做"的知识来源都能走这条编译管线。把它跟RAG组合在一起，就形成了一种**过程性记忆 + 声明性记忆的双通道架构**。前者提供已验证的操作路径，后者提供当前任务的事实背景。

局限也实实在在。分类体系是手工定义的，新领域要扩展分类树。技能生命周期管理依赖LLM判断，长尾场景的可靠性还得验证。但核心方向很有说服力：**让 Agent既能检索知识，也能积累技能。** 这也是ANYTHING2SKILL这个名字的含义——从任何形式的"东西"里，编译出可复用的能力。
