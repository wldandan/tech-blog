# SKILLWEAVER：Agent 面对 2209 个技能，怎么选出对的那几个

> 一个问题需要三个技能串起来解决，技能库里有 2209 个候选。选对一串比选对一个难得多。

## 选一个不难，选一串才是真问题

LLM Agent 越来越依赖技能来做事。Anthropic 的 Agent Skills 规范出了之后，社区里流通的技能已经有几千个了——Stripe 支付、Slack 通知、AWS 部署、PDF 解析，各种工具都被打包成了标准化的技能文件。

但技能路由（Skill Routing）的研究一直有一个盲区。现有工作全在解决"给一个需求，选一个最匹配的技能"。真实世界里，大部分用户请求需要多个技能串起来才能做。

比如："从上个月的销售数据库里拉数据，分析趋势，然后把报告发到 Slack。"这不是一个技能能搞定的。数据库查询、数据处理、消息通知——三个技能，按顺序协同。

阿里云的 Xueping Gao 在 SKILLWEAVER 里把这个问题叫 **Compositional Skill Routing（组合式技能路由）**：给定复杂查询和大型技能库，拆成原子子任务，为每个子任务检索技能，最后编排成可执行的 DAG。

## COMPSKILLBENCH：一个真实的评测场

论文先搭了评测场。从公开 MCP 生态爬了 2228 个注册服务器，清洗去重后得到 2209 个真实技能，24 个功能类别。Developer Tools 357 个，Finance 270 个，Integrations 229 个——前三名。

然后生成了 300 个组合式查询，分三个难度。Easy：2 个技能 2 个类别，150 条。Medium：3+3，100 条。Hard：4-5+4-5，50 条。每条标注了 ground-truth 子任务、技能 ID、类别和顺序。

核心指标：Decomposition Accuracy（DA）看拆出来的步数对不对，Category Recall@k（CatR@k）看检索的技能类别对不对，还有链级的 Chain Category Match。

## 框架怎么搭的

分三步。

先把复杂查询拆成原子子任务序列。每个子任务恰好配一个技能。decomposer 用 Qwen2.5-7B-Instruct。拆多拆少都翻车。

然后对每个子任务，用双编码器（all-MiniLM-L6-v2，384 维）做语义检索，FAISS 索引。值得注意的细节：编码的只是技能的名称和描述，没碰正文。2209 个技能，纯元数据向量检索，CatR@10 能到 69.0%。信号够强了。索引 15 秒，检索每个 batch 不到 15ms。

最后把各子任务的 top 候选按技能间兼容性（I/O 类型匹配、类别 Jaccard、关键词共现）选最优组合，形成 DAG，能并行的就并行。

## 瓶颈不是检索，是分解

这是论文最核心的发现。

标准 LLM 分解在 2209 个技能池上，CatR@1 只有 34.2%。但按 DA 分组一看就清楚了：当分解步数对的时候（DA=1），CatR@1 跳到 41.2%，CatR@10 到 81.6%。检索器够用。你知道该找几个技能，它就能帮你找到。

问题是 DA 在全体查询上只有 51.0%。超过一半的查询，LLM 连"该拆几步"都搞错了。主要是过度分解。Qwen2.5-7B 平均拆 4.09 步，ground truth 是 2.73。一步"发 Slack 通知"被拆成了"连接 API → 构造消息 → 发送 → 验证结果"。每一步都对，但粒度错了，后面的检索就全偏了。

## SAD：让 decomposer 看一眼"可选技能长什么样"

Skill-Aware Decomposition（SAD）的设计很聪明。它是一个检索增强反馈回路：先标准分解，然后对每个子任务跑检索，把 top-15 候选技能名作为提示词喂回 decomposer，让它重新分。

效果：单轮 SAD 把 DA 从 51.0% 拉到 67.7%（+32.7% 相对，p < 10^-6），DA_±1 从 71.3% 到 84.3%。更大模型受益更明显——qwen-max 的 DA 从 66.0% 飙到 92.0%。

论文做了一个关键实验来搞清 SAD 到底在做什么。把 vanilla 和 SAD 按 DA 是否匹配分组。在两者 DA 都对的 128 个查询上，CatR@1 一模一样（41.7% vs 40.9%，p=0.97）。但 SAD 修好了 75 个（25%）vanilla 步数错误的查询，这些查询上 CatR@1 从 23.6% 跳到 37.0%。

结论很干净。SAD 是个粒度修正器，不是词汇对齐器。它让 decomposer 看到了"技能库里大概有什么"，把步数调对了。检索精度本身没变。

还有一个更直接的验证：直接告诉 vanilla decomposer ground truth 步数（"拆成恰好 K 步"），DA 飙到 99.3%，CatR@1 到 39.8%，跟 SAD 在 DA=1 条件下的 41.2% 几乎一样。

还有个有意思的发现。Qwen2.5-14B 的 vanilla DA 是 32.0%，远低于 7B 的 51.0%。不是 14B 更差，是它过度分解更严重——14B 平均拆 4.72 步（ground truth 2.94），7B 拆 3.62 步。SAD 把 14B 压回 3.18 步，DA 跳到 68.0%。分解粒度问题是跨模型都有的，跟模型大小不完全相关。

## 不是靠背词汇表

SAD 是靠记住了技能池的词汇才有效的吗？

两个迁移实验。类别级留出：从 24 个类别抽掉 2 个（security + code-execution，191 个技能），SAD 仍然 +35.6% 相对 DA 提升。目标类别不在检索池里，关联类别的词汇就够了。技能级留出：随机抽 20% 技能（442/2209），+23.2%，比完整池的 +32.7% 略有退化但依然稳健。再加 200 条人类风格改写查询（零文本重叠），DA_±1 从 30.5% 到 50.5%（+66% 相对）。

不是模板记忆。

## Token 账

全量 2209 个技能塞进上下文要 88.4 万 token。SKILLWEAVER 把每个子任务暴露给执行 LLM 的技能数压到 2-5 个，平均 2.9 个。**上下文消耗降低 99.9%。** 加上 SAD Pass-2 的 15 个提示技能（约 1,100 token），总开销比全量技能小两个数量级。

## 还有哪些空间

CatR@10 在 70%，但 CatR@1 徘徊在 37-41%。这个 gap 是下一个目标。论文在附录里试了用 Qwen2.5-7B 做 listwise reranker，把 SAD top-10 重排，CatR@1 从 37.1% 升到 40.9%（+10.3%，p<0.01）。换成 BGE-base，进一步到 45.1%。gap 是表征层面的，换更强 encoder 或加 reranker 就能继续收窄。

组合阶段的评测还缺 ground-truth 兼容性标注。但 30 条查询的 pilot 执行实验里，76.7% 的技能链从头跑到了尾，pipeline 产出的计划是可执行的。

## 最后

这篇论文的看点不是"又出了一个技能路由框架"。它做对了一件事：**先定位真正的瓶颈，然后集中攻击它。**

组合式技能路由的直觉可能是"检索最难"——毕竟 2209 个技能里挑。数据说不是。检索器用最轻量级的元数据编码就能到 70% CatR@10。真正卡住的是 decomposer 不知道技能库里有什么，拆出来的子任务词汇跟可用的技能对不上。

SAD 的解法不需要换模型、不需要微调、不需要重训编码器。就是让 decomposer 在第二次分解时多看一张菜单——"你可以用的技能大概有这些"——它自己就把粒度调对了。

一个直接能用上的启发：搭 Agent 系统时，发现分解不对，别急着换更大模型。给 decomposer 看一眼可用的工具清单。
