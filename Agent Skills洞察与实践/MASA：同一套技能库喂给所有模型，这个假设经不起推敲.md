# MASA：同一套技能库喂给所有模型，这个假设经不起推敲

> Qwen3-8B 跑 ALFWorld，不装任何技能得分 32.1%。装了 Base Skill 之后掉到 25.0%。装了 DS-Adapter 之后 27.1%。装上 MASA 之后 57.9%。技能跟模型之间的适配问题，之前几乎没人认真讨论过。

## 一条沉默的假设

LLM Agent 领域有一条默认假设：**技能库是模型无关的。**

不管是手工写的还是从 trajectory 中蒸馏出来的，那些在决策时刻从外部库中检索出来的短文本程序指令——论文里叫 skill——通常被当作一套通用知识。给 Qwen3-4B 用是它，给 Qwen3-32B 用也是它。很少有人在部署之前停下来问一句：一套技能写法，对不同体量、不同行为模式的模型，效果真的一样吗？

华东师范大学的研究团队在 **MASA** 的实验部分做了这样一件事。拿同一套技能库，只改一个变量：表达粒度。保持底层策略原则不变，只调整措辞的详略程度，做成三个版本。Concise 约 370 token，Moderate 约 1092 token，Detailed 约 2737 token。然后在四个 Qwen3 模型（4B/8B/14B/32B）上跑 ALFWorld。

结果很扎眼。

**Qwen3-4B 的最优粒度是 Moderate，Qwen3-14B 和 32B 的最优粒度是 Detailed。Qwen3-8B 呢？所有三个技能版本的表现都比不加技能更差。** 不加技能时 Qwen3-8B 自然沿短路径高效完成任务，加错粒度的技能反而引入了多余的过程性推理模式，导致过度探索和不必要的谨慎。

换成 Gemma3 家族（4B/12B/27B）跑同样的实验，同样的趋势。粒度和效果的曲线不是单调的，不是"大模型就需要详细技能"或"小模型就需要简洁技能"这种简单规则能概括的。**同一个模型同一个粒度下，跨任务类型的得分差距可以超过 60 个百分点。** Qwen3-14B 在 Concise 技能下 PICK 任务做到 74.2%，COOL 任务只有 13.7%。

结论很清楚：**技能的有效性不仅取决于写了什么，还取决于怎么表达、对谁表达。** 内容写对的技能，如果表达方式跟目标模型不匹配，同样会在推理中内生噪声。

## MASA 怎么解决

MASA 的核心路线从三个设计需求出发，每条需求都直接来自实验发现：

1. **技能对齐必须是模型条件化的。** 不存在"最好的技能写法"，只有"对某个特定模型最好的写法"。目标模型的能力档案——参数规模、训练来源、能力强弱项——必须作为显式条件参与技能改写。
2. **对齐必须是搜索驱动的。** 粒度和效果的关系非单调，模型特异性强，不存在一条简单的经验规则。必须靠搜索来找到最优表达方式。
3. **对齐必须在任务类型粒度上操作。** 同一个模型内不同任务类型对同一组技能的反应差异巨大，只做全任务平均层面的优化不够。

基于这三点，MASA 分成两个部分：一个搜索时的分层技能进化管道，和一个部署时的轻量技能改写器。

### 分层技能进化

技能库的结构参照了 SkillRL，分成两层。**通用技能管跨任务的行为先验**，比如"始终检查动作是否正确解析"。**任务特定技能管领域内的操作步骤**，比如"先检查冰箱再看柜台"。两层分开优化，既减少了每步的搜索空间，也避免了归因混淆：一个编辑影响了全任务平均分，没法判断是通用策略对了还是某类任务的局部适配对了。

MASA 用一个更强的教师模型（DeepSeek-V4-Pro）来驱动搜索，目标模型全程冻结不动。

**Stage 1，通用技能的爬山搜索。** 每次迭代：用当前最佳通用技能集在全任务集上 rollout，教师模型 T 分析失败轨迹中通用行为层面的系统性问题，结合目标模型的能力档案和历史上高奖励技能集，输出改写后的通用技能。全任务评估，只接受比当前分数高的改写。跑若干轮或连续多轮不涨就停。

**Stage 2，任务特定技能的 UCB 树搜索。** 通用技能在这一阶段固定不变。每个任务类型跑一棵独立的 UCB 树，节点存该任务的技能，边代表一次教师改写。节点选择用 UCB1，平衡高奖励路径的开采和少探索分支的探索。被选中的节点在该类任务上 rollout，教师分析失败，输出新的子节点技能。新节点的奖励一路反向传播到根节点。各任务类型的树完全独立，可全部并行。

贯穿两个阶段的关键信号是**目标模型的能力档案（model card）**：包含模型架构元数据（族、参数规模、注意力配置、上下文窗口）、训练来源（训练数据规模、多语言支持）和能力画像（强项与弱项）。教师模型在每次改写时都以这份档案为条件——被提示"你正在为这样一个模型改写技能"。这份档案让改写不再是"教师觉得怎么写最好"，而是"教师判断对这个特定模型怎么写最可能有效"。

训练目标上，MASA 定义了一个调整后的每轮奖励 R：任务成功（SR）减去一个加权的 inactivity 惩罚。后者是 NHR——nothing-happens rate——环境状态完全不变的步骤比例。这些不产生任何环境反馈的"空转"，往往是技能表达诱导模型过度审慎或不知所措的直接信号。惩罚它的意义在于引导搜索远离那些"无害但无用"的技能表达。

### 技能改写器：把搜索压缩进一次前向传播

进化管道效果好，但需要几百到上千轮完整环境 rollout 和教师 LLM 的推理成本。MASA 做了第二件事：**训练一个轻量的模型条件化技能改写器，把搜索过程学成一个一次前向推理的 policy。**

训练数据从进化管道的搜索轨迹中采集。每条训练样本是一组三元组：(目标模型能力档案 M_F, 一组输入技能 S_in, 任务描述 d) → 进化的最优技能 S*_F。关键是 S_in 的来源被刻意分散在质量分布的各段：进化管道各阶段的中间产物、从其他模型的进化最优结果跨模型迁移过来的技能、教师的单轮改写（无迭代搜索）、以及各类加噪、截断、冗长变体。**这个多样性暴露确保改写器学到的是"不管进来的技能质量多差都能改好"，而不是"只对高质量输入有效"。**

改写器用 Qwen3-4B 做底座，标准 SFT 训练。部署时，给定目标模型卡、一组输入技能和任务描述，改写器在一次前向传播中输出适配技能，不需要环境交互，不需要迭代搜索，也不需要教师模型参与。

## 实验

三个环境：ALFWorld（文本具身任务）、WebShop（模拟在线购物）、Search-augmented QA（知识检索与推理）。四个 Qwen3 底座，全部用非思考模式。

### 技能进化：全环境全模型最优

**ALFWorld 上 MASA 在每个底座上都是最高平均成功率：31.4（4B），57.9（8B），64.3（14B），65.7（32B）。** 相比最强基线，gain 分别是 +4.3、+25.8、+20.0、+20.7 点。Qwen3-14B 和 32B 上 MASA 在所有六个任务类型上同时排第一，说明提升不是靠平均拉分，没有牺牲任何一类任务的覆盖率。

效率侧：**MASA 一致降低了平均交互步数。** 8B 从 39.1 降到 29.2，14B 从 36.7 降到 25.7。适配的技能让模型更准确地定位目标对象和执行动作序列，减少了冗余探索和失败重试。

**WebShop 暴露了一个更深的问题。** 不装技能时，8B/14B/32B 的基线表现全面低于 4B——14B 只有 2.8% 的成功率，4B 是 23.0%。原因是大模型在 WebShop 环境下会生成过于冗长的推理前导，大量的步骤预算被消耗在思考上而不在交互上。这是模型特定行为模式的一个典型标本：给 4B 好用的 Base Skill，在 14B 上不仅帮不上忙，还可能进一步恶化。**MASA 通过模型条件化进化，把 14B 从 2.8% 拉到了 29.2%，步骤数从 12.7 降到 8.0。**

Search-augmented QA 上，技能进化只在 NQ 和 HotpotQA 两个数据集上进行，但收益强力泛化到了五个 out-of-domain 数据集。Bamboogle 上 4B 从 12.9（最佳基线）跳到 61.3。

### 技能改写器：跨环境跨任务泛化，4B 改写器压住大教师

改写器在两种迁移设定下被测试。

**跨环境迁移**：改写器只在 WebShop 和 Search-augmented QA 的进化轨迹上训练，然后直接应用于 ALFWorld 的任务特定技能改写，不做任何环境内微调。**在所有四个底座上超过 DS-Adapter**（单轮教师改写），gain 在 +1.5 到 +3.0 之间。改写器学到的是模型特定的改写模式，不绑定具体环境。

**跨任务迁移**：在跨环境数据基础上，再加入三个 ALFWorld 任务类型的进化轨迹，但刻意排除评估所用的三个任务类型。gain 大幅提升，8B +8.8，14B +13.2，32B +7.4，全部超过 DS-Adapter。

最值得注意的是：**MASA-Rewriter 只有 4B 参数，却一致超过了由 DeepSeek-V4 支持的 DS-Adapter。** 一个小模型训练成的改写器，在没见过的新任务上改写得比它的教师更好，推理成本几乎可以忽略。

## 几个问题

**技能库的部署异质性是真实需求，但讨论远远不够。** 论文开篇点出：由于延迟预算、推理成本和硬件限制，现实部署中不同场景会使用不同规模的模型，不是全往最强的模型上挤。"一套技能适配所有模型"在生产中是有害的，四个 Qwen3 底座的实验已经表明 Base Skill 在某些模型上是负收益。但当前的技能库开源生态对此几乎没有关注。

**进化管道的成本限制了迭代轮次。** 全环境 rollout、教师模型调用、目标模型评估，每一步都是真实计算开销。论文把搜索中积累的中间状态转化为改写器的训练数据，这个"把搜索压缩为 policy"的设计抵消了一部分成本，但进化管道本身的边际成本仍然是瓶颈。

**能力档案（model card）的设计空间还很大。** 当前的能力档案是静态的预定义文档。如果目标模型在进化过程中表现出未曾在档案中记录的特定行为模式——WebShop 中那种冗长推理倾向就是典型——这些新发现的行为特征能否反向注入到能力档案中，指导后续改写？论文没有展开这层。

## 最后

MASA 做了一件事：**把"技能应该适配模型"这个直觉，用结构化的实验数据和可操作的工程方案立了起来。**

一种技能措辞能让一个模型受益却让另一个模型退化，这个现象在生产中一直在发生，只是很少被量化过。MASA 用四个 Qwen3 底座、三个环境、一组系统的实验把这件事摆到了明面上。然后给出了解法：不是让技能更好，是让技能跟得上模型。
