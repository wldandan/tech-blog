# 把SKILL.md"结构化"：解耦三层信号，技能发现+12%，风险识别+25%

SKILL.md正在变成Agent世界的通货。每个技能一个Markdown文件，里面塞着指令、控制流、工具调用、权限声明、输入输出契约……什么都往里头写。人读起来还算顺畅，但机器想搞清楚三件事——**这个技能什么时候该调用？分几步执行？会碰哪些资源？**——就得反复解析同一团自然语言。

这事正在成为一个瓶颈。不是推理能力不够，是表示层面就不对。

北大计算语言学团队最近发了《From Skill Text to Skill Structure:
The Scheduling-Structural-Logical Representation
for Agent Skills》论文，提出了**SSL（Scheduling-Structural-Logical）表示法**，一个专为Agent skills设计的三层结构化表示。论文的核心论点简单直接：**SKILL.md把三种性质完全不同的信号纠缠在了单一文本面上，应该解耦开**。

效果怎么样？在6,184个公开技能的语料上，技能发现的MRR@50从0.649拉到0.729；在 252个技能的风险评估上，宏观F1从0.409拉到0.509。这两个数字背后的东西值得展开聊聊。

# 一、一团文本，三种信号

典型的SKILL.md里有什么？名称、描述、标签、适用场景、触发条件、输入输出参数、执行步骤（夹杂工具调用和条件分支）、权限声明、资源依赖、注意事项、已知局限……

对人来说，这种文档没问题。但对机器来说，**三种信号被揉在了一起**：

1. **调度信号**：技能做什么、何时调用、输入输出是什么 → 用于技能发现和路由
2. **结构信号**：执行分几个阶段、阶段间怎么流转 → 用于执行编排和检查点
3. **逻辑信号**：每一步做什么操作、访问了什么资源、有什么副作用 → 用于风险评估和权限审查

现在通用做法是：下游系统需要什么信号，就把整个SKILL.md扔给LLM去理解。信息没丢，但**信号被稀释了**。而且每次都要重新理解一遍——没有持久化的结构化中间层。

这事我越看越觉得眼熟。Web当年从HTML乱炖进化出语义网和结构化数据，Agent Skills生态迟早也要走这条路。

# 二、SSL：三层解耦

![alt text](image.png)

图1 SSL（技能结构化语言）表示概览</center>

SSL的设计灵感来自一个你可能想不到的地方 —— **Schank和Abelson在1970-80年代的认知语言学理论**。一开始也觉得这个引援有点"为了理论而理论"，但读完三层的对应关系，确实挺贴切：

- **Memory Organization Packets（MOP）**：目标导向的经验组织 → 启发Scheduling层
- **Script Theory**：定型活动 = 有序场景序列 → 启发Structural层
- **Conceptual Dependency（CD）**：语义分解为基本动作结构 → 启发Logical层

一份SKILL.md经SSL映射后变成一个带类型的JSON图：

$$G_d = (r_{sch}, G_{str}, G_{log}, R_{cont}, R_{entry})$$

下面拆开看每一层。

## 2.1 Scheduling层：技能名片

这一层回答调度问题：技能干什么、什么时候调用、输入输出长什么样。

字段包括技能名称、目标描述、标签、意图签名（intent signature）、控制流特征（长时运行？有副作用？需要人工确认？），外加一个聚合的场景概要。

定位很明确：**一张不用展开全文就能快速比较的"技能名片"**。在几千个技能的注册中心里做检索，不需要每次都翻完整SKILL.md。

## 2.2 Structural层：执行剧本

这一层是**一个有向图**，节点是"场景"（scene），边是阶段转移。标准阶段序列是 **prepare → acquire → act → verify → finish**，也可以有recovery等。阶段类型使用受控词汇表，跨技能可比。

有了这层，执行引擎知道"当前在哪""下一步去哪""异常时回退到哪"——不需要重新理解整篇文档。

## 2.3 Logical层：原子动作与资源边界

最细粒度的一层。每个节点是逻辑步（logic step），边是步间微观转移。每个逻辑步从预设动作类型里取值：**READ、WRITE、CALL、EXPORT、CHECK**，并记录参数、效果和资源边界。

SSL做了一个重要取舍：**"原子性"是相对于源文档而言的**——源文档支持的最小可区分操作单元，不要求对应运行时真实原子操作。这避免了过度推断。

举个例子：一个文件操作技能在Logical层可能是`READ(local_file) → WRITE(local_file) → EXPORT(result)`，每一步都带typed资源标记。风险评估器直接看到"这个技能会读本地文件、写本地文件、导出结果"，不用从自然语言里推断。

# 三、怎么标准化：LLM提取 + 硬验证

SSL配了一个完整的标准化流水线，基于DeepSeek-V3.2：

1. 提取Scheduling记录（技能级接口信息）
2. 分解为场景序列（结构层）
3. 展开逻辑步（每个场景内的原子操作）
4. 硬验证（结构完整性、标识符一致性、枚举值合法性、包含关系、转移目标）

核心原则：**只提取，不发明**。每个字段必须有源文档证据。输出解析失败或硬验证不通过就重试。

论文对100个技能做了人工审计：**83%的SSL输出被判定为有源文档支撑**。17%的误差空间主要在模糊或underspecified的技能上，这也合理——原文本身就写得模糊，标准化器不该替它"脑补"。

# 四、实验：两个任务，两种用法

### 技能发现

6,184个技能做候选池，431个意图查询，同一个正确答案。所有方法用相同embedding模型（Qwen3-Embedding-0.6B）和FAISS索引，只变技能表示。

![alt text](image-1.png)

几个发现：

- **SSL-Shallow（仅规范化名称+标签+目标）就已经显著超过纯文本**。
- SSL-Rich（加场景类型、依赖、输入输出签名）达到最优——场景级和接口级结构信号对检索确实有用。
- 完整SKILL.md + SSL反而不如简短描述 + SSL —— **结构信号浓缩版比冗余全文更有效**。这事挺反直觉的，但仔细想想也合理：检索要的是信号的纯度，不是信息量。

### 风险评估

从语料分层抽样252个技能，标注六个风险维度：数据外泄、破坏性行为、权限提升、隐蔽执行、资源滥用、凭证访问。

标注流程用了Gemini-3.1-pro、Claude-Sonnet-4.5、GPT-5三模型投票 + 不一致维度人工复核。最终评估固定DeepSeek-V3.2做裁判，只变给裁判看的表示。

![alt text](image-2.png)

这里最关键的发现是**SSL不能单独用**。Full SSL的F1（0.422）仅略高于Full SKILL.md（0.409），但两者组合（0.509）远超任一单独使用。

为什么？因为风险评估需要的不只是"有没有写操作"（SSL能告诉你），还需要判断"这个写操作严重吗"（只有源文档能告诉你）。这个WRITE是守卫在人工审核后面的？还是一种假设场景？还是一次直接的文件覆盖？**SSL负责浮现线索，源文档负责提供上下文。**

### 消融实验的反直觉结论

在技能发现中，浓缩结构信号比完整全文更有效。但在风险评估中，两者必须搭配使用。

这揭示了一个更一般的原则：**不同任务需要不同粒度的信号**。意图匹配靠接口层面的线索就够了，文书中的旁白反而是噪音；风险判断需要完整的上下文，结构信号和原始文本缺一不可。

SSL-Shallow（仅接口字段）在风险评估上甚至比纯文本更差——因为它只提供了名称和标签，这些信息与风险无关，反而变成了噪声。这个反面结果其实很重要：**不是说加了结构化信息就一定更好，要看加的是什么信息。**

# 五、设计哲学：别替代源文档，站在它旁边

论文反复强调一个边界：**SSL是证据接口，不是源文档的替代品**。

这个定位我很喜欢。太多结构化表示方案一上来就想着"替代"自然语言文档，结果两头不讨好 —— 结构化不够完整丢了信息，文档又因为要迁就结构化格式变得难读。SSL的选择是保持与源文档的**邻接关系**：结构指向关键证据，原文提供完整上下文。

论文把SSL定位为技能系统中的**共享清单层（shared manifest layer）**。技能注册中心、路由器、策略检查器、人工审核者，各自不再需要从同一份SKILL.md中反复恢复相同的事实 —— SSL把这些事实持久化了下来。

远期来看，同一个表示可能支持更多场景：技能构建时让预期输入和阶段边界显式化；技能执行时用来做候选筛选、检查点追踪、敏感操作标记。

# 六、总结

当然SSL也存在局限性，如仅支持静态Skills分析；归一化结果依赖LLM能力，可能遗漏信息或者过度正则化模糊行为；评估只覆盖了技能发现和风险评估，没碰执行、规划、监控、事后改进等；风险标签来自三模型投票，不是安全专家的独立审计等。

但是，SSL解决了一个正在急速膨胀的问题：Agent Skills越来越多，光靠SKILL.md这种"文本优先"的表示已经不够用了。需要一个结构化的、机器友好的中间层。

三层解耦（调度-结构-逻辑）直接对应技能管理的三个核心需求：**发现路由、执行编排、风险审查**。实验数据支撑了这个设计。

但我觉得更有参考价值的是论文的定位感。SSL不声称要取代SKILL.md，不声称是标准，甚至不声称是端到端的技能管理方案。它就是 **"实用的一步"（a practical step）**。这种对自身边界的清醒。

如果你正在搭Agent Skills系统，这篇论文值得细读。在把几百上千个SKILL.md扔进向量数据库之前，也许该先想想——是不是缺了一个结构化的中间表示层。
