# SelSkill：技能不是"相关就该用"，而是"必要才该调"

> 在一个 78% 的情况下技能调用既没帮助也没损害的实验中，8% 的情况下调用技能反而让结果变差了。SelSkill 提了一个之前没被认真问过的问题：**就算一个技能跟当前任务相关，Agent 在这个时刻真的应该调用它吗？**

## 相关不等于必要

LLM Agent 调用技能的典型流程是任务来了，检索相关技能，调用，把结果注入推理，继续执行。这条链路里有一个问题很少有人停下来想：**为什么相关就应该调用？**

美团和复旦、上海交大、南大、北大的联合研究团队做了一个反事实分析。在多个基准上，启用技能访问只在约 14% 的配对轨迹中提升了最终结果。78% 的轨迹没有明显变化。**约 8% 的轨迹里，调用技能反而让结果变差。** 进一步分析发现，有害的技能调用往往跟同一上下文中有效的技能使用在语义上非常接近。表面看起来"对"的技能调用也可能搞砸。

这里面有两个结构性问题。

**第一，技能价值的分布极度集中。** 一个技能的效用不是"但凡相关就有收益"，而是只在一小撮特定的状态窗口里有效。在这个窗口之外，同一项技能要么是冗余的噪音，要么直接干扰了模型本来能走通的路径。

**第二，轨迹级反馈的信噪比太低。** 一轮调用最后任务成功了，不代表这轮调用是有效的，可能是后续某一步修正了它的影响。一轮调用最后失败了，也不能直接归因到它，可能是链上其他环节导致的。轨迹级别的成败标签，对于每一步"该不该调"这个局部决策来说，分辨力远远不够。

## SelSkill 怎么做：把"调还是不调"变成一个偏好学习问题

SelSkill 的定义很简洁：**在每个决策点，Agent 面对一组可用的技能，它为每一个技能做一个二选一的判断，调（invoke）还是跳（skip）。**

为了学这个判断，SelSkill 构造了两种不同粒度的偏好信号。

### 轨迹级偏好：整体的好坏

同一个任务跑出多条完整轨迹。如果一条成功、一条失败，就组成一个偏好对，成功的优于失败的。这个信号并不直接标定"哪个技能的哪次调用"是否有效，但它为整体行为序列的价值提供了一个约束：**模型起码要知道，走的这条路比走的那条路更好。**

### 决策点级偏好：局部调了还是没调

这是 SelSkill 的核心设计。

在每次 rollout 中，模型记录下每个 token 的 log-probability，然后计算每个候选技能决策点上的**预测熵**。熵高的位置说明模型不太确定下一步该怎么走，而这恰好是最值得分行做对比实验的岔路口。

SelSkill 做了一个诊断性实验验证这个直觉：在实际的技能调用位置创建 invoke/skip 分叉，然后比较两条路径在分叉之后的平均 token 熵。**注入技能的那条路径在后续动作预测中一致展现出更高的熵值**，技能注入本身就增加了模型在整合返回信息时的不确定性。高熵点因此成了优先做 invoke/skip 对比的候选。

确定了分叉点之后，SelSkill 从这个点同时向下跑两条路径：一条强制调用技能 s，一条强制跳过。两条路径共享完全相同的前缀轨迹 h_t，唯一的差别就是分叉点的那个局部决策。然后按照"结果-效率"规则排序。成功的路径优于失败的。两条都成功了，步数少的优于步数多的。两条都失败，丢弃不参与训练。

**这层设计的价值在于把"该不该调"的因果信号从轨迹级噪音中拎了出来。** 两条对比路径只差了一个 invoke/skip 决策，其他一切相同，效果差异可以更干净地归因到这个决策。

DPO 优化的时候，SelSkill 还额外对决策点级偏好对做了一个**局部窗口掩码**：损失只在分叉点之后的 n 个 assistant turn 上计算，不把整条长轨迹都拿来做梯度传播。这让梯度更直接地对准 invoke/skip 决策的短期后果，不被下游几百步之外的无关环境动态稀释。

### 双粒度联合优化

轨迹级偏好和决策点级偏好被合并进同一个训练集，用同一个 DPO 目标优化。两者的分工很清楚：决策点级信号管"这个技能在此时该不该调"，轨迹级信号管"这条整体行为序列好不好"。双粒度联合之后，模型学到的不仅是调用质量的提高，还有调用频率的自然收缩。不该调的时候学会跳过。

## 实验结果

两个主基准：ALFWorld（多步具身决策）和 BFCL（多轮函数调用），外加 PopQA 和 Tau-bench 做域外迁移。

### ALFWorld

**Qwen3-8B 上，SelSkill 把任务成功率提高了 10.9 个百分点，执行精度提高了 29.1 个百分点。** 同时，技能调用频率从每轮 2.55 次降到 0.44 次，而技能调用成功率（SR@Invoke）达到 97.0%。模型不再是什么看起来相关都调，调的质量也大幅提高。Qwen3-4B 上同样是任务成功率上升而调用频率几乎不变，更小模型在学会了更好的调用判断后保持了收敛的调用策略。

### BFCL

**任务成功率从 18.5% 提升到 24.2%，执行精度从 44.0% 跳到 73.5%。** 一个值得注意的细节：SelSkill 把每轮技能调用数从 0.73 提到了 1.01，跟 ALFWorld 上的下降相反。原因在于 BFCL 场景中，技能调用成功率在基础模型中极低（44%），说明模型在调用格式、参数填写、前提条件检查上本身就存在严重缺陷。SelSkill 学会了让能调对的时候才调，调不对就跳过，所以有效调用次数反而上升了，无效调用被筛掉了。平均步数从 18.2 降到 14.2。

### 域外迁移

SelSkill 在 ALFWorld 上训练后，不做任何再训练，直接装到 Tau-bench 和 PopQA 上评估。Tau-bench 上任务成功率从 10.1% 提升到 20.2%，PopQA 上从 41.7% 提升到 55.5%。**学到的选择性调用策略转移到了从未见过的技能和领域上**，没在目标域上跑任何一轮训练。

### 消融

把双粒度拆开看。**只看轨迹级偏好（Episode only），BFCL 上任务成功率 17.3%，仍然低于 SelSkill 的 24.2%。** 只用熵引导的决策点级偏好（Entropy-local only），成功率 21.7%。再加一段分析：如果把决策点级偏好的构造方式从"基于熵"换成"基于技能调用位置"（Skill-call-local only），不是用熵来选高价值决策点，而是所有技能调用点都做 invoke/skip 分叉，任务成功率反而从 21.7% 下降。**不是所有分叉都有信息量，用熵来筛选决策点是有增益的。**

## SelSkill 没有做什么

这篇论文有一个很清晰的边界。**它不研究技能库怎么构建、怎么维护、怎么改进。它只做一件事：给定一个固定的技能库，Agent 在每个决策点判断该调还是该跳。**

这个边界划定本身就隐含了一个判断：技能调用的优化空间不在技能本身的内容里，而在 Agent 对"什么时候用"的判断里。目前为止，大部分技能优化工作都在前者，做更好的检索、做更强的技能蒸馏、做更精细的库维护。SelSkill 开的是后一条路。

## 最后

SelSkill 的核心洞察：**"有技能"不等于"该用技能"。** 这个判断在人类操作者手中是直觉，但在 LLM Agent 的自动化流水线里，之前从未被视为一个独立的优化目标。

双粒度偏好学习从两个方向夹住了这个问题：轨迹级信号管整体效用，决策点级信号管局部必要性。联合训练之后，模型学会了在不确定性低的时候相信自己的推理跳过技能，在不确定性高的时候精准调用技能兜底。调用次数不一定降，但调对的比例一定会升。
