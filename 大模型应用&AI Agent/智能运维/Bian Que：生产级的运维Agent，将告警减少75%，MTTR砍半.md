# Bian Que：生产级的运维Agent，将告警减少75%，MTTR砍半

每天几十次上线，上百号人盯着，告警响了就得跳起来 —— 做在线引擎运维的兄弟都懂这种滋味。

快手的搜索团队干了件实在事。他们在电商搜索引擎上跑了一个叫**扁鹊（Bian Que）**的Agent框架，已经稳定运行了六个多月。效果怎么样？**触发告警量下降75%，非 actionable告警绝对量下降约95%，根因分析准确率80%，MTTR压缩超过50%**。

Bian Que的代码也开源了，下面聊聊这个框架到底做了什么，以及为什么它和市面上那些"LLM + 运维"的demo不一样。

# 一、真正的瓶颈不是推理，是编排

聊框架之前，先想一个问题：给运维Agent接上LLM，最难的是什么？

很多人会说是推理能力不够强。但是，难的并不是推理本身。上下文给够了，中等规模的开源模型做运维推理也不差。真正难的是：每次事件，到底该把**哪些数据和知识**塞给模型。快手团队认为 —— **瓶颈不在推理，在编排（Orchestration）**。

为什么？一个真实的在线引擎系统，数据源可能有上千个：时序指标、结构化日志、变更事件、调用链追踪、业务指标……每次告警或版本发布，你把所有数据一股脑塞给LLM，模型根本不知道该看什么。关键信号被噪音淹没，幻觉就来了。

反过来，如果让工程师手工为每种事件指定"查哪些数据、用哪些知识"，面对每天几十次上线、几十个业务模块的组合爆炸，这事根本不具备可操作性。

扁鹊解决的就是这个问题：**如何自动地为每个运维事件精准匹配正确的数据和知识**。

# 二、三条防线：把问题前移

扁鹊把日常运维工作抽象成了三种标准模式，他们叫"三条防线"：

| 防线 | 做什么 | 一句话理解 |
|------|--------|-----------|
| **发布拦截** | 版本上线时逐项检查，拦下不稳定的发布 | 把问题挡在上线之前 |
| **主动巡检** | 定期扫描系统各项指标，发现潜伏风险 | 在告警之前发现异常 |
| **告警根因分析** | 告警触发后，自动做关联归因并给出诊断 | 告警来了能快速定位 |

这个排列顺序本身就是一种态度。大多数运维Agent一上来就盯着告警根因分析做文章，但扁鹊把**自动化的发布监控和主动巡检放在了告警前面** —— 先尝试把问题消灭在萌芽阶段，告警驱动的诊断是最后一道防线，不是第一反应。告警量 75% 就是这么降下来的：问题还没来得及触发告警就被发现和解决掉了。

三种模式共享一个统一的计算公式：

$$o = f_{LLM}( D(e), K(e), p_s )$$

`o`是结构化输出（诊断、判定、建议动作），`D(e)`是从生产信号里组装的事件相关数据，`K(e)`是相关运维知识，`p`是编码场景逻辑的Agent提示。

这件事有个我很欣赏的设计选择：**框架不造新的监控设施，直接复用已有的数据栈**。挑战从来不在于"能不能采到数据"，而在于"该把哪些数据摆到LLM面前"。

# 三、Agent矩阵 × 技能池：正交分解

跨业务线、跨服务模块的运维复杂度，扁鹊没做那种一个大而全的Agent，而是用矩阵设计从两个正交维度去拆问题。

<center>

![alt text](image-1.png)

图2 智能体矩阵与技能池</center>

**Agent承担场景逻辑**：三种Agent分别对应发布拦截、主动巡检、告警分析，各自封装场景逻辑。三种Agent共用同一个底层LLM，只在场景知识上和所调用的Skill上不同。

**Skill承担业务模块的上下文**：每个Skill针对一个具体的（业务线 + 模块）组合，定义该组合需要查哪些数据和知识。

**正交组合把它们拼在一起。** 一次Agent执行 = Agent的场景逻辑 + 匹配到的Skill。运维事件来了，框架先匹配到对应的Agent类型，再通过事件元数据的关键词匹配选一个或多个相关Skill。这种拆分让Agent和Skill可以各自独立迭代 —— 改Agent的推理策略不用碰Skill，调Skill的数据配置也不用动Agent。

# 四、最核心的创新：灵活技能编排

这是整个框架里我觉得最巧妙的部分。

每个Skill是一个结构化的YAML文档，三个字段：

- **LoadDataSchema**：一份JSON规范，声明该Skill要调哪些数据源指标、日志、变更事件、链路追踪）、查哪些知识库、带上什么参数、哪些是必选。
- **Prompt**：一个结构化模板，告诉LLM拿到数据后按什么步骤分析，出结构化诊断。
- **Meta**：名称、版本、描述、标签，用来做关键词匹配。

关键在于——**这些Skill不是人写的，是LLM自动生成的，而且可以通过自然语言反馈持续修改**。

## 4.1 自动生成

当一个新业务场景首次遇到运维事件、但没有匹配的Skill时，框架自动触发生成。LLM拿到两份"种子说明书"：

1. **源描述符**：这个平台上有哪些数据源可用，每个数据源是什么语义。
2. **能力描述符**：对于当前这个场景，哪些数据源是相关的。

LLM生成Skill后，框架会**真的去执行LoadDataSchema里的调用** —— 如果返回的数据格式不对、字段缺了，错误信息会喂回给LLM，让它重来。最多三轮。已有的相关Skill也会作为参考样本传入，帮助跨场景迁移。

## 4.2 人工修正

运维同学如果发现Agent分析有偏差，直接在消息界面用自然语言描述问题和期望行为就行了。框架会自动判断是该调整LoadDataSchema（数据没查对），还是该改Prompt（数据查对了但推理逻辑有问题）。**整个过程零代码改动**。

# 五、一条反馈，两条进化路径

这个设计也很有意思。扁鹊里每一个运维事件跑完后都会产生一条案例记录，如果运维人员给了反馈，这条反馈会同时走两条路：

```
f → MemoryWrite(c) → KnowledgeDistill(c)   （知识进化）
f → SkillRefine({s_i}, f)                   （技能进化）
```

## 5.1 知识路径

短期：每个案例写入记忆库，下次类似事件来了，Agent先翻最近的案例当"工作记忆"参考。

长期：每个新案例触发LLM自动提炼可复用的经验 —— 模块A影响模块B的规律、某个指标的异常模式意味着什么、哪种告警组合通常指向同一个根因——这些被沉淀为**持久的领域知识条目**。每天一次离线任务做去重和冲突消解，防止无限膨胀。

知识库用了**三种索引**：KV索引查确定性知识，KKV索引查关系知识（"模块A对模块B在GMV指标上影响有多大"），向量索引做模糊语义检索。

## 5.2 技能路径

同一条反馈还被用来定位Skill的缺陷：
- 如果问题是**数据没查全或查错了** → 改LoadDataSchema
- 如果问题是**数据查对了但分析跑偏了** → 改Prompt

第三种情况是知识条目本身写错了或过时了，这事归知识通路管，不归Skill路径。

两条路径天然耦合。知识库越丰富，新生成的Skill质量越高；Skill越好用，产生的案例记录越干净，知识提取的质量也越高。这是一个**正向飞轮**。

# 六、生产环境六个月，数据说话

扁鹊在快手电商搜索引擎上跑了六个多月。这个系统服务数亿DAU，几十个核心服务模块，约100人的跨职能团队维护，日均数十次上线。LLM基座用的是Qwen3.5-35B-FP8，跑在一张L20 GPU上。

### 生产环境核心指标

![alt text](image-3.png)

几个数字值得展开说一下。

**75%的告警量下降**主要受益于前两道防线——大量问题在触发告警阈值之前就被发布拦截和主动巡检解决了。非actionable告警比例从80%降到15%，再叠加告警总量的缩减，**运维人员实际面对的噪音只有原来的约5%**。这个95%的降幅意味着什么？意味着以前PagerDuty一晚上响十次，现在可能一周才响一次。

**80%的RCA准确率**还有提升空间（论文也坦诚，误差主要集中在新上线的服务，知识库还没攒够经验），但配合95%的案例在5分钟内出诊断，实际价值已经很扎实了——运维同学不再是盯着满屏监控大盘盲猜，而是拿到一份有证据支撑的结构化诊断建议。

### 离线评估

![alt text](image-4.png)

在104个真实运维事件（44个告警、60个巡检）上：

- Skill自动生成：pass@1就达到**78.8%**，五次采样后**94.2%**，仅6个案例没搞定
- 对剩余6个案例做人工自然语言反馈修正：端到端pass@1拉到**97.1%**，pass@3达到**99.0%**

### 消融实验

两个关键结论：

- **把Flexible Skill换成静态手工配置**：pass@5从94.2%跌到83.7%（−10.5pp）。而且有16.3%的案例在pass@5下也解不了——不是采样不够，是静态规则有**不可约的盲区**
- **禁用历史知识检索**：pass@5掉到86.5%（−7.7pp），并且多采样几次也补不回来——LLM缺少业务先验，多试几次只是从同一个信息不足的分布里重复采样

### 反馈机制的长期价值

这个实验让我印象最深。他们在线上做了一个对照：

- **不开反馈机制**：准确率13天内从75%跌到32%——新出现的告警类型直接暴露了冻结Skill的覆盖盲区
- **开反馈机制**：系统自动从每次错误中总结学习，准确率稳定在80%以上；偶尔因为全新故障模式出现的下滑会自我纠正

简单说，**没有自进化能力的运维Agent，部署后就开始慢慢失效**。这也解释了为什么静态规则方案走不远。

### LLM基座怎么选

在36个事件子集上对比了7个模型。结论很明确：**30B以上参数量的模型表现接近**，DeepSeek-V3.2在pass@5达到100%。从35B降到9B，pass@1直接掉了25个百分点。再往下到4B和0.8B，性能灾难性退化。**生产部署建议35B起步**。

# 七、结论

扁鹊这个工作，核心贡献不是"又一个用LLM做运维的系统"，而是把问题的焦点从"怎么让LLM推理得更准"扭到了"**怎么让LLM每次都能看到该看的数据**"。

三个设计让这件事跑通了：
- **三条防线范式**：把零散的运维动作收敛到统一框架里，而且把拦截时机前移了。
- **灵活技能编排**：数据路由本身变成LLM生成的、可演化的工件，人只需要用自然语言给反馈。
- **统一自进化**：知识和技能在一条反馈闭环里协同改进，越用越好而不是越用越差。

一些坦诚的局限和未来方向

扁鹊目前覆盖推理和诊断，还存在一些局限：

- 框架目前做的是诊断和推理，**修还没自动化**（回滚、扩缩容这些执行动作还是人来做）。
- Skill匹配目前用的是关键词，换成基于事件元数据的embedding路由可能会更好。
- 单个事件内多Agent的协同还没解决——遇到级联故障时主要靠人协调。
- Skill和知识的协同飞轮效应目前只是定性观察，要定量测出来需要一年以上的数据积累。
- RCA剩下的20%误差集中在刚上线还没攒够经验的服务上，这个要看自进化机制长期能不能填补。
