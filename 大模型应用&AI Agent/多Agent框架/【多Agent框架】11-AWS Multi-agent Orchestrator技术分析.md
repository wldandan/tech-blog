## 1 概述

[多智能体系统](https://zhida.zhihu.com/search?content_id=256175253&content_type=Article&match_order=1&q=%E5%A4%9A%E6%99%BA%E8%83%BD%E4%BD%93%E7%B3%BB%E7%BB%9F&zhida_source=entity)是指由多个相互作用且基于LLM构建的智能体所组成的系统，多智能体之间通过框架实现信息共享、通信和协调行动，从而共同解决问题或完成特定任务。如何对智能体进行有效的编排、管理与利用，是当前业界的一大难题。当前，各大厂商都开始布局多智能体框架，如OpenAI的Swarm、微软的[AutoGen](https://zhida.zhihu.com/search?content_id=256175253&content_type=Article&match_order=1&q=AutoGen&zhida_source=entity)、以及亚马逊的Multi-Agent Orchestrator。

**Multi-Agent Orchestrator提供了一整套多智能体系统编排的解决方案，该框架的主要目的是合理管理多智能体** ，并智能地根据用户查询匹配最合适的智能体，同时在交互过程中提供上下文信息。该框架主要有意图分类、灵活的智能体响应（支持流式与非流式输出）、上下文管理、架构可拓展、通用部署、智能体重叠分析、提供默认智能体等特性。

## 2 Multi-Agent Orchestrator框架关键技术

### 2.1 框架架构

Multi-Agent Orchestrator框架的核心模块是**编排器** ，负责**协调分类器、智能体、存储器和检索器等模块，以完成对用户的响应，同时提供回避机制以处理各种错误场景** 。编排器由分类器（Classifier）、智能体集合（Agents）、检索器（Retrievers）以及会话存储器（Conversation Strorage）四大模块构成。

  * **分类器** ：根据用户请求、已注册智能体的描述信息以及用户历史会话信息，选择用于处理用户请求的智能体。
  * **智能体集合** ：用于处理用户的不同请求。各智能体功能相互独立，避免影响分类器的执行结果。
  * **检索器** ：为智能体补充额外知识，除检索功能外，内置检索器支持根据知识路径完成数据到向量库的转换。
  * **会话存储器** ：存储用户与智能体间的历史对话信息，并根据场景提供给不同的模块。



图1 Multi-Agent Orchestrator框架运行流程

如图1所示，编排器的核心工作流程可分为如下六步：

  1. 编排器接收用户请求，并将用户请求转发给分类器。
  2. 分类器提取用于分类的详细信息（包括从会话存储器中提取该用户与所有智能体的历史交互信息，以及智能体的描述信息）。
  3. 分类器识别用户意图，并根据历史对话信息与智能体的描述信息，选择与用户意图最匹配的智能体。
  4. 框架通过分类器选择的智能体处理用户请求。此步骤中，智能体可根据配置，通过检索器检索外部知识库或通过API调用外部工具，增加智能体的能力。
  5. 会话存储器将用户与智能体间的交互信息保存至存储媒介中。
  6. 框架返回对用户请求的执行结果。



总的来说，编排器接收用户的输入，能够结合会话存储器的历史对话信息，使用分类器识别用户意图，并选择最匹配的智能体处理用户请求。同时，在各个步骤中，编排器能根据配置项，调整存储器的存储行为以及对各种异常行为的进行处理。

### 2.2 分类器

**分类器可简单理解为特殊的智能体** ，其**核心通常是一个可调用工具的大语言模型，主要用于根据用户的历史行为以及用户输入，判断用户意图，并从已注册的智能体中选择与其最为匹配的智能体** ，是编排器的关键模块。分类器作为编排器的总入口，是用户输入和智能体之间的纽带，其效果直接决定了框架的最终性能。框架支持内置分类器和自定义分类器两种。

**内置分类器**

框架内置的分类器共有以下三种类型：

  * 基于亚马逊[Bedrock](https://zhida.zhihu.com/search?content_id=256175253&content_type=Article&match_order=1&q=Bedrock&zhida_source=entity)的分类器（默认模型为 Claude 3.5 Sonnet）。
  * 基于[Anthropic](https://zhida.zhihu.com/search?content_id=256175253&content_type=Article&match_order=1&q=Anthropic&zhida_source=entity)的分类器（默认模型为 Claude 3.5 Sonnet）。
  * 基于OpenAI的分类器（默认模型为 [GPT-4 Turbo](https://zhida.zhihu.com/search?content_id=256175253&content_type=Article&match_order=1&q=GPT-4+Turbo&zhida_source=entity)）。



这三种分类器仅在模型上存在差异，执行流程均保持一致：

  1. 首先，分类器接收来自编排器的用户输入和所有智能体的对话历史信息，并从中分析用户此次请求的意图。
  2. 然后，分类器读取编排器已注册的智能体的配置信息（包括智能体描述、可使用工具等），并返回最适合用于处理用户意图的智能体。



若注册的智能体均不适合处理用户请求，分类器会根据OrchestratorConfig的相关配置，选择使用默认的通用智能体处理用户请求，或是返回特定告警信息。此外，以上三种内置分类器均支持对模型、模型可用的最大token数、温度、生成多样性等超参数进行配置。

**自定义分类器**

编排器除三种内置的分类器以外，还支持用户自定义分类器，以增强框架在特定领域上的编排能力。

自定义分类器主要有以下两个步骤：

  1. 创建自定义分类器类，并继承Classifier类。
  2. 重写process_request方法，该方法是自定义分类器的核心，通过该方法，用户可自定义对输入的意图分析以及处理。



### 2.3 智能体集合

智能体是框架的基础模块，用于处理用户请求并帮助用户完成任务。智能体的核心属性如下：

  * name：表示智能体名称的字符串。
  * id：智能体的唯一标识符，根据名称自动生成。
  * description：描述智能体的功能和专业知识的字符串。
  * save_chat：一个布尔值，表示框架是否保存此智能体与用户的历史对话信息。
  * callbacks：一个可选的AgentCallbacks对象，用于处理流式响应中的事件。



由于分类器使用智能体的描述与用户的意图进行匹配，因此，智能体的描述对框架至关重要，需要对智能体的描述信息应尽可能详细和全面。此外，智能体响应用户请求的返回值支持非流式响应的ConversationMessage或流式响应的AsyncIterable类的对象。同时，框架为智能体提供了工具类，支持智能体调用工具。

框架提供了多种内置的智能体，并提供了让用户自定义智能体的能力，内置智能体和自定义智能体构成一组可供开发者选用的智能体集合。

**内置智能体**

框架支持以下几种内置智能体，基本支持亚马逊Bedrock提供的所有智能体，以及以OpenAI和Anthropic的语言模型为基础的智能体：

  * Bedrock LLM Agent：利用相关 API 与 Bedrock 支持的多种语言模型进行交互
  * Bedrock Agent：可直接使用亚马逊 Bedrock 中实现的所有智能体。
  * LexBot Agent：可直接使用亚马逊 LexBot 相关的智能体。
  * Lambda Agent：可直接调用AWS Lambda 上的各种函数。
  * OpenAI/Anthropic Agent：使用 OpenAI 或 Anthropic 提供的语言模型作为智能体。
  * Comprehend Filter Agent：亚马逊提供的智能体，可根据情绪、个人身份信息等对文本内容进行分析和筛选。
  * Bedrock Translator Agent：使用 Amazon Bedrock 的语言模型在不同语言之间翻译文本。
  * Bedrock Inline Agent：亚马逊提供的动态创建特定用途的智能体。
  * 链式智能体：可链式执行上述智能体。



**自定义智能体**

智能体在框架中仅用于处理用户输入，即只需要实现对用户输入的处理即可。因此，自定义智能体可以为：语言模型、外部服务、用于数据处理和分析的类、规则处理输入的系统以及前面几种的混合模式。

自定义智能体可通过以下两个步骤实现：

  1. 创建自定义智能体类，并继承Agent类。
  2. 重写process_request方法，该方法是自定义分类器的核心，用于接收用户输入和历史会话信息。通过该方法，用户可自定义对用户输入的处理。



### 2.4 检索器

受限于训练数据，智能体在专有领域的效果往往难以达到生产要求。因此，框架提供了检索器，以补充智能体的知识储备。**检索器用于从大规模数据集或特定数据库中提取与查询相关的信息，并将检索信息提供给智能体，从而实现对智能体知识的拓展** 。框架中内置了亚马逊 Bedrock的知识库检索器。为该检索器指定对应数据位置后，其可自动完成将数据构建为矢量数据库的整个流程。智能体可以通过检索器检索知识库中的信息，从而更好的响应用户需求。

检索器和智能体的协作通常由以下步骤完成：

  1. 智能体通过processRequest方法接收用户查询。
  2. 智能体在将用户查询发送到语言模型之前，会调用检器的retrieve_and_combine_results方法，从数据源中获取与用户查询相关的领域知识信息。
  3. 智能体将检索到的信息与用户输入合并，并一同发送到语言模型中进行理解与处理。
  4. 语言模型根据用户查询、检索到的知识以及会话历史信息对用户请求进行处理，并将结果返回。



### 2.5 会话寄存器

**会话存储器用于管理对话历史信息，是框架实现多轮对话以及智能体感知历史行为、实现自我演进能力的核心模块** 。框架提供的默认存储器为内存存储，适用于短期会话场景，同时框架提供了DynamoDB存储和SQL存储，用于实际应用场景。对于特定任务需要，框架同样允许用户自定义存储模式。自定义存储类需要继承ChatStorage类，并通过相应的方法进行实现。

在框架中，会话存储器为分类器提供与用户相关的所有历史对话信息，但对单智能体仅提供用户与该智能体相关的对话信息。

### 2.6 智能体重复性分析工具

框架除提供基础的编排能力以外，还提供了**智能体重复性分析工具，用于帮助用户分析框架已注册的智能体，识别冗余智能体、提高分类器准确率以及优化框架执行效率** 。在理想状态下，框架中可供用户使用的智能体在功能上，应该是相互独立且互补的，从而能够更好的处理用户的不同意图。若智能体间重复性大，分类器将难以匹配处理用户输入的智能体，影响后续执行流程。

## 3 总结

总的来说，Multi-Agent Orchestrator是面向生产设计的，提供多种内置模块给用户。对于特定条件下不可用的情况，也支持用户根据实际应用场景自定义更适合的模块，并对框架进行替换升级。这些特性也使得该框架适用于广泛的应用程序，如从简单的聊天机器人到复杂的AI系统。但该框架也存在一定的局限性：

  1. 该框架仅支持Bedrock内置的动态编排智能体，大部分场景下仅支持对智能体的静态编排。
  2. 智能体仅能通过记忆的方式进行自我调整，自我演进能力有限。
  3. 只可使用已注册的智能体处理用户需求，框架无法按需自动生成智能体，分类器失效场景无法有效处理用户请求。
  4. 重复性分析工具比较简单，只可通过词频判断，没有考虑词性。
