# 你的Agent上线了，但评估它的代码还没写 —— AWS的EvalAgent想解决这件事

![alt text](image-3.png)

先说个你可能经历过的事。

你花了两周搭了一个Agent，能调工具、能多轮推理、能自己纠错。要上线了，老板问你："这玩意儿到底靠不靠谱？"你说："应该还行。"老板问："怎么评估的？"你沉默了。

**Agent评估是这个行业里最尴尬的瓶颈。** 每个人都在疯狂造Agent，但几乎没人在认真想"怎么评估Agent"这件事。AWS AI Labs最近发了篇论文，第一次系统地研究了一个问题：**能不能让AI自己写Agent的评估代码？**

答案是可以，但不是你想象的那种方式。

# 一、直接把Agent代码扔给编码助手？翻车了

最直觉的想法：把Agent源码和几条执行Trace扔给Claude Code，让它"写一个评估脚本"。能有多难？

**AWS团队试了，结果惨不忍睹。** 前沿编码助手直接生成评估代码，首次执行成功率只有30%。不是90%，是30%。而且平均每个Agent产出12个以上的指标 —— 什么p99延迟、token数、每分钟调用次数，花花绿绿一大堆，但几乎没有一条触及核心问题：**这个Agent到底完成任务了没有？**

论文把失败归纳成三种系统性的翻车模式：

**指标泛滥** ：没有评估领域的先验知识，模型会默认生成一大堆操作导向的指标。延迟、吞吐、token 消耗……这些对运维有价值，但对"Agent质量好不好"几乎不说明任何事。

**代码过度设计** ：生成的评估代码比需要的长2-3倍，散落在5-7个文件里，带着各种提前抽象和不必要的设计模式。一个评估脚本而已，搞得像微服务架构。

**计划和代码脱节** ：这是最精妙的一个失败模式。模型在规划阶段会写出很好的评估准则——"语义化评估任务完成质量"，到了实现阶段却变成了`"success" in output.lower()` 这种关键字匹配。**它知道该怎么评，但写不出来。**

这三种失败不是编码能力的问题。Claude Code写代码很强。**问题在于评估这个活儿需要的不是通用代码能力，而是"评估领域的专有知识" —— 测什么、怎么测、什么叫好的评估。**


# 二、EvalAgent怎么做：把评估知识编码成"技能"

<center>

![alt text](image.png)

EvalAgent评估流水线</center>

AWS团队搞了一个系统叫**EvalAgent**。核心思路很有意思：**通用编码助手不缺能力，缺的是评估领域的知识。把缺失的知识打包成"技能"（Skills），在每个阶段按需加载。**

技能分三类：

**流程指令** ：每个阶段的step-by-step工作流和约束。比如规划阶段的技能要求模型"先分析Agent架构和执行Trace中的行为模式，然后识别2-4个关键指标，每个指标捕捉一个独立的行为维度"。代码生成阶段的技能则规定了一个固定流程：`实现指标 → 构建主入口 → 自审 → 更新依赖`，以及一条硬约束v——v**先做最小可行版本，不要展开范围。**

**可复用的代码和模板** ：规划模板（含Agent 规格、指标、测试场景等固定结构）、报告模板（含执行摘要和故障分析结构）、Trace解析和DeepEval集成代码片段。这些东西减少了每次运行的方差，也阻止了范围放大。

**动态API文档** ：通过**Context7**实时拉取DeepEval、LiteLLM等库的最新API文档。这个决策有一个很现实的理由：**库的API进化速度远快于模型的训练周期。** 没有实时文档，生成的代码里充满了已废弃的API调用——这是无约束条件下执行失败的最大来源。

这三类技能组合成一条六阶段流水线，拿Agent源码和执行Trace进来，产出可执行的评估代码和可读的评估报告：

| 阶段 | 做什么 |
|------|--------|
| **1. 评估规划** | 分析Agent架构和Trace行为，产出结构化的评估计划（指标、评分标准、测试场景） |
| **2. 测试用例生成** | 按JSONL格式生成测试用例，每条含输入查询、场景类别、预期行为 |
| **3. Agent插桩** | 给原Agent加几行OpenTelemetry/Traceloop代码，轻量无侵入 |
| **4. Trace收集** | 插桩后的Agent执行测试用例，产出OTEL兼容Trace，过滤到Agent相关字段 |
| **5. 代码生成** | 把评估计划翻译成可执行Python代码：指标实现（确定性检查 + LLM-as-Judge）+ 编排器 + 依赖文件 |
| **6. 生成报告** | 结果合成结构化报告：执行摘要、逐指标分析、失败根因、优先级建议 |

每个阶段只加载相关的技能，渐进式披露，不相关的技能不出现在上下文里。


# 三、怎么评估"评估系统"本身？

这引出了一个元问题：**你用什么来判断一个评估系统好不好？** 没有标准答案，也不知道"正确的评估"长什么样。

<center>

![alt text](image-1.png)

元评估框架</center>

AWS团队搞了一个元评估框架，核心思路是把问题变成**成对比较**（pairwise comparison）。同一套Agent源码和Trace，两个方法各生成一套评估方案，让一个专门的元评估器（用Claude Opus 4.5）从五个维度做对比判断：

| 维度 | 权重 | 问的是什么 |
|------|------|-----------|
| **指标相关性** | 30% | 指标的信号噪声比高吗？测的是有意义的东西还是凑数？ |
| **代码质量与复杂度** | 25% | 实现正确吗？组织清楚吗？有没有过度设计？ |
| **需求满足度** | 15% | 用户要测的东西测了吗？ |
| **计划质量** | 15% | 计划是否完整、可执行、不注水？ |
| **计划-代码对齐** | 15% | 代码是否忠实地实现了计划？ |

他们还明确加了一条**反长度偏见**规则：代码长不代表质量好。计划超过 1000 行需要特殊理由，超过 1500 行大概率输给更短的等价方案。

为了系统性地测这个，他们还搞了**AgentEvalBench** —— 据他们所知是第一个自动化Agent评估的Benchmark：**20个Agent，覆盖9个框架、14个领域、3个复杂度等级**。从TikTok小代理到多Agent数据标注系统都有。每个Agent配两种需求（泛化需求："评估这个Agent"和具体需求："评估医疗实体提取的正确性"）和5个测试场景。

元评估器的可靠性怎么验证？三个专家人工标注了40对评估方案（200个维度级判断），跟元评估器对比。**Fleiss' κ = 0.923（近乎完美的一致），案例级胜者匹配率 97.5%（39/40）。** Claude Opus 4.5和Sonnet 4.5分别当元评估器，交叉模型一致性86.6%。


# 四、关键实验结果

实验用2×2因子设计：两个主干模型（Claude Haiku 4.5和Sonnet 4.5）× 两种需求类型（泛化和具体），跑AgentEvalBench上20个Agent。

四个Baseline：B1单轮LLM直出、B2 Agent仅看源码不看Trace、B3 Agent看源码 + Trace直接生成代码、B4 Agent看源码 + Trace先规划再实现。

![alt text](image-2.png)

### 找Trace比看源码强，但代码更容易崩

Trace-based（B3）对纯源码（B2）的胜率是 73-77%，指标相关性维度尤其明显（80%）。Trace 能暴露 Agent 实际干了什么——运行时工具调用、错误恢复、决策链路——这些静态代码分析根本看不到。

但 Trace 解析本身引入了额外复杂度。B2 的 Eval@1 是 60%（Sonnet），B3 只有 35%。不过 B2 的分数有水分——它生成的有些评估跑的是单元测试桩而不是真实的 Agent 行为。Trace 评估虽然实现成本更高，但对于捕获行为失败和决策级错误，目前没有替代方案。

### 没有结构约束时，规划不如不规划

B3（直接生成代码）对 B4（先规划再实现）的胜率是 74%。不是规划没价值，是**没有领域知识的规划等于给范围放大发了一张通行证。** B4 生成的计划是 EvalAgent 的 4.4 倍（584 行 vs 134 行），代码是 6.6 倍（1902 行 vs 289 行），还在超过一半的 Agent 上生成了从未被调用的死代码文件。EvalAgent 的规划只有 134 行，指定了 3 个指标和具体方法，代码就刚好实现了这三个——19/20 的 Agent 上保持一致的 2 文件结构，零死代码。

### 需求越模糊，EvalAgent越有优势

用户只给一句"评估这个 Agent"（没给具体指标），EvalAgent 对 Baseline 的胜率到 **95-100%**。这个场景恰好最常见。**大部分人做评估之前自己都不知道该测什么。** EvalAgent 能从 Trace 行为里推断出合理指标，Baseline 在没明确指引时直接掉进指标泛滥的坑里。

### 复杂度上去了，评估技能顶得住

论文做了个干净实验：固定 Prompt 让系统生成 1 个、3 个、5 个指标，看 Eval@1 怎么变。Baseline 随指标增加一路往下掉，B3 从 55% 跌到 30%。**EvalAgent 反而从 60% 升到 65%。** 指标多了意味着更多 Trace 解析逻辑和跨指标依赖，Baseline 的失败概率成倍叠加。评估技能把 Trace 解析模式提取成了可复用组件，复杂性被控制住了。

### 实时API文档不是加分项，是必选项

关掉 Context7，Sonnet 的 Eval@1 从 65% 掉到 20%。45 个百分点。没有实时文档时，生成的代码用错误的模型 ID、过时的类继承方式、不匹配的构造函数。DeepEval 的 API 迭代比模型训练周期快太多了。

### 同一个通过率，失败的原因完全不同

Haiku 4.5 和 Sonnet 4.5 在 EvalAgent 上的 Eval@1 差不多（62.5% vs 65.0%），但失败类型几乎不重叠。**Sonnet 有 64% 的失败卡在同一个问题上**：DeepEval 默认 `async_mode=True`，Sonnet 只实现了同步 `measure()` 没写 `a_measure()` 协程。代码逻辑上没问题，加一行 `async_mode=False` 就行。修掉这一处，Eval@1 从 65% 直接跳到 **87.5%**。Haiku 的问题就杂了：Python API 误用（40%）、时间戳当字符串算（33%）、import 路径搞混——典型的小模型代码质量天花板。


# 五、说清楚局限

论文列了四个局限，不藏着掖着：

**Agent覆盖。** 20个Agent虽然跨了9个框架和14个领域，但全部是文本为中心的。具身Agent、多模态Agent不在范围内。

**模型依赖。** 所有实验都在Claude家族上跑。跨模型泛化性还需要验证 —— 虽然这个框架的设计本身是模型无关的。

**首次执行成功率还有空间。** 65%意味着大约三分之一的评估仍然需要人工介入调试。修掉DeepEval async那个问题能跳到87.5%，剩下12.5%的尾巴主要来自Trace解析的复杂性。

**元评估本身也是难题。** "这个评估好不好"本来就是主观判断，即使是人类专家在某些维度（计划质量 50%、计划-代码对齐 60%）上也不完全一致。


# 六、最后说两句

这篇论文触及了一个真实且被严重低估的问题。每个人都在造Agent，但Agent的评估还停留在"开发者自己写几个测试用例然后手动看Trace"的阶段。

EvalAgent做对的最关键的一件事是：它没有试图造一个"万能评估器"，而是承认**通用编码能力不等于评估能力**，然后把评估领域知识系统地编码为可组合的技能模块。评估技能的消融实验说明了一切：去掉技能，Eval@1 从 65% 掉回 30%。

如果你在维护甚至只是偶尔写Agent，这篇论文至少给了三个可操作的结论：**评估一定要看执行Trace，不要只看源码**（Trace对质量判断的提升是 73-77%）。**投时间打磨评估技能或模板**，别指望通用代码生成能替你搞定。**实时API文档检索是刚需**，不打开的话生成的代码基本跑不起来，差了 45 个百分点。
