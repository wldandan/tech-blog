# 评测挂了？先别判死刑，可能是评测Agent走错了路

> GUI Agent被越来越多地用来评测交互式软件，但它自己也会犯错。一次失败的测试，该归咎于被测软件真有Bug，还是评测Agent自己操作失误？DeepWisdom和港科大的DiagEval提出了一套诊断协议：不盲目重试，而是分析失败轨迹、定位不确定性来源、定向探测，把"软件真挂了"和"评测者误判了"区分开。WebDevJudge-Unit上准确率从 69.9% 提到 78.3%，RealDevBench上从 65.0% 提到 81.6%。

# 当评测者自己也会犯错

用GUI Agent评测交互式软件正在成为趋势。和传统静态代码分析不同，GUI Agent能像真人一样点击按钮、填写表单、滚动页面，检查软件是不是真的能跑通。这个方案有个自带的麻烦：**评测者本身不可靠。**

一个大模型驱动的GUI Agent在测网页应用时，可能因为自己没找到按钮、看错了页面状态、或者推理逻辑出错，就下结论说"功能不可用"。实际软件没问题，评测者操作有问题。这在自动化评估中产生的后果很恶劣：**一个能跑通的功能被误判为有Bug，开发者追着不存在的缺陷浪费时间。**

论文给这个问题起了个名字：**单轨迹可辨识性缺口（single-trajectory identifiability gap）。** 意思是，软件正确性是一个图级别的可达性属性。只要存在至少一条合法路径能到达目标状态，软件就是对的。但GUI Agent只能看到自己走过的那一条路径，一次失败只能排除这一条路径，不能证明目标状态不可达。从单条失败轨迹中，你无法判断到底是**环境真挂了（ENVFAIL）**还是**Agent自己失误了（AGENTFAIL）**。

现有应对方案都太粗糙。朴素重试是再跑一次看看，但如果Agent在同一类错误上反复栽跟头，重试多少次都没用。Best-of-N采样是多跑几次取最好的，烧钱且低效。自修正是让 Agent反思哪里错了再试，但LLM的自我批评往往强化原错误而不是修正它。这些方法本质上都在**添加更多尝试次数**，而不是**定向收集新证据**。

# DiagEval怎么做：把失败诊断变成定向证据采集

DiagEval的核心思路是：**失败轨迹里藏着诊断该往哪探的线索。** 从失败轨迹出发，分析不确定性类型，生成针对性的诊断探测，再根据探测结果更新对失败原因的判断。整个过程是一个**轨迹条件化的诊断协议**，不是无差别的重复执行。

## 失败解析：找到一个可以重新出发的点

拿到一条失败轨迹之后，DiagEval首先定位**分叉节点（fork node）**。这是轨迹中可以重新出发的最早点位，从这个状态开始尝试不同操作路径，最有可能把AGENTFAIL和ENVFAIL区分开。

这个点位不是随便选的。DiagEval用一个LLM裁判分析整条轨迹的压缩诊断摘要（Failure Diagnostic Summary），同时输出分叉位置、主导不确定性类型和诊断解释。FDS还包括从分叉点恢复所需的环境上下文，这让DiagEval可以**热恢复**，直接从分叉点启动诊断而不是重跑整条前缀。

## 四种不确定性，三种探测方式

DiagEval把导致Agent误判的不确定性归纳为四类（SOU）：

- **SOU-1 感知定位错误**：Agent没找到正确的UI元素，点错了按钮或看漏了图标；
- **SOU-2 不完全观测**：界面有折叠菜单、屏外内容或隐藏区域，Agent以为功能不存在；
- **SOU-3 推理幻觉**：Agent根据弱UI信号过早下结论，或者把已成功的操作误判为失败；
- **SOU-4 运行时不稳定**：网络超时、后端延迟抖动、UI状态偶发失败。

不同类型对应不同的诊断策略，DiagEval设计了三种探测分支（diagnostic probes）：

**Type A（替代路径）**：用不同的动作、模态或路径去实现同一个子目标。上次点了图标没反应，这次换用菜单入口。它回答的问题是"是不是上次操作方式的问题"。Type A一旦成功，强烈暗示AGENTFAIL：换个方式就能过，说明软件没问题。

**Type B（观测扩展）**：通过滚动、展开菜单、切换标签页来暴露之前不可见的UI区域。它回答的问题是"是不是还有东西没看到"。Type B一旦成功，同样暗示AGENTFAIL：功能一直在那，只是Agent视野太窄没发现。

**Type C（可复现性测试）**：在相同或微扰条件下重复执行，测试失败是否稳定。它回答的问题是"这是偶发波动还是真的不行"。Type C的失败强烈暗示ENVFAIL：怎么试都不过，说明确实有缺陷。

三类探测有一个关键的设计巧思：**归因不对称性。** Type A/B偏AGENTFAIL，Type C偏ENVFAIL。这使得多轮探测的结果可以累积为有方向的证据，而不是简单的"过了/没过"。

## 信息增益排序：只跑最有区分度的探测

每轮诊断中DiagEval生成一批候选探测分支（默认 5 个），但不全跑。它用**期望信息增益（EIG）**来排序，只执行 Top-3。

EIG的计算逻辑很直觉化：模拟这个分支跑出"成功"和"失败"两种结果后，归因不确定性各自下降多少，加权求期望。下降越大的分支越优先执行。结果就是，那些最能区分两种失败原因的探测被优先调度，Type C如果失败了就偏向ENVFAIL，Type A如果成功了就偏向AGENTFAIL。

## 贝叶斯更新 + 非对称停止

DiagEval维护一个内部归因信号 $p$，代表当前对ENVFAIL的倾向（从 $p=0.5$ 出发，中性）。每跑完一个探测，根据结果和分支类型用贝叶斯公式更新 $p$。

停止规则利用了结果不对称性：**一旦某个探测返回$verified_{success}$，立即停止并判定PASS**，因为一次成功就证明了目标状态可达。如果多次失败导致 $p$ 超过阈值 $τ_{env}$，提前判定 FAIL。否则继续诊断直到预算耗尽。这里的直觉是，成功是确定性的证据（它证明了有路可走），而失败只是概率性的证据（它降低了有路的可能性）。DiagEval 的设计让一次成功拥有最终的否决权。

# 实验：两个benchmark，全部基线，全面对比

所有方法在WebDevJudge-Unit（502 个单元级网页开发任务）和 RealDevBench（429 个可执行案例）上评测，报告端到端准确率和每案例成本。DiagEval仅当初始判定 FAIL 时触发诊断，每轮诊断从 N=5 候选分支中 EIG 排序取 Top-3 执行，起点中性先验 p=0.5，ENVFAIL 阈值 $τ_{env}$=0.7。

### 基线对比：DiagEval 加到任何框架上都能提

![alt text](image-2.png)

**用Gemini 3 Flash Preview作为GUI执行骨干时，DiagEval（×2）在WDJ-U上把AppEvalPilot的 69.9% 拉到 78.3%（+8.4），在RDB上从 65.0% 拉到 81.6%（+16.6）。** RDB的涨幅更大，因为它的任务更复杂、交互步长更长，评测Agent出错的概率更高，DiagEval纠正误判的空间也就更大。

RDB上趋势一致。同一Gemini Flash骨干，DiagEval（×2）比AppEvalPilot高了 16.6 个百分点。**RDB的案例交互链更长、页面状态更多，评测Agent更容易触发 SOU-2 不完全观测和 SOU-3 推理幻觉**，DiagEval的诊断价值因此更大。

### 假阴性恢复：诊断 vs 重试

![alt text](image-3.png)

只看FN子集（软件本来可用但被误判为失败的案例），DiagEval（×2）在WDJ-U上恢复了 59.6%，在RDB上恢复了 62.1%。朴素重试（NR ×2，不分析直接重跑两轮）只恢复了 28.1% 和 46.2%。

RDB上NR的恢复率（46.2%）远高于WDJ-U（28.1%），因为任务步骤更长、随机性更大，重试有一定概率走出不同路径。但即使在这个利好重试的场景下，**DiagEval仍高出 15.9 个百分点**。WDJ-U上差距更极端（59.6% vs 28.1%），因为WDJ-U的任务更短更确定，重试大概率重复相同失败，诊断收益被放到最大。

### 消融：拆开看每一层贡献

如表2所示，消融实验在FN子集上拆解三个组件：

- **NR（朴素重试）**：无FDS、无分叉点、无SOU引导。WDJ-U FN恢复 17.5%/28.1%（×1/×2）。
- **NR+IE**：加上FDS解析和分叉点定位，但没有SOU引导的分支选择。FN恢复 20.2%/30.7%。比NR好一点，因为从正确位置恢复至少比从头重跑强。但增益有限，知道在哪错了却不知道该用什么方式验证。
- **DiagEval完整版**：加上SOU引导的定向分支和归因更新。FN恢复到 45.6%/59.6%。**SOU引导的分支选择是增益的主要来源**，它把"再试一次"变成了"针对最可能的误判原因做实验"。

归因分数 p 与实际情况高度一致：恢复的FN案例平均 p 在 0.33-0.34（倾向 AGENTFAIL），保留的TN案例平均 p 在 0.80-0.87（倾向 ENVFAIL）。ROC-AUC在RDB上单轮 0.620 两轮 0.761，WDJ-U上单轮 0.720 两轮 0.763。p 未经校准不是真实后验，但排序质量足够支撑诊断。


# 这到底意味着什么

DiagEval最关键的洞察是：**当软件正确性是图级别的而评测证据是路径局部的，评测的可靠性不取决于你试了多少次，而取决于你能否从失败中定向收集区分性证据。**

它把GUI Agent评测从"跑 → 挂 → 判错"的简单流程升级为"跑 → 挂 → 分析不确定性 → 定向探测 → 归因判定"的闭环诊断。朴素重试是在重复采样同一条路径附近，诊断探测是在主动降低不确定性。

这个思路的适用性不止于网页评测。App测试、API调用链验证、端到端自动化测试，只要评估者本身有失败概率，就需要一个机制区分**评估者失误**和**被测系统的真实缺陷**。DiagEval给了一条可操作的路径。

局限是诚实的。当前似然参数手工设定，分支排序依赖 prompt，归因是二元的。但框架本身 —— FDS解析、SOU分类、定向探测、非对称更新——是通用的，精度可以通过学习参数来提升。
