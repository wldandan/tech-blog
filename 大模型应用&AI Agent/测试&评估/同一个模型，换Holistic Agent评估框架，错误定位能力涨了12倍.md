# 同一个模型，换Holistic Agent评估框架，错误定位能力涨了12倍

![alt text](image-2.png)

先看一组数据。

**GPT-5.4当"单体评委"** —— 把整条Agent执行Trace一次性扔给模型，让它找出哪里出错了。在SWE-bench上，错误定位准确率：**7%**。

**GPT-5.4装进Deepchecks的框架里** —— 拆成一个个独立的小任务，逐个Span评估。同样的模型，同样的Trace，错误定位准确率：**86%**。

差了12倍。不是模型变了，是评估方法论变了。

Deepchecks最近发了篇论文，系统地把这个现象拆了一遍。结论就一句：**Agent评估的瓶颈不是模型能力，是方法论。** 把长Trace一股脑塞给一个LLM让它找出哪里错了，这件事的天花板比你以为的低得多。


# 一、为什么单体评委搞不定长Trace

Agent跑一次任务，产出的是结构化Trace：一棵Span树。LLM调用是Span，工具调用是Span，每个Span有输入、输出、耗时、状态。一条稍复杂的执行，几百个Span很正常。

现在的评估做法，主流两种：

**结果导向**：只看最终答案对不对。GAIA、SWE-bench、AgentBench都是这个路线。好处是简单，坏处是输了也不知道输在哪。正确答案可能掩盖推理过程的问题，错误答案也看不到中间哪一步翻的车。

**Trace级单体评委**：把整条Trace扔给一个LLM，让它标注错误。TRAIL基准测试就是这么设计的。几个前沿模型（Gemini、Claude、GPT、OpenAI o系列）全跑了一遍。**同一个 Trace里找错误，最好的单体模型也就 54.6% 的定位准确率。** 更致命的是，Trace一长，表现直接崩，Claude 3.7 Sonnet、OpenAI o3、OpenAI o1在SWE-bench上直接爆上下文，整条数据集跑不了。

论文管这个叫"**上下文窗口问题是结构性的，不是过渡性的。**"模型窗口在涨，但Agent Trace也在变长、变密、变复杂。单体评委这条路的尽头，不是更强的模型能解决的。

# 二、怎么破：自上而下 + 自下而上，两个视角互补

Deepchecks框架的做法一句话就能说清楚：**把评估拆成独立的小任务，每个Span单独审，然后把结果拼回来。**

但光拆开不够。有些问题只在单个Span层面可见，有些问题只有站在全局视角才能发现。所以框架用了两套评估，互相补位。

## 2.1 自下而上：逐Span打分，向上传播

框架从最底层的叶子Span（LLM 调用、工具调用）开始，逐个评估。

![alt text](image-3.png)

对每个**LLM Span**，评估四个维度：指令遵循（输出跟Prompt要求对得上吗）、推理完整性（逻辑站得住吗、有没有跳过关键步骤）、回避检测（是不是输出了一个"我不知道"而不是真的去推理）、错误检测（输出是正常结果还是系统/API 报错）。

对每个**工具Span**，评估两个维度：完整性（这个工具调用的目标达成了吗）和错误检测。

每个维度产出两类输出：一个 1-5 的分数（5 代表没问题，1 代表严重失败），加一段自然语言 rationale（为什么打这个分，具体哪出了问题）。

然后，叶子层的失败状态向上传播。默认规则很简单：**一个非叶子节点，如果任何一个子节点有失败指标，它就标记为失败。** 框架也支持更复杂的传播策略——阈值式（失败子节点比例超过 X 才传播）、类型过滤式（只传播 LLM 失败、忽略工具错误）、联合式（所有子节点都失败才传播）。

这就实现了一个很实用的能力：**Trace被判为失败时，你能沿着传播链一路追到最底层 —— 哪几个Span、哪几个指标出了问题、Judge的原话是什么。**

## 2.2 自上而下：站在Agent层面看全局

![alt text](image-4.png)

有些问题，盯着单个Span永远发现不了。比如：

- Agent搜索了 10 次，每次搜索单独看都没问题。但整体来看，它一直在绕圈子 —— 这就是**工具滥用**。
- Agent计划了 6 个步骤，执行了其中 4 个，最后两个被跳过了。单个步骤看都执行对了，但**任务没做完**。
- Agent说要去查三个数据源，最后只用了一个，产出里却声称"已全面核查" —— **计划偏离**。

框架在Agent层面评四个维度：**计划效率**（执行跟计划对齐吗）、**工具覆盖度**（该查的信息源都查了吗）、**工具滥用**（有没有反复调同一个东西、失败后换策略了没）、**完整性**（最终输出有没有完整回应用户的问题）。

论文给了一个具体的例子来说明两种视角的互补性：Agent被要求从一个YouTube视频里提取两个CFM数值，最终产出是编造的。

- **自下而上**：抓住了格式化错误（工具调用参数写错了）和幻觉（LLM Span 里编造了数值）。但漏掉了"搜索策略不对"和"重复失败后不换策略"——这两个需要看全局才知道。
- **自上而下**：抓住了搜索覆盖不足（工具覆盖率 1.0 分）、计划偏离和编造行为。但漏掉了格式化错误 —— 因为参数错误发生在单个Step内部，全局视角覆盖不到。

四个ground-truth错误，单一视角各漏两个。**两者合在一起，全部抓住。**


#  三、TRAIL基准上的结果

TRAIL是目前唯一公开的、具有Span级错误标注的Agent评估基准：148条OpenTelemetry 格式的Trace，841个Span级错误标注，20+错误类别分类法。来自两个底层数据集 —— GAIA（多Agent系统，层级结构明显）和SWE-bench（单Agent，Trace更长、工具调用更密）。

三个评估指标：**定位准确率**（预测出错的 Span ID 跟标注一致的比例）、**加权类别 F1**（按类别加权的分类 F1）、**联合准确率**（Span 和类别同时对上的比例，最严格）。

![alt text](image-1.png)

### 全指标最优，差距在变宽

Deepchecks 框架在 GAIA 和 SWE-bench 上**每一个指标都是最好**。GAIA 上：类别 F1 0.547（最强基线 0.434，+26%），定位准确率 0.823（最强基线 0.657，+25%），联合准确率 0.616（最强基线 0.239，+158%）。

**SWE-bench 上差距拉得更开。** 类别 F1 0.698（+38%），定位准确率 0.860（最强基线 0.250，+244%），联合准确率 0.638（最强基线 0.051，+1152%）。

SWE-bench 的 Trace 比 GAIA 长得多，工具调用也更多。更强的基线和更弱的基线之间的差距，在长 Trace 上不是缩小了，是拉大了。

### 更强的单体模型救不了这个局

论文对比了 TRAIL 原始报告的 6 个前沿模型，外加用 GPT-5.4、Gemini-3.1-Pro、Claude-Sonnet-4.6 复现的结果。**没有一个单体评委能把定位准确率拉到跟框架同一个量级。**

最有说服力的对比：GPT-5.4——框架内部用的就是这个模型。当单体评委时，GAIA 定位准确率 0.292，SWE-bench 0.070。装进框架后：0.823 和 0.860。**模型没换，差距在方法论。**

### 多阶段流水线在长Trace上一样崩

FAGI-AgentCompass（目前最强的框架级基线）用了一个多阶段流水线 —— 聚类 + 跨执行记忆。在 GAIA（较短的多 Agent Trace）上能做到 0.657 定位准确率，跟 Deepchecks 的 0.823 差距不至于绝望。但切到 SWE-bench 直接掉到 0.250。**流水线式分解 + 聚类 + 跨执行记忆在短 Trace 上有用，但到长 Trace 就失灵了。** 逐 Span 独立评估 + 预定义指标的方案在两种场景下都稳住了。

# 四、框架的局限

论文没有回避自己的问题。

**基准标注质量有坑。** TRAIL的Span级错误标注存在一系列不一致：同样的错误（LLM 给工具传了畸形的参数），有时标在LLM Span，有时标在Tool Span；分类体系中"目标偏离"和"指令不遵守"的定义实际重叠，都属于"系统没按任务要求做"，但归入了不同分支，导致同一行为被同时标为两类；还有标注理由跟Span实际证据直接矛盾的情况。

**聚合策略太粗暴。** 目前默认用"存在即失败"的传播规则 —— 任意一个Span有一项指标不达标就向上传播，最终整条Trace被判定为有问题。论文发现了 44 条Trace被人类给出了 Overall≥3.5 的高分（甚至有 3 条是满分 5.0），却因为有Span级标注错误而被判"有问题" —— 比如少了 `<end_plan>` 标签这种不影响推理质量的格式问题。框架能抓到这些问题是对的，但把它们等同于逻辑错误传播到 Trace 级评分则不对。未来需要引入严重度加权和结果相关性过滤。


# 五、最后说两句

这篇论文跟之前聊过的Agentic CLEAR和AWS EvalAgent是同一个大方向的不同切面。Agentic CLEAR解决"自动发现评估维度"，EvalAgent解决"自动生成评估代码"，Deepchecks的Holistic Agent解决"怎么评估长Trace才准"。

三个工作指向同一件事：**Agent评估这个领域，方法论瓶颈比模型能力瓶颈大得多。** 哪个模型更强当然重要，但你把模型放在什么框架里、给它设计了什么评估流程，比模型本身更能决定结果。

如果你在搞Agent评估或者选型，这篇论文有一条非常具体的行动建议：**别用单体LLM Judge评估长Trace。**拆开，逐Span审。自下而上定位问题，自上而下发现模式。同一个模型，换一种用法，差距是12倍。
