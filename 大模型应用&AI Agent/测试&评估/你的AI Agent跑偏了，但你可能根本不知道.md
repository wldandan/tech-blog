# 你的AI Agent跑偏了，但你可能根本不知道

![alt text](image-4.png)

让一个AI Agent帮你做任务。它跑了100步，每一步看着都还行。回头检查结果，发现中间有一步偷偷错了 —— 调了一个不存在的API，引用了一个编造的数据。后面 50 步全在这个错误上接着跑。

**你怎么发现这种事？**

现在的答案很简单：靠人肉看。IBM Research最近发了篇论文，做了一个叫**Agentic CLEAR**的工具，专门解决这个问题。能自动告诉你Agent在执行过程中到底哪里掉链子了。

# 一、Agent越来越强，也崩得越来越安静

好消息是AI Agent现在确实能干活了。写代码、做科研、浏览网页，端到端的复杂任务也能搞定。

**坏消息是它们很不稳定。** 论文列了几种经典翻车姿势：

- 陷入**重复循环**，同一个工具调了一遍又一遍；
- 子 Agent 之间行为**对不齐**，一个干完了另一个还不知道；
- 错误在步骤间**悄悄传播**，看最终输出根本发现不了；
- 干到一半**不吭声放弃了**，任务没做完就停了。

而且这真不只是大厂的问题。论文特意提到，个人开发者也在大量用Agentic Workflow做日常自动化。**Agent的"民主化"跑得飞快，但评估和诊断的工具还在地上爬。**

现在市面上的工具就两类，各自有硬伤：

**观测平台**（LangSmith、LangFuse这些）。它们能记下Agent每一步的执行轨迹，这点没问题。但到了"评估"，基本就停留在算几个聚合指标，或者用一个Prompt让另一个LLM给整条Trace打个总分。**想发现系统性问题？还得人一条一条翻Trace。** 一个项目跑几百条Trace，谁有时间翻？

**学术界的错误分类法**。研究者花了很多精力给Agent的错误分门别类，什么"推理错误""规划错误""工具调用错误"。听起来很系统，问题是**这些分类是死的。** 换个任务、换个模型、换个领域，之前那套分类可能完全对不上。

于是现状就很尴尬了。Agent跑出来的Trace堆成山，但能告诉你"哪里出问题了、为什么会出问题"的工具几乎没有。

# 二、整体思路：让LLM自己审LLM

IBM团队的想法很直接。人翻Trace 太慢，错误分类又太死，那就**让LLM自己来评估LLM Agent干了什么。**

工具名叫**Agentic CLEAR**。CLEAR是之前他们做的一个错误分析框架（Yehudai et al., 2025b），这次升级成了全自动的Agent评估系统。整体分两个阶段：**第一阶段逐Trace评估，第二阶段跨Trace聚合洞察。**

## 2.1 第一阶段：一条Trace，三个角度审

对每条Trace，Agentic CLEAR用一个LLM Judge做三种评估。注意这三种评估用的是不同的Prompt，各自关心不同的东西：

**（1）Step-wise评估：每一步拉出来单独审**

把Trace里的每一步（输入 + 输出）单独拎出来。Judge关注三个维度：**正确性**（这一步的推理对不对）、**完整性**（该干的事干完了吗）、**清晰度**（格式和逻辑清不清楚）。每步给一个质量分，外加一段自然语言的"评语"。这评语不是泛泛的"不错""还行"，而是具体指出问题。

所有step级的评语会**按节点分组**。比如规划节点产生的所有步骤的评语归到一组，执行节点归到另一组。这是后面做节点级归因的基础。

**（2）Trace-wise评估：看整条Trace的全局质量**

只盯着单个步骤会丢失上下文。这步Judge退后一步看全局：步骤之间有逻辑断裂吗？衔接合理吗？最终交付物完整吗？

**（3）Rubric评估：让Judge自己出题、自己判卷**

这是整个系统里最巧妙的设计，分两小步：

- **第一步，先出题**：只给Judge任务描述，让它**自己生成一套评估标准**。比如一个购物任务，Judge可能自动生成"检索了商品列表吗？→ 正确解析了每个商品吗？→ 加入购物车了吗？→ 完成结算了吗？"标准数量、内容、粒度全由Judge自主决定。零人工。

- **第二步，再判卷**：拿着刚生成的评估标准，逐条对照Trace检查每一条是否满足，给出判断和理由。

**三种评估的Prompt都要求Judge先给出自然语言推理，再给分数** —— 等于内置了Chain-of-Thought。论文Prompts全部开源在代码仓库。

## 2.2 第二阶段：跨Trace聚合，找"通病"

单条Trace评估只是开始。假设跑了200条任务，第一步会产出几百条step评语和trace评语。**真正有价值的是从这几百条碎片化评语里，提炼出反复出现的系统性模式。**

这里用的是**CLEAR算法**（他们2025年的前置工作，专门做LLM Judge评语的聚类和摘要）。分两个方向聚合：

**节点级聚合**：把第一步里按节点分好组的step评语，分别喂给CLEAR。CLEAR对同类型评语聚类、做摘要，输出**每个节点独有的高频问题清单**，每条问题下面挂着触发它的具体 Trace 实例——点了就能跳回去看原始步骤。

**系统级聚合**：把所有trace级的全局评语和rubric级评语汇总，同样过CLEAR，输出**跨整个系统的全局问题模式**。这些"通病"不会告诉你"规划节点有问题"，而是告诉你"你的 Agent 整体上存在冗余工具调用"或"输出格式不遵守要求"。

最终每条Trace的完整评估记录都关联到上述两个层次的洞察，形成一条从`全局模式 → 具体节点 → 单条Trace → 单个步骤`的可追溯链。

## 2.3 交互式 UI：三层视图

论文还做了一个仪表盘，分三个视图，层次上从宏观到微观：

**系统视图（System View）**：**直接从Trace数据里反向重建出Agent的拓扑结构**（节点之间怎么连的、调用关系什么样）。这一页还展示全局得分、节点使用频率、系统级的Top-N高频问题。是全局诊断的入口。

**节点视图（Node View）**：按节点切。每个节点下面列出Agentic CLEAR自动发现的该节点专属问题（动态生成，不是预定义的），可以按问题类型和分数区间过滤。**点击任何一条问题，展开看到所有受影响的Trace实例以及Judge的原始评语。** 这是定位到具体组件故障的入口。

**Trace视图（Trace View）**：单条Trace的深度解剖。展示整体评估结论、每个step的维度得分（正确性/完整性/清晰度）、Rubric逐条检查结果、Judge的完整推理过程。还支持按分数、关键词、问题类型搜索和过滤Trace。

## 2.4 工程落地细节

几个工程上的选择值得一说：

- **Trace格式**：用的**OpenTelemetry兼容格式**，具体是LangFuse的schema。如果是其他平台（比如LangSmith或自建的）的Trace，只需要写少量预处理代码把LLM调用的输入输出 + 元数据映射到它的中间表示。这个中间表示的核心字段就是 (input, output, node_id)。

- **Judge模型选择**：论文用了两个Judge，**GPT-5**和**OSS-120B（高思考模式）**。前者代表闭源旗舰，后者代表开源前沿。两个都能跑，但产出的问题深度和风格差异很大。

- **一键运行**：整个流水线通过一条CLI命令启动，YAML配置。跑完后出ZIP 包，交互式UI从命令行启动，加载ZIP就能看。

- **Python包**：`pip install clear-eval`，Apache 2.0协议。每个评估阶段（Step-wise / Trace-wise / Rubric）都可以单独调用，也可以组合，用户按需裁。


# 三、在四个Benchmark上跑出了什么

论文实验做得挺扎实：**4个Benchmark**（GAIA、SWE-Bench Verified Mini、AppWorld、τ2-Bench），**7种Agent配置**（不同Agent架构 × 不同模型，有CUGA、HAL Generalist、HF Open Deep Research等），两个Judge模型，几万次LLM调用。

捡几个最值得说的：

### 不管什么Agent，有些坑人人踩

跨所有配置，Agentic CLEAR挖出了四类通用错误：

1. **冗余低效的工具使用**：同一个搜索调 6 遍，文件在眼前还要去网上搜
2. **错误处理拉胯**：工具报错之后蒙了，不会换条路走，没有 fallback
3. **任务跑不完**：干着干着停了，该做的事没做完
4. **输出格式不对**：要JSON不给JSON，缺字段，夹废话

### 不同Benchmark的专属翻车姿势

除了通用错误，每个Benchmark还暴露了领域特有问题。**有意思的是，Agentic CLEAR没加任何Benchmark特定的Prompt，自己就把这些问题挖出来了。**

GAIA（研究型任务）被"信息来源验证不足""缺乏交叉验证"支配。AppWorld（多步API编排）出现了"操作被污染的购物车""丢失邮件附件"这种很具体的失败。SWE-Bench（代码修复）是monkey-patching、diff输出损坏。τ2-Bench（客服）则是越权支付、虚构费用。

### 同一个系统，规划和执行的犯错方式完全不一样

这是我最喜欢的一个发现。拿CUGA Agent来说：

**规划节点（TaskDecompositionAgent）**的典型错误：任务分解不合理、API选择错误、幻觉出不存在的API。

**执行节点（APICodePlannerAgent）**的错误：忘记处理分页、没做pagination。

**两种错误只看最终输出，都长一个样——"任务没完成"。但修复它们要干的事完全不一样。** 前者要调规划Prompt，后者要加强代码生成逻辑。不分层分析的话，你根本不知道该改什么。

还有一个细节值得注意：**幻觉主要出现在规划阶段**（比如ShortlisterAgent列出了API目录里不存在的API），执行阶段几乎不出现幻觉，但容易出现功能性bug。这对调优方向的指导非常具体。

### 用什么Judge，结果差很多

论文对比了**GPT-5**和**OSS-120B**当Judge的效果。差距不小。

GPT-5发现的东西更具体、更细粒度、更针对领域。"因为SQL缺少别名重映射导致查询正确性受损"，这种。平均每条issue长度130个字符。OSS-120B的比较泛、更操作导向，"冗余搜索导致低效"这种，平均67个字符。

跟人类标注的错误分类（TRAIL标准）做语义对齐：**GPT-5的Macro-F1到了0.459，覆盖了12个类别里的10个；OSS-120B的Macro-F1是0.374，覆盖了10个。** 注意这是没有任何预定义分类的前提下做到的。映射过程用了Claude Opus 4.6，由作者验证。

### 还能预测任务成功率

论文用Agentic CLEAR的评分去预测Trace是否成功（算AUC）。

![alt text](image-3.png)

AppWorld上，GPT-5的Trace-wise AUC最高到**0.89**。综合来看，Trace-wise式是最强的预测器。但不同Benchmark差异很大：AppWorld上三种方式都超过0.75，τ2-Bench上没超过0.62。**这说明评估方式和Agent架构/Benchmark特性之间有交互效应：模块化架构（如CUGA）更容易用step-wise分解评估，而策略合规性强的场景（如 τ2-Bench）用rubric更合适。**

# 四、这东西怎么上手

Agentic CLEAR已经在PyPI上了，Apache 2.0协议，`pip install clear-eval`直接装。

接入方式挺务实的。用OpenTelemetry兼容的Trace格式，具体是LangFuse格式。用其他Trace格式的话，做一点预处理，把LLM调用的输入输出和元数据对齐到它的中间表示就行。

一条CLI命令跑完整条分析流水线，YAML配一下。跑完出个ZIP包，用它的交互式UI打开就能看。

我觉得最实用的几个场景：

- 改完Prompt或者换了模型，跑一遍全量Trace，看系统级和节点级的问题变化。
- 同一个 Agent 架构分别接 GPT 和 Claude 跑同一批任务，用 Agentic CLEAR 对比失败模式。
- 线上定期采样Trace跑一轮评估，看Agent可靠性有没有随时间漂移。

# 五、说实话，也有局限

论文自己也没藏着。

**Rubric自动生成还不够靠谱。** Judge只能看任务描述来生成评估标准，看不到Benchmark内部的"正确答案"。τ2-Bench这种对抗性任务（正确做法是拒绝用户请求），生成的 Rubric会给"完成了用户请求"打高分 —— 但正确答案恰恰是拒绝用户。在AppWorld 上，生成的Rubric是"检索列表 → 解析商品 → 加购物车 → 结算"这种流程性的描述，而黄金标准是"恰好创建了一个新订单""没有修改任何地址记录"这种程序化的状态断言。**任务描述和正确评估标准之间，确实存在信息差。**

**不同Judge的差距太大了。** GPT-5和OSS-120B产出的问题类型、深度、侧重点都不一样。你今天用这个Judge找到的问题，换一个可能就找不到了。Judge本身的可靠性还是个没解决的问题。

**目前只覆盖了推理和规划层面的错误。** 系统执行层的东西（环境问题、超时、资源不足）还没纳入分析范围。论文明确把这个列为future work。

# 六、最后说两句

过去一年所有人都在卷模型、卷框架、卷工具链。但如果你连 Agent 到底在什么地方出错都不知道，优化基本靠猜。

Agentic CLEAR做对了一件事：把评估从"人定义规则"变成了"LLM自己发现模式"。**没有预定义的错误分类法，不需要手工写评估标准，能自动适应不同的任务和领域。** 这个思路本身比它现在跑出来的分数更重要。

开源、pip 直接装、一条命令跑通。在 Agent评估这个方向上，这是目前我见过的把"好用"和"有深度"平衡得最舒服的一个工具。

**如果你在做Agent开发，花半小时试试。你可能会发现一些让你想拍桌子的问题——那种在你眼皮底下跑了无数遍但你从来没注意到的坑。**
