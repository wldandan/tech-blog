### 背景

虽然[LLMs](https://zhida.zhihu.com/search?content_id=250442343&content_type=Article&match_order=1&q=LLMs&zhida_source=entity)在理解以自然语言表达的用户意图方面表现出色，但它们不足以满足需要比文本到文本转换更多的应用程序特定的用户意图，往往存在以下一些需要解决的技术问题：

  1. 大模型虽然擅长理解文本，但是确不擅长将文本转换为具体的应用。
  2. 大模型生成内容可解释性不足，黑盒，无法应用于对稳定性、可靠性要求高的系统。
  3. 如何在完成任务的条件下，控制token的使用，降低成本。
  4. 大模型生成代码的能力很强，但是发生错误如何纠错。
  5. 自然语言的描述存在歧义，每种可能对应不同的实现方式。



本文核心思想是提出了**一个语义解释器系统，该系统可以将人类的自然语言转换为自定义的[ODSL](https://zhida.zhihu.com/search?content_id=250442343&content_type=Article&match_order=1&q=ODSL&zhida_source=entity)（office DSL）。**这是一种简洁的高级语言，专门用于在Office应用程序中执行操作并与实体进行交互。作者的讨论集中在将ODSL应用到 PPT的生成中。

## 方案与关键技术

上图展示了整个ODSL的设计思路。

语义解释器将自然语言用户话语和相关文档上下文作为输入。文档上下文被编码为轻量级的 JSON 树形数据结构，其中包括与用户话语相关的文档实体、内容和属性。

然后，语义解释器使用一种分析检索方法（[ARM](https://zhida.zhihu.com/search?content_id=250442343&content_type=Article&match_order=1&q=ARM&zhida_source=entity)）来生成以用户话语和文档上下文为条件的提示。ODSL 解释器将 ODSL 程序解析为抽象语法树（AST），并进行分析以验证程序和识别错误。然后，AST 将通过代码校正层，该层利用特定领域的知识，自动修复程序中的错误（如果可以明确地修复或使用一些意见规则）。

最后，解释器将 ODSL AST 转换成用本地应用程序接口（如 Office-JS）编写的程序，然后由应用程序执行，以实现用户意图。

ODSL包含的关键技术点主要包含以下几方面：

**办公领域特定语言 (ODSL)**

其作为 LLM 与应用程序之间的接口，用于表达和执行用户意图。

设计原则包含：

  * **表达性和可扩展性** ：通过实体和语句构建，易于表达各种用户意图，并可轻松扩展新功能。
  * **ML 可感知性** ：语言设计简洁、统一、紧凑，并减少冗余，便于 LLM 学习和生成程序。
  * **鲁棒性** ：限制语言范围，只包含安全操作，并通过静态类型和**代码校正** 层提高代码的可靠性。



特点包含：

  * 由于PPT场景特点，本文将所有操作抽象成4个类别，即选择、插入、更新、删除，这四种操作即可满足用户的大多数要求，对应到 ODSL语言如， select_text(scope=shapes, text="hello")。  
所有的ODSL 语言都由 select\format\insert\delete 开头。
  * 语句名为 snake写法，变量为 驼峰写法



**分析-检索方法 (ARM)**

优化 LLM 的提示构建过程，使其在有限的提示长度内生成更准确的 ODSL 程序。

其过程包含分析和检索两个步骤：

  1. **分析** ：通过实体分类器和需要上下文分类器确定与用户意图相关的 ODSL 实体集合，并判断是否需要文档上下文。
  2. **检索** ：基于实体和语义搜索，从 ODSL 样本库中选择最相关的 ODSL 程序样本。



特点包含：

  * 将ODSL分类：由于之前已经将操作分成了4类，因此在此可以先确认操作的类别，缩小范围
  * 输入标准化：ODSL库里的样本是成对的 标准化过的描述-ODSL示例，用户输入的内容也会被标准化之后再去和ODSL库匹配。如，所有的 insert create generate都会标准化为 add 



**程序比较**

评估 ODSL 程序是否满足用户意图。

其方法包含：

  * 程序规范化：通过程序转换，使程序比较对无关紧要的差异变得不敏感。
  * 子程序分析：判断生成的程序是否包含所有可接受程序，即使它包含额外的步骤。 



通过ODSL，解决了如下几方面的问题：

  * **LLM 缺乏应用特定知识** ：通过 ODSL，将用户意图转换为可执行的程序，并利用领域专家知识进行代码分析和校正，弥补了 LLM 的不足。
  * **LLM 提示长度限制** ：通过 ARM 方法，动态构建简洁且相关的提示，充分利用 LLM 的能力，同时避免超长提示带来的成本和风险。
  * **代码生成准确性** ：通过 ODSL 的简洁和一致性设计，以及 ARM 的实体感知语义搜索，提高 LLM 生成的代码的准确性。
  * **代码安全性** ：通过 ODSL 的安全设计，以及代码校正层的应用，确保生成的代码安全可靠。



## 实验与数据

本文通过消融研究来评估 Semantic Interpreter 各个组件的贡献，并分析不同参数设置对系统性能的影响。

**实验设置**

  * 模型：使用 [OpenAI GPT-3.5](https://zhida.zhihu.com/search?content_id=250442343&content_type=Article&match_order=1&q=OpenAI+GPT-3.5&zhida_source=entity) 模型系列中的 text-davinci-003 模型。
  * 数据集：构建 197 个测试用例，涵盖 PowerPoint 相关的各种用户意图，例如创建演示文稿、添加新幻灯片、插入文本内容、修改现有内容、插入图片、格式化文档实体等。
  * 评估指标：使用程序匹配等级来评估生成的 ODSL 程序与可接受程序之间的匹配程度，并将通过率定义为程序匹配等级为 Subprogram Normalized 或以上的测试用例比例。



**消融研究**

  * 不同 k 值的影响：研究 k 值（语义搜索中选择的 ODSL 样本数量）对系统性能的影响。结果表明，few-shot prompting 对系统性能至关重要，并且存在一个最优的 k 值，在本文实验中为 5。
  * 分析步骤的影响：比较是否包含实体分类器和需要上下文分类器的系统性能。结果表明，分析步骤可以显著提高系统性能，这表明分析步骤可以有效缩小搜索空间，提高语义搜索的准确性。
  * 代码校正的影响：比较是否包含代码校正层的系统性能。结果表明，代码校正层可以有效提高系统性能，这表明代码校正层可以有效纠正 LLM 生成的代码中的错误，提高代码的可靠性。



**实验结果**

最佳配置的系统通过率为 96.06 ± 2.69%，表明 Semantic Interpreter 能够有效地理解和执行用户意图。

消融研究结果表明，ARM 方法和代码校正层对系统性能至关重要，而实体分类器和需要上下文分类器也有助于提高系统性能。

## 总结与展望

本文作者贡献了ODSL方法，并提供了一个将程序合成自然语言命令系统的方法。

  * **Office 领域特定语言 (ODSL)** ：一种简洁、高级的语言，专门用于在 Office 应用程序中执行操作和与实体交互。
  * **语义解释器架构** ：利用 LLM 进行程序合成的分析-检索提示构建方法，将自然语言用户命令转换为 ODSL 程序，然后将其转换为应用程序 API 并执行。
  * **评估程序合成自然语言命令系统的方法** ：将评估问题重新定义为分析程序等价性，从而解决用户查询抽象和定义不明确的问题。



其价值主要包含如下几方面：

  * **提高生产力软件的用户体验** ：使用户能够通过自然语言与软件交互，无需学习复杂的命令或界面。
  * **利用 LLM 的强大能力** ：将 LLM 的理解用户意图的能力与 ODSL 的精确性和安全性相结合，实现更准确和可靠的程序合成。
  * **促进自然语言命令系统的发展** ：为评估和改进自然语言命令系统提供了一种新的方法。
