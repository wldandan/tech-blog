# Code as Agent Harness：代码不再是AI生成的产品，而是AI的操作系统

> 一篇来自UIUC、Meta和Stanford联合发布的百页综述，重新定义了代码在Agent系统中的位置。

![alt text](image-9-1.png)

如果你对AI编程的认知还停留在"帮我写一个排序函数"或者"生成一个登录页面"，这篇文章可能会让你对代码的角色产生完全不同的理解。

今年5月，来自UIUC、Meta、Stanford的联合研究团队发了一篇百页综述，名字叫《**Code as Agent Harness: Toward Executable, Verifiable, and Stateful Agent Systems**》。核心论点只有一句话，但信息量极大：

**代码正在从LLM的"最终产出物"变成AI Agent的"操作系统级基础设施"。**

翻成大白话：**代码不再是你要的东西，而是Agent靠它活着的东西。**

这篇文章会把这个框架拆开来讲：三个层级怎么划分、每层到底在干什么、落地到了哪些场景，以及现在还卡在哪些坑里。篇幅不短，但读完你会对"AI Agent到底是怎么工作的"有一个结构化的理解。

# 一、为什么代码的角色变了？

先想一个场景。

2023年你用ChatGPT写代码，流程大概是：`描述需求 → 复制生成的代码 → 自己跑一遍 → 报错了丢回去让它改 → 再复制粘贴`。代码在这里是一个**产物**：你用自然语言下单，它交货。

2025年你用Claude Code或者Codex，流程变成了：`描述需求 → Agent 自己翻仓库文件 → 自己定位要改哪里 → 自己改 → 自己跑测试 → 失败了看 trace → 自己修复 → 再跑`。整个过程你只是在旁边看着，偶尔点个Approve。

区别在于：代码的角色从"终点"变成了"介质"。

这背后发生了什么？研究团队从系统的角度给出了一个框架：**所谓Agent，本质上是一个软件层（他们叫harness），把无状态的LLM包裹起来，给它配上工具、沙箱、记忆、权限边界、执行循环和反馈通道**。这个底盘的核心材料，不是配置文件，不是prompt模板：**是代码本身**。

他们把这个视角叫做**Code as Agent Harness**，Harness Interface（接口层）、Harness Mechanisms（机制层）和Scaling the Harness（规模化层）三层结构如下：

# 二、第一层：Harness Interface —— 代码作为连接一切的接口

<center>

![alt text](image-1.png)

图2 代码在推理、行动和环境接口层示意</center>

这一层回答一个最基础的问题：模型跟外部世界之间，用什么东西当桥梁？

**答案是代码，也只能是代码。**

论文给了三个理由：
- **可执行（executable）**：代码能被解释器、编译器、测试框架跑起来，产生确定性的反馈。
- **可检查（inspectable）**：中间变量、执行轨迹、控制流都可以暴露出来被读取和审计。
- **有状态（stateful）**：代码仓库是一个持久化的、可修改的实体，任务的进展可以写在里面。

这三个属性让代码天然适合做 Agent 的"接口层"。具体体现在三个子方向上：

## 2.1 推理介质：让思考变成可执行的计算

传统Chain-of-Thought在自然语言里推理，模型得同时负责"分析问题"和"执行计算"。但LLM做算术不可靠是公开的秘密，你让它算个17×23它可能犹豫半天。

**把计算部分外包给代码解释器**，模型只管"怎么算"的意图，解释器负责"算出结果"。这就是Program-of-Thoughts（PoT）和PAL的核心思路。

但后面还有更猛的。两条路线：

一条是**形式化验证路线**。用Lean、Isabelle这类证明助手，每一步推理都必须被验证器检查通过，不是模型自己说对了就算。DeepSeek-Prover-V2在Lean里搜索证明路径，Goedel-Code-Prover把代码验证也塞进了形式化框架。

另一条是**迭代推理路线**。`模型生成代码 → 执行 → 看结果 → 修正 → 再执行`。CodePRM在每一步执行后给予过程奖励，R1-Code-Interpreter在一个交互式会话里自己决定什么时候跑代码、什么时候继续推理。

**这里的关键是：代码执行的结果成了推理质量的"客观裁判"。** 不是另一个LLM来评价你答得好不好，而是编译器/trace/lint 的明确信号。

## 2.2 行动接口：把意图翻译成可执行操作

推理完了得干活。机器人要抓杯子，GUI Agent要点按钮，代码助手要改文件。

**Code-as-Policies（CaP）直接用LLM生成的Python程序作为机器人控制策略**，不是让模型选预定义的技能，而是让它写控制逻辑本身。RoboCodeX更进一步，生成的是树形结构的多模态代码。

Voyager在Minecraft里搞了一个**自我进化的技能库**，每次成功完成一个新任务就把对应的程序存下来，下次遇到类似情况直接用。这不是"训练"的概念，而是**代码作为持久化记忆**。

SayCan和KnowNo则解决"可行性"问题：LLM说"拿起远处的杯子"，但机器人胳膊根本够不到，怎么办？**用可行性模型和保形预测来给LLM的计划加一层物理约束**，不确定的时候就触发澄清，而不是瞎执行。

## 2.3 环境模型：用代码表示世界状态

这可能是最被低估的一点。

大多数Agent系统把环境当黑盒，给一段文字描述当前状态，模型据此决策。问题是文字描述会丢信息、有歧义、难以跨步骤保持一致性。

Code-for-Environment的思路是：**用可执行的程序来显式建模环境状态和动态。**

WorldCoder让Agent把学到的世界知识写成Python程序，不是藏在模型参数里，而是作为可编辑、可执行、可复用的代码产物。CWM（Code World Model）从程序执行轨迹中直接学习预测性世界模型。SWE-bench整个评测体系就是用单元测试来定义"代码仓库目标状态"：**通过测试 = 到达目标状态**。

**翻译成人话：代码仓库 + 测试套件 = 可验证的"数字世界地图"。**

# 三、第二层：Harness Mechanisms —— 让Agent跑长跑不摔跤

接口层接入之后，下一个问题是：**长任务怎么办？**

单次生成过得了，来回改十个文件、跑三十次测试、中间还要纠错，只靠模型能力是扛不住的。这一层扛起了"可靠地跑长程"的责任。

## 3.1 规划：不只是"先写计划再执行"

<center>

![alt text](image-3.png)

图4 规划机制概览 </center>

早期方案就是让模型先列个numbered list，然后按顺序干活。这个东西在简单任务上够用，但碰到仓库级复杂任务就不行了。

综述把规划分成了四种范式：

**线性分解**：就是你想象的那种"先写计划"模式。不过有意思的是，工业界实践里，Claude Code等系统把PLAN.md之类的文件当成了**文件系统级别的控制对象**。它可以被review、被Git版本化、被子Agent消费。规划不再只是一段prompt里的文本，而是一个持久化的、可审计的控制产物。

**结构锚定**：CodePlan不凭空规划，而是先建一张"编辑义务图"：改A文件会影响哪些其他文件？B的依赖链反着走能走到哪？**规划的依据不是语意联想，而是仓库的物理结构。** VerilogCoder在硬件设计领域做的是类似的事：用"任务-电路关系图"把每个子任务锚定到具体的信号、转换和示例上。

**搜索式规划**：灵感来自AlphaGo。ReThinkMCTS在推理空间中做蒙特卡洛树搜索，CodeTree把策略探索、方案生成、精化组织成统一树结构。**代价就是算力**，但好处是"如果当前路径走不通，回溯试另一条"。Meta-Harness把搜索从模型层面往上升了一级：它搜索的是harness代码本身，给Agent访问之前尝试过的好方案、执行轨迹和评分。

**编排式规划**：MapCoder把任务拆成多阶段流水线，AgentCoder用程序员→测试者→执行者的循环。最激进的一步是Natural-Language Agent Harnesses：**直接把角色、阶段、合约、适配器写成自然语言规范，由一个 Intelligent Harness Runtime在运行时解释和约束执行**。规划变成了一种"运行时解释"问题，而不是一个prompt设计问题。

## 3.2 记忆：context window永远不够，怎么管理？

<center>

![alt text](image-4.png)

图5 记忆机制概览 </center>

如果你做过Agent开发就会知道：**长任务的杀手不是模型不够聪明，是上下文炸了。**

一个典型的SWE-bench任务，Agent要翻几十个文件、跑多次测试、反复修 bug，所有这些东西塞进prompt里，不仅超token限制，还会引入大量噪声。

综述把记忆分成了五个层次：

- **工作记忆**：SWE-agent精心设计了"当前trajectory里该给模型看什么"的管理策略。CodeMem用槽位预算来管理多步编辑中的上下文分配，不是什么东西都往里塞，而是**按信息效用分配有限的注意力空间**。
- **语义记忆**：AutoCodeRover和RepoCoder不是简单做"语义检索"，而是做**结构感知检索**，基于类继承关系、调用链、文件依赖来找证据。关键往往是那个跟你当前代码有调用关系的函数，而不是那个语义相似但八竿子打不着的文件。
- **经验记忆**：跨任务复用。MemGovern的结论是反直觉的：**不加筛选的"积累历史"反而有害**，会引入噪声、错误传播和虚假关联。质量比数量重要得多。
- **长期记忆**：MemCoder把commit记录和人工验证的修复方案存成持久记忆，用来做仓库特定的经验积累。TALM在多Agent场景下检索过去的"问题→解决方案"轨迹。
- **多 Agent 记忆**：ChatDev、MetaGPT里，记忆不是个人笔记本，而是**共享黑板**：哪个 Agent 做了什么、当前状态是什么、测试结果怎么影响下步决策。

## 3.3 工具使用：不只是"调用函数"

<center>

![alt text](image-5.png)

图6 规划机制概览 </center>

工具在综述里被提升到了一个更高的层面：**治理接口**。

工具调用 = 权限边界。调用前检查参数合法性，执行后清理和压缩输出，记录审计日志，遇到高危操作触发人工审批。工具不只是"扩展能力"，它也是"限制能力"的载体。

## 3.4 lan-Execute-Verify闭环：Agent的控制论

<center>

![alt text](image-6.png)

图7 Harness工程的自适应Harness优化机制概览 </center>

这一部分是整篇综述里最漂亮的概念抽象。

传统认知里，调试是"代码写完了修bug"。但在Agent系统中，**调试就是Agent的核心工作循环本身**。

综述把这个循环重新定义为Plan-Execute-Verify：

- **Plan**：不只分解任务，还要明确哪些文件可以改、校验标准是什么、如果出错回到哪一步。
- **Execute**：在沙箱里跑。只读模式可以浏览仓库；沙箱编辑模式可以改代码、跑测试、装依赖；全权限模式则需要人工审批。
- **Verify**：不靠 LLM 判断"改得对不对"，而是靠**确定性传感器**：lint、编译器、类型检查、单元测试、fuzzer、覆盖率工具。这些传感器的输出是客观信号，不会"幻觉"。

**整个系统是一个控制论意义上的调节器：观察执行效果，决定下一步状态转换。**

# 四、第三层：Scaling the Harness —— 多Agent协作

单Agent瓶颈在哪？上下文窗口装不下整个仓库 + 没有分工导致效率低 + 无法自我纠错。

多Agent的思路很自然：把软件工程里的角色分工 —— 架构师、程序员、测试、审查 —— 映射成多个Agent，共享同一个代码仓库作为协作底板。

<center>

![alt text](image-7.png)

图8 通过基于代码的多智能体协同扩展Agent Harness </center>

## 4.1 角色怎么分？

综述从文献中梳理出了五种核心：

- **编码Agent**：写代码的。
- **理解Agent**：读代码的。MAGIS有个Repository Custodian专门负责"这坨代码到底什么意思"。
- **验证Agent**：找茬的。AgentCoder的Test Designer设计上故意不接触代码实现，避免"自己写测试放水自己过"。这个设计原则叫**防模式坍缩**：如果测试和代码是同一个Agent写的，它能给自己开绿灯。
- **执行Agent**：跟沙箱/解释器交互的。AgentCoder里的Test Executor甚至不是LLM，是一个**确定性Python脚本**，严格把"推理"和"执行"分在两个不同性质的模块上。
- **规划Agent**：SoA的Mother Agent根据子函数复杂度动态决定拆成几个子任务。EvoMAC的Gradient Agent看执行日志来判断哪个Agent的锅。

## 4.2 协作模式

**Critique-and-Repair**是主流模式：A写 → B审 → A改 → 重复。AgentCoder是多Agent循环的经典实现。

**Adversarial Validation**更有意思：Fuzzer Agent不是为了"看代码好不好"，而是**主动生成让它崩溃的输入**。你不是说没问题吗？拿这个跑一下试试。

**Debate**是目前最重的模式：CANDOR用三个独立Panelist评审oracle正确性，Curator多数投票决定。MAGIS让部署前开会讨论，各Agent各自陈述依赖关系和潜在冲突。

## 4.3 关键发现：共享状态的形式化程度决定一切

综述在这一层有一个很有意思的观察：

**L2MAC用最简单的顺序链拓扑，但它有最显式的共享状态（持久文件存储 D + 显式上下文调度）。而EvoMAC、SEW这些缺乏形式化共享状态的系统，却在发展极其复杂的自适应拓扑（动态DAG、workflow突变）。**

这意味着什么？**复杂的协作结构可能是"缺乏好用的共享状态"的补偿机制。** 如果你有一个清晰、可查询的共享状态，Agent 之间的协作可以很简单；如果没有，你就得设计各种精巧的通信和协调机制来弥补信息缺失。

另一个让人意外的发现：**LLM模拟执行可以达到 98%+ 的精度预测真实执行结果**（Self-Collaboration、QualityFlow 实验数据）。但这不等于不需要真实执行，模拟对模式化的 bug 有效，真实执行的价值在于抓那些"语言推理结构上不可能想象到"的边界情况：内存溢出、竞态条件、性能在特定数据规模下的不可预期衰退。


# 五、五个落地场景，了解一下

这个框架不是凭空搭的。综述在五个领域逐一展开了案例：

<center>

![alt text](image-8.png)

图9 代码作为Agent Harness在五个新兴领域的应用概述</center>

**编码助手**

从GitHub Copilot的内联补全到SWE-agent的自主issue修复，跨度极大。

LingmaAgent在阿里云实际部署，**全自主解决16.9% 的内部工单，43.3%需要人工介入后解决。**这说明"自主修bug"虽然还不是100%，但已经从实验室走向了生产环境。

更重要的是，**Harness正在成为训练数据的来源。** Cursor的Composer用真实的用户交互轨迹做在线RL，OpenAI的Codex模型在Codex harness循环中收集长周期多轮交互数据做训练。Harness不只是在"运行Agent"，它在"教Agent怎么做更好"。

**GUI/OS Agent**

按按钮、填表单、拖文件，所有这些操作本质上都是**对代码的调用**。

综述把GUI/OS环境抽象成一个POMDP：**隐状态**是浏览器DOM/JS堆、Android Activity栈；**观察**是DOM树/accessibility树/截图标注；**动作**是element.click()或pyautogui.click()；**状态转移**不是学出来的，而是浏览器引擎和操作系统确定的。

Claude Computer Use、OpenAI Operator、Google Project Mariner 都是这个方向。

**科学发现**

Agent动态构建"假设 → 实验 → 分析"的流水线，实验室协议、仿真控制、数据分析都用可执行代码表达。每一步都**可追溯、可复现**，这在传统科研里是极其稀缺的东西。

**个性化推荐**

用户偏好被编码为可编辑的策略状态，交互反馈持续驱动调整。

**具身智能**

Voyager在Minecraft里用技能库实现终身学习；CaP和RoboCodeX直接把LLM输出变成机器人控制程序；GenSwarm协调多机器人的策略生成和部署。


# 六、开放挑战：目前还有哪些悬而未决的问题

这一节是整篇综述最有"问题意识"的部分。坦率讲，读到这里会意识到：**整个领域对Agent的理解还处于很早期的阶段。**

**验证信号的充分性危机**

**通过测试 ≠ 正确修复。** 这是一条工程常识，但在Agent评测里被大规模忽视了。

PatchDiff发现SWE-bench上很多"solved" issue实际上依赖了issue文本里的答案泄露。Aardvark和Codex Security揭示了安全属性跟功能正确性之间存在巨大鸿沟。**仅用单元测试做验证oracles 是不够的**，需要lint规则、静态分析、fuzzer、甚至形式化验证的组合来作为verdict信号。

**长周期故障归因**

Multi-Agent系统里，"到底哪个Agent在什么时候把事情搞砸了"，现有方法的最佳归因准确率只有14-53%。连责任人都分不清，还谈什么优化？

**共享状态的一致性**

SyncMind 把这个问题形式化了：定义 |B-S| 为**"Agent 信念状态"和"真实仓库状态"之间的偏差**。当多个 Agent + 人类开发者 + CI 系统同时修改同一个仓库时，这种偏差是常态而非例外。

**自我进化与回归控制**

Agentic Harness Engineering让harness本身成为优化对象：更改prompt、调整工具描述、修改上下文打包策略。但如果Agent可以改自己的"底盘"：
- 怎么保证改完了不比之前更差？
- 怎么回滚？
- 权限边界被无声无息地放宽了怎么办？

这些问题在生产环境里是致命级别的。

**人机协作的信任校准**

多Agent + 多人类同时对同一个仓库操作时，什么时候该暂停等人类确认？什么时候可以放心让 Agent自己改？什么时候必须保留checkpoint便于回滚？目前这些决策还**没有清晰的设计原则**。


# 七、写在最后

读完这篇综述，有一个比喻反复浮现：

过去两年，整个行业在模型参数上狂飙突进：GPT-4、Claude 3.5、Gemini 2.5、DeepSeek-V3。但越来越清楚的是，**瓶颈正在从"模型够不够聪明"转移到"Agent能不能稳定可靠地干活"。**

这不是说模型能力不重要了，而是说：**一个聪明的模型没有好的底盘支撑，就像一辆法拉利引擎装在拖拉机底盘上，开快了会散架。**

这篇综述的价值，在于把"Agent基础设施"这个话题从零散的工程直觉，变成了一套**有分类学、有因果逻辑、有开放问题的理论框架**。Harness Interface、Harness Mechanisms、Scaling the Harness这三层分类，不一定是最终答案，但提供了一个非常有用的思考坐标系。
