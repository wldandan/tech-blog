# 【Agent实践】03-构建Agent的12原则

原文链接：https://zhuanlan.zhihu.com/p/1951370323745284347

​

目录

最近，[GitHub](https://zhida.zhihu.com/search?content_id=263142404&content_type=Article&match_order=1&q=GitHub&zhida_source=entity)上发布了一个项目12-Factor Agents，连续多天在趋势榜上，并且获得了8.2k stars。该项目主要介绍了构建AI智能体时，应该遵守的12条原则。这12条原则不是理论，而是实用的工程指导，目标是：

  * 让AI智能体更加健壮。
  * 让它们的决策更加透明。
  * 让开发者能完全控制它们的行为。
  * 帮助开发者从原型到生产安全地进行扩展。



## 1 原则1：[自然语言](https://zhida.zhihu.com/search?content_id=263142404&content_type=Article&match_order=1&q=%E8%87%AA%E7%84%B6%E8%AF%AD%E8%A8%80&zhida_source=entity)到工具调用

在智能体构建中，最常见的模式之一就是把自然语言转化为结构化的工具调用。这是一种强大的模式，有助于构建能理解任务并实际执行的智能体。

✅良好的智能体行为：“用户说‘我冷’ → 调用天气工具 → 获取温度。”  
❌不良的智能体行为：“用户说‘我冷’ → AI写了一首关于冬天的诗。”

## 2 原则2：拥有自己的[Prompt](https://zhida.zhihu.com/search?content_id=263142404&content_type=Article&match_order=1&q=Prompt&zhida_source=entity)

将Prompt视为重要的指令，不要将提示工程交给外部框架。掌控Prompt的好处：

  * 完全控制：准确地编写智能体需要的指令，没有黑盒抽象。
  * 测试和评估：为提示词构建测试和评估。
  * 迭代：根据实际性能快速修改提示。
  * 透明度：确切了解智能体正在处理的指令。



✅保存、修改和测试提示词。  
❌不要随机地将提示词硬编码到应用程序中。

## 3 原则3：拥有自己的[上下文窗口](https://zhida.zhihu.com/search?content_id=263142404&content_type=Article&match_order=1&q=%E4%B8%8A%E4%B8%8B%E6%96%87%E7%AA%97%E5%8F%A3&zhida_source=entity)

只向AI提供真正需要的信息。如同寻求技术帮助时，只需描述问题和设置，而不是讲述你的整个人生故事。创建良好的上下文包括以下内容：

  * 提供给模型的提示和指令。
  * 检索的任何文档或外部数据（例如RAG）。
  * 任何过去的状态、工具调用、结果或历史记录。
  * 来自相关但独立的历史/对话中的任何过去消息或事件（记忆）。
  * 有关输出何种结构化数据的指令。



✅精心选择传递给[LLM](https://zhida.zhihu.com/search?content_id=263142404&content_type=Article&match_order=1&q=LLM&zhida_source=entity)的输入数据。  
❌不要盲目地传递所有数据（这会浪费token并让模型困惑）。

## 4 原则4：工具只是结构化的输出

工具不需要复杂。本质上，它们只是LLM生成的结构化输出，用来触发确定性的代码。

✅`LLM returns: { tool: “search”, args: { query: “cheapest flights” } }`  
❌LLM writes: “Hey, I think you should maybe try looking it up somewhere…?”。

## 5 原则5：统一执行状态和业务状态

将AI应用所有步骤记录在一个清晰的日志中，如当前步骤、下一步、等待状态、重试次数等。就像将机器人完成任务时的每一步记录在一本笔记本，方便回顾和重现。

  * 执行状态：当前步骤、下一步、等待状态、重试次数等。
  * 业务状态：智能体工作流中到目前为止发生的事情（例如OpenAI消息列表、工具调用和结果的列表等）。



其好处为：

  * 简单：所有状态的真理都有一个来源。
  * 序列化：线程是可序列化/反序列化的。
  * 调试：在一个位置可以看到整个历史。
  * 灵活性：只需添加新事件类型即可轻松添加新状态。
  * 恢复时间：只需加载线程即可从任意点恢复。
  * 分叉：通过将线程的某个子集复制到新的上下文/状态ID中，可以在任何时间分叉线程。
  * 人机界面和可观察性：将线程转换为人类可读的markdown或富Web应用UI的简单方法。



## 6 原则6：使用简单的[API](https://zhida.zhihu.com/search?content_id=263142404&content_type=Article&match_order=1&q=API&zhida_source=entity)启动/暂停/恢复

智能体本质上是程序，应该能够随时启动、查询、停止或者恢复。

## 7 原则7：通过工具调用联系人类

LLM API通常依赖于一个关键的高风险token选择：是返回纯文本内容，还是返回结构化数据？当AI不确定时，让它向人类提问，避免瞎猜，导致错误输出。

✅“嘿，用户，我应该继续退款吗？”  
❌AI：未经确认直接输出：“退款已批准”。

## 8 原则8：拥有自己的[控制流](https://zhida.zhihu.com/search?content_id=263142404&content_type=Article&match_order=1&q=%E6%8E%A7%E5%88%B6%E6%B5%81&zhida_source=entity)

可以自定义构建适合特定用例的控制流，如某些类型的工具调用可能是跳出循环并等待来自人类或其他长期运行任务（如训练流水线）的响应。还可以实现以下自定义功能：

  * 工具调用结果的摘要或缓存。
  * 用LLM对结构化输出进行裁决。
  * 上下文窗口压缩或其他内存管理。
  * 日志记录、追踪和指标分析。
  * 客户端速率限制。
  * 持久休眠/暂停/“等待事件”。



✅代码指定：“尝试X → 如果失败，执行Y → 然后询问用户”。  
❌AI自行决定，你不知道怎么做，也不知道为什么。

## 9 原则9：将[错误压缩](https://zhida.zhihu.com/search?content_id=263142404&content_type=Article&match_order=1&q=%E9%94%99%E8%AF%AF%E5%8E%8B%E7%BC%A9&zhida_source=entity)进上下文窗口

智能体有一个优势是“自愈”能力。当执行任务过程中出错时，其能够通过读取错误信息，调整后续的工具调用。其价值点在于：

  * 自愈：LLM可以读取错误消息并确定在后续工具调用中要更改的内容。
  * 持久性：即使一个工具调用失败，智能体也可以继续运行。



✅“工具因超时而失败 → 5秒后重试。”  
❌AI失败后，隐藏错误并盲目重试。

## 10 原则10：小型、专注的智能体

一个智能体 = 一个任务。与其构建试图完成所有任务的单一大型智能体，不如构建小型、专注的智能体，每个智能体专注于完成一件事。这些智能体只是一个更大确定性系统中的构建块之一。其好处为：

  * 可控的上下文：较小的上下文窗口意味着更好的LLM性能。
  * 明确的职责：每个智能体都有定义清晰的范围和目的。
  * 更高的可靠性：复杂工作流中出错的可能性更小。
  * 更简单的测试：更容易测试和验证特定功能。
  * 改进的调试：更容易在问题发生时定位并修复。



✅  
智能体A：获取用户意图。  
智能体B：搜索文档。  
智能体C：总结结果。  
❌一个试图完成所有任务的大型智能体，最终大多数情况下都会失败。

## 11 原则11：从任何地方触发，在任何地方满足用户

用户应该能够通过聊天、远程、电子邮件、短信或任何其他们想要的渠道触发智能体，智能体也可以通过相应的渠道进行响应。其好处为：

  * 在用户所在的地方遇见用户：帮助构建看起来像真人，或者至少像数字同事的AI应用。
  * 外部循环智能体：允许智能体由非人类触发，例如事件、定时任务、故障等。
  * 高风险工具：通过快速引入各种人类参与，可以赋予智能体更高风险的操作权限，例如发送外部邮件、更新生产数据等。



## 12 原则12：让智能体成为无状态的reducer

每一步都应该是：输入 + 状态 = 新输出。

✅ 示例：智能体像纯函数一样工作，无隐藏记忆。 ❌ 示例：智能体行为依赖于之前不明来源的记忆。

## 总结

12条构建智能智能体的原则，其核心意义绝非引入额外复杂度，而是旨在​​提升开发过程的严谨性和系统性​​。助力开发者将创意从​​快速原型​​顺利落地为​​真实、可靠的生产系统​​。遵循这些原则，你将能够：

  * 显著减少隐藏缺陷和不可预测的行为。
  * 打造出更易于调试、扩展及长期维护的智能体。
  * 在强大的模型能力与必要的工程控制之间，划定清晰的职责边界。
