# JiuwenSwarm自演进机制：让蜂群越用越聪明

> 本文是JiuwenSwarm系列技术博客的第三篇。前两篇我们讲清了「单个Agent怎么被驾驭」和「一群Agent怎么协作」。本篇再往前推一步：**这套系统能不能在使用过程中自己变好？** JiuwenSwarm的答案是一整套自演进机制，从Skill级别的经验沉淀，一直延伸到Harness级别的「后训练」。

大多数Agent系统都藏着一个不太被人提起的缺陷：它的能力上限，从部署那天就被锁死了。

工具调用出错，留下一条日志，没人看；用户说"你理解错了"，被当成普通对话略过；同一个坑，Agent一遍遍踩，从不长记性。模型不会因为你今天纠正了它，明天就记住。它的权重是冻结的，它的技能文档是写死的。**能力定义一旦交付，就基本不再变化。**

JiuwenSwarm想打破这块天花板。它的回答不是某个单点功能，而是一整套**自演进机制**：从最轻量的Skill经验沉淀，一路延伸到对模型权重的在线增训。这套机制贯穿一个朴素的判断：**能力的上限，不该在部署那天就被固定。**

这篇文章把这套机制拆开讲清楚：它分成哪几层、每一层解决什么问题、为什么需要快慢两条路径协同。

![alt text](images/overview.png)

# 一、一张全景图：快慢两条路径

先把结论摆在前面。JiuwenSwarm的自演进不是单一闭环，而是**两条互补的路径**：

一条是**快速闭环演进**，信号检测、经验生成、Skills更新，**分钟级生效**，不重启、不重新部署。它擅长解决"操作层面"的问题：上次这个工具超时了，下次先重试；上次把"上海"理解成了"北京"，下次记住用户的真实意图。

另一条是**深度模型演进**，轨迹诊断、在线RL、LoRA热加载，**训练后生效**。它处理的是更深的问题：Leader拆解任务的策略对不对？子任务该并行还是串行？审校该在哪一步介入？这些不是操作失误，而是**策略选择**，靠静态的经验沉淀解决不了，需要让模型从真实协作轨迹里学。

![JiuwenSwarm 自演进全景](images/01_overview.png)

两条路径覆盖了从"操作失误"到"策略偏差"的完整问题谱系。**快的负责立刻止损，慢的负责改变直觉。** 接下来从快的这条线讲起。


# 二、Skill自演进：把"踩过的坑"变成"活的文档"

快速闭环的核心，是让单个技能从真实使用里积累经验。它基于openJiuwen的自演进框架，由`SkillCallOperator`这个算子统一管理所有Skill的读写与演进分发。整套机制是一个由**四个组件构成的闭环**。

![Skill 自演进四组件闭环](images/02_skill_loop.png)

**第一环是SignalDetector（信号检测器）。** 它是整条链路的起点，关键设计是**基于规则工作、不调用LLM**，所以响应快、成本低。它盯两类信号：
- 一类是**执行异常**，捕捉`error`、`timeout`、`permission denied`、`command not found`这类失败关键词，并归因到当前正在执行的Skill；
- 另一类是**用户纠错**，识别"不对""应该换个方式""你理解错了"这种负反馈。用户纠错往往比报错日志更有价值：异常只告诉你"出错了"，纠错直接告诉你"哪里理解错了"。

**第二环是SkillOptimizer（优化器）。** 它接住信号，结合对话上下文判断"这个问题值不值得记录"，偶发的网络抖动不必写进Skill，再调用LLM生成具体的改进建议。你敲下的`/evolve`命令，背后就是它在干活。

**第三环是SkillEvolutionManager（生命周期管理者）。** 它协调整个流程：`.scan()`扫描信号，提取需要演进的事件 → `.generate()`让LLM将信号转化为可执行的改进方案 → 存储，维护evolutions.json文件的读写 → `.solidify()`把待定演进内容固化进SKILL.md。

**第四环是SkillCallOperator（运行时入口）。** 每次调用Skill时，它会检查是否存在`evolutions.json`，存在就自动加载，把最新的演进经验和原始SKILL.md合并后再返回给Agent。**演进经验是实时生效的。**

这套闭环里有两个工程细节，特别能体现设计的克制。

第一个是**信号到改进的分类映射**。不同来源的信号，自动落到文档里最合适的位置：执行异常补进Skill的**Troubleshooting（排障）**段，下次遇到同样场景主动提示风险点；用户纠错则写入**Examples（正确用法示例）**段，让后续调用更容易理解用户的真实意图。

第二个是**经验"先待定、再固化"**。演进内容先写进Skill目录下的`evolutions.json`，字段`applied: false`表示待固化、`true`表示已合并进SKILL.md。**所有变更都不会静默落盘**，需要经过审批确认。于是Skill从一次性写死的静态文档，变成了持续迭代的活文档，同时又不会失控。

![信号分类映射与待定—固化机制](images/03_signal_mapping.png)

触发方式上既支持自动，也支持手动。开启`evolution.auto_scan`后，系统会在每次工具执行和对话结束后自动扫描，让演进变成一种"背景进程"。值得注意的是**集群模式下的`/evolve`必须带明确的演进意图**：团队技能影响所有成员，不能含糊。

在团队场景里，这套自演进会在**两个层面同时展开**：团队技能层根据执行轨迹自主演进Team Skills（增加成员角色、补充约束规则、优化协作流程），成员技能层让每位成员的Skill单独进化。演进采用**补丁架构**：经验以独立条目附加到Skills上，不直接改原始文件，每条经验都带触发来源、上下文、时间戳和质量评分，可以单独审查与淘汰。就像人类团队里"经验手册"和"操作规程"分开管理。


# 三、找得到、补得上：技能检索与自动创建

Skill自演进解决的是"已有技能怎么变好"。但还有两个相邻的问题：技能多了，怎么找到对的那个？技能不存在，又怎么补上？

**先说"找"。** 当系统装了几十上百个技能时，把它们全量塞进上下文会占用过高、分散注意力。JiuwenSwarm 的做法是离线构建一棵**已安装技能树索引**，运行时让模型像查目录一样**逐层披露**。它提供三个工具：
- `skill_branch_explore`：展开分支，主检索工具，优先用；
- `skill_branch_peek`：只看轻量摘要、不展开；
- `skill_index_build`：构建或复用索引，仅在索引缺失时调用。

它的边界很清晰：**只负责找技能，不替代执行或派发，也不安装新技能。**

**再说"补"。** 通过开关`evolution.skill_create`启用后，系统会注册`SkillCreateRail`（集群模式下是`TeamSkillCreateRail`），当判断某个任务值得沉淀为新技能时，主动提出创建建议。

找得到、补得上、用得好，三者合起来，技能体系才真正成为一个能**自我扩展**的能力库。

# 四、Auto Harness：给Harness装上"后训练"

如果说Skill自演进是局部优化，那**Auto Harness**就是快速演进的最高形态。

要理解它，得先接受一个等式：**`Agent = Model + Harness`**。Harness是把LLM变成自主Agent的编排基础设施：system prompt、工具定义、Rail（生命周期钩子）、Skill文档、子Agent 编排、长期记忆。没有Harness，LLM只是个会说话的概率模型；套上Harness，它才成为能行动的Agent。

这里有个被长期忽视的矛盾：**Model早就有了后训练**，从RLHF到DPO到GRPO，后训练让模型从"会说话"变成"好用"。**而Agent的另一半，Harness，它的后训练至今几乎完全依赖人工。** 调prompt、改tool desc、写skill文档、配rail钩子，每一次优化都是人在试、在猜、在调。

Auto Harness就是openJiuwen给出的工程化回答：**不训练模型权重，而是让Harness在实战中自动学习、自动优化。** 它被形象地称为给Harness装上"后训练"。

## 4.1 双层架构：Meta与Expert

Auto Harness的设计基础，是把Harness分成两层。这个分层的关键在于**安全边界的差异化**。

![Auto Harness 双层架构与评测驱动闭环](images/04_auto_harness.png)

**Meta Harness是基座**：通用编排引擎，所有Agent共享，定义控制流、路由、权限、上下文管理、生命周期、错误恢复。动它会影响所有Agent，所以必须走最严谨的路：**修改源码 → 提PR → 人工审查 → 合入主干。**

**Expert Harness是扩展**：领域特化的配置包，按需加载，包含领域提示词、技能定义、工具配置、MCP插件、Rail 扩展、角色画像等八类组件。它影响范围可控，可以**生成 Package → 热加载 → 即插即用**，迭代更快。一个Agent实例，就由Meta Harness加上 0 个或多个Expert Harness组合而成。

## 4.2 评测驱动的闭环

两层共享同一套核心流程，一个**评测驱动的闭环**：**评测 → 规划 → 实施 → 再评测**，循环迭代直到达到预期或超出最大次数。

落到工程上，每个环节都有具体机制：接收自然语言优化目标后，LLM分析任务维度、生成覆盖不同场景的评测集；用当前配置创建Agent跑评测，收集成功率、耗时、质量评分；从结果里识别问题根因，检索经验库里的历史经验，输出带依赖关系的优化动作清单；每个动作在**独立git worktree** 中隔离执行，按拓扑排序分波并行，完成后经过两阶段fix loop修复（CI直接修复 + 评审质量修复）；优化生效后，经验自动写入经验库，形成**飞轮效应**：越优化越精准。

整个过程都有严格的兜底：预算控制、编辑安全、不可变文件保护、失败自动回滚。Meta Harness优化还在此之上多了三步：**定时任务**（按周期自动触发，比如每 48 小时调研一次竞品）、**业界调研**（竞品差距分析，搜索业界较好实现作为参考）、**提交PR**（代码变更交人工审核）。举一个典型场景就很说明问题：**对标Claude Code的上下文压缩特性，自动吸收方案并提交PR**。Agent能主动对标业界最佳实践来改进自己。


# 五、深度演进：在线RL改变模型的"直觉"

快速闭环再快，也有它够不着的地方。

有些失败是更深层的：Leader Agent拆解任务的**策略**不对。应该先研究再写作，还是边研究边写作？子任务该并行还是串行？Teammate被阻塞时该继续等待，还是触发Leader修订计划？这些不是"上次报错下次规避"能解决的，它们是**策略选择**问题，固定的手写规则覆盖不全。要解决，只能让模型从真实协作轨迹中学习"怎样协同更有效"，也就是 **在线RL增训**。

## 5.1 离线RL的三个局限

经典 Agentic RL发生在模型研发阶段，是**离线**的：算法专家定方向 → 收集数据 → RL 训练 → 冻结权重 → 部署。对个人助理类Agent来说，这有三个局限：**优化目标固定**（以通用任务为目标，覆盖不到个性化需求）、**数据缺乏个性化**（来源泛化，缺乏用户偏好）、**迭代周期长**（厂商主导，难以快速响应用户侧反馈）。

**在线Agentic RL** 换了个思路：直接把处于部署工作状态的Agent接入RL训练系统，**采集真实用户交互轨迹，利用系统闲时动态训练更新模型。** 训练数据来自真实场景，能兼顾通用能力和个性化适配。优化后的权重即时生效，缩短了迭代周期。更妙的是，训练中的Rollout也真实走JiuwenSwarm Runtime，缓解了训推环境不一致的老问题。

JiuwenSwarm结合 **Uni-Agent**（verl社区最新的Agentic RL训练框架）实现了这套能力。

![在线 RL 闭环与实验验证](images/05_online_rl.png)

整条链路是这样跑通的：用户请求进入JiuwenSwarm，规划任务工作流，通过**Agent Gateway**发请求给推理模块，同时**Gateway采集轨迹数据**；轨迹经过奖励计算（内置PRM过程奖励，缓解在线RL奖励不稳定的问题），触发**Model Trainer**启动训练；训练生成**LoRA**存入仓库；Agent Runner监听到新版本，通知推理模块**加载LoRA、无停机完成在线优化**。整套流程**原生训推异步**，闲时/睡时训练，资源利用率高，还支持LoRA实时热加载和灵活回退。

### 5.2 让训练数据自己长出来

在线RL最大的难点，是高质量训练轨迹的持续构建。真实场景里，用户不会反复提交失败的任务。JiuwenSwarm的解法是**轨迹诊断模块**：把每次任务执行保存为可回放的多轮交互轨迹（含Prompt、中间决策、工具调用、Observation、团队信息、输出结果、用户反馈），先定位失败诱因发生在哪个阶段（规划、信息检索、证据绑定、结构化输出、Swarm协同），再针对薄弱项构造专项数据和奖励。奖励计算用的是**混合评测逻辑**，融合规则评分和基于Rubrics的生成式评分。

更关键的是，在线RL的优化对象不只是单Agent的行动选择（基座模型决策），还包括Agent Swarm的**分工协同策略**（Swarm Policy模型决策）。一次成功的协作，沉淀的不只是最终答案，还有任务拆分、依赖安排、消息沟通、审校返工的全过程；一次失败的协作，则沉淀暴露的问题：任务拆分粒度过小、审校环节缺失、成员重复工作、Leader介入过迟。**协同策略由"专家手写规则"进化成了"数据驱动的策略优化"。**


# 六、底层支撑：自演进是怎么挂进生命周期的

把镜头拉到最底层会发现，这些自演进能力其实都是一组挂在DeepAgent生命周期钩子上的**Rail**：`SkillUseRail`把演进经验注入技能描述，`EvolutionRail`采集模型与工具调用轨迹，还有`SkillCreateRail`、`TrajectoryRail`等。

openJiuwen把"可演进性"明确列为Harness的设计原则之一：**轨迹采集、Skill使用、Skill创建和Skill演进，形成经验反馈闭环。** 换句话说，**自演进不是事后打的补丁，而是从架构原则层面就内建好的能力。**


# 七、小结：一个飞轮，两种节奏

JiuwenSwarm的自演进，本质上是一个分层递进、快慢协同的飞轮：

**快速闭环**（分钟级）：SignalDetector捕信号、SkillOptimizer生成改进、经验先待定再固化，让技能越用越好；再往上是Auto Harness，用Meta / Expert双层和评测驱动闭环，让Agent自主优化驾驭层本身。它解决的是操作失误、规则缺失、经验空白。

**深度演进**（训练级）：轨迹诊断、在线RL、LoRA热加载，把部分用户反馈固化成模型的"直觉"。它解决的是策略偏差、协同低效、能力短板。

两者协同，覆盖从"上次这里报错了"到"这类任务该怎么协同"的完整问题谱系，最终指向同一个目标：**越用越聪明，越用越懂你。**

而所有这一切的前提，是系统得"记得住"用过的经验——这恰好是另一个值得单独展开的话题了。
