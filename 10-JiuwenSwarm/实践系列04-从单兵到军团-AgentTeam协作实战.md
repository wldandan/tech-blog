# 从单兵到军团：AgentTeam 协作实战

> 前三篇你让一个 Agent 越来越强。最后一篇，让一群 Agent 组队。这才是 JiuwenSwarm 要解决的那个真正的问题。

*JiuwenSwarm 实践系列 · #04 · ~4000 words*

## 这篇做什么

论单个 Agent 的能力你已经走到头了。但真实世界里最难的那些任务——跨领域调研、大型软件交付、多角色联合决策——从来不是一个人能搞定的。

软件交付要产品、研发、测试、SRE。会诊要分诊师加多个科室专家。一份两百页技术报告，十个人并行铺开再汇总。这些场景缺的不是更强的单兵，是一支配合默契的团队。

这篇组队：Leader 怎么带团队。SwarmFlow 怎么编排。TUI 怎么监控。读完你手里的不再是一个强 Agent——是一支可以被工程化管理的 Agent 团队。

---

## 什么时候组队

不是所有任务都该用团队。判断标准一条：**能不能拆成多个互不依赖的子任务，并行跑。**

- "今天天气怎么样"——一个 Agent 够。
- "分析这个项目的代码结构，给改进建议"——结构分析、安全审计、性能评估彼此独立。可以组队。
- "调研三个行业的竞争格局"——三个调研互不依赖。最该组队。
- "先读论文，再写摘要，最后发邮件"——每步依赖上一步，只能串行。别组队。

并行度是团队效率的来源。串行依赖多的任务拆成团队反而更慢——多了一层 Leader 调度的开销。

不确定要不要组队？看规划模式下 Agent 拆出来的 plan。多个步骤之间没有箭头连接（没有先后依赖）的，就可以并行交给团队。

## Leader 怎么带队

`/mode team`。现在你面前的不再是一个 Agent。Leader 就位了。

Leader 像一个项目经理。职责：理解需求 → 组建团队 → 拆任务 → 分配成员 → 汇总结果。Leader 不替 Teammate 干活。Teammate 认领自己擅长的任务，独立执行，完成汇报，碰到没法决策的问题向上求助。

12345 热线收到一个投诉最能说清这套机制："某工地夜间施工噪音扰民且扬尘污染。"一句话跨三个部门：

- 城管管噪音——夜间施工是否超标
- 环保管扬尘——排放是否合规
- 住建管资质——有没有夜间施工许可

Leader 拆成三个子任务，三个 Teammate 并行推进。城管查标准、环保查法规、住建查许可——互不阻塞。最后 Leader 汇总，拼成一份联合处置方案。

这整条链路上有一个刚性约束：**关键决策权始终在 Leader 手里。** 环保 Teammate 建议罚款——敏感操作，自动提交 Leader 审批。城管想查封设备——Leader 先查有没有"情节显著轻微可整改"的替代。住建要调跨部门敏感数据——请求经 Leader 审批并记审计日志。Teammate 自主建议可以，收尾的决策不放手。

## 协作的物理层：共享工作区 + 团队记忆

多个 Agent 协作需要两个东西：共同的桌面，和共同的记忆。

**共享工作区**就是桌面。`team-workspace/` 目录所有成员可见。Teammate A 写完调研数据，Teammate B 直接读——不用传文件，不用发消息说"你去看那个路径"。共享区有文件级锁定：A 在写 `report.md` 时 B 只能读。各成员自己还有独立工作区，放临时工具输出，互不干扰。

**团队记忆**是所有成员的共享知识库。关键设计：**所有成员只读，Leader 在每个协作轮次结束后统一写入。** "只读 + Leader 一个人写"这个约束，从根上消灭了多成员并发写同一份记忆的冲突和污染。Leader 写完，提取 Agent 用四类标签分拣：`decision`（为什么选 A 不选 B）、`lesson`（什么有效、什么导致返工）、`member`（谁擅长什么）、`context`（业务约束、截止日期）。

记忆还分临时和持久两种。临时记忆任务结束就清。持久记忆跨轮次累积。跟人类团队把经验手册和操作规程分开管理，一模一样。

## SwarmFlow：让协作跑稳

没有 SwarmFlow 时，Leader 的调度全靠临场反应：边理解需求，边拆任务，边建成员，边等结果，边决定下一步。任务简单时灵活。任务变长变大后，三个问题马上冒出来：

1. Leader 变瓶颈。每份中间结果都回 Leader，上下文被流程信息淹了。
2. 过程不稳定。同一个任务跑两次，两次路径不同——因为每步都是 Leader 当场拍板。
3. 执行不可靠。谁先谁后、何时汇总、失败怎么处理——全看 Leader 临场发挥。

SwarmFlow 的解法直接：**编排归系统，智能归 Agent。** 谁先做、谁并行、谁把结果给谁、何时汇总、失败怎么处理——这些编排逻辑，系统按程序稳定跑。每个子任务具体怎么理解、怎么推理——这些智能，Agent 来做。

Leader 不再每轮手动调度。启动工作流，后面 SwarmFlow 自动推。

### 什么时候用，什么时候不用

一条标准：**编排能不能提前确定。**

能确定——"论文分析 → 生成总结 → 撰写邮件"，每步输入输出明确。固化成工作流，SwarmFlow 稳定执行。不能确定——"多专家圆桌研讨、战略讨论"，谁回应谁、谁质疑谁、谁补充谁，得看上下文随时调整。用不带脚本的 Swarm Skill，保留开放协作。

确定性跟开放性在同一套体系里各自安好。SwarmFlow 增加的不是 Agent 数量，是协作的确定性。

### 动手：用一句话生成工作流

在 TUI 里：

```
帮我写一条工作流，逐篇分析这批论文、汇总成技术分享报告，最后发邮件给团队成员。
```

内置的 SwarmSkill Creator 自动判断——这个任务需要工作流。生成最小 `SKILL.md` 入口和 `workflow.py` 脚本。脚本里的算子把协作编排固化为阶段：搜论文 → 理素材 → 分析内容 → 生成报告 → 发邮件。阶段内 Agent 并行，阶段间串行推。

不用手写脚本，不用理解算子。一句话。

### TUI 监控面板

工作流跑起来了，怎么看过程？`/swarmflows`，三层交互：

```
Swarm workflows
1 running

  ● running  技术调研与邮件分享    3/8 agents
```

按 Enter 进阶段详情：

```
Phases                    Agents · 搜索
  ✓ 搜索论文   2/2         ✓ completed 资料搜索员
  ● 整理素材   1/2         ✓ completed 论文筛选员
  ○ 分析内容   0/2
  ○ 生成报告   0/1
  ○ 发送邮件   0/1
```

选中 Agent 按 Enter——完整 Prompt、Outcome。失败了还能看 Error。不翻日志，不 grep。三层逐层下钻。

## 这篇完了。系列也完了。

这是第四篇，也是最后一篇。从装好第一个 JiuwenSwarm 实例开始，到这篇让一群 Agent 组队协同——你走过了一条完整的线。

AgentTeam 不是更强的单兵，是一支会协作的团队。Leader 管规划和汇总，Teammate 各做各的并行推。共享工作区和双层记忆提供了物理底座。SwarmFlow 把确定的编排固化成稳定的工作流——编排归系统，智能归 Agent。

JiuwenSwarm 从第一天就在做同一件事：**不造一个超级个体，造一群各有所长、彼此协同、不断进化的群体智能。**
