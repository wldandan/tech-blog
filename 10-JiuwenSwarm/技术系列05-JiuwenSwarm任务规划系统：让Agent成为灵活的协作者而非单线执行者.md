# JiuwenSwarm任务规划系统：让Agent成为"灵活的协作者"而非"单线执行者"

> 本文是JiuwenSwarm系列技术博客的最后一篇。前几篇讲了驾驭、协作、演进和记忆，本篇聚焦「执行的骨架」，也就是任务规划系统。它要解决的核心矛盾是：**真实世界的任务总在变，Agent得在动态变局中既不丢目标，又能随时插队，还能自己定时醒来干活。**

![alt text](images/overview.png)

# 一、从一个发票场景说起

一个特别生活化的场景点出了任务规划的必要性：

你正让Agent处理 12 月的发票，处理到一半，临时想起 1 月的也要一起弄，于是插一句"把 1 月的也加进来，最后合并发我"。

一个"单线执行者"式的Agent会怎么办？要么排队等 12 月处理完再从头来过，要么干脆把新需求当噪声忽略。这两种都不是人想要的。**人真正想要的，是Agent能动态打断当前流程，把新任务插进合适的位置，最后按新目标合并交付。** 在政务这类场景里，这种动态性更突出 —— 领导临时批示加急、上级政策突发调整，Agent得在不重置上下文的前提下响应变更。

为此JiuwenSwarm引入了**"任务规划模式"**，把智能体从"单线执行者"升级为"灵活的协作者"。它依托openJiuwen的智能体中断恢复与任务调度能力，保证任务流在被打断、被插入后依然连贯可控。

这篇文章拆解这套系统的五个支点：双层任务循环、待办工具包、运行中交互、定时与心跳的自驱、以及统一的命令入口。

![任务规划：从单线执行者到灵活协作者](images/01_overview.png)


# 二、执行的骨架：双层任务循环

要理解任务规划，得先看Agent是怎么执行任务的。JiuwenSwarm的任务执行是**双层嵌套**的，一个ReAct内环套在一个Task Loop外环里。

![双层任务循环：ReAct 内环 + Task Loop 外环](images/02_double_loop.png)

:::info
**内层是ReAct循环**，最基本的执行单元：思考当前状态，调用工具，观察结果，循环直到当前子任务完成。

**外层是Task Loop**，负责把一个大目标推进为多轮任务执行。它由五个组件协作：
- **TaskLoopController**，提交每轮任务、接收结果；
- **LoopCoordinator**，综合停止条件、中断信号、追加需求、运行时引导等判断该继续还是停止；
- **TaskLoopEventHandler**，做事件分发；
- **TaskLoopEventExecutor**，执行单轮迭代；
- **LoopQueues**，管理运行中交互的队列。

什么时候用哪种？**单轮模式**，开销低，适合单次问答和短工具任务；**Task Loop模式**，适合需要任务计划、异步子代理、运行中干预和明确完成条件的复杂场景。
:::

那"任务算不算做完了"由谁判断？是TaskCompletionRail，它统一承载所有停止条件，采用OR语义（任一满足就停）：最大轮数（不想Agent无限烧token，设 8 轮就行）、超时（默认 1200 秒）、token预算、Agent承诺完成的项全部完成、以及自定义停止条件。

# 三、计划怎么表示：待办工具包

双层循环是执行的引擎，而任务规划的核心载体，是一份待办清单（todo）。它以JSON持久化到会话专属的存储目录下，**按会话隔离，支持并发安全读写**。

围绕这份清单，有四个工具构成任务的完整生命周期。

:::info
![待办工具包与任务状态机](images/03_todo_tools.png)

- **创建工具**把模糊的复杂需求拆成子任务列表；
- **修改工具**负责更新、删除、取消、插入、追加；
- **列表工具**展示状态概要；
- **详情工具**取单个子任务详情。

每个子任务的状态机是 **"待执行 → 执行中 → 已完成"**，外加随时可达的"已取消"，而且**同一时间只有一个任务在执行中**。
:::

把开头那个发票场景对应到工具调用，逻辑就清楚了：模糊需求用创建工具拆成清单，用户中途插入"1月也加进来"用修改工具在合适位置插入新子任务，某子任务做完用修改工具更新状态，不需要了就取消或删除。

这套工具包要解决的根本问题，是长周期任务里的"目标丢失与执行断层"。有了一份持久化、可随时增删改的显式清单，Agent就不会在第 20 步忘了第 1 步要干嘛，被打断后也能准确回到正轨。

值得强调的是，**任务规划不是写死的内置功能，而是一条按模式动态挂载和卸载的Rail**。

- **默认的规划模式**：会注册任务规划和子代理Rail；
- **快速模式**：则主动卸载**这些"重编排"能力，换取快速响应；
- **集群模式**：成员同样挂载任务规划Rail，由Leader负责分解、调度、汇总。

需要复杂编排时挂上，需要极速响应时卸掉，这正是整个系列反复出现的主线。

:::info
# 四、人怎么插手：运行中的三种交互

光能拆任务还不够。人得能在Agent跑的过程中介入。JiuwenSwarm支持三类运行中交互，它们的差别在于"多急、多快生效"。

![运行中交互：追加需求 / 运行时引导 / 取消](images/04_interaction.png)

**追加需求**不打断当前执行，把新需求排进Task Loop的下一轮。好处是不破坏当前正在执行的工具调用的一致性，Agent先把手头这步做完，下一轮再处理补充需求。

**运行时引导**即时生效，Agent在当前轮次的下一步推理中就考虑这个引导，适合"方向偏了，赶紧纠正"。

**取消任务**取消当前任务循环，尝试中断内层ReAct循环做到干净退出。

三类交互通过TaskLoopController的事件队列进入TaskLoopEventHandler，在每轮迭代边界被消费。设计思路很克制：**不中断当前原子操作的一致性，但在最近的迭代边界响应人类意图。** 既不让人干等，也不在工具调用一半时硬生生掐断。

配套还有状态持久化兜底：当前迭代次数、TaskPlan完整快照、停止条件状态、待处理的追加需求队列、规划模式状态，都会保存在会话里。再加上分支会话（从当前会话创建分支）和回退（回退到指定轮次并可选恢复文件），长程任务即使跑几十分钟也不怕丢状态。
:::

# 五、不用问也会动：定时任务与心跳

到这里，Agent还是"你问一句它答一句"。要让它真正成为协作者，得给它**自驱能力**，不用人提问，自己也会按时醒来干活。JiuwenSwarm给了两条互补的自驱入口。

![自驱的两条入口：定时任务与心跳](images/05_cron_heartbeat.png)

**定时任务解决"在特定时间点做特定的事"** 

它持久化到本地任务文件，调度器自动生效，有几个设计细节很见功夫：
- 时间表达式支持简洁的 5 字段格式和精确的 7 字段格式（5 字段自动补齐）；
- 提前唤醒秒数（默认 300 秒）让Agent在推送时刻之前就预先开跑，等到点时结果已经备好，而不是到点才慢悠悠启动；
- 运行模式字段直接复用任务规划能力，简单提醒模式只做轻量回复，先规划再执行模式则是"复杂任务先拆解再推进"。

多通道推送里还有个有意思的例外：推送到终端时故意不绑定会话ID，而是广播到所有已连接的终端窗口，因为终端会重启、会切会话，按会话过滤会让重启后的推送被丢掉，广播才能保证送达。

**心跳解决"以固定节奏持续自检和推进"**

网关按固定间隔（默认 3600 秒）向AgentServer发探活请求。关键在于它怎么处理这次探活：若工作区根目录存在心跳任务文件，Agent 会解析其中的活跃任务项，**把它拼成一条聊天请求、走正常的对话流程执行**；否则只返回一个正常的确认响应。把任务拼成普通聊天请求走正常管线，是个相当省事的复用，心跳不需要一套独立的执行引擎。它还支持生效时段（可跨午夜）、结果回传到指定渠道，且环境变量优先级高于配置文件。

一句话区分：**定时任务是"按日历表干活"，心跳是"按固定脉搏自驱"。** 两者一起，Agent才不再是只在你提问时才醒来的被动工具。

# 六、统一的入口：Slash命令架构

最后，所有这些任务和控制能力，都需要一个统一、一致的交互入口，这就是Slash命令体系。它的核心设计是"单一事实来源"加三层分类。

三层分类对应命令在系统里"走多远"：
- **第一类是通道控制命令**，在Gateway就被识别，可能只改会话、模式或路由而根本不进入Agent对话，比如新建会话、切换模式；
- **第二类是纯客户端命令**，仅终端或 Web 本地解析，比如清屏、换主题；
- **第三类是混合类**，名称跨端对齐但执行分端。

单一事实来源的具体落点是一份集中维护的常量表，**其它管线必须引用同一份常量，不能各自维护一个子集**。这是为了避免"集合漂移、同名命令语义不一致"这类典型的多端bug。和任务直接相关的命令包括定时任务管理、自动化流水线、分支会话、回退等。


# 七、小结：四种入口，一套可控的执行骨架

把本篇串起来，JiuwenSwarm的任务规划系统其实提供了四种互补的任务入口，共同支撑起一套可控的执行骨架：

- **待办工具包**：管"这一次对话"里的任务拆解、插入、状态流转，解决目标丢失与执行断层；
- **任务规划Rail**：把规划能力做成按模式挂载和卸载的可插拔组件；
- **定时任务**：管"特定时间点"的任务，可直接复用规划能力，还有提前唤醒、广播推送等细节；心跳管"固定节奏"的自驱任务，复用了正常对话管线；
- **Slash命令**：用单一事实来源加三层分类，给所有任务和控制提供统一一致的接口。

**这套系统的共同目标，是让Agent从"你问一句它答一句"的单线执行者，变成一个能动态响应变化、能自己定时醒来、被打断后也能稳稳回到正轨的"灵活协作者"。**
