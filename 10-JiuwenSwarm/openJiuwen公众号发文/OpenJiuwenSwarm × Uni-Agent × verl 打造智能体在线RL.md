# OpenJiuwenSwarm × Uni-Agent × verl 打造智能体在线RL
openJiuwen openJiuwen 2026年6月11日 20:20 上海 听全文

当前主流的个人助理类Agent，如JiuwenSwarm、OpenClaw、Hermes等，已经具有很强的任务处理能力，能够通过调用工具，自主规划执行复杂任务。然而，由于应用场景的多样性，用户在遭遇以下使用问题时，可能需使用RL技术增强模型能力：
• **模型垂域能力欠缺**：现有基座模型普遍采用通用数据集训练，导致Agent在部分垂域应用场景下表现欠佳，需定向增训；
• **Swarm协同策略需调优**：固定的Swarm协同策略难以应对复杂多样的应用场景，需通过在线调优Swarm协同策略，实现基座模型和Harness系统的联合优化，提升Agent Swarm的适应能力；
• **Skills数量激增**：Skills数量存在收益边界，无序增长可能导致Agent表现下降、上下文膨胀、延迟增加，需探索将其内化为模型能力，提升Agent系统效率。
为了缓解这些问题，JiuwenSwarm基于Uni-Agent × verl 的Agentic RL 能力，实现了模型在线RL自演进能力，让个人助理越用越聪明，越用越懂用户。
# 一、什么是在线 RL
经典的Agentic RL发生在大模型研发阶段，通常采用离线训练模式：算法专家确认大模型优化方向和训练策略后，从多来源收集训练数据，用于RL训练；经训练优化的模型，权重参数冻结后，被用于驱动Agent的推理服务。

对于JiuwenSwarm为代表的个人助理类Agent，离线Agentic RL存在以下局限性：
1. **优化目标固定**：训练目标通常以增强LLM在若干类Agent通用任务场景下的能力为目标，未充分覆盖不同用户和场景；
2. **数据缺乏个性化**：训练数据通过预先离线收集获取，来源广泛通用，缺乏对用户偏好的定制，导致模型缺乏个性化；
3. **模型迭代周期长**：“数据收集->RL训练->模型发布部署”的开环演进模式，由模型或Agent厂商主导，迭代周期长，难以快速捕捉和敏捷满足用户侧反馈需求。
区别于离线Agentic RL，在线Agentic RL直接将处于部署工作状态的Agent，接入到RL训练系统中，在在线采集Agent与用户、环境/工具的交互轨迹数据的基础上，利用系统闲时动态训练更新模型。一方面，在线Agentic RL训练所用的轨迹数据来源于真实的Agent应用场景，可在提升特定任务准确率的同时，满足用户个性化诉求；另一方面，训练优化后的模型权重参数，可即时更新，有效缩短了迭代周期。此外，在线Agentic RL训练过程中，Rollout服务通过Gateway和JiuwenSwarm进行交互，使得训练中Rollout也真实使用了JiuwenSwarm Runtime，缓解了训推环境不一致的问题。

# 二、在线RL适用于哪些场景
此前，JiuwenSwarm已先后发布上下文、记忆、Skills等自演进能力，支持基于轨迹数据所携带的用户反馈、执行结果等信息，实现Agent应用的数据闭环和自我强化。现有的上下文、记忆、Skills等自演进能力，具有闭环快、可解释、易回滚等优势，能够有效改善众多场景下用户的Agent使用体验。作为Agent自演进能力的有机组成部分，在线RL进一步聚焦解决由于长程交互（Long Horizon）、策略敏感等因素，所导致的复杂Agent任务重复失败的问题。同时，作为上下文、记忆、Skills等自演进能力的有益补充，在线RL可将部分用户反馈信息固化为模型“直觉”，从而提升Agent的基座能力。
JiuwenSwarm期望使用基于在线RL的自演进技术，增训个人助理类Agent的基座模型（Agent Policy）和协同策略模型（Swarm Policy），提升以下应用场景用户体验：
### 1. 高频垂类任务个性化
个人助理类Agent的主要应用场景之一，是重复处理特定垂类任务，如资料检索、数据实时分析、视频制作、博文生成/发布等。这些垂类任务的执行流程相对固定，但依赖相关专业领域的知识或能力。驱动个人助理类Agent的模型服务通常采用通用大语言模型，特定专业领域的知识或能力相对薄弱，可通过积累高频任务轨迹数据、挖掘用户反馈或评测执行结果，实现对指定任务目标的个性化增强。
### 2. 通过端云模型协同降低Token成本
长期持续使用个人助理类Agent的Token消耗量惊人，业界正在探索借助模型服务的动态路由分发技术，将部分简单任务由云端模型服务，卸载至本地部署的4B/8B/14B量级小规格大模型。然而，小规格大模型处理长程交互任务的能力有限，容易由于自规划、工具调用、多轮状态管理、停止条件等中间环节结果的不确定性，导致任务执行失败。借助在线Agentic RL，针对性地提升小规格大模型在特定任务中策略规划能力和行为稳定性，可使其成为云端大模型的低成本可靠“助手”。
### 3. Swarm协同策略演进
JiiuwenSwarm支持多个专业Agent协同工作，以实现群体智能与协作涌现。其由单Agent的“自主规划任务”，演进为“协调团队合作”：Leader Agent需要拆解目标、安排成员、管理依赖；Teammate Agent需要认领任务、独立执行、汇报结果；最后Leader Agent需要审校任务是否达标，并触发回退、重试、轮转。其中所涉及的大量策略选择，包括但不限于：
• 当前任务是否需要组队；
• 是否需要，及如何拆解子任务；
• 先研究再写作，还是边研究边写作；
• 子任务采用并行或串行策略；
• Leader Agent介入审校的时机；
• Teammate被阻塞时应继续等待，还是触发Leader修订计划。
上述策略共同构成了Swarm协同策略，仅靠固定模板，难以覆盖所有Agent Swarm协作场景。在线Agentic RL通过驱动协同策略模型从Agent Swarm轨迹数据中学习“如何协同有利于完成任务”，使协同策略由专家手写规则进化为数据驱动的策略优化。

# 三、Uni-Agent × verl 使能JiuwenSwarm在线RL
作为尚处于技术探索阶段的创新方向，在线Agentic RL存在以下技术挑战：
1. **数据稀疏**：RL训练依赖大量数据，但在线采集的Agent轨迹数据数量少，数据样本和奖励信息稀疏，算法设计难度高；
2. **奖励不稳定**：个人助理类Agent处理的大部分任务，不存在“标准答案”或标注信息，无法给出明确正确或者错误的奖励信号，可能导致训练效果欠佳；
3. **环境难复现**：离线Agentic RL可通过GRPO等算法，重复处理同一任务，在较大范围内探索策略空间，而在线Agentic RL仅能记录单次用户真实环境下的轨迹数据，缩小了策略探索范围，提升了训练难度。
Uni-Agent是verl社区最新发布的Agentic RL训练框架，旨在提升Agentic RL的端到端训练效率，其包含以下模块：
• **Agent Gateway**：支持Agent应用零修改、黑盒接入RL框架，提供Agent轨迹数据管理机制，可根据RL算法灵活适配轨迹记录。
• **Agent Runner**：负责端到端执行Agentic RL的训练流程，包括Agent Rollout，奖励计算，收集轨迹，触发RL训练。
• **Agent Infer**：Agentic RL推理加速模块，通过优化多轮交互场景的KVCache命中率，提升Rollout效率。
• **Agent Env**：Agent环境管理模块，通过沙箱实现Agent执行环境/资源的隔离和复用，优化多实例执行效率。
• **Model Trainer**：基于verl实现的Agentic RL高性能训练优化器，实现训推异步等加速特性。

JiuwenSwarm结合Uni-Agent的Agent Gateway，实现了基座模型或Swarm协同策略模型的在线Agentic RL训练，其主要流程包括：
1. 用户交互请求进入JiuwenSwarm；
2. JiuwenSwarm规划任务工作流，通过Agent Gateway发送LLM请求给Agent Infer进行推理计算，并调用环境/工具处理任务；
3. Agent Gateway采集轨迹数据，将轨迹发送给Redis模块；
4. Agent Runner从Redis模块获取轨迹数据，按配置调用在线RL专用的奖励计算模块，计算轨迹奖励，完成数据预处理后触发Model Trainer启动RL训练；
5. Model Trainer调用优化器进行RL训练，生成LoRA并存入LoRA仓库。
6. Agent Runner监听到训练生成新版本LoRA，通知Agent Infer加载LoRA文件，完成无停机的模型在线优化。

JiuwenSwarm和Uni-Agent在真实的用户环境下，采集包含用户请求、工具返回/沙箱状态等信息的失败任务轨迹数据，用于增训基座模型或Swarm协同策略模型，其主要的优势包括：
1. **原生训推异步**：JiuwenSwarm任务执行过程中生成轨迹数据，闲时/睡时触发模型训练，训推过程异步，无需Rollout，计算资源使用率高。
2. **多种轨迹模式**：Agent Gateway支持多种轨迹收集模式，包括session级、用户级，可根据场景或算法扩展数据粒度。
3. **内置PRM奖励**：默认支持Process Reward Model模式奖励，可解决或缓解在线RL奖励不稳定问题。
4. **低成本/灵活**：采用低参微调训练，减少GPU资源需求，支持LoRA实时热加载和灵活回退。
# 四、训练数据动态构建：从专项能力到Swarm协同策略能力
在线Agentic RL的核心难点之一，是高质量可复盘训练轨迹的持续构建。在真实Agent应用场景中，用户通常不会反复提交执行失败或结果不符合预期的任务，因此，真实Agent应用场景下沉淀任务执行过程、诊断能力短板，并将失败轨迹转换为专项训练数据，是提升Agent长程任务处理能力的关键。JiuwenSwarm引入轨迹诊断模块，从任务执行轨迹中分析模型能力短板，然后针对性地构造RL专项能力训练数据，用以增训提升模型的长程任务的处理能力。相较于业界基于细粒度过程奖励模型的方案，该方案实现更轻量，原理更简明，也能降低过程级奖励建模成本。


### 1. 任务轨迹诊断与专项数据构造
Uni-Agent的Agent Gateway组件，将每次任务执行过程，分别保存为一条可回放的多轮交互轨迹数据。每条轨迹数据包含用户Prompt输入、Agent中间决策、工具调用、Observation、团队信息、输出结果、用户反馈等详尽信息。轨迹诊断模块在分析轨迹数据的基础上，首先定位任务失败的诱因，确认偏差发生于Agent工作流的具体阶段，如规划阶段、信息检索阶段、证据绑定阶段、结构化输出阶段、Swarm协同阶段等。然后，轨迹诊断模块根据轨迹数据，抽象任务Prompt及其奖励，并针对失败诱因所对应的模型能力薄弱项，构造专项数据及其奖励。其中，奖励计算采用混合评测逻辑，即融合规则评分和基于Rubrics的生成式评分。

### 2. 协同策略模型优化
JiuwenSwarm多Agent协同场景下，RL优化对象不仅包括单Agent的行动选择（由基座模型决策），还包括Agent Swarm的分工协同（由Swarm协同策略模型决策）。Swarm协同策略模型负责生成Agent Swarm执行过程中的可观察决策，如：
• 如何拆解任务和设置依赖；
• 何时执行审校；
• 失败后如何调整团队策略。
在线Agentic RL可以把Agent Swarm执行轨迹转换为训练信号：一次成功的协作，不仅沉淀最终答案，也会沉淀任务拆分、依赖安排、消息沟通、审校返工等过程；一次失败的协作，会沉淀Swarm协同过程中暴露的问题，如任务拆分粒度过小、审校环节缺失、成员重复工作、Leader Agent介入过迟等。在Swarm Policy训练环节中，Leader Agent以带领Swarm团队完成任务为学习目标，Teammate Agent侧重学习如何从团队信息中明确子任务并执行，生成正确且可靠的子任务响应。
# 五、Meeting Analysis 实验
本文从PinchBench meeting_analysis 任务中，随机抽取了5个任务样本作为测试集，并基于未进入测试集的23条样本构建用户任务数据和专项能力数据。实验中采用Qwen3-4B模型作为Agent的基座模型，采用GRPO算法对其进行在线训练。单Agent场景下，直接RL增训基座模型，提升其任务处理能力；Agent Swarm协作场景下，增训Leader Agent的Swarm Policy，提升其筛选融合Teammate Agent结果的能力。由于模型和数据集的规模均较小，上述模型训练收敛快且存在波动，单Agent的任务平均得分由47.8%提升至54.2%（+6.4%），Agent Swarm协作场景的任务平均得分由51.4%提升至58.5%（+7.1%），且Agent系统输出的会议纪要结果，有较显著的主观质量提升。

# 六、小结
在线Agentic RL作为智能体自演进能力的有机组成部分，与上下文、记忆、Skills等自演进能力相结合，可实现“慢速+快速”协同的智能体持续演进，最终达成“越用越聪明、越用越懂你”。JiuwenSwarm和Uni-Agent相结合，实现了面向单Agent和Agent Swarm的在线Agentic RL，验证了模型增训可有效提升长程交互、策略敏感任务的执行效果。
个人助理类Agent由于应用场景的复杂性，其能力自演进仍然是一个前沿和开放的命题。在线Agentic RL仍然存在数据样本少、奖励稀疏、环境难重现等挑战，限制了其模型增训收益。
JiuwenSwarm后续计划探索以下技术方向，以进一步提升在线Agentic RL效果：
1. 环境重现：利用Uni-Agent的Agent Env实现环境拷贝能力，在训练环境中复现真实用户环境（非私有数据），使能多路径采样，提升GRPO类算法在线RL训练收敛速度。
2. 上下文优化：在线Agentic RL在多轮交互任务中，上下文冗长，且思考过程（<think>）等非结论性内容占比高，需通过轨迹采集优化，去除冗余内容，提升RL训练稳定性
3. Swarm RL：面向Agent Swarm协作场景，构建基于Uni-Agent的在线Multi-Agent RL能力，探索轨迹拆分、奖励分配等关键技术。
