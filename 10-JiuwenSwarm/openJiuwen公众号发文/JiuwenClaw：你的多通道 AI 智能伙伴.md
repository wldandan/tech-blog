## 1. 任务规划：为智能体装上“任务管理器”，做一个懂你的任务管家

## 2. 自演进：自动识别智能体轨迹中的异常错误，实现Skills自主演进
Agent 系统普遍存在一个问题：能力定义一旦写好，就基本不会再变了。工具调用出错，记录一条日志；用户反馈说理解有误，下次还是同样的逻辑。能力的上限，从部署那天就已经固定了。
JiuwenClaw 基于 **openJiuwen 自演进框架**，以 SkillCallOperator 算子统一管理所有 Skills 的读写与演进分发。在此基础上，系统内置了一套演进信号检测机制，持续检测执行过程和对话内容，将真实使用中遇到的问题转化为 Skills 的改进输入。
**识别哪些信号？**
信号来源主要分两类：
**执行异常**：包括工具调用超时、接口返回报错、代码执行中的异常中断等。只要任务执行中出现明确的失败字样，系统会自动识别并将其归因到当前正在执行的 Skill 上。
**用户纠错**：当你说“不对”、“应该换个方式”、“你理解错了”这类话语时，系统不会将其当作普通对话略过，而是会识别为一次有效的负反馈。这类信号往往比报错日志更有价值——它直接点出了 Skill 在理解或处理逻辑上的偏差。
**信号捕获之后做什么？**
系统会全程追踪当前活跃的 Skill 模块，确保每个信号都能准确对应到具体的技能文档。具体的改写逻辑如下：
**异常案例 → 排障建议**：执行失败的现场记录会被整理成具体的操作建议，补充进 Skills 的 Troubleshooting（已知问题与处理方式）部分。下次遇到相同场景，Skill 可以主动提示已知的风险点和应对方式。
**纠错交互 → 示例补充**：用户纠错的对话片段会作为新的 Example（正确用法示例）写入 Skills 文档，让后续的调用更容易理解用户的真实意图。
**重复失败 → 优先修订**：如果同一个工具在同一会话中多次出错，系统会将其识别为高优先级的演进信号，标记为需要优先关注和修订的 Skill。

## 3. 上下文瘦身：上下文卸载技术保证智能体长时运行，守护你的账单
在日常使用智能体的过程中，我们常常会遇到这样一个问题：随着对话越来越长，AI的“记忆力”似乎开始跟不上了，要么反应变慢，要么干脆忘了最开始说过什么。这背后其实是技术上的一个核心挑战：​**上下文窗口的容量限制**​。就像人的短期记忆有限一样，AI在同一轮对话中能处理的信息量也是有上限的。
为此，JiuwenClaw **使用了openJiuwen的​上下文卸载（Context Offload）机制**​，核心思路可以类比为“整理办公桌”：当桌面上的文件（消息数量）或纸张厚度（Token数量）超过一定限度时，系统会自动将那些“不太重要的大文件”归档压缩，只保留一份简短的索引放在桌面上，从而为当前正在进行的任务腾出空间。
#### 这套机制是如何配置和工作的？
为了让AI能够灵活应对不同类型的对话场景，JiuwenClaw 提供了一套可调节的“上下文管理规则”，核心参数包括：
**触发条件设置**​：你可以设定当消息数量超过messages_threshold（如3条），或累计Token数超过tokens_threshold（默认20000）时，启动“上下文瘦身”。
**识别超长消息**​：通过large_message_threshold（如1000 Token）来定义什么样的消息算“大消息”，系统会优先处理这些占内存的信息。
**选择性归档**​：通过offload_message_type，你可以指定只对某些角色的消息进行压缩（例如只压缩工具返回的冗长数据，保留用户和助手的核心对话）。
**压缩方式**​：当触发卸载时，系统会通过采样或摘要的方式保留关键信息，并保留索引信息[[OFFLOAD:...]]，既节省空间又保留了上下文的可检索性。
**重要内容保护**​：为了避免误伤近期的关键对话，你还可以设置messages_to_keep强制保留最近若干条消息，或开启keep_last_round确保最新一轮的用户-助手对话完整保留。

## 4. 记忆随行：分层记忆，让智能搭档越来越懂你的心
JiuwenClaw 搭载**分层持久化记忆系统**，以结构化文件体系为核心载体，实现身份、场景、操作轨迹的全维度长效存储与智能检索，保障跨会话交互的连贯与精准。
系统采用**文件分层存储**设计，所有记忆数据落地本地文件系统：
**稳定身份层**：存储核心身份、专属称呼等固定信息；
**长期背景层**：沉淀角色设定、核心关系等跨会话背景；
**动态轨迹层**：按日记录交互轨迹、临时关键信息。
交互过程中，系统会主动捕捉并记录关键决策、专属称呼、重要事项等内容，通过后台周期性维护整理，让记忆随使用持续沉淀、动态迭代。

## 5. 浏览器操控优化：本地浏览器打破人机验证困局
在当前的浏览器操控场景中，大多数解决方案（如Playwright）采用启动空白浏览器实例的方式进行操作。这种方式虽然标准化，但在实际应用中却面临着一系列天然缺陷：**无历史记录、无Cookie、缺乏用户身份信息**，导致每一次访问都如同一个“陌生人”首次登录。其直接后果是——频繁遭遇人机验证拦截、账号状态丢失、以及重复登录的繁琐流程，严重影响了自动化任务的流畅度与成功率。
针对这一痛点，JiuwenClaw采用**操控本地浏览器**的思路。用户只需在设置中完成浏览器路径的配置，并启动浏览器服务，JiuwenClaw即可接管用户原有的浏览器环境——包括已登录的账号、缓存的Cookie、浏览历史乃至用户偏好。这意味着，自动化操作不再是“裸奔”式的未知访问，而是带着完整用户身份的“无缝延续”。
启动JiuwenClaw服务后，需要在前端配置好浏览器的执行路径，并拉起浏览器服务。
