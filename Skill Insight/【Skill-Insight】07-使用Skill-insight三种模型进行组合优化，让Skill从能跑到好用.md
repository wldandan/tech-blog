# 使用Skill-insight三种模型进行组合优化，让Skill从能跑到好用


> 写完一个Skill，很少改一次就到位。前几篇拆解了Feedback、Static和Dynamic各自怎么用，但在真实项目里，三种模式不是"三选一"的单选 —— **它们是三层互补的过滤器**：Static过滤规范硬伤，Dynamic过滤实战偏差，Feedback过滤方法论盲区。这篇用一个完整的组合优化案例，把"什么时候用哪种、怎么搭着用"讲清楚。

# 一、从单模式到组合：优化能力的完整版图

本篇要回答的是：**一个Skill既有规范硬伤、又有实战偏差、还有方法论盲区的时候，三种模式怎么排顺序，怎么组合收益最大。**

![alt text](image.png)

下面先用一张表速览三种模式的定位。

| 模式 | 看什么 | 核心问题 | 适用时机 |
|:----|:------|:---------|:---------|
| **Static**（静态） | Skill 文本本身 | "按规范该有的，你写了吗？" | 刚写完、没跑过、想先扫一遍硬伤 |
| **Dynamic**（动态） | Agent 实际运行轨迹 + 评测扣分 | "按你写的，Agent 做到了吗？" | 有运行数据、效果不稳定、得分卡住 |
| **Feedback**（反馈） | 用户的主观发现 | "你发现它漏了什么，但文本和数据都没体现？" | 你有明确的修改意见，能说清"在哪一步加什么" |

**简单记：Static看规范、Dynamic看运行、Feedback听你的。** 三者管的层互不重叠，一个补文本缺陷、一个补执行偏差、一个补方法论盲区。可以单独用，也可以叠加用。叠加时的顺序和原因见第四节。

# 二、实操：vmcore-analysis三模式组合优化全流程

下面以一个真实案例，演示Static → Dynamic → Feedback三模式叠加优化的完整过程。

**案例背景**：`vmcore-analysis`是一个Linux内核崩溃转储分析Skill，覆盖OOM、Panic、Deadlock等常见故障。它跑过若干次任务，静态文本写得比较完整，有故障模式表、诊断模板、兜底策略，但评测得分一直卡在 60%~70%，而且完全没写安全约束。

这种场景天然适合三模式组合：**文本层有规范缺失，平台上跑出了扣分数据，用着用着还发现了方法论上的改进空间。**

## Step 1：Static体检 — 先补"规范该有但你漏了"的部分

**为什么先跑Static：** 它不需要运行数据，任何时候都能跑。先让它把文本层的硬伤扫出来修掉。这些问题是后面两轮修不了的，而且底子干净了，Dynamic和Feedback定位也更准。

在openCode中触发：

```
使用static（静态优化）模式，优化一下vmcore-analysis Skill。
```

跑完static，它给出1条诊断，落在**运维可靠性**这一维：

> 这个Skill涉及高危操作（用sudo执行脚本、操作vmcore文件），但**没写清楚哪些操作是禁止的、违规会怎样、出错怎么回滚**。脚本失败时也没强制要求向用户报告。

静态优化过后其新增了一段「安全约束」章节：

**Before**：技能内容中完全没有关于安全合规的任何描述。

**After**：

```markdown
## 安全约束 (约束) 🛡️

> 本技能涉及高危操作（sudo、直接分析内核内存），必须严格遵守以下约束。

1. **禁止操作**：
   - 禁止直接修改原始 `vmcore` 文件。
   - 禁止在未获用户明确授权的情况下分析非当前问题的 vmcore。
   - 禁止执行非技能自带的第三方脚本。

2. **违规处理**：
   - 任何试图触发上述禁止操作的行为，AI 必须立即拒绝，并明确报告"安全违规"及原因。

3. **状态回滚**：
   - 执行 `collect.sh` 等脚本后，如需恢复任何临时改动，
     应指导用户从原始备份（如系统快照）恢复。
```

涉及root权限和内核数据这种高危操作，原Skill里完全没写边界 —— 这是规范上的硬伤。静态优化把它补上了。

### static在改什么

它在帮你补"规范上该有，但你没写"的部分 —— 比如安全边界、错误报告、回滚机制这种容易漏的东西。但这些改的都是"规范层面的缺失"，对于`为什么实际跑起来得分只有60%+`这种问题，还得靠Dynamic来回答。

## Step 2：Dynamic复盘 — 再修"跑过才知道不对"的部分

**为什么Static之后跑Dynamic：**Dynamic要运行数据，这些数据是Static修复之前产生的。但Skill结构层面的改动（加安全章节、加预检）不会影响评测模型对"代码级分析深度"和"采集策略选择逻辑"的判分，所以Dynamic分析的扣分项在Static修完之后依然有效，可以直接在这轮处理。

继续用vmcore-analysis。假设你已经跑了几次任务，平台上能看到运行轨迹和扣分记录。跑完dynamic后，它会改一些static永远找不出的问题。

在openCode中触发：

```
对vmcore-analysis这个Skill进行动态优化
```

最典型的一个改动是**故障分类的整体重排**。

原版列了6类故障：OOM/Panic/Deadlock/Hardware/NULL Pointer/Stack Overflow。看着挺全，规范上挑不出毛病。

但运行数据告诉它：

- NULL Pointer和Stack Overflow在真实场景里几乎都从Panic入口进入，单独列出来基本是浪费。
- 反过来，**网络故障、文件系统故障、硬件Bit Flip**才是常见但漏了的场景。

它就把分类直接重排了：

```diff
- 6 类：OOM / Panic / Deadlock / Hardware / NULL Pointer / Stack Overflow
+ 7 类：硬件 Bit Flip / Panic / OOM / Deadlock / 网络 / 文件系统 / 硬件组件
+ 优先级：硬件故障 > Panic > OOM > 死锁 > 网络 > 文件系统
```

### dynamic 在改什么

**它在帮你改"跑过才知道不对"的部分**——分类不实用、命令不顺手、判断逻辑模糊。这些问题文本上看不出来（每条都写得规规矩矩），但实战里就是别扭。

## Step 3：Feedback打磨 — 最后补"机器看不出来、只有你发现了"的事

**为什么放在最后**：前两轮已经把规范和实战层修得差不多了。Feedback要加的东西，通常是一条全新的分析路径、一个机器不可能自己发现的方法论改进。这些如果放在前面加，可能被Dynamic的分析过程干扰，因为运行数据里还没有这条新路径的轨迹。放在最后，前面两轮不会误伤它，加完立刻就能跑一次验证。

继续拿 vmcore-analysis 举例。vmcore-analysis用过几次后，你可能会注意到一件事：

> 它每次都是先猜一个故障类型，再跑对应的`check_*.sh`脚本去验证。可有时候我看 vmcore，panic信息明明已经把崩溃位置告诉我了——直接看那行内核代码就能知道问题。它从来不这么干。

这就是一个典型的Feedback 时刻——**你看到了一种它从没用过的分析方法**，而且能说清楚什么时候该用、怎么用。

### 把它写成反馈

直接把心里这件事告诉Agent，**用大白话**就行：

```
你: vmcore-analysis 这个 skill 我用了几次发现——它每次都直接挨个
    跑 check_*.sh 脚本来猜故障类型。但很多时候 panic 信息已经把崩
    溃位置告诉我了，明明可以直接读源码看出问题。能加一条路径吗：
    先看能不能从 panic 定位到代码行，能就读源码，不能再去跑那些
    脚本。

Agent: 收到，针对反馈"加一条源码分析作为优先尝试"来优化
       vmcore-analysis。开始准备环境……
```

不用想着把每一步都写成命令——你只要说清楚"是什么问题、想怎么改"，剩下的具体怎么落地（用什么命令、加在哪一节、怎么写判定条件）交给优化器去想。

反馈也不一定是"加东西"，**改、删都行**：

- "把诊断结论模板里'修复建议'那段砍掉，太啰嗦"
- "采集脚本的超时改成 600 秒"
- "去掉 NULL Pointer 这个故障类别，没用过"

想让它改什么，就直接说。反馈越具体，改得越准。

### 它改了什么

跑完之后，Step 2之前多了一节「链路选择」：

```diff
  ### Step 1: 信息采集
  ...
+ ### Step 2: 链路选择 (Track A vs Track B)
+
+ 进入故障模式识别前，先做链路判定：
+ - Track A — 能从 panic 定位到函数+行号 → 直接读源码分析
+ - Track B — 不能定位时 → 按故障模式表跑 check_*.sh
+
+ Track A 入口命令：bt -l → dis -r → sym → addr2line
+ Track A 退出条件（任一命中就回退到 Track B）：
+   1. 符号无法解析   2. 调用栈被破坏   3. 没有源码
+   4. 第三方驱动     5. 硬件故障
  
- ### Step 2: 故障模式识别
+ ### Step 3: 故障模式识别 (Track B)
  ...（这一段原内容一字未改，只是顺延了编号）
```

诊断结论模板里加了一行：

```diff
+ 分析链路: Track A (源码分析) | Track B (模式匹配)
```

并且新建了`references/code_analysis.md`，把Track A用到的命令、解析步骤、内核常见缺陷模式都写下来。

注意一点：你只让它"加一条路径"，它就只动加路径需要的部分——原有的故障模式表、check_*.sh 调用、追问原则没被牵连。如果你说的是`删掉X`或`把 X 改成 Y`，它也只会动 X 那一块，**不会自作主张去改你没提的地方**。

### Feedback在改什么

**它在帮你改"你看出来了但机器看不出来"的事**。

机器再聪明，也只能看Skill文本是否合规、运行轨迹是否符合预期。但"应该再加一条全新的分析路径"这种判断——它不知道有这个路径存在，运行数据里也没有"用户预期但Agent没做的事"。这种事只有你能告诉它。

# 三、三种模式怎么选：场景速查指南

三种模式的选择可以归结为两个问题：

**问题一：你的Skill跑过吗？**（平台上有运行记录和评测数据）

**问题二：你心里有具体想改的地方吗？**

按答案直接选：

| 你的情况 | 选哪种 | 说明 |
|:--------|:------|:-----|
| 没跑过、也没具体想法 | **Static** | 先做一次全面的六维体检，补规范硬伤 |
| 跑过几次、用着不太顺、说不清哪里不对 | **Dynamic** | 让评测模型基于运行轨迹帮你诊断 |
| 不管跑没跑过，你有明确修改意见 | **Feedback** | 直接说"在哪一步加/删/改什么" |
| 三个条件都具备 | **Static → Dynamic → Feedback** | 按从粗到细的顺序叠加 |

### 为什么叠加时是这个顺序

```
Static → Dynamic → Feedback
```

**这个顺序不是强制的，但最省事。** 原因是：

1. **前置条件从少到多**：Static任何时候都能跑；Dynamic要先有运行数据；Feedback 要先有想法。这个顺序不会卡前置。
2. **改动从粗到细**：Static改规范层（骨架问题）→ Dynamic改实战层（跑出来的偏差）→ Feedback改方法论层（盲区补齐）。反过来跑会有重复工作。Feedback加的新路径，Dynamic阶段没有轨迹数据分析不了——但也不会误伤它。
3. **每一步都能单独看**：每跑完一步自动开Diff页面，满意再进下一步，不满意直接停。

**大多数情况其实只跑其中一个或两个：**

| 你的情况 | 跑什么 |
|:--------|:------|
| 刚写完，没跑过 | 只跑Static |
| 上线一阵了，得分卡住 | 只跑Dynamic |
| 想加一段东西 | 只跑Feedback |
| 跑过 + 有想加的 + 想顺便体检 | Static → Dynamic → Feedback |

要让Agent一步到位跑完三轮，直接说清楚即可：

```
vmcore-analysis我都要：跑一遍自动诊断、再结合运行日志看看，
最后我有一段想加的内容。
```

Agent会按Static → Dynamic → Feedback顺序依次执行，每步完成后展示Diff供你确认。


# 四、小结

回顾本篇和前两篇优化文章构成的完整路径：

```
第4篇（优化入门）：四种优化模式总览 → Feedback 入门演示
第5篇（Static）  ：六维评估 → 安全约束补全 → 发布前质量门禁
第6篇（Dynamic） ：执行轨迹 → 评测数据 → 归因分析 → 定点改写
第7篇（组合实战）：Static → Dynamic → Feedback 三模式叠加全流程
```

**三个核心要点：**

- **三种模式不是三选一，是三层互补的过滤器。** Static 补规范硬伤（"按规范该有的，你没写"），Dynamic 修实战偏差（"按你写的，Agent 没做到"），Feedback 填方法论盲区（"你发现它该做但没做，文本和数据都没体现"）。三层各管一摊，互不重叠。
- **叠加时按 Static → Dynamic → Feedback 顺序最省事。** 从粗到细、从少前置到多前置，每一步改在干净的底子上，每步可独立验收。
- **每一处改动都有据可依。** 从 62% 到 93% 不是靠盲改。Static 改什么看六维评分，Dynamic 改什么看扣分归因，Feedback 改什么看你的主观发现。改得明白，回滚也有迹可循。

三层都过一遍，Skill 才算真正从"能跑"走到了"好用"。

# 五、常见问题

**Q1：必须按Static → Dynamic → Feedback这个顺序吗？**

不必须。每种都能单独跑。三种都要跑时按这个顺序最省事，反过来跑会有一些重复工作。

**Q2：Feedback会不会把前面Static/Dynamic改的东西覆盖掉？**

不会。Feedback只改你明确提到的部分，你说"加 X"它就加 X，"删 X"就删 X，没说的就不动。本文例子里Feedback加了"源码分析路径"，Dynamic阶段补的故障分类、采集策略、代码分析流程全都没受影响。如果你的Feedback本身和前面的改动冲突，比如让它删掉前一轮加的东西，那以你说的为准。

**Q3：三轮跑下来Skill会不会变得很冗长？**

有这个可能，尤其是Feedback反复加新东西时。两个应对办法：

- **当场删**：每跑完一轮Diff页面会自动打开，看到某段太啰嗦，直接Revert掉。
- **再发反馈让它瘦身**：比如"把安全约束章节砍到5行以内"、"删掉冗余的命令注释"。Feedback不光能加东西，也能删。

**Q4：Dynamic优化对Skill的"运行数据量"有最低要求吗？**

理论上一次完整执行就能触发优化。但执行次数越多，扣分点的归因越准确——建议至少3-5次执行后再启动Dynamic优化，避免单次偶发因素导致优化方向偏移。

**Q5：扣分项归因为"Agent"的问题，动态优化能解决吗？**

解决不了。归因为Agent说明是执行模型能力不足，比如模型本身就不擅长某类反汇编分析。这类问题得换更强的模型，光改Skill没用。Skill-insight的归因机制就是帮你把这两种情况分开：哪些"改Skill有用"，哪些"换模型才有用"。
