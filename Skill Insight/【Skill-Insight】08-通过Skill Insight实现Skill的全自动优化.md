# 通过Skill Insight实现Skill的全自动优化

> 反馈优化、静态优化、动态优化——你已经掌握了三种打磨 Skill 的"招式"。但每一种都还需要你亲自下场。今天，让机器把"反复迭代"这件事也接管过去。

***

读到这一篇的同学，相信对Skill-insight的优化能力已经不陌生了。

**反馈优化**让你基于执行后的评测反馈针对性修改Skill；**静态优化**不用跑任务，单看Skill文本就能给出改进建议；**动态优化**则能根据Agent的实际运行轨迹精准定位瓶颈。三种手段各有侧重，组合起来已经能覆盖绝大多数场景。

但它们有一个**共同的局限**：

> 每一次优化都是"单次动作"——人手动触发，人查看结果，人决定要不要再跳一轮。

如果你的Skill一次就能改到位，这没问题。但现实是，把一个Skill从"能跑"打磨到"好用"，往往需要3~5轮甚至更多次的迭代。每一轮都要切工具、看报告、判断方向、再触发 —— 一套流程下来半天就过去了。改到第4轮还想回滚到第2轮？早就找不到了。

**Skill-insight的「自动多轮优化」能力，就是把这个迭代过程本身也自动化。** 你只需要设定一个目标——比如"准确率达到90%"，剩下的执行、评估、修改、再执行，系统全包了。喝杯咖啡的功夫，达标的版本就摆在你面前。

![alt text](Skill自动多轮优化手绘图.png)

简单类比一下：如果说前三种优化是"招式"，那么多轮自优化就是把招式连成"组合拳"，并且自己出拳、自己复盘、自己调整——你只需要在场边告诉它"打到对手投降为止"。


# 一、为什么需要"自动迭代"？

在讲怎么用之前，先把问题讲透。

手动迭代Skill的痛苦，凡是认真打磨过Skill的同学都深有体会：

- **痛点一：循环操作繁琐。** 一轮"执行 → 评估 → 修改"至少要切换 3 个工具、执行 5~6 个命令。迭代 5 轮就是几十次操作，注意力很容易在切换中耗散。
- **痛点二：版本混乱难追溯。** 改到第 4 轮发现还不如第 2 轮，但第 2 轮的版本早就被覆盖了——这种事几乎每个调过 Skill 的人都干过。
- **痛点三：终止判断靠感觉。** 跑到第几轮就该停？再跑一轮还会不会更好？没有自动化机制，全凭经验拍脑袋，要么提前收手错过更优解，要么过度优化浪费时间。

自动多轮优化把这三个痛点一次性解决：**用户只设定目标和上限，系统自动执行「执行任务 → 收集结果 → 优化Skill → 再执行」的完整循环，每一轮自动备份版本、自动评估、自动判定是否达标，直到达成目标或触达最大轮数。**

整个过程，你只需要做两件事：**配置目标**，和**等待结果**。

# 二、它是怎么工作的？

自动优化的核心机制可以用一张图说清楚：

```
   ┌─────────────┐
   │ 设定优化目标 │
   └──────┬──────┘
          ↓
   ┌─────────────┐
   │ 备份当前版本 │ ←──┐
   └──────┬──────┘    │
          ↓           │
   ┌─────────────┐    │
   │ 执行测试任务 │    │
   └──────┬──────┘    │
          ↓           │
   ┌─────────────┐    │
   │ 评测得分判定 │    │
   └──────┬──────┘    │
          ↓           │
       达标？─否─→ 优化 Skill
          │           │
         是           │
          ↓           │
   ┌─────────────┐    │
   │ 输出优化报告 │    │
   └─────────────┘    │
                      │
   达到最大轮数────────┘
```

每一轮循环包含4个关键动作：

1. **Step 1：版本快照。** 每轮开始前，系统会把当前Skill完整备份到`iteration-logs/round-N/skill-snapshot/`目录。这意味着你随时可以回滚到任意一轮的历史版本，再也不用担心"改回不去"。
2. **Step 2：执行任务。** 系统使用配置的模型（如deepseek/deepseek-chat）按照测试任务描述完整执行一遍Skill，记录全过程数据。
3. **Step 3：自动评估。** 执行完成后，系统轮询Skill-insight平台的评测API（每30秒一次，最多10分钟），等待评测模型给出得分。得分会和你设定的阈值（如0.9）做对比。
4. **Step 4：智能优化或终止。** 达标则结束，输出报告；未达标则调用`skill-optimizer`基于本轮的执行结果生成优化建议，自动应用到Skill文件，然后进入下一轮。

整个循环由`iterative-optimizer`这个Skill编排，全程无需人工介入。


# 三、实操：让一个Docker故障排查Skill自动优化到达标

下面用一个真实场景演示完整流程：你已经生成了一个Docker故障排查Skill（假设叫`openeuler-docker-fault`），但实际跑下来效果不太理想——经常漏掉cgroup限制检查、网络namespace排查等关键步骤。我们用自动多轮优化把它的准确率提升到0.9以上。

## 前置条件确认

开始之前，请确认以下几点都已就绪：

| 检查项 | 说明 |
|:------|:-----|
| Skill-insight平台 | 已部署并正常运行 |
| 待优化的Skill | 已存在于`.opencode/skills/`目录下 |
| 评测模型 | 平台已配置好（如DeepSeek、OpenAI等） |
| opencode工具 | 已安装并可正常调用 |
| 执行模型 | 已配置可用的模型（如deepseek/deepseek-chat） |

## 方式 A：自然语言一句话触发（推荐）

最轻量的方式——直接告诉opencode你的优化需求：

```
帮我迭代优化/user/work/.opencode/skills/openeuler-docker-fault 这个skill，
目标是准确率达到0.9以上，最多跑5轮
```

`iterative-optimizer`会自动识别"迭代优化"、"自动优化skill"、"循环优化"、"多轮优化"等关键词，然后引导你补充缺失的配置（比如测试任务描述）。整个交互过程像聊天一样自然 —— 你回答完最后一个问题，它就开始跑了。

## 方式 B：配置文件精细控制

如果你想对优化过程做更精细的控制（比如配置故障注入、交互预设等），可以先创建一个`iter-config.yaml`配置文件：

```yaml
framework: opencode

skill:
  name: openeuler-docker-fault
  path: /user/work/.opencode/skills/openeuler-docker-fault

optimization:
  score_threshold: 0.9          # 达标分数：0.9
  goal: 提升 Docker 故障排查准确率
  max_rounds: 5                 # 最多跑 5 轮

# 可选：每轮测试前注入故障
fault_injection:
  inject: docker stop test-container
  cleanup: docker start test-container

tasks:
  query: |
    排查并解决容器无法启动的问题，容器名称为 test-container

# 可选：预设交互场景，避免人工干预
interactions:
  - scenario: 用户确认继续
    trigger: 是否继续|是否确认
    response: 确认继续
```

配置文件的几个关键字段说明：

- **`score_threshold`**：达标阈值，0~1之间。设 0.9意味着只有得分 ≥ 0.9才算达标。
- **`max_rounds`**：最大轮数兜底。哪怕一直不达标，也不会无限跑下去。
- **`fault_injection`**：故障注入（可选）。每轮测试前注入故障、测试后自动清理，特别适合验证 Skill 在真实故障场景下的表现。
- **`interactions`**：预设交互（可选）。如果测试任务涉及多轮对话（比如需要确认操作），预设关键词和回复，让流程全自动跑完不用人盯。

配置完成后，在opencode终端中触发：

```bash
/si-optimizer
```

系统会自动读取`iter-config.yaml`并启动迭代循环。


# 四、看懂优化报告：每一轮发生了什么

整个迭代结束后，你会拿到一份结构化的优化报告。以Docker故障排查Skill为例，典型的报告长这样：

```
========================================
迭代优化报告
========================================
Skill 名称:     openeuler-docker-fault
优化目标:       准确率达到 0.9 以上
使用模型:       deepseek/deepseek-chat
终止原因:       达成优化目标

----------------------------------------
各轮次详情:
----------------------------------------
第 1 轮:
  执行得分:        0.62
  达标:            否
  评判摘要:        排查步骤缺少 cgroup 限制检查

第 2 轮:
  执行得分:        0.85
  达标:            否
  评判摘要:        缺少网络 namespace 排查

第 3 轮:
  执行得分:        0.93
  达标:            是
  评判摘要:        覆盖率达标，报告结构清晰

----------------------------------------
得分趋势:  0.62 → 0.85 → 0.93
----------------------------------------
最终生效 Skill:  /user/work/.opencode/skills/openeuler-docker-fault
历史版本保留在:  ./iteration-logs/round-N/skill-snapshot/ 目录下
========================================
```

这份报告的价值在于——它不仅告诉你"成功了"，更告诉你"是怎么成功的"。

**得分趋势`0.62 → 0.85 → 0.93`**让你看到优化的曲线：第一轮得分仅0.62，评判摘要明确指出"缺少cgroup限制检查"；第二轮系统针对性补充了这一项，得分提升到0.85，但又暴露出"缺少网络namespace排查"；第三轮再次定向修补，最终突破0.9阈值达标。

这种"评测 → 归因 → 修补"的螺旋上升过程，正是手动优化最难复现的部分——人很难每一轮都精准定位到瓶颈，但机器可以。


# 五、版本管理与回滚：你永远有"后悔药"

自动优化最让人安心的一点是：**任何一轮的版本都不会丢。**

每一轮优化前，系统会把当前Skill的完整目录（包括`SKILL.md`、`scripts/`、`references/`等所有文件）备份到`iteration-logs/round-N/skill-snapshot/`。

举个例子：假设跑了5轮之后你发现，最终生效的第5版虽然得分最高，但生成的内容过于冗长不符合团队规范，反而第3版更简洁实用。这时只需要一行命令就能回滚：

```bash
cp -r ./iteration-logs/round-3/skill-snapshot/* /user/work/.opencode/skills/openeuler-docker-fault/
```

回滚后，第3版会成为新的基准版本，后续任何优化都将在此基础上继续。这套机制让"敢于迭代"成为可能——反正改坏了随时能撤回。


# 六、什么场景最适合用自动优化？

不是所有Skill都需要走自动优化流程。根据实践经验，以下三类场景收益最大：

**场景一：故障排查类Skill。** 这类Skill的特点是步骤多、覆盖维度广，第一版生成时很容易遗漏某些排查项。配合`fault_injection`，可以在受控环境下反复验证Skill在真实故障下的表现，把准确率打磨到生产可用水平。

**场景二：从大量历史案例提炼出的归纳型Skill。** 这类Skill在初期往往泛化能力不足 —— 某些边缘案例没覆盖。通过自动优化迭代，可以让Skill逐步吸收更多模式。

**场景三：批量提升存量Skill质量。** 如果你的团队已经积累了几十个Skill，可以用配置文件批量启动优化任务，统一拉齐质量基线，比逐个手动优化高效得多。

反之，对于单一职责、流程极其简单的Skill（比如只做一次API调用的工具型Skill），收益就比较有限——因为本身没有太多优化空间。


# 七、小结

从反馈优化、静态优化、动态优化，再到今天的自动多轮优化，Skill-insight的优化能力完成了从"招式"到"组合拳"的升级：

```
反馈优化 ┐
静态优化 ├─→ 单次动作（人工触发 + 人工迭代）
动态优化 ┘
                ↓
          自动多轮优化（人工设定目标 + 自动迭代到达标）
```

**前三种优化解决的是"每一轮怎么改得更准"，自动多轮优化解决的是"多轮迭代怎么不再耗人"。** 它们不是替代关系，而是叠加关系——多轮自优化在底层依然会调用反馈、静态、动态优化的能力，只是把"人决定何时停止"这一环也交给了机器。

回到开头的那个比喻：你不需要再亲自下场打每一拳，**只需要在场边告诉它"打到对手投降为止"。**

# 八、常见问题

**Q：优化过程中可以中断吗？**

A：可以。随时手动终止即可，已完成的轮次结果会被完整保留——你可以在中断点继续，也可以回滚到任意一轮的版本。

**Q：跑了5轮还没达标怎么办？**

A：几种思路：① 检查测试任务描述是否准确反映了实际场景；② 适当降低达标阈值（比如从 0.9 调到 0.85）；③ 增加 `max_rounds`；④ 手动介入查看每轮的评判摘要，判断是 Skill 设计问题还是模型能力问题——如果是后者，光靠优化 Skill 解决不了，要换更强的执行模型。

**Q：评估结果显示超时怎么办？**

A：评估默认轮询 10 分钟（30 秒一次，最多 20 次）。如果超时，先检查 Skill-insight 平台是否正常运行、网络是否畅通。也可以登录平台手动查看评测结果。

**Q：详细的执行日志保存在哪里？**

A：每一轮的所有日志都在 `iteration-logs/round-N/` 目录下，包括：执行日志、优化日志、上传日志、故障注入日志（如果配置了的话）。需要复盘时随时可查。

**Q：当前支持哪些Agent框架？**

A：当前自动多轮优化只支持opencode框架。Claude Code、OpenClaw等其他框架的支持正在规划中。

**Q：每一轮都能看到具体改了什么吗？**

A：可以。对比相邻两轮的`skill-snapshot/`目录就能看到差异，配合评判摘要，能清楚理解每一轮优化的方向和效果。
