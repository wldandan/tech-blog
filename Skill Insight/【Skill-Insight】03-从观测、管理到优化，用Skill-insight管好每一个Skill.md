# 管理篇 | 观测、管理、优化——Skill评测与全生命周期管理

# 从生成到管理：Skill上线之后怎么办

第2篇里，我们用扩展和归纳能力从各种形态的输入生成了高质量的Skill。不管是"一句话"还是"几十份问题单"，都能稳定产出可用的Skill制品。

但Skill生成出来只是起点。真正的问题是：**Skill上线之后，怎么知道它执行得好不好？发现问题了怎么改？改完之后怎么验证？多个Skill之间怎么管理版本和依赖？**

第1篇解决"怎么生成Skill"，第2篇解决"输入形态各异怎么办"。第3篇要回答的是——**Skill进了生产环境之后，怎么让它持续可靠地运行**。

Skill-insight在"生成→执行→评测→归因→优化"这个闭环上，提供了三块功能：

- **观测**——采集执行数据，量化评估Skill的质量和效率
- **管理**——控制Skill的版本、流程和生命周期
- **数据集**——用标准答案驱动自动化评测，让优化有据可依

三块功能递进而非独立：先通过观测发现偏差，再用管理定位和修复问题，最后用数据集验证改进效果，形成持续迭代的循环。

# 一、为什么需要管理

## 1.1 规模化之后的三个痛点

当Skill数量从十几个增长到上百个，三个问题会逐渐浮出来：

- **规模失控**。大量相似的Skill挤在库里，Agent不知道该调哪个。实测数据显示，Skill数量超过40-50个后，召回率从95% 骤降到30% 以下，Token消耗反而涨上去了。冗余的Skill不仅没帮助，净添乱。
- **评测失真**。传统评测只看"任务完没完成"，但很多时候结果对了，中间过程却走了弯路——跳过了安全检查、遗漏了关键步骤。没有过程级的数据，这些隐患永远发现不了。
- **优化失效**。没有执行数据，优化只能停留在"改改文字描述"的层面。到底是Skill写得不好，还是模型能力不够？靠直觉猜，大概率改错方向。

## 1.2 Skill-insight的应对思路

Skill-insight针对这三个痛点，对应三条能力线：

| 痛点 | 解决方案 | 实测效果 |
|---|---|---|
| 规模失控 | 语义聚合去冗余、合相似 | 磁盘故障场景：召回率20% → 100%，Token 460k → 355k |
| 评测失真 | 过程级可追溯，执行流程图逐项比对 | 偏差节点标得清清楚楚 |
| 优化失效 | 数据驱动归因，区分Skill缺陷vs模型问题 | Docker卡顿场景：准确率0.7 → 0.8 |

后面三节对应这三条能力线的具体操作。

# 二、安装和配置Skill-insight

## 2.1 安装Skill-insight

打开终端，一行命令搞定：

```bash
npx @witty-ai/skill-insight install
```

安装完成后，打开浏览器访问`http://localhost:3000`，你会看到Skill-insight的可视化看板。默认账号是`admin`，直接登录即可。

接下来，在OpenCode中注册Skill工具包：

```bash
npx skills add https://gitcode.com/openeuler/witty-skill-insight.git
```

至此，环境准备完毕。

## 2.2 配置模型服务

评估模型用于对Agent执行结果进行自动评分。需要配置一个LLM作为评估模型。Skill-insight支持DeepSeek、OpenAI、Anthropic等多个模型提供商。

1. 登录看板，点击左上角**⚙️Eval Config**按钮。
2. 在弹出的对话框中，单击**+ Add New Configuration**。
3. 填写以下信息：

   | 字段 | 说明 | 示例 |
   |:---|:---|:---|
   | Config Name | 配置名称 | 我的 DeepSeek |
   | Provider | 模型提供商 | DeepSeek / OpenAI / Anthropic / Custom |
   | Base URL | API 基础地址（可选） | `https://api.deepseek.com` |
   | API Key | API 密钥 | sk-xxxx-xxxx-xxxx |
   | Model Name | 模型名称 | deepseek-chat / gpt-4o |

4. 点击**Test Connection & Save**，系统会自动测试连接并保存配置。
5. 回到配置列表，点击**Activate**激活该配置。

# 三、观测：看得见才能管得住

观测是管理的基础。这一节拿"Docker应用卡顿排查"来演示：数据怎么来、指标怎么看、观测结果能用来做什么。

## 2.1 数据怎么来

Skill-insight与Agent框架的集成有两种方式：

- **原生集成**：OpenCode等支持插件的框架，安装时自动配好数据上报通道。Agent执行任务，数据自动上报，不用额外配置。
- **日志旁路**：Claude Code、OpenClaw等框架通过日志文件旁路监听采集数据，Agent侧不用做任何改动。

### 为Claude Code、OpenClaw配置数据采集通道（可选）

如果采用Claude Code、OpenClaw时，数据采集配置方式如下：

**第一步：获取API Key**。

1. 打开看板页面（如http://192.168.1.100:3000）。
2. 点击右上角用户图标。
3. 在弹出的对话框中获取或生成API Key。

**第二步：执行安装命令**.

```Bash
curl -sSf http://192.168.1.100:3000/api/setup | bash
```

> 192.168.1.100:3000为示例IP地址和端口，实际请替换为平台部署服务的实际地址和端口。

安装过程会提示输入 API Key：

```
🔑 WITTY_INSIGHT_API_KEY is not set.
👉 Please enter your API Key: sk-xxxx-xxxx-xxxx
```

完成后，系统会自动写入配置文件 ~/.skill-insight/.env，并自动安装相关数据上报组件。

### 执行任务，获取数据

安装Skill-insight完成后，执行一个任务试试：

```bash
opencode run "我的docker应用有时会卡住，帮我分析下原因"
```

任务跑完之后，登录Skill-insight平台就能看到完整的执行记录。

## 2.3 执行详情：不止看结果，还能追溯过程

点击某条记录进入详情页，可以看到三个层次的信息：

**评测结果摘要**：状态判定（Skill调用是否符合预期、执行结果是否正确）、判题理由（每个评分项的匹配程度和扣分原因）。

**执行流程分析**：点击"流程分析"生成Mermaid执行流程图，再点"Skill对比"把实际执行轨迹与预期流程做逐步骤比对。四种状态用不同图标标识：
- ✅ 符合预期：执行与Skill规定的步骤一致
- ⚠️ 部分偏离：操作与规定存在偏差
- ❌ 非预期调用：执行了Skill未规定的操作
- ⭕ 跳过：Skill规定的步骤未被执行

**Skill有效性分析**：系统自动判断扣分项能不能通过优化Skill来改善。判定为"Skill缺陷"就给出具体修订建议；判定为"模型推理缺陷"则为模型选型提供数据支撑。

### 2.4 核心指标速查

Skill-insight提供的评测指标覆盖效果、效率、成本和稳定性四个维度：

| 类别 | 指标 | 说明 |
|---|---|---|
| **效果** | 准确率 | ∑(评分项得分×权重) / ∑权重 |
| **效果** | Skill召回率 | 实际调用Skill中属于期望Skill的数量 / 期望Skill总数 |
| **效果** | Skill提升率 | (使用Skill准确率 - 未使用准确率) / (1 - 未使用准确率) |
| **效率** | 时延 | 任务从开始到完成的完整耗时 |
| **效率** | 模型调用次数 | 完成任务所需的推理次数 |
| **效率** | 工具调用次数 | 完成任务所需的工具调用次数 |
| **成本** | Token总量 | 输入Token + 输出Token |
| **成本** | 模型费用 | Token × 单价估算 |
| **成本** | CPSR | 获得一个正确答案的平均成本 |
| **稳定性** | 工具调用错误率 | 调用错误次数 / 调用总次数 |

## 2.5 观测数据能做什么

三个层次的应用：

- **诊断执行问题**：通过流程对比发现偏差，定位是Skill设计缺陷还是模型推理问题。还能识别冗余步骤和安全风险操作。
- **评估Skill效果**：准确率和召回率量化Skill的实际价值，CPSR指标衡量成本效益。
- **驱动持续优化**：趋势分析追踪优化效果，对比优化前后的指标变化，形成数据驱动的迭代闭环。

如何基于数据实现Skill的持续迭代优化，我们将在后续的文章中进行介绍。

# 三、管理：Skill的版本与生命周期

观测发现问题之后，就该管理Skill本身了 —— 注册、解析、版本控制、验证、迭代。

## 3.1 Skill注册

进入"技能管理"页面，上传包含 `SKILL.md` 的Skill文件夹。Skill文件中的YAML Frontmatter包含名称和描述等元信息：

```markdown
---
name: server-troubleshooter
description: 服务器故障诊断技能，支持CPU/磁盘/内存故障诊断
---
```

![alt text](image-3.png)

上传后，系统自动解析Skill内容。

## 3.2 流程解析与可视化

点击"解析流程"，系统识别Skill中的步骤结构：

- **必选步骤**（required）：必须执行的核心步骤
- **条件分支**（conditional）：按条件选择执行路径，及各路径的分支动作
- **循环动作**（loop）：预期多次执行的重复操作
- **可选步骤**（optional）：按场景决定是否执行

解析结果用Mermaid流程图展示，不同控制流类型用不同颜色标注。Skill的执行逻辑一目了然，方便和实际执行轨迹做对比。

## 3.3 版本管理

Skill修改后，可以创建新版本保留修改历史。支持：

- **版本对比**：查看不同版本之间的差异
- **版本切换**：回退到历史版本
- **版本删除**：清理废弃版本

## 3.4 验证与迭代

Skill注册或更新后，用Agent执行真实任务来验证效果。在详情页中对比实际轨迹与预期流程，分析偏差和归因。基于归因结果修改Skill内容，创建新版本，再次执行验证——这就是一个完整的优化循环。

# 四、数据集：定义什么是"正确"

观测和管理回答的是"Skill跑得怎么样"，数据集回答的是**"怎样才算跑对了"**。数据集定义了标准答案，自动化评分才有参照系。

## 4.1 核心能力

- **低门槛配置**——提供问题（Query）和标准答案，系统自动提取评分点（Root Causes、Key Actions）。
- **多模式输入**——支持手动填写和文档上传（`.md`/`.txt`/`.pdf`）。上传案例文档后，系统自动解析提取评分点。
- **自动化评分**——基于LLM的评测引擎，替代人工逐条排查。每个扣分点都可回溯到具体操作指令，评分客观可重复。

## 4.2 操作步骤

**Step 1**：进入"数据集管理"页面，点击"+ 新增配置"。

**Step 2**：填写评测内容。两种方式任选：
- **手动输入**：填写问题（如"排查Docker应用卡顿原因"）和标准答案（预期诊断结果、操作步骤等）。
- **上传文档**：填写问题，上传案例文档，系统自动解析并提取评分点。

**Step 3**（可选）：关联Skill。如果已有对应Skill，系统会自动从中抽取关键动作填充评测基准，省去手动填写Key Actions的步骤。

**Step 4**：保存后查看解析状态。状态包括：
- 🔄 parsing：正在解析
- ✅ completed：解析完成
- ❌ failed：解析失败

解析完成后，可以查看自动提取的评分项——Root Causes（关键结论、诊断结果）和Key Actions（关键操作步骤及权重）。

## 4.3 多Skill场景

如果任务涉及多个Skill，在数据集配置时可以选择关联多个Skill。系统自动合并关键动作，按调用顺序拼接，Skill之间自动插入handoff衔接动作。评测时按完整动作链评分，覆盖多Skill协作的复杂场景。

## 五、小结

回顾整个系列三篇文章的路径：

```
第1篇：一份文档 → 一个Skill（入门）
第2篇：一句话 → 扩展 → 完整Skill（进阶）
       多份问题单 → 归纳 → 通用Skill
第3篇：观测 → 发现问题 → 管理迭代 → 数据集验证（管理）
```

三篇放在一起，构成了一个完整的Skill生命周期视角：生成只是开始，持续观测和管理才能让Skill真正发挥作用。

要点回顾：

- **先观测，再优化**。没有数据就做优化等于蒙眼走路。观测数据帮你定位瓶颈在Skill还是在模型。
- **用版本管理兜底**。改坏了可以回退。每个版本保留修改历史，对比可见差异。
- **用数据集固化标准**。什么是"正确执行"不应靠拍脑袋，而应由数据集来定义。
- **循环迭代**。观测 → 归因 → 修改 → 验证 → 再观测，每一次循环都让Skill离"好用"更近一步。

如果你是从第1篇一路读到这里的，现在你已经有能力用Skill-insight完成Skill的全生命周期管理了：从生成到上线，从观测到优化，形成闭环。

目前三篇覆盖了"生成 → 进阶 → 管理"的主线。未来可以展开的方向包括：团队协作下的Skill共建共享、自定义评测模型与评分策略调优、大规模Skill库的自动化治理。有感兴趣的方向，欢迎在社区提Issue或PR。

## 常见问题

**Q1：准确率显示"--"是什么原因？**

三种可能：未配置评估模型、该问题未在数据集管理中配置标准答案、评估模型调用失败。

解决：在Settings页面配置评估模型；在数据集管理中为该问题添加配置；点击"重评"重新触发评分。

**Q2：如何提高评测准确性？**

使用能力较强的评测模型（如DeepSeek-V3），设计清晰明确的标准答案，合理设置评分项权重，定期校验评测结果。评测模型的选择直接影响评分的质量——如果判题理由经常和你的判断不一致，换一个更强的模型通常是最直接的改善方式。

**Q3：Skill对比图中"不计入总分"是什么意思？**

对于条件分支类型的动作，如果实际执行未匹配到该分支，则该分支的动作不计入评分计算。例如：Skill定义了网络故障、磁盘故障、内存故障三条分支，实际执行走了磁盘故障分支，则另外两条分支的动作不会影响评分。

**Q4：观测数据如何导出？**

进入执行记录详情页，点击右上角"导出"按钮，导出的HTML文件可离线查看。

**Q5：执行中断后平台观测不到数据怎么办？**

直接Ctrl+C关闭进程会导致数据上报中断。建议先连按两次ESC键，等待数据上报完成后再Ctrl+C退出。

**Q6：如何判断评测模型是否合适？**

看四点：评分结果是否符合预期、判题理由是否合理可解释、多次评测结果是否一致（稳定性）、成本是否可接受。

**Q7：多Skill场景如何管理？**

在数据集配置时选择关联多个Skill，系统自动合并关键动作并按调用顺序拼接，Skill之间自动插入handoff衔接动作。评测时按完整动作链评分。

**Q8：Skill优化后如何验证效果？**

保存优化后的Skill新版本，用Agent执行相同任务，在执行详情页选择优化前后的记录进行对比。查看准确率、时延、Token等指标的变化，通过流程对比验证执行路径的改进。
