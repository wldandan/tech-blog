# 使用Skill-insight动态优化模式，把每次“轨迹”都变成精准优化信号

> 静态优化是"对着Skill文本找毛病"，动态优化是"看Skill实战表现找毛病"。前者像审稿，后者像复盘——两者都重要，但后者往往能挖出文本审查永远发现不了的问题。

如果你已经看过Skill-insight的静态优化篇，应该理解了"基于Skill文本本身做优化"的玩法——通过分析Skill的描述、结构、关键词等元素，找出可以改进的地方。这种方式快、轻、见效直接，但有一个本质局限：

**静态优化看不到Skill真正"跑起来"是什么样。**

举个例子：你的Skill文档写了"运行采集脚本"，从文本上看完全没问题。但实际执行时，Agent因为某些场景下脚本会超时，索性绕过脚本直接调命令——这种执行偏差在静态文本里是看不出来的。

**Dynamic动态优化** 就是为了解决这类问题而生的。它不看Skill写了什么，**它看 Agent实际做了什么**——把执行轨迹、评测结果、扣分原因连起来分析，定位到具体哪一段指令出了问题、为什么Agent没按预期执行，再针对性地改写Skill。


# 一、什么是Dynamic动态优化？

一句话定义：**Dynamic动态优化是基于Agent实际运行轨迹和评测结果，反向定位Skill设计缺陷并进行针对性改写的优化模式。**

![alt text](image.png)

它的工作流程可以简单概括为：

```
执行任务 → 评测打分 → 归因分析 → 定位Skill缺陷 → 针对性改写
```

与静态优化的核心差异：

| 维度 | Static静态优化 | Dynamic动态优化 |
|:-----|:--------------|:--------------|
| **依据** | Skill文本本身 | Agent实际执行轨迹 + 评测结果 |
| **能发现的问题** | 描述模糊、结构缺失、关键词不全 | 执行偏差、指导深度不足、判断逻辑模糊 |
| **优化方向** | 让Skill更"规范" | 让Skill更"实战可用" |
| **适用阶段** | Skill初始打磨 | Skill已上线、积累了运行数据 |

简单说：**静态优化让Skill看起来更专业，动态优化让Skill用起来更靠谱。** 在实际项目中，两者通常配合使用——先静态打底，再动态调优。

### 适用场景

动态优化最适合以下三类情况：

- **Skill已经在用，但效果不稳定**：有时表现好、有时表现差，需要找到不稳定的根源。
- **评测得分卡在某个区间上不去**：明明Skill写得很全，但准确率就是过不了某个阈值。
- **执行行为不符合预期**：Agent总是绕过某些步骤，或者输出格式经常不到位。

# 三、实操步骤

下面以一个真实的Skill优化案例演示完整流程。

**案例背景：** 我们有一个`vmcore-analysis` Skill，用于Linux内核崩溃转储分析。它跑过几次任务后，评测得分一直卡在60%~70%之间。看Skill文本本身写得相当完整——有故障模式表、有诊断模板、有兜底策略，但实际效果就是不达标。这正是Dynamic动态优化最擅长的场景。

## Step 1：环境依赖安装

确保Skill-insight平台和opencode都已就绪（参考入门篇的安装步骤）。如果还未安装动态优化所需的skill包，执行：

```bash
npx skills add https://gitcode.com/openeuler/witty-skill-insight.git
```

## Step 2：数据准备

动态优化的"原材料"是 **Agent的执行轨迹** 和 **评测扣分记录**。这两份数据由Skill-insight平台自动收集——你只需要确认平台上能查到目标Skill的至少一次完整执行记录即可。

打开Skill-insight看板，找到`vmcore-analysis` Skill 的最近一次执行记录，可以看到类似这样的扣分明细：

```
评测得分: 62%
扣分项明细:
- RC-0  代码级根因分析未到位     扣 1.0 分  归因: Skill
- RC-2  逻辑缺陷识别不深入       扣 2.4 分  归因: Agent
- RC-3  修复建议层次不清         扣 0.6 分  归因: Skill  
- KA-0  采集脚本选择逻辑模糊     扣 0.5 分  归因: Skill
- KA-3  排查脚本未明确执行       扣 0.5 分  归因: 未明
```

注意右侧的 **「归因」** 列——这是Skill-insight评测模型自动判定的扣分责任方：

- **归因为Skill** → Skill设计缺陷，可以通过动态优化改进。
- **归因为Agent** → 执行模型能力不足，光改Skill解决不了，需要更换更强的模型。

我们重点关注归因为Skill的那些扣分项——这正是动态优化的发力点。

## Step 3：执行优化

在opencode终端中触发动态优化：

```
对vmcore-analysis这个skill进行动态优化
```

Skill-insight会自动拉取该Skill最近的执行轨迹和扣分记录，对每一个**归因为Skill**的扣分点，分析具体是Skill文本的哪一部分指导不到位，然后生成针对性的修改建议并自动应用。

下面以两个最具代表性的扣分点为例，看看动态优化具体改了什么。

### 优化示例一：RC-0 — 代码级根因分析的"指导深度"问题

**扣分原因（来自动态分析）：**

> Skill在Step 3.4中确实提到了"代码级根因分析"，列出了"BUG_ON误用、空指针解引用"等缺陷类型，**但只是笼统列举，没有提供详细的判断标准和分析方法**。Agent因此只能给出"检测到非法状态"等表面推测，无法精确定位到"`set_pid_unused()` 错误触发BUG_ON 而非直接返回"这种具体的逻辑缺陷。

**优化前的Skill片段：**

```markdown

3. **执行验证**：运行对应的排查脚本：
   
   bash scripts/check_{selected_mode}.sh ...
   
4. **结果解读**：
   - **命中 (HIT)** → 结合reference中的排查建议分析根因
   - **未命中 (MISS)** → 返回Step 2，选择下一个候选模式

```

可以看到，原Skill对"代码级分析"几乎没有具体指导——只在Step 4提了一句"上报至内核/硬件支持团队"。

**优化后的Skill片段：**

```markdown
#### 3.4 代码级根因分析（内核BUG场景）

当定位到具体崩溃位置时，需进行源码级分析：

1. **阅读代码分析方法论**: 参考 `references/code_analysis.md`
2. **提取崩溃位置**: 从bt输出获取RIP地址和函数偏移量
3. **反汇编分析**: 使用 `dis -r <函数名>` 分析崩溃指令
4. **检查关键变量**: 使用 `struct`、`p`、`rd` 等命令检查指针和变量状态
5. **定位逻辑缺陷**: 对照常见内核代码缺陷模式
   （空指针解引用、越界访问、BUG_ON 误用、资源清理顺序异常等）

**代码级分析输出要求**：
当定位到内核代码缺陷时，必须在诊断结论中包含：

代码级根因:
  崩溃函数: {函数名}()
  崩溃位置: {文件}:{行号}
  崩溃指令: {ud2/mov/cmp等}
  条件分析: {条件表达式}={实际值}，预期={预期值}
  逻辑缺陷: {具体缺陷描述}
```

**改动的关键点：**

1. **从"提一句"变成"五步流程"**：把模糊的"代码级分析"拆解为可执行的5个具体步骤。
2. **指明具体工具命令**：`dis -r`、`struct`、`p`、`rd`这些crash工具命令直接给出，Agent不用再猜。
3. **强制输出结构化结论**：通过模板字段（崩溃函数、崩溃指令、条件分析、逻辑缺陷）反向倒逼分析深度。

### 优化示例二：KA-0 — 采集脚本的"判断逻辑"问题

**扣分原因（来自动态分析）：**

> Skill在Step 1写的是"标准采集（推荐）"和"大文件降级策略"，但**没有明确要求Agent先检查文件大小再决定执行策略**，也没有强制要求记录判断依据。Agent实际执行时直接用了crash命令，跳过了标准采集脚本，但又没记录"为什么跳过"——评测系统无法判断这是合理选择还是错误偏离。

**优化前的Skill片段：**

```markdown
### Step 1: 信息采集

运行基础信息采集，获取vmcore元数据和系统状态：

bash scripts/collect.sh {{VMCORE_PATH}} {{VMLINUX_PATH}}
```

只有一个标准采集命令，没有任何分支判断、降级策略、判断依据要求。

**优化后的Skill片段：**

```markdown
### Step 1: 信息采集

**大文件预警**: 对于大型vmcore文件（>1GB），脚本执行可能耗时较长。
若脚本执行超时，应立即采用降级策略。

#### 1.1 标准采集（推荐）
bash scripts/collect.sh {{VMCORE_PATH}} {{VMLINUX_PATH}}
- `--timeout SECONDS`: 可选，设置超时时间（默认180秒）

#### 1.2 大文件降级策略（超时时使用）
若脚本执行超时或vmcore文件较大（>1GB），直接使用crash命令增量获取：

# 快速获取核心元数据（每次命令独立执行，避免超时）
crash {{VMLINUX_PATH}} {{VMCORE_PATH}} -e 'sys' > /tmp/vmcore_sys.txt
crash {{VMLINUX_PATH}} {{VMCORE_PATH}} -e 'panic' > /tmp/vmcore_panic.txt
crash {{VMLINUX_PATH}} {{VMCORE_PATH}} -e 'bt' > /tmp/vmcore_bt.txt

#### 1.3 利用第三方诊断报告
若已有第三方诊断报告（如Kuafu报告），应优先参考其中的关键结论，
再通过crash命令验证和补充细节。
```

**改动的关键点：**

1. **从"单一命令"变成"分级策略"**：标准采集、大文件降级、第三方报告三种路径，覆盖真实场景的多样性。
2. **明确触发条件**：写清楚什么情况下走哪条路径（>1GB走降级、超时180秒走降级）。
3. **承认 Agent 的合理偏离**：把Agent实际行为（绕过标准脚本）转化为Skill中的合法路径，避免"跑偏却扣分"的情况。

> 💡 **第二个示例特别值得注意**：动态优化不仅会让Skill更严格，**还会让Skill更"贴近真实使用习惯"**。Agent在实战中"自发"采用的合理做法，应该被吸纳进Skill而不是被判为偏离。

## Step 4：调试与验证

优化完成后，Skill-insight会自动触发L1/L2两级质量检查（参考入门篇）。检查通过后，新版本Skill会自动加载到OpenCode的Skill目录下。这一步全程自动，你只需要等结果。

## Step 5：执行优化后的技能

重启OpenCode，让新版Skill生效，然后执行同样的任务：

```
分析这个 vmcore 文件 /var/crash/xxx/vmcore，定位崩溃根因
```

可以重点观察 Agent 的执行行为变化：

- **代码级分析部分**：Agent是否按5步流程执行，是否使用了`dis -r`等命令，是否填充了完整的"逻辑缺陷"字段。
- **信息采集部分**：Agent是否先检查文件大小，是否记录了选择策略的依据。

## Step 6：评测与对比

任务跑完后，进入Skill-insight看板对比两次执行结果：

```
====================================================
              动态优化前后对比
====================================================
评测维度          优化前      优化后      变化
----------------------------------------------------
RC-0  代码根因    50%         100%       ↑ 50%
RC-3  修复建议    40%         100%       ↑ 60%
KA-0  采集脚本    50%         100%       ↑ 50%
KA-3  排查脚本    50%         100%       ↑ 50%
----------------------------------------------------
综合得分          62%         93%        ↑ 31%
====================================================
```

可以看到，针对动态分析定位到的扣分点进行精准改写后，对应的评测维度全部回到100%，**综合得分从62%跃升到93%**，一次优化解决多个长期卡点。

更重要的是，**这次提升不是靠盲改实现的——每一处改动都对应一个明确的扣分原因和归因依据。** 下次遇到类似 Skill，可以复用相同的优化思路。


# 四、对比Skill Creator：同样是"测试驱动"，差在哪里？

如果你熟悉Anthropic官方的**Skill Creator**，可能会有个疑问：Skill Creator也支持evaluation、benchmark、iterate，看起来同样能"基于运行结果"优化 Skill，那它和 Skill-insight的动态优化有什么区别？

这个问题非常值得说清楚——**两者都依赖测试用例驱动评测，但在"评测的深度"和"优化的可操作性"上有本质差异**。

### 共同点：都需要测试用例

先澄清一个事实：**两者都需要测试用例**，没有"零前置成本"这种说法。

- **Skill Creator**：需要用户手写evals/evals.json，每个case包含prompt、expected output、assertions。
- **Skill-insight**：同样需要测试用例，由**关键观点（RC）**和**关键动作（KA）** 组成——其中RC从用户提供的案例文档自动提取，KA从Skill本身自动提取，全自动生成，用户只需提供原始材料。

测试机制本身大同小异，真正的差异在于"**测试结果如何转化为可操作的优化建议**" —— 这恰恰是Skill优化里最关键、也最容易被忽视的环节。

### 核心差异：从"看结果"到"看过程"

**Skill Creator的评测视角**：以最终输出为主 —— assertion主要验证"输出是否符合预期"，给出pass/fail二元判断。虽然grader可以查看transcript，但assertion设计上仍以"结果对不对"为主，不深入到"过程哪一步出了问题"。

**Skill-insight的评测视角**：把执行过程拆解为**关键观点（RC）**和**关键动作（KA）**两个维度，分别打分：

- **RC（关键观点）**：评估Agent是否提取到了案例文档中的核心结论（如`内核BUG在mm/pin_mem.c:2726`、`set_pid_unused()错误触发BUG_ON`等具体事实）。
- **KA（关键动作）**：评估Agent是否按Skill定义的步骤执行（如`是否运行采集脚本`、`是否读取子场景分类`、`是否生成诊断结论`等具体行为）。

这种拆解带来一个关键能力：**当某个扣分发生时，能精确定位到是"哪一类失败" —— 是没看到正确的事实（RC 缺失）？还是没按流程做（KA 缺失）？** 这正是单纯看 pass/fail 看不到的。

### 详细对比

| 维度 | Skill Creator | Skill-insight动态优化 |
|:-----|:-------------|:----------------------|
| **测试用例来源** | 用户手写evals.json | RC从案例文档自动提取，KA从Skill自动提取 |
| **评测维度** | assertion二元pass/fail | RC（事实提取）+ KA（流程执行）双维度细粒度评分 |
| **过程可见性** | 能看transcript，但assertion仍以最终输出为主 | 直接对每个执行步骤打分，过程偏差一目了然 |
| **责任归因** | 不区分（只看整体pass rate） | 自动区分**Skill问题** vs **Agent模型能力问题** |
| **优化粒度** | 主要优化description（trigger准确率）和整体重写 | 针对每个具体扣分点做点状定位 + 针对性改写 |

### 两个场景下的实际差异

**场景一：Skill在生产中得分波动，但说不清原因。**

- **Skill Creator**：assertion只能告诉你"输出是否符合预期"。如果Agent跳过了某个排查步骤但最终蒙对了答案，assertion会判pass —— 你根本看不到这个潜在风险。
- **Skill-insight**：KA维度会明确扣分（"未执行collect.sh标准采集脚本"），告诉你流程上有偏差，即使这次结果对了，下次可能就翻车。

**场景二：扣分了，但不知道是Skill的问题还是模型的问题。**

- **Skill Creator**：只告诉你assertion没过，但不会告诉你是Skill写得不到位、还是模型能力不够。你可能反复改Skill但症状依旧——本质上是模型本身做不到。
- **Skill-insight**：评测自动归因 —— **Skill问题**就建议改写，**Agent模型能力问题** 就建议换更强的模型，从根本上避免"改错方向"。

### 一句话总结

Skill Creator是 **"输出对不对"** 的视角 —— 保证Skill在测试场景下能产出正确结果；Skill-insight动态优化是 **"过程对不对"** 的视角——保证Skill在每个步骤、每个事实点上都按预期工作，并能告诉你"为什么没做到"。

两者不是替代关系，而是互补关系——前者让Skill"结果可信"，后者让Skill"过程可控"。在实际项目中，理想的组合是用Skill Creator验证整体效果，用Skill-insight动态优化定位过程瓶颈，两手都要硬。


# 五、小结

动态优化的核心价值不在于"改了多少地方"，而在于**改的每一处都有据可依**：

```
执行轨迹（Agent实际做了什么）
    +
评测扣分（哪里没做到位）
    +
归因分析（是Skill还是Agent的问题）
    ↓
精准定位Skill文本中需要改写的具体片段
    ↓
针对性改写，避免误伤正常部分
```

回到开头那个比喻：**静态优化是审稿，动态优化是复盘。** 真正想让一个Skill在生产环境稳定可用，两者缺一不可——先用静态优化把Skill文本打磨规范，再用动态优化基于实战数据持续打磨细节。

如果你已经掌握了静态优化和动态优化两种手法，那么下一篇的「自动多轮优化」会进一步把这两种优化的迭代过程也自动化——你只需要设定目标，剩下的交给机器。

# 六、常见问题

**Q：动态优化对Skill的"运行数据量"有最低要求吗？**

A：理论上一次完整执行就能触发优化。但执行次数越多，扣分点的归因越准确——建议至少3~5次执行后再启动动态优化，避免因为单次偶发因素导致优化方向偏移。

**Q：扣分项归因为"Agent"的问题，动态优化能解决吗？**

A：解决不了。归因为Agent意味着是执行模型能力不足，比如模型本身就不擅长某类反汇编分析。这类问题需要换更强的模型（如GPT-4、Claude）来解决，光改Skill没用。Skill-insight的归因机制就是为了帮你区分清楚哪些问题"改Skill 有用"、哪些"换模型才有用"，避免做无用功。

**Q：动态优化会不会把Skill改得越来越冗长？**

A：会有这个风险，特别是当扣分点很多时。建议每轮动态优化后留意Skill文件大小变化，如果发现某些部分被过度膨胀，可以手动精简。后续我们也在规划"压缩优化"模式，专门解决Skill体积膨胀问题。

**Q：动态优化和反馈优化的区别是什么？**

A：反馈优化更轻量——基于一次执行的反馈做单点修补，速度快但深度有限。动态优化更系统——会综合多次执行的轨迹数据、扣分归因等信息，做结构化改写。简单场景用反馈优化够了，复杂的稳定性问题需要动态优化。

**Q：能否针对单个扣分项进行优化，而不是一次改所有项？**

A：可以。在触发优化时指定具体的扣分项ID（如 RC-0、KA-0）即可。这种"定点优化"特别适合排查某个具体行为偏差，避免一次改动太多导致其他部分受牵连。
